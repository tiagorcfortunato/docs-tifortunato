# Payment Flow Journeys

This page details the two distinct payment flows implemented in Odys: the client-to-professional payment process using PIX (entirely off-Stripe) and the professional-to-Odys subscription payment process via Stripe Checkout. Understanding these separate mechanisms is crucial for comprehending the platform's financial architecture.

## Flow 1 — Client pays professional (PIX, off-Stripe)

This flow facilitates session payments using PIX, Brazil's instant payment system. Stripe is not involved in this process.

1.  The professional configures their PIX key within the `/dashboard/settings` interface.
2.  For a confirmed appointment with `paymentType: "upfront"`, the booking widget renders a `<PixQR>` component. This component generates an EMV BR Code payload using the pure-TypeScript implementation in `src/lib/pix.ts`. This implementation includes inline CRC-CCITT (polynomial `0x1021`, initial `0xffff`) and ID-length-value fields, adhering to the Banco Central's "Manual de Padrões para Iniciação do Pix". The `buildPixPayload` function in `src/lib/pix.ts` constructs the payload, including fields for merchant name, city, and an optional amount.
3.  The client scans the generated QR code with their Brazilian banking application and makes a **bank-to-bank** payment directly to the professional.
4.  The professional manually updates the appointment status to `paid` from their dashboard by sending a `PATCH` request to `/api/appointments/[id]` with `action: "paid"`. This action updates the `appointments.paymentStatus` column to `captured`.

Odys's infrastructure and Stripe do not handle the movement of these funds. The platform functions solely as a QR code generator and a status tracker.

### Why PIX-first for Brazilian SMBs

The decision to prioritize PIX for client-to-professional payments is driven by several factors:

*   **Zero processing fees**: PIX transactions are free for bank-to-bank transfers in Brazil.
*   **Instant settlement**: Funds are transferred and settled within seconds, 24/7.
*   **Universal adoption**: PIX is widely adopted, with approximately 90% of the Brazilian population using it.
*   **Avoids Stripe's Brazilian fee structure**: This approach bypasses Stripe's transaction fees (typically ~4-6% plus FX) on small-ticket service payments.

Refunds for PIX payments are marked as `refunded` in the database, but the actual money transfer is a manual bank operation performed by the professional; Odys does not facilitate fund movement for refunds.

## Flow 2 — Professional pays Odys (Stripe Checkout, subscription)

This flow manages the professional's subscription payments to Odys for Basic, Pro, or Premium plans, leveraging Stripe.

### Plan definition

The `src/lib/stripe/plans.ts` file serves as the single source of truth for all plan-related data, including pricing, feature limits, and human-readable descriptions:

*   **Free**: Limited to 10 clients and 20 appointments per month, with `priceId: null` as it does not correspond to a Stripe product.
*   **Basic / Pro / Premium**: Each paid plan maps to a specific Stripe Price ID, retrieved from environment variables like `process.env.STRIPE_BASIC_PRICE_ID`, `process.env.STRIPE_PRO_PRICE_ID`, and `process.env.STRIPE_PREMIUM_PRICE_ID`. If an environment variable is missing, it explicitly defaults to `null`.

Each plan object in `PLANS` includes a `limits` object (e.g., `clients`, `appointmentsPerMonth`) and a list of `features` for display on the `/dashboard/plans` page. The `PLAN_FEATURES` array defines the master list of all features.

### Checkout

The subscription checkout process is initiated as follows:

1.  A professional clicks the "Upgrade" button on the `/dashboard/plans` page.
2.  The client-side application sends a `POST` request to the `/api/stripe/checkout/route.ts` API route.
3.  The handler authenticates the user via `supabase.auth.getUser()` and retrieves the corresponding professional record from the `professionals` table using `db.select().from(professionals).where(eq(professionals.userId, user.id))`.
4.  It then calls `stripe.checkout.sessions.create()` with the following key parameters:
    ```ts
    stripe.checkout.sessions.create({
      mode: "subscription",
      payment_method_types: ["card"],
      line_items: [{ price: PLANS[plan].priceId!, quantity: 1 }],
      metadata: { professionalId, plan },
      customer_email: user.email,
      locale: "pt-BR",
      subscription_data: trialEnd ? { trial_end: trialEnd } : undefined,
      payment_method_collection: trialEnd ? "if_required" : "always",
      // ... other parameters
    })
    ```
5.  If the professional is within their 14-day Pro trial period (determined by `trialDaysLeft(professional.trialEndsAt)`), `subscription_data.trial_end` is set, and `payment_method_collection: "if_required"` is used. This configuration means a payment method is not required upfront, reducing friction for trial activation.
6.  The API route returns `session.url`, and the browser redirects (`window.location.href`) to the Stripe-hosted checkout page.

This approach uses a **hosted, not embedded**, Stripe Checkout page. This keeps Odys entirely out of PCI scope, as no sensitive card data touches Odys's servers. The `locale: "pt-BR"` parameter ensures the Stripe checkout page is rendered in Portuguese for Brazilian professionals.

### Webhook — the state-update mechanism

The `/api/stripe/webhook/route.ts` endpoint processes events sent by Stripe to update the professional's subscription status:

1.  **Critical first step**: The handler reads the **raw request body** using `req.text()`. Using `req.json()` would strip whitespace and invalidate the HMAC signature.
2.  It extracts the `stripe-signature` header. If this header is missing, the route immediately returns a `400` status before any `constructEvent` call.
3.  `stripe.webhooks.constructEvent(body, sig, process.env.STRIPE_WEBHOOK_SECRET!)` is called within a `try/catch` block. A signature verification failure results in a `400` status **before any database write**. This "fails closed" mechanism prevents unauthorized updates.
