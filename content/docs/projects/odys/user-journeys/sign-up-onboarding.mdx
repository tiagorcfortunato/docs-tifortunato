# Professional Onboarding

The professional onboarding process is a foundational user journey, guiding new professionals from initial registration to a fully configured profile ready to serve clients. This multi-stage flow is designed to be intuitive while capturing essential information, ensuring that each professional can quickly establish their presence and define their service offerings on the platform.

## Overview

The end-to-end professional onboarding experience is orchestrated across several key components. It begins with user authentication via Supabase, progresses through an email verification or OAuth callback, and culminates in a dedicated onboarding API route where professionals define their services and availability. Throughout this process, rate limiting is applied to safeguard against abuse, and database interactions are carefully managed using Drizzle ORM to persist professional and availability data.

The core of this flow involves three primary API routes:
*   `/api/auth/register`: Handles the initial user account creation.
*   `/auth/callback`: Processes Supabase authentication callbacks, including email verification and OAuth.
*   `/api/onboarding`: Facilitates the collection and persistence of detailed professional profile information.

These routes interact with the `professionals` and `availability` tables in the database schema, which together store the comprehensive details of a professional's profile and their working hours.

## Registration

The journey begins with the `POST` endpoint at `src/app/api/auth/register/route.ts`. This route is responsible for creating a new user account. Before any processing, it applies a rate limit using `getRegisterLimiter`, allowing a maximum of 5 registration attempts per hour from a single IP address to prevent brute-force attacks or spam registrations.

Input validation is performed using `zod` with the `registerSchema`, which mandates a valid `email` and a `password` of at least 8 characters, aligning with Supabase's default security requirements. Optional `name` (minimum 2 characters) and `type` fields allow for initial user categorization.

A notable design choice here is the conditional logic for user creation based on the `NODE_ENV` environment variable. In `production`, the system uses `supabase.auth.signUp()`. This method is crucial because it automatically triggers Supabase's email verification flow, sending a confirmation link to the user. The API response includes a `requiresEmailConfirmation` flag, indicating whether a session was immediately established or if verification is pending. Conversely, in `development` environments, the `getAdminClient().auth.admin.createUser()` method is employed with `email_confirm: true`. This bypasses the email verification step, allowing developers to rapidly test the onboarding flow without needing to check inboxes, significantly accelerating the development cycle.

Any errors during this stage, such as invalid credentials or Supabase-related issues, are captured and returned with appropriate HTTP status codes, and internal errors are logged via `logger.error`.

## Authentication Callback

Following registration, users typically interact with the `GET` endpoint at `src/app/auth/callback/route.ts`. This route serves as the destination for Supabase's authentication callbacks, primarily for email verification links or OAuth redirects. It extracts a `code` parameter, which Supabase uses to exchange for a user session, and a `type` parameter, which indicates the context of the callback.

A critical security measure implemented here is the `VALID_TYPES` `Set`. This explicitly defines the only accepted values for the `type` parameter (`professional`, `client`, `recovery`). This prevents an attacker from injecting arbitrary data into the user's metadata via a crafted callback URL, addressing a potential vulnerability.

Upon successful code exchange, the route handles several scenarios:
*   If the `type` is `recovery`, the user is redirected to the `/reset-password` page, initiating the password reset flow.
*   If the `type` is `professional` or `client`, the user's metadata in Supabase is updated to reflect their designated role. This ensures that the user's profile is correctly categorized within the application.
*   A "smart-linking" mechanism is implemented for `client` types. If a user's email is verified and they are identified as a `client`, the system attempts to link their new `userId` to any existing entries in the `clients` table that share the same email but currently have a `NULL` `userId`. This is particularly useful for professionals who might have added clients by email before those clients registered their own accounts, providing a seamless experience by associating pre-existing client data with the newly authenticated user.

Finally, the user is redirected to `/auth/redirect` on success or `/login?error=auth` if the authentication process encounters an issue.

## Professional Onboarding Profile Setup

Once a professional has registered and verified their email, they proceed to the `POST` endpoint at `src/app/api/onboarding/route.ts` to complete their profile. This route is protected by `getOnboardingLimiter`, which permits 3 attempts per hour per IP address, mitigating potential abuse during profile creation.

The route first verifies the user's authentication status using `getUser()`. If no user session is found, an `unauthorized` response is returned. To ensure idempotency, the system checks if a `professionals` record already exists for the authenticated `userId`. If one is found, the existing professional data is returned, preventing duplicate profile creation if the user submits the form multiple times.

The core of this route involves validating the professional's details using `onboardingSchema`, a `zod` schema that enforces constraints on fields like `name`, `profession`, `phone`, `sessionDuration`, `sessionPrice`, `serviceType`, `city`, `state`, and `availability`. The `availability` field is an array of at least one object, each specifying a `dayOfWeek`, `startTime`, and `endTime`.

A unique `slug` is generated from the professional's `name` using the `toSlug` and `generateUniqueSlug` functions. The `generateUniqueSlug` function is designed to create a URL-friendly identifier, handling potential conflicts by appending numeric suffixes (e.g., `john-doe-2`) and preventing collisions with a predefined `RESERVED_SLUGS` `Set` that includes application routes like `api`, `auth`, and `dashboard`. This ensures that professional profile URLs are clean and do not interfere with core application paths.

After successful validation and slug generation, a new record is inserted into the `professionals` table. If `availability` data is provided, multiple entries are then inserted into the `availability` table, each linked to the newly created professional via `professionalId`. The user's Supabase metadata is also updated to set their `type` to `professional` and their `name`.

Finally, in `production` environments, a welcome email is sent to the professional using `sendProfessionalWelcomeEmail`. This is a fire-and-forget operation, meaning its failure does not block the onboarding process, though warnings are logged.

## Design decisions

The design of the professional onboarding flow reflects a deliberate approach to user experience, security, and maintainability:

*   **Modular Authentication Stages**: Separating registration (`/api/auth/register`), callback handling (`/auth/callback`), and profile setup (`/api/onboarding`) into distinct API routes allows for a clear, step-by-step user journey. This modularity simplifies debugging and makes it easier to introduce new steps or modify existing ones without affecting the entire flow.
*   **Supabase as the Auth Backbone**: The system fully delegates user authentication, session management, and email verification to Supabase. This choice offloads the complexities and security considerations of building an authentication system from scratch, allowing the development team to focus on core business logic. The distinction between `signUp()` for production (triggering email verification) and `admin.createUser()` for development (bypassing verification) is a pragmatic trade-off that significantly enhances developer velocity.
*   **Robust Input Validation with Zod**: The use of `zod` schemas (`registerSchema`, `onboardingSchema`) at the API layer provides strong type safety and data validation. This prevents malformed or malicious data from reaching the database, improving data integrity and application stability.
*   **Idempotent Onboarding**: The `/api/onboarding` route's check for an existing `professionals` record based on `userId` makes the operation idempotent. This means that if a user accidentally or intentionally submits the onboarding form multiple times, it will not result in duplicate professional profiles, providing a more resilient user experience.
*   **User-Friendly and Conflict-Free Slugs**: The `toSlug` and `generateUniqueSlug` functions are crucial for creating human-readable and unique profile URLs. The inclusion of `RESERVED_SLUGS` demonstrates foresight in preventing URL conflicts with application-level routes, which could lead to routing errors or unexpected behavior.
*   **Smart Client Linking**: The logic in `src/app/auth/callback/route.ts` to link a newly registered client's `userId` to existing `clients` records by `email` is a thoughtful design for user convenience. It anticipates a common scenario where professionals might add clients by email before those clients create their own accounts, ensuring data continuity.
*   **Rate Limiting for Stability**: Implementing rate limits via `@upstash/ratelimit` for critical endpoints like registration and onboarding (`getRegisterLimiter`, `getOnboardingLimiter`) is a fundamental security and stability measure. It protects the application from denial-of-service attacks, brute-force attempts, and excessive resource consumption.

## Potential improvements

1.  **`professionals.email` Column Redundancy**: The `professionals` table includes an `email` column, which is nullable. However, the `userId` column is non-nullable and unique, directly linking to a Supabase user who inherently has a verified email address. To reduce data duplication and potential inconsistencies, the `professionals.email` column could be removed or made non-nullable and automatically populated from the associated Supabase user's email, ensuring it always reflects the source of truth.
2.  **Dynamic `RESERVED_SLUGS` Management**: The `RESERVED_SLUGS` `Set` in `src/app/api/onboarding/route.ts` is hardcoded. As the application grows and new routes are added, this list will require manual updates, increasing the risk of oversight and potential slug conflicts. A more maintainable approach would be to dynamically generate this list from the application's route configuration or manage it through a centralized, version-controlled configuration file.
3.  **Enhanced Availability Time Validation**: The `onboardingSchema` in `src/app/api/onboarding/route.ts` validates `startTime` and `endTime` using a regex `^\d{2}:\d{2}$`. While this ensures the format, it does not prevent logically invalid times (e.g., `25:00` or `10:00` as `endTime` when `startTime` is `11:00`). Implementing custom `zod` refinements or using a dedicated time parsing library could enforce valid time ranges and ensure `endTime` is always after `startTime`, improving data quality.
4.  **Robust Welcome Email Delivery**: The welcome email sending in `src/app/api/onboarding/route.ts` is a fire-and-forget operation with a `console.warn` for errors. For critical communications like welcome emails, a more resilient delivery mechanism could be beneficial. This might involve integrating with a dedicated email service that offers retry logic, dead-letter queues, or detailed delivery status tracking, rather than relying solely on a `catch` block.
