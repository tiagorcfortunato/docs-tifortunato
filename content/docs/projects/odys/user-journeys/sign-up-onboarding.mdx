# Professional Onboarding

The professional sign-up and onboarding flow is a critical user journey in Odys, guiding new service providers from initial registration to a fully configured profile ready to accept clients. This process involves several distinct steps, each handled by dedicated API routes and interacting with the core database schema to establish a professional's identity, availability, and service details. Understanding this flow is key to grasping how new professionals are integrated into the platform and how their initial data is managed.

## Overview

The journey for a new professional begins with account creation, typically via email and password. Once an account is established and verified, the professional proceeds to an onboarding phase where they provide detailed information about their services, pricing, and availability. This structured approach ensures that all necessary data is collected before a professional can be listed or booked.

The process is orchestrated across three primary API endpoints:
*   `/api/auth/register`: Handles the initial user account creation with Supabase.
*   `/auth/callback`: Processes the Supabase authentication callback, particularly important for email verification and linking user accounts to existing client profiles.
*   `/api/onboarding`: Collects and persists the professional-specific details, creating their public profile and setting up their initial availability.

The `/api/onboarding` route interacts primarily with the `professionals` and `availability` tables, while `/api/auth/register` creates a Supabase user and `/auth/callback` handles session exchange and client smart-linking. The `clients` table also plays a role in the callback, enabling "smart-linking" for users who might have interacted with the platform as clients before becoming professionals.

## Registration Flow

The initial step for any new user, including a professional, is to register an account. This is managed by the `src/app/api/auth/register/route.ts` API endpoint.

Upon receiving a `POST` request, this route first applies a rate limit using `getRegisterLimiter`. This limiter is configured to allow only 5 registration attempts per hour per IP address, a measure designed to mitigate brute-force attacks and prevent abuse of the registration system. The IP address for rate limiting is extracted using the `getIp` utility.

The incoming request body is then validated against the `registerSchema`, a Zod schema that enforces the presence and format of `email` (must be a valid email string), `password` (minimum 8 characters, aligning with Supabase's default), and optionally `name` (minimum 2 characters) and `type` (either "client" or "professional"). This strict validation prevents malformed data from reaching the authentication backend. The `name` and `type` fields, if provided, are stored in the user's Supabase metadata during account creation.

A key design choice in this route is the conditional handling of user creation based on the `NODE_ENV` environment variable. In a production environment, the `supabase.auth.signUp` method is used. This method automatically triggers Supabase's email verification flow, sending a confirmation email to the user. If no session is returned, it indicates that email confirmation is required, which is communicated back to the client. In a development environment, to accelerate testing cycles, the `getAdminClient().auth.admin.createUser` method is employed with `email_confirm: true`. This bypasses the email verification step, instantly confirming the user's email and allowing developers to proceed directly to onboarding. The `_adminClient` itself is lazy-initialized to ensure that environment variables like `SUPABASE_SERVICE_ROLE_KEY` are only accessed when needed, which is important for build environments where secrets might not be present.

If user creation is successful, the route returns a success response, indicating whether email confirmation is required and the user's specified `type`. Any errors from Supabase or internal server issues are caught and returned with appropriate HTTP status codes, such as `400` for invalid data or `500` for internal errors, with logging via `logger.error`.

## Authentication Callback

After a user completes an action like email verification or password recovery through Supabase, they are redirected to the `src/app/auth/callback/route.ts` endpoint. This route is crucial for finalizing the authentication process and performing any necessary post-authentication setup.

The callback extracts a `code` and an optional `type` parameter from the URL's search parameters. The `type` parameter is particularly important for the onboarding flow, as it can indicate whether the user is registering as a "professional" or "client", or if they are in a "recovery" flow. A security measure, identified as an L4 finding, is the `VALID_TYPES` set, which strictly limits the accepted values for `type`. This prevents an attacker from injecting arbitrary data into the user's metadata via a crafted callback URL.

The route then uses the `supabase.auth.exchangeCodeForSession` method to exchange the provided code for a user session. If successful:
*   For a `recovery` type, the user is redirected to the `/reset-password` page.
*   For `professional` or `client` types, the user's Supabase metadata is updated to reflect their chosen `type`.
*   A "smart-linking" mechanism is implemented for `client` types. If a user registers with an email that matches an existing entry in the `clients` table where `userId` is null, their newly verified `user.id` is linked to that `clients` record. This is a thoughtful design choice to connect pre-existing client data (perhaps from an unauthenticated booking) with a newly registered user account.

Finally, the user is redirected to `/auth/redirect` to complete the client-side authentication flow, or to `/login?error=auth` if an error occurred during the session exchange. The `origin` for redirection is carefully constructed using `x-forwarded-host` headers to ensure correct redirects in production environments, especially when deployed on platforms like Vercel.

## Professional Onboarding

Once a professional has registered and verified their account, they proceed to the onboarding phase, handled by the `src/app/api/onboarding/route.ts` endpoint. This is where the detailed professional profile is created.

Similar to registration, this route is protected by `getOnboardingLimiter`, allowing only 3 attempts per hour per IP address to prevent abuse during the profile creation process. It also requires an authenticated user, verified by `getUser()`, returning `unauthorized()` if no user session is found.

The route implements an idempotency check: if a `professionals` record already exists for the authenticated `user.id`, it simply returns the existing professional data, preventing duplicate entries and allowing users to safely re-submit the form without adverse effects.

The core of the onboarding process involves validating the incoming data against the `onboardingSchema`. This comprehensive Zod schema captures essential professional details:
*   `name`: The professional's full name.
*   `profession`: Their area of expertise.
*   `phone`: Contact phone number.
*   `bio`: An optional biography.
*   `sessionDuration`: The standard length of a session in minutes (integer, positive, max 480).
*   `sessionPrice`: The price of a session (integer, non-negative).
*   `serviceType`: Whether services are `in_person`, `online`, or `both`.
*   `city`, `state`: Optional location details.
*   `availability`: A crucial array of objects, each defining a `dayOfWeek` (0-6) and `startTime`/`endTime` (validated by regex `^\d{2}:\d{2}$`). This array must contain at least one entry.

A unique `slug` is generated for each professional based on their `name` using the `toSlug` and `generateUniqueSlug` functions. The `toSlug` function normalizes the name into a URL-friendly format. `generateUniqueSlug` then checks for conflicts in the `professionals` table and against a `RESERVED_SLUGS` set (which includes routes like "api", "auth", "dashboard", etc.). If the initial slug is empty or conflicts with a reserved route, `generateUniqueSlug` uses 'user' as a base before appending numeric suffixes. If a conflict exists, it appends a numeric suffix (e.g., "john-doe-2") until a unique slug is found. This ensures clean, human-readable URLs for professional profiles (`/p/[slug]`) that do not clash with application routes.

Once validated, the professional's data is inserted into the `professionals` table. If availability data is provided, it is then inserted into the `availability` table, linked to the new professional's `id`. The user's Supabase metadata is also updated to set their `type` to "professional" and store their `name`.

Finally, in production environments, a welcome email is sent to the professional using `sendProfessionalWelcomeEmail`. The `sendProfessionalWelcomeEmail` function is called with the professional's email, name, and generated slug. This is a fire-and-forget operation, meaning the onboarding process doesn't block on its completion, and any email sending errors are merely warned about rather than causing the entire request to fail.

## Design Decisions

The design of the professional sign-up and onboarding flow reflects several deliberate choices aimed at user experience, data integrity, and system security:

*   **Separation of Registration and Onboarding**: The initial `/api/auth/register` route focuses solely on creating a user account with minimal information (email, password). The `/api/onboarding` route then handles the collection of extensive professional-specific data. This separation allows for a multi-step user journey, where account creation is quick, and detailed profile setup can be completed at the user's pace, potentially even after initial login. It also cleanly separates authentication concerns from application-specific profile data.
*   **Conditional Supabase User Creation**: The use of `supabase.auth.signUp` in production and `admin.createUser` in development for `/api/auth/register` is a pragmatic trade-off. It prioritizes a secure, verified user experience in production while enabling rapid development and testing by bypassing email verification locally. The lazy initialization of `_adminClient` further supports this by not requiring sensitive environment variables during build steps.
*   **Strict Type Validation in `auth/callback`**: The `VALID_TYPES` set in `src/app/auth/callback/route.ts` is a direct response to a security finding. By explicitly whitelisting accepted `type` values, the system prevents malicious actors from injecting arbitrary data into user metadata, which could have unforeseen consequences. This demonstrates a commitment to hardening the application against common web vulnerabilities.
*   **Unique Slug Generation**: The `toSlug` and `generateUniqueSlug` functions in `src/app/api/onboarding/route.ts` are designed to create user-friendly and unique URLs for professional profiles. The inclusion of `RESERVED_SLUGS` is a forward-thinking measure to prevent conflicts with existing or future application routes, ensuring that a professional's public profile URL (`/p/[slug]`) remains distinct and functional. The iterative suffixing (`-2`, `-3`, etc.) is a simple yet effective way to handle name collisions.
*   **Rate Limiting**: The application of rate limits via `getRegisterLimiter` and `getOnboardingLimiter` (defined in `src/lib/ratelimit.ts`) on these critical endpoints is a fundamental security and stability measure. It protects against automated attacks like brute-forcing registration attempts or overwhelming the onboarding service, ensuring fair usage and system resilience.
*   **Referential Integrity with `onDeleteCascade`**: Many tables, such as `availability`, `clients`, `clientNotes`, `recurringSchedules`, `appointments`, `messages`, `follows`, and `reviews`, include a `professionalId` column with an `onDeleteCascade` action. This is a crucial database design choice. It means that if a professional's record is deleted from the `professionals` table, all associated data (their availability slots, clients, notes, appointments, messages, followers, and reviews) will be automatically removed. This maintains data consistency and prevents orphaned records, simplifying data management and ensuring privacy compliance if a professional decides to leave the platform.

## Potential Improvements

While the current implementation is functional and addresses key concerns, several areas could be considered for future enhancements:

*   **`professionals.email` Column Usage**: The `professionals` table includes an `email` column that is currently nullable. However, the welcome email in `src/app/api/onboarding/route.ts` relies on `user.email` from Supabase. If the professional's email address is a critical piece of contact information or needs to be displayed on their public profile, making the `professionals.email` column non-nullable and ensuring it's populated during onboarding (perhaps from `user.email`) would create a more self-contained and consistent professional record within the application's own database schema. This would reduce reliance on Supabase for this specific piece of data after initial registration.
*   **Optimized Slug Generation**: The `generateUniqueSlug` function in `src/app/api/onboarding/route.ts` currently performs a database query in a `while` loop to find a unique slug. While effective, for a very large number of professionals with similar names or under extremely high concurrent onboarding traffic, this iterative querying could become a performance bottleneck. An alternative approach might involve generating a slug with a random suffix initially (e.g., `name-randomstring`) or using a more sophisticated collision resolution strategy that minimizes database roundtrips, perhaps by attempting a few common suffixes first before resorting to a loop.
*   **Stronger Typing for Time Fields**: The `startTime` and `endTime` columns in the `availability` table are stored as `text` and validated using a regex (`^\d{2}:\d{2}$`). While this works, using a more specific time-oriented data type (if supported by Drizzle ORM and PostgreSQL, such as `time` or `interval`) could offer better type safety, enable more efficient time-based queries directly in the database, and simplify time arithmetic in application code. This would reduce the need for string parsing and validation at the application layer for every interaction with these fields.
