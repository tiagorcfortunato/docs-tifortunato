# Sign-up & Onboarding Flow

This document details the end-to-end journey for professionals signing up and completing onboarding in the Odys platform. It covers authentication, rate limiting, data modeling, and post-registration setup, explaining how the system ensures security, scalability, and a smooth user experience.

## Overview

The professional sign-up and onboarding flow spans multiple systems: Supabase for authentication, Drizzle ORM for database interactions, Upstash for rate limiting, and background processes for email and WhatsApp notifications. The flow begins at `/api/auth/register`, continues through OAuth callback handling, and concludes with profile creation via `/api/onboarding`. Multiple database tables support this workflow, with `professionals` as the central entity linked to `availability`, `clients`, and `appointments`. Multiple cron jobs and API routes—including MCP tools—reflect a mature, event-driven architecture.

## Account Creation and Type Safety

Registration starts with a POST to `/api/auth/register`, which enforces a rate limit of 5 attempts per hour per IP using Upstash Redis. The endpoint accepts `email`, `password`, `name`, and `type`. The `type` value is passed to Supabase user metadata during registration. During the OAuth callback, the `type` query parameter from the URL is validated against a closed set of "professional", "client", or "recovery" to prevent metadata injection attacks. If valid, the user's metadata `type` is then updated with this validated value. In production, `supabase.auth.signUp()` triggers email verification; in development, the service role client creates confirmed users for faster iteration.

After email confirmation, Supabase redirects to `/auth/callback` with a one-time code. This route exchanges the code for a session and validates the `type` query parameter against a `Set` of allowed values, preventing arbitrary data injection into `user_metadata`. If the user is registering as a client and their email matches a pre-existing `clients` record (without `userId`), the system links them automatically—a "smart-linking" pattern that preserves client history across account creation.

## Onboarding and Profile Setup

Once authenticated, professionals submit their profile via `/api/onboarding`. The `/api/onboarding` endpoint is designed to be idempotent: if a professional record already exists for the authenticated user, the existing record is returned immediately without further processing. This endpoint is rate-limited to 3 requests per hour per IP and validates input using Zod. The schema requires `name`, `profession`, `phone`, `sessionDuration`, and `sessionPrice`, with optional `bio` and mandatory weekly `availability`. The `slug` is derived from the name using Unicode normalization and hyphenation, then made unique by appending a number if it conflicts with existing slugs or reserved routes like `/dashboard` or `/api`.

Upon successful insertion into the `professionals` table, the system writes availability blocks into the `availability` table and updates the user’s metadata to `type: professional`. A welcome email is sent in production using `sendProfessionalWelcomeEmail`, fire-and-forget style to avoid blocking the response.

## Design decisions

The use of Supabase’s service role in development allows rapid iteration without email verification delays, while production relies on standard `signUp()` to enforce security. The `VALID_TYPES` set in `/auth/callback/route.ts` prevents metadata pollution—a direct response to a prior security finding. Rate limiting is granular: `/onboarding` is capped at 3/hour to deter spam, while `/register` allows 5/hour. Slug generation avoids collisions with app routes via a hardcoded `RESERVED_SLUGS` set, ensuring URL integrity.

## Potential improvements

1.  **Slug generation performance**: `generateUniqueSlug` in `/api/onboarding/route.ts` uses a loop with sequential DB queries. For high-traffic systems, this could cause contention. A probabilistic approach with random suffixes or batch checks could reduce round trips.

2.  **Availability validation**: The onboarding schema does not validate that `startTime` is before `endTime` or that time ranges are non-overlapping. Adding this in `onboardingSchema` would prevent invalid schedules from being stored.

3.  **Error handling in email sending**: The `sendProfessionalWelcomeEmail` call uses `.catch()` but only logs warnings. In `/api/onboarding/route.ts`, consider retry logic or queuing via a durable function to ensure delivery.
