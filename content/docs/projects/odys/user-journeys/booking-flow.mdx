# Booking Flow

The booking flow is a central interaction within Odys, enabling clients to discover professionals and schedule appointments. This process encompasses the client's journey from viewing a professional's public profile to selecting an available time slot, providing their details, and receiving confirmation. It involves several interconnected components, including a server-rendered professional page, a client-side booking widget, and a robust API for handling availability checks and appointment creation.

## Overview

The booking journey begins when a client navigates to a professional's dedicated public page, typically at `/p/[slug]`. This page, rendered on the server, displays the professional's profile, reviews, and embeds the interactive `BookingWidget`. The widget, a client-side component, dynamically fetches available time slots for a selected date by querying the `/api/booking` endpoint. Once a client chooses a slot and provides their contact information, the widget dispatches a `POST` request to the same `/api/booking` endpoint. This API handler performs critical validations, checks for appointment conflicts, manages client records, creates the appointment, and triggers various notifications. Finally, the system handles post-booking actions, such as confirming or rejecting appointments, through the `/api/appointments/[id]` endpoint.

## Professional Public Page

The `src/app/p/[slug]/page.tsx` file defines the server-rendered public profile page for each professional. This page serves as the primary entry point for clients to learn about a professional and initiate a booking.

Upon loading, the page fetches comprehensive details about the professional from the `professionals` table using their unique `slug`. It also retrieves associated `reviews` and `clients` data to display ratings and comments, providing social proof to potential clients. The page also identifies if the visitor is the professional themselves (`isOwner`) or a logged-in client, which influences UI elements like the presence of a `FollowButton` or pre-filling client form data.

A key element of this page is the `BookingWidget` component. This client-side component is responsible for the interactive date and time selection process. The server component passes essential professional data to the widget, such as `slug`, `professionalName`, `sessionDuration`, `sessionPrice`, and payment details like `paymentType`, `paymentPercentage`, `pixKeyType`, and `pixKey`. If a client is logged in, their `name`, `phone`, and `email` are also passed as `defaultName`, `defaultPhone`, and `defaultEmail` to streamline the booking form experience.

The page also displays related professionals based on the current professional's `profession`, fetching their basic details and aggregated statistics using `getProStats`. This encourages exploration and provides alternatives if the primary professional is unavailable.

## Booking Widget

The `src/app/p/[slug]/booking-widget.tsx` is a client-side React component that orchestrates the interactive booking experience. It guides the user through three main steps: "pick" (date and time selection), "form" (client details), and "success" (confirmation).

When a user selects a date on the `Calendar`, the widget triggers a `GET` request to `/api/booking`, passing the professional's `slug` and the `date`. The `useEffect` hook manages this data fetching, setting `loadingSlots` to `true` while the request is in flight. The response from the API includes the professional's `availability` rules and `existingAppointments` for that day.

The `generateSlots` utility function, located in `src/lib/slots.ts`, then processes this data. It takes the selected `date`, the professional's `availability` rules (which define the `dayOfWeek`, `startTime`, and `endTime`), any `existingAppointments`, and the `sessionDuration` to produce a list of available time slots. This function carefully checks for conflicts with existing appointments and ensures that generated slots do not extend past the professional's `endTime` for the day. The `isBefore` and `isAfter` functions from `date-fns` are used to accurately compare time ranges. **`generateSlots` also skips slots that are in the past** using `isAfter(current, now)`.

Once a slot is selected, the widget transitions to the "form" step, presenting a client details form. This form uses `react-hook-form` with `zodResolver` for client-side validation against the `clientSchema`. The schema defines required fields like `name` and `phone`, and an optional `email`. Crucially, it includes a `website` field, which acts as a honeypot to detect and deter bot submissions.

Upon form submission, a `POST` request is sent to `/api/booking` with the selected `startsAt` time, professional `slug`, and client details. If the booking is successful, `posthog.capture` records a `booking_created` event for analytics, and the widget moves to the "success" step. This final step displays a confirmation message and, if applicable, a `PixQR` code for immediate payment. It also offers a "Create free account" option, pre-filling registration details from the submitted booking data to encourage user conversion.

## Booking API

The `src/app/api/booking/route.ts` file defines the API endpoint responsible for handling both the retrieval of available slots and the creation of new appointments.

### GET Request

The `GET` handler processes requests for a professional's available slots on a specific date. It expects `slug` and `date` as query parameters. The endpoint first retrieves the professional's `id` and `sessionDuration` from the `professionals` table. A deliberate design choice here is to `select` only these necessary columns, rather than `SELECT *`, to prevent sensitive professional data (like `email`, `phone`, `userId`, `plan`, `trialEndsAt`, `stripeCustomerId`, `stripeSubscriptionId`) from being exposed to unauthenticated clients.

It then fetches the professional's `availability` rules and `existingAppointments` for the requested date. The `existingAppointments` query specifically excludes appointments with a `status` of `"rejected"` or `"cancelled"`, ensuring that only genuinely occupied slots are considered. The collected `rules` and `existingAppointments` are then returned to the client-side `BookingWidget` for slot generation.

### POST Request

The `POST` handler is the core of the booking process, responsible for creating a new appointment.

1.  **Rate Limiting**: The request is first subjected to rate limiting using `getBookingLimiter()`, which allows a maximum of 5 booking attempts per IP address every 10 minutes. This helps prevent abuse and denial-of-service attacks.
2.  **Timing Oracle Mitigation**: To counter potential timing attacks, the system introduces a response-time floor. The `respondAfterFloor` utility ensures that all responses take at least 250ms. This blunts the signal an attacker might gain by timing responses to distinguish between existing and new client records, which would otherwise complete faster due to a `SELECT` being quicker than a `SELECT` followed by an `INSERT`.
3.  **Validation and Honeypot**: The incoming request body is validated using `bookingSchema` from Zod. A critical security measure is the honeypot: if the hidden `website` field is populated, the request is silently dropped, returning a fake success response to mislead bots.
4.  **Professional Lookup**: The professional is retrieved from the `professionals` table using the provided `slug`.
5.  **Plan Limits**: The system checks the professional's active plan (`effectivePlan`) against configured limits for `clients` and `appointmentsPerMonth`. If a limit is exceeded, and the client is not an existing one (for client limits), the booking is rejected with a `403` status.
6.  **Atomic Transaction**: The most critical part of the `POST` handler is the atomic transaction, executed with `db.transaction({ isolationLevel: "serializable" })`. This transaction wraps three operations:
    *   **Conflict Check**: It checks for any overlapping appointments for the professional within the proposed `startsAt` and `endsAt` timeframe, excluding `rejected` or `cancelled` appointments. If a conflict is found, it throws a `"SLOT_TAKEN"` error.
    *   **Client Upsert**: It attempts to find an existing client by `phone` or `email` for the given professional. If no client is found, a new entry is inserted into the `clients` table. If an existing client is found but lacks an `email`, and one is provided, the client's `email` is updated.
    *   **Appointment Insert**: Finally, the new appointment is inserted into the `appointments` table with a `status` of `"confirmed"` if `professional.autoConfirm` is true, or `"pending_confirmation"` otherwise. **New appointments are inserted with a `paymentStatus` of `"none"`**.
    The `serializable` isolation level is crucial here. It prevents race conditions where two concurrent booking requests might both pass the conflict check before either commits, leading to a double-booking. If a `40001` SQLSTATE (serialization failure) or `"SLOT_TAKEN"` error occurs, the system returns a `409` Conflict status, informing the user that the slot was just taken.
7.  **Notifications**: After a successful appointment creation, the system sends various notifications:
    *   An entry is inserted into the `notifications` table for the professional.
    *   WhatsApp messages are sent to the client (confirmation or pending) and the professional (request or receipt), using templates like `msgBookingConfirmed`, `msgBookingRequest`, and `msgBookingConfirmedReceipt`.
    *   An email notification is sent to the professional if their `email` is available.
8.  **Analytics**: `posthog.capture` and `setOnce` are used to record `booking_received_server` events and the `first_booking_at` timestamp, providing valuable insights into professional activity and conversion funnels.
9.  **Equalized Response**: The response to the client is a generic `{ success: true }`, deliberately omitting the newly created appointment details. This is another security measure to prevent information leakage that could aid attackers in probing for client data.

## Appointment Management API

The `src/app/api/appointments/[id]/route.ts` endpoint handles status updates for existing appointments. While not directly part of the initial booking creation, it's integral to the post-booking lifecycle.

This `PATCH` endpoint allows authenticated users (either the professional or the client, depending on the action) to modify an appointment's status or payment status. It enforces rate limiting via `getApiLimiter()` and performs strict authorization checks using `getUser()` to ensure only authorized individuals can perform actions.

A predefined `ALLOWED_ACTIONS` set (`"confirm"`, `"reject"`, `"cancel"`, `"paid"`, `"complete"`, `"no_show"`) ensures that only valid state transitions are attempted. Any unknown action results in a `400` error.

Depending on the `action` and the user's role (`isProfessional` or `isClient`), the endpoint updates the `appointments` table and triggers a cascade of notifications. For instance, a client can `cancel` their own `pending_confirmation` or `confirmed` appointment, which then notifies the professional via WhatsApp and a system notification. Professionals can `confirm` or `reject` pending bookings, mark them as `paid`, `complete`, or `no_show`, each action triggering specific WhatsApp messages (e.g., `msgBookingConfirmed`, `msgBookingRejected`, `msgPaymentConfirmedToClient`) and, where applicable, email notifications to the client. If a client is not registered, the confirmation message includes a `msgRegistrationInvite` to encourage them to create an account.

**Confirming an appointment sets its `paymentStatus` to `"authorized"`, while rejecting sets it to `"refunded"`**.

## Design decisions

*   **Server-side Rendering for Professional Pages**: The choice to render `src/app/p/[slug]/page.tsx` as a server component allows for efficient initial page load, better SEO, and direct database access for fetching professional details and reviews without client-side API calls. This provides a fast and complete initial view for potential clients.
*   **Client-side Booking Widget**: The `BookingWidget` is a client component (`"use client"`), enabling rich interactivity for date and time selection without full page reloads. This separation of concerns allows for a dynamic user experience while keeping the initial page load lean.
*   **Data Minimization in GET API**: The `GET` request to `/api/booking` explicitly selects only `id` and `sessionDuration` from the `professionals` table. This was a deliberate change from a previous `SELECT *` to prevent sensitive professional data from being exposed to unauthenticated users, enhancing data privacy.
*   **Atomic Transaction for Booking POST**: The `POST` request to `/api/booking` uses a Drizzle `serializable` transaction for the conflict check, client upsert, and appointment insertion. This is a critical design choice to prevent race conditions and ensure data integrity. Without it, two simultaneous booking attempts for the same slot could both pass an initial availability check and then both insert, leading to double-bookings. The `serializable` isolation level forces the database to detect and prevent such phantom reads.
*   **Timing Oracle Mitigation**: The `respondAfterFloor` function in the `POST /api/booking` handler is a security measure. By ensuring a minimum response time, it makes it harder for attackers to distinguish between existing and new client records based on subtle differences in database query execution times, thus protecting against enumeration attacks.
*   **Honeypot for Bot Detection**: The `website` field in the `bookingSchema` and its server-side check is a simple yet effective anti-spam mechanism. Bots often fill all form fields, including hidden ones, allowing the system to silently discard their submissions without revealing the trap.
*   **Comprehensive Notification System**: The booking flow integrates deeply with WhatsApp and email for notifications. This multi-channel approach ensures clients and professionals receive timely updates, whether for booking requests, confirmations, rejections, or cancellations, improving communication and reducing no-shows. The conditional `msgRegistrationInvite` for non-logged-in clients is a thoughtful touch for user acquisition.
*   **Rate Limiting on API Endpoints**: The use of `Ratelimit` from `@upstash/ratelimit` on both `/api/booking` and `/api/appointments/[id]` protects the backend from excessive requests, preventing abuse and ensuring service availability.

## Potential improvements

1.  **Unified Client Email Handling**: In `src/app/p/[slug]/booking-widget.tsx`, the `clientSchema` defines `email` as `z.string().email().optional().or(z.literal(""))`. However, the `clients.email` column in the database schema is `nullable: true` and of type `text`. While the current code handles `null` and empty strings correctly, standardizing on `null` for missing email addresses (e.g., by converting `""` to `null` before database insertion) could simplify logic and ensure consistency across the application.
2.  **Enhanced Error Handling for External Services**: The `sendWhatsApp` and `sendBookingRequestEmailToProfessional` calls in `src/app/api/booking/route.ts` and `src/app/api/appointments/[id]/route.ts` use `.catch(console.warn)`. While this prevents booking failures due to notification issues, it means notification failures are only logged as warnings. For critical notifications, consider integrating a more robust error reporting system (e.g., Sentry, dedicated logging service) or implementing retry mechanisms to ensure messages are eventually delivered, rather than silently failing.
3.  **Dynamic Session Duration/Price Display**: The `BookingWidget` in `src/app/p/[slug]/booking-widget.tsx` receives `sessionDuration` and `sessionPrice` as props. If a professional offers multiple service types with varying durations or prices, the current widget would need to be extended to allow clients to select a service, which would then dynamically update these values. Currently, it assumes a single `sessionDuration` and `sessionPrice` per professional.
4.  **Refine Plan Limit Checks**: In `src/app/api/booking/route.ts`, the plan limit checks for `clients` and `appointmentsPerMonth` are performed by fetching `count()` from the database. While functional, for very high-traffic professionals, these counts could be cached (e.g., in Redis) and invalidated periodically to reduce database load on every booking request. This would trade off some real-time accuracy for improved performance.
