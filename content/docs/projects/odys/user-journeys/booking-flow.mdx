# Booking Flow

This document details the end-to-end customer booking journey in Odys, from landing on a professional’s public page to submitting a confirmed appointment. It covers data flow, validation, conflict detection, and post-submission actions.

## Overview

The booking flow spans frontend and backend components across 10 database tables, including `professionals`, `availability`, `appointments`, and `clients`. It begins when a user visits a professional’s public page (`/p/[slug]`), selects a time slot, and submits their details via `/api/booking`. The system validates input, enforces plan limits, and notifies the professional. The core booking logic—checking for scheduling conflicts, upserting the client record, and inserting the appointment—is performed within a serializable database transaction to prevent race conditions.

## Flow Breakdown

The journey starts in `src/app/p/[slug]/page.tsx`, where the page fetches professional data using the `slug` param. The `BookingWidget` component renders a calendar and time slots by calling `GET /api/booking` with a date, which returns availability rules from the `availability` table and existing appointments from `appointments`. Slots are generated client-side via `generateSlots` in `src/lib/slots.ts`, which respects `sessionDuration` and avoids overlaps.

On submission, `POST /api/booking` validates input with Zod, checks plan limits (e.g., `clients` or `appointmentsPerMonth`), and performs an atomic transaction: it checks for slot conflicts, upserts the `clients` record (matching by phone or email), and inserts the `appointments` record. If `autoConfirm` is enabled, the status is set to `confirmed`; otherwise, it remains `pending_confirmation`.

After success, the system creates a notification for the professional and sends WhatsApp alerts using `msgBookingRequest` via `sendWhatsApp`. Email fallbacks are triggered if the professional has an email. The client sees a success screen, optionally with a Pix QR code if `paymentType` requires upfront payment.

## Design decisions

The API uses a serializable transaction to prevent double-booking, a critical choice given that separate queries could allow race conditions. The honeypot field `website` silently drops bot submissions without revealing the trap. Sensitive fields like `email` and `phone` are excluded from public `GET /api/booking` responses to prevent data leakage. Rate limiting via Upstash (`rl:booking`) caps submissions at 5 per 10 minutes per IP, reducing spam.

## Potential improvements

1. **Client deduplication logic in `src/app/api/booking/route.ts`** uses OR conditions on phone/email but doesn’t enforce uniqueness constraints in the DB. Adding a partial unique index on `(professionalId, phone)` and `(professionalId, email)` would prevent duplicates at the schema level.

2. **Slot generation in `src/lib/slots.ts`** runs client-side after fetching raw rules and appointments. Moving this to the API would reduce client-side logic and allow consistent handling of edge cases like timezone gaps.

3. **WhatsApp message context in `src/app/api/appointments/[id]/route.ts`** passes `professionalId` and `appointmentId` to `sendWhatsApp`, but this data isn’t persisted. Storing `context` in the `notifications` table would enable audit trails and message replay.
