# Customer Booking Flow

The customer booking flow is a core interaction within Odys, enabling clients to discover professionals and schedule appointments efficiently. This document explores the journey from a customer landing on a professional's public profile to successfully booking a time slot, detailing the underlying technical mechanisms, design choices, and potential areas for enhancement.

## Overview

The booking process is orchestrated across several components: a public-facing professional profile page, a client-side booking widget, and a backend API endpoint. The system relies heavily on the `professionals`, `availability`, `appointments`, and `clients` tables to manage professional details, define available time slots, record bookings, and store client information. The `/api/booking` route serves as the central hub for both fetching available slots and submitting new appointment requests.

## Professional Profile and Initial Load

A customer's journey begins by navigating to a professional's public profile page, handled by `src/app/p/[slug]/page.tsx`. This page dynamically fetches the professional's details using their unique `slug`. The database query targets the `professionals` table, retrieving all columns, which include `name`, `profession`, `sessionDuration`, `sessionPrice`, `bio`, and `avatarUrl`. A check ensures the professional exists and is `active`; otherwise, the user is redirected to a "not found" page.

The page also intelligently pre-fills client information if the visitor is a logged-in user who is not the professional themselves. It queries the `clients` table using the `userId` to retrieve `name`, `phone`, and `email`, streamlining the booking form experience for returning users.

Crucially, the `BookingWidget` component, located in `src/app/p/[slug]/booking-widget.tsx`, is rendered on this page. This client-side component is responsible for the interactive date and time selection, acting as the primary interface for the booking process.

## Slot Selection and Availability

Within the `BookingWidget`, customers first select a date using a calendar interface. Upon selecting a date, a `useEffect` hook triggers a `GET` request to the `/api/booking` endpoint. This request includes the professional's `slug` and the chosen `date`.

The `GET` handler in `src/app/api/booking/route.ts` performs two key database queries in parallel:
1.  It fetches the professional's `availability` rules for the selected day of the week, which define the `startTime` and `endTime` for that day.
2.  It retrieves all `existingAppointments` for that professional on the selected date, filtering out any appointments with a `status` of `"rejected"` or `"cancelled"` to ensure only active bookings are considered for conflicts.

A significant design choice here is the selective column fetching for the `professionals` table. Instead of `SELECT *`, the query explicitly requests only `id` and `sessionDuration`. This prevents sensitive professional data like `email`, `phone`, `userId`, `plan`, `trialEndsAt`, `stripeCustomerId`, and `stripeSubscriptionId` from being exposed to anonymous visitors, enhancing data privacy.

The fetched `rules` and `existingAppointments` are then passed to the `generateSlots` utility function in `src/lib/slots.ts`. This function calculates all potential time slots for the day, taking into account the professional's `sessionDuration` and filtering out any slots that conflict with `existingAppointments` or fall in the past relative to the current time. The `BookingWidget` then displays these available slots, grouped by morning, afternoon, and evening, allowing the customer to pick a specific time.

## Booking Submission

Once a date and time slot are selected, the customer proceeds to a form to enter their `name`, `phone`, and optionally `email`. This data, along with the `slug` and `startsAt` time, is submitted via a `POST` request to the `/api/booking` endpoint.

The `POST` handler initiates several critical steps:
1.  **Rate Limiting**: The `getBookingLimiter()` from `src/lib/ratelimit.ts` is applied, restricting booking attempts to 5 per IP address every 10 minutes (`rl:booking`). This helps prevent abuse and spam.
2.  **Response-time Floor**: A deliberate 250ms delay is introduced before responding. This "timing oracle" mitigation helps obscure whether a client's phone or email already exists in the system, making it harder for attackers to infer information based on response times.
3.  **Input Validation**: The incoming `body` is validated using `bookingSchema`, a Zod schema. This ensures that `slug`, `startsAt`, `clientName`, and `clientPhone` meet minimum requirements, and `clientEmail` is a valid email format if provided.
4.  **Honeypot Detection**: The `bookingSchema` includes a `website` field, which is a hidden input in the `BookingWidget`. If this field is populated, the request is identified as a bot submission and silently dropped, returning a fake success response to deter further attempts.
5.  **Professional Lookup**: The professional is retrieved from the `professionals` table using the `slug`.
6.  **Plan Limit Checks**: The system verifies if the professional's current plan (`PLANS` from `src/lib/stripe`) allows for new clients or appointments. It queries the `clients` and `appointments` tables to check against `plan.limits.clients` and `plan.limits.appointmentsPerMonth`. If limits are exceeded and the client is new, the booking is rejected.
7.  **Atomic Transaction for Booking**: The core of the booking logic is wrapped in a Drizzle `db.transaction` with `isolationLevel: "serializable"`. This is a crucial design choice to prevent race conditions.
    *   **Conflict Check**: Inside the transaction, a final check is performed to ensure the chosen slot has not been taken by another concurrent booking. It looks for any `appointments` for the same `professionalId` that overlap with the proposed `startsAt` and `endsAt` times, excluding `rejected` or `cancelled` appointments. If a conflict is found, the transaction is rolled back with a `SLOT_TAKEN` error, which is then surfaced as a 409 Conflict response.
    *   **Client Upsert**: The system attempts to find an existing client for the professional using the provided `clientPhone` or `clientEmail`. If found, the existing client record is used; otherwise, a new client is inserted into the `clients` table. If an existing client is found but lacks an email, and one is provided, the client's email is updated.
    *   **Appointment Insertion**: Finally, the new appointment is inserted into the `appointments` table with a `status` of `"confirmed"` if `professional.autoConfirm` is true, or `"pending_confirmation"` otherwise. The `paymentStatus` is set to `"none"`.

## Confirmation and Notifications

After a successful booking, the system triggers a series of notifications:
*   A new entry is added to the `notifications` table for the professional, indicating a `booking_confirmed` or `booking_request`.
*   WhatsApp messages are sent using `sendWhatsApp` (from `src/lib/whatsapp`) to both the client and the professional, using templates like `msgBookingConfirmed`, `msgBookingConfirmedReceipt`, or `msgBookingRequest` depending on the `autoConfirm` setting.
*   If the professional has an `email` configured, an email notification is sent via `sendBookingRequestEmailToProfessional` (from `src/lib/email`).
*   Analytics events (`booking_received_server`) are captured using Posthog, attributing the booking to the professional and setting a `first_booking_at` property for cohort analysis.

The `BookingWidget` then transitions to a "success" step, displaying a confirmation message. If the professional's `paymentType` is not `"after"` and a `pixKey` is configured, a `PixQR` component is shown for immediate payment. For unauthenticated clients, an invitation to register for an Odys account is presented, with pre-filled data from their booking submission.

## Appointment Management

While the booking flow primarily focuses on the client's perspective, the `src/app/api/appointments/[id]/route.ts` endpoint allows professionals (and in some cases, clients) to manage the status of an appointment. This `PATCH` endpoint supports actions like `"confirm"`, `"reject"`, `"cancel"`, `"paid"`, `"complete"`, and `"no_show"`. Each action triggers specific updates to the `appointments` table and sends corresponding WhatsApp messages and notifications to both parties, ensuring consistent communication throughout the appointment lifecycle.

## Design Decisions

*   **Atomic Booking Transaction**: The use of `db.transaction` with `isolationLevel: "serializable"` in `POST /api/booking` is a critical design choice. It ensures that the conflict check, client upsert, and appointment insertion operations are treated as a single, indivisible unit. This prevents race conditions where two clients might attempt to book the same slot simultaneously, guaranteeing data integrity and preventing double-bookings. Without this, a simple `SELECT` followed by an `INSERT` could allow both requests to pass the `SELECT` check before either `INSERT` commits, leading to an overlap.
*   **Honeypot for Bot Mitigation**: The `website` field in `bookingSchema` and its corresponding hidden input in `BookingWidget` serve as a simple yet effective honeypot. Bots often fill all available form fields, and by detecting a value in this hidden field, the system can silently discard spam submissions without revealing to the bot that it has been detected. This reduces server load and keeps the database clean.
*   **Response-time Floor for Privacy**: The intentional 250ms delay in `POST /api/booking` before returning a response is a subtle but important privacy measure. It helps to mitigate timing attacks, where an attacker might try to infer whether a client's phone or email already exists in the database by measuring the difference in response times between existing and non-existing records. By equalizing the response time, this signal is blunted.
*   **Selective Column Fetching**: In `GET /api/booking`, the decision to `select` only `id` and `sessionDuration` from the `professionals` table, rather than all columns, is a strong adherence to the principle of least privilege. It prevents the accidental exposure of sensitive professional data (like `email`, `phone`, `userId`, and payment-related IDs) to unauthenticated users who are merely querying for slot availability.

## Potential Improvements

1.  **Stronger Typing for Availability Times**: The `startTime` and `endTime` columns in the `availability` table are currently defined as `text`. While functional, using a dedicated `time` type in the database schema (if supported by Drizzle and the underlying database) could offer better semantic clarity, enable database-level validation for time formats, and potentially optimize time-based queries or calculations directly within SQL. This would be a change in `src/lib/db/schema.ts` and reflected in `src/lib/slots.ts`.
2.  **Refined Client Upsert Logic**: The current client upsert logic in `POST /api/booking` uses `or(eq(clients.phone, clientPhone), eq(clients.email, clientEmail))` to find an existing client. This approach could inadvertently merge two distinct clients if they happen to share a phone number (e.g., a family phone) or an email address. A more explicit strategy, such as first attempting to find by `phone`, and if not found, then by `email`, or enforcing unique constraints on `(professionalId, phone)` and `(professionalId, email)` could prevent unintended client data merging.
3.  **Granular Client-Side Error Reporting**: The `BookingWidget` in `src/app/p/[slug]/booking-widget.tsx` currently displays a generic `toast.error("Erro ao realizar agendamento. Tente novamente.")` when the `POST /api/booking` request fails. The server-side `route.ts` provides more specific error messages (e.g., "Limite de clientes atingido", "Este horário acabou de ser preenchido"). The client-side code could be enhanced to parse the `res.json()` for these specific `error` or `details` messages and present more informative feedback to the user, improving the user experience.
