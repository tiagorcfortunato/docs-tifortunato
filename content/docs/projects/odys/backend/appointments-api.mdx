# Appointment Action Endpoint: confirm/reject/cancel/paid/complete/no_show

This endpoint powers the core lifecycle management of appointments in Odys, enabling both professionals and clients to drive state changes through a single, action-driven `PATCH` request. It reflects the real-world flow of scheduling interactions—confirmation, cancellation, payment capture, and session completion—while enforcing ownership and business rules.

## Overview

The `/api/appointments/[id]` route is one of 21 API endpoints in the Odys backend, uniquely responsible for mutating appointment states. It interacts primarily with the `appointments`, `professionals`, `clients`, and `notifications` tables from the 10-table schema. The endpoint leverages Drizzle ORM to query and update records, validates user permissions based on JWT-derived `userId`, and triggers side effects like WhatsApp messages using predefined templates such as `msgBookingConfirmed` or `msgPaymentConfirmedToPro`.

## State Transition Logic

The endpoint accepts an `action` parameter that determines the transition: `confirm`, `reject`, `cancel`, `paid`, `complete`, or `no_show`. These actions map directly to updates on the `status` and `paymentStatus` columns in the `appointments` table. For example, `paid` sets `paymentStatus` to `"captured"`, while `complete` sets `status` to `"completed"`. Only professionals can mark payments or session outcomes, whereas both parties can cancel within allowed states (`pending_confirmation`, `confirmed`).

Notifications are inserted into the `notifications` table when relevant—such as informing a client that their booking was rejected—and WhatsApp messages are dispatched using context-aware templates. The use of `context` in `sendWhatsApp` allows tracking via `professionalId` and `appointmentId`, linking messages back to core entities.

## Design decisions

The `ALLOWED_ACTIONS` set prevents invalid or typo-prone actions from silently failing, addressing a past bug where `"cancle"` defaulted to rejection. Permission checks are centralized: `isProfessional` and `isClient` determine access before action routing. This avoids scattered authorization logic and reduces duplication. The endpoint also avoids over-fetching by limiting selects to one row and uses `.catch(console.warn)` on message sends to prevent notification failures from rolling back database updates.

## Potential improvements

1. **Move WhatsApp message construction into dedicated service functions** — In `src/app/api/appointments/[id]/route.ts`, the inline template calls (e.g., `msgBookingConfirmed(...)`) clutter the main logic. Extracting these into a `notificationService` would improve readability and testability.

2. **Add audit logging for status changes** — Currently, there’s no record of *who* performed an action or *when* it occurred beyond `updatedAt`. Introducing an `appointment_logs` table would support accountability and debugging.

3. **Use database transactions for atomic updates** — The current code updates `appointments` and inserts `notifications` in separate queries. Wrapping related operations in `db.transaction()` would ensure consistency, especially when sending notifications conditionally after a status change.
