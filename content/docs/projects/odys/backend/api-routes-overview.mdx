# API Routes Overview

This page documents the backend API structure of the Odys platform, detailing the available endpoints, shared utilities, and authentication patterns. Understanding this layer is essential for extending functionality, debugging integrations, or securing routes—especially as the system scales across professionals, clients, and third-party services like Stripe and WhatsApp.

## Overview

The backend exposes **23 API routes** across public, authenticated, and cron-triggered contexts. These routes interact with **10 database tables**, including `professionals`, `appointments`, `clients`, and `messages`, forming a cohesive system for managing bookings, payments, and communications. Key operations—like creating appointments, handling Stripe webhooks, or sending WhatsApp reminders—are encapsulated in dedicated endpoints, many protected by rate limiting and cron authorization.

Notably, the API supports both user-facing actions (e.g., `POST /api/booking`) and automated workflows (e.g., `GET /api/cron/reminders`), with shared helpers ensuring consistent responses and secure authentication. The presence of MCP (Model Context Protocol) tools like `get_appointments_today` and `update_appointment_status` suggests integration with AI agents, while 19 WhatsApp message templates indicate deep messaging automation.

## Shared Helpers: Consistency and Security

The `src/lib/api.ts` file provides foundational utilities used across all routes. Response helpers like `ok`, `badRequest`, and `serverError` standardize JSON output and status codes, ensuring clients receive predictable payloads—even during errors, where internal details are logged but never exposed.

Authentication is handled via `getUser` and `getProfessional`, which chain Supabase auth with a Drizzle ORM lookup to resolve a professional’s profile from their user ID. This pattern ensures that only registered professionals can access protected routes, and it’s reused across endpoints like `/api/settings` and `/api/messages`.

Security-critical utilities include `getIp` for rate limiting and `isCronAuthorized`, which uses constant-time comparison (`safeEqual`) to validate cron secrets—resisting timing attacks whether the header is `Authorization: Bearer <secret>` or `x-cron-secret: <secret>`. This dual-header support allows both Vercel cron jobs and manual testing.

## Design decisions

The API design prioritizes **reusability** and **security**. By centralizing response and auth logic, the code avoids duplication and enforces consistency. Using `NextRequest` and `NextResponse` aligns with Next.js App Router patterns, while Drizzle ORM offers type-safe database access.

Rate limiting is implicitly supported via `getIp`, enabling middleware integration (not shown). The cron authorization model is flexible, supporting multiple invocation methods without compromising security.

The use of Portuguese error messages (e.g., "Não autorizado") suggests a Brazil-focused initial market, aligning with Pix payment support in the `professionals` table.

## Potential improvements

1. **`src/lib/api.ts`: `getProfessional` lacks error handling** — If the database query throws, it will bubble up uncaught. Wrap in a try/catch and use `serverError` to maintain consistency.

2. **`src/lib/api.ts`: `safeEqual` could misbehave with non-ASCII secrets** — Since `Buffer.from(string)` defaults to UTF-8, but timing-safe comparison assumes uniform encoding, consider validating input or documenting encoding requirements.

3. **`src/lib/api.ts`: Response helpers don’t support optional metadata** — Adding fields like `timestamp` or `requestId` would aid debugging. Consider enhancing `ok(data, meta?)` to accept optional context.
