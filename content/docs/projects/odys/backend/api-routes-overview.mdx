# API Routes Overview

This document maps the entire backend API surface of Odys, a practice management platform for professionals. Understanding the API structure is essential for extending features, debugging flows, and maintaining consistency across auth, error handling, and data access. Every endpoint adheres to a shared set of helpers that enforce uniform responses and secure user resolution.

## Overview

Odys exposes **21 API routes** across core domains like appointments, messaging, payments, and user management. The backend operates on a **10-table Drizzle ORM schema**, centered around the `professionals` table, which links to clients, appointments, messages, and reviews. Key subsystems include Stripe integration (`/api/stripe/*`), cron-triggered automation (`/api/cron/*`), and WhatsApp messaging via webhooks. Two cron jobs run daily: reminder dispatch at 8 AM and a WhatsApp watchdog at 9 AM. Rate limiting is enforced on critical paths like booking and registration, using Redis-backed limits to prevent abuse.

The API design emphasizes consistency through shared utilities. Every route imports response helpers like `ok`, `badRequest`, and `serverError` from `src/lib/api.ts`, ensuring standardized JSON payloads and status codes. Authentication is abstracted via `getUser` and `getProfessional`, which resolve the current user from Supabase auth and their associated professional profile. This pattern reduces boilerplate and enforces security at the edge.

## Design decisions

The current structure favors **composability and maintainability** over fragmentation. By centralizing response logic and auth resolution, the codebase avoids duplication and ensures that changesâ€”like updating error formatsâ€”propagate globally. The use of Drizzle ORM with explicit schema definitions enables type-safe queries while keeping SQL intent clear. Routes are grouped by concern (e.g., `/api/messages`, `/api/recurring`) rather than HTTP method, improving discoverability. Cron endpoints are isolated and secured, reflecting their scheduled nature and need for external access (e.g., via Vercel Cron Jobs).
