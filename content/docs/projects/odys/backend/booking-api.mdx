# Booking GET + POST endpoint internals

The `/api/booking` endpoint is the core interaction point between clients and professionals in Odys, enabling real-time availability checks and secure appointment booking. This document explores how the GET and POST handlers leverage the 10-table schema to deliver reliable, rate-limited, and plan-aware scheduling while maintaining data consistency across `professionals`, `availability`, `appointments`, and `clients`.

## Overview

Odys’s booking system operates on a schema with 10 tables, where `professionals`, `availability`, `appointments`, and `clients` form the scheduling backbone. The `/api/booking` route supports two operations: **GET** for fetching a professional’s availability on a given date, and **POST** for creating a new appointment. Both are protected by rate limiting (`rl:booking`, 5 requests per 10 minutes) and integrate with WhatsApp and email notifications. The logic enforces plan-based limits from the `PLANS` configuration—Free, Basic, Pro, and Premium—via `effectivePlan`, ensuring tenants stay within their tier’s constraints.

## GET: Fetching Availability

The GET handler retrieves a professional’s weekly availability rules and existing appointments for a specific day. It first looks up the professional by `slug`, then concurrently fetches their `availability` (weekly time slots) and `appointments` (daily booked slots). The query filters out `rejected` or `cancelled` appointments using `notInArray`, ensuring only active bookings block time. This two-layer model—static rules + dynamic appointments—allows flexible scheduling while preventing double-booking.

## POST: Creating Appointments

The POST handler orchestrates client upserting, conflict detection, plan limit enforcement, and notifications. It validates input via `bookingSchema`, ensuring required fields like `clientPhone` and `startsAt` are present. If the client doesn’t exist, it’s created via `insert(clients)`; otherwise, their email is updated if missing. The system checks for time overlap using `lt(appointments.startsAt, endDate)` and `gt(appointments.endsAt, startDate)`, rejecting conflicting slots. After insertion, a notification is sent to the professional via `notifications` table, WhatsApp (`sendWhatsApp`), and optionally email.

## Design decisions

The code uses optimistic conflict detection instead of transactions to avoid database contention, accepting a small race condition risk in favor of simplicity. Client identity is resolved by phone or email, prioritizing usability over strict uniqueness. Rate limiting is applied at the IP level using `getIp(req)`, protecting against abuse without requiring authentication. The use of `returning()` ensures immediate access to inserted records, reducing round trips.

## Potential improvements

1. **Move client deduplication into database** — In `src/app/api/booking/route.ts`, the client upsert uses application-level logic. Adding a partial unique index on `(professionalId, phone)` and using `onConflictDoUpdate` in Drizzle would make this atomic and safer under concurrency.

2. **Prevent notification spam on retries** — The `sendWhatsApp` and `sendBookingRequestEmailToProfessional` calls are fire-and-forget. If the client retries the POST due to timeout, multiple notifications may send. Storing notification state in the `notifications` table and checking before sending would prevent duplicates.

3. **Enforce plan limits consistently** — The plan limit check in `POST` handles `clients` and `appointmentsPerMonth`, but the schema’s `PLANS` object shows other potential limits. Centralizing this logic in `effectivePlan` or a dedicated guard would ensure all endpoints apply limits uniformly.
