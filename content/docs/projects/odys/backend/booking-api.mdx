# Booking API Internals

This document explores the implementation of the `/api/booking` endpoint, which powers client scheduling in the Odys platform. It handles both querying available time slots (GET) and creating new appointments (POST), enforcing business rules around availability, plan limits, and concurrency.

## Overview

The booking system interacts with `professionals`, `availability`, `appointments`, `clients`, and `notifications` tables. Two HTTP methods—GET and POST—serve distinct but related purposes: fetching scheduling rules and booking slots. The endpoint integrates with WhatsApp and email notifications, respects rate limits via `getBookingLimiter`, and enforces plan-based quotas defined in `PLANS`. It also uses a honeypot field (`website`) to silently reject bot submissions.

## GET: Fetching Availability Rules

The GET handler returns time rules and existing appointments for a given professional (identified by `slug`) and date. It avoids over-fetching by selecting only `id` and `sessionDuration` from `professionals`, preventing leakage of sensitive fields like `email`, `userId`, or `stripeCustomerId`. It then retrieves weekly availability patterns from the `availability` table and booked slots from `appointments`, filtering out `rejected` or `cancelled` entries. This two-query pattern is efficient and avoids N+1 issues.

## POST: Creating Appointments

The POST handler validates input using Zod, checks for bot activity via the `website` honeypot, and verifies the professional exists. It enforces plan limits—client count and monthly appointments—by querying the `clients` and `appointments` tables with `count()`. The client count limit specifically applies to *new* clients; existing clients can continue to book even if the professional's client limit is reached. The core scheduling logic runs in a serializable transaction to prevent race conditions: it checks for time conflicts, upserts the client (matching by phone or email), updating the client's email if an existing client is found without one and a new email is provided, and creates the appointment. If the professional has `autoConfirm` enabled, the status is set to `confirmed`; otherwise, it remains `pending_confirmation`.

Upon success, it creates a notification, sends a WhatsApp message using `msgBookingRequest`, and optionally emails the professional. WhatsApp and email notifications are fire-and-forget, with errors caught and logged but not blocking the response. Notification persistence **is awaited** to ensure it is reliably recorded before responding to the client.

## Design decisions

The code prioritizes data safety and concurrency control. The switch from `SELECT *` to explicit column selection in GET prevents accidental PII exposure. The use of a serializable transaction in POST eliminates race conditions during booking, a critical safeguard in high-traffic scenarios. The honeypot pattern avoids challenging bots directly, instead feeding them a fake success to reduce spam. Plan limits are evaluated dynamically using `effectivePlan`, allowing trial states to influence access without duplicating logic.

## Potential improvements

1. **Avoid redundant plan lookup in POST**: The `effectivePlan` function is called, but `professional.plan` is already known. Since `trialEndsAt` is nullable, the logic could be inlined unless trial checks are complex—currently, it adds indirection without clear benefit (`src/app/api/booking/route.ts`, `effectivePlan` call).

2. **Client upsert logic could be simplified**: The OR condition on phone/email in client lookup may cause mismatches (e.g., same phone, different email). Consider standardizing on phone as the primary key, as it's required and unique per professional (`src/app/api/booking/route.ts`, client upsert block).

3. **Unified error handling for post-booking side effects**: While notification persistence is correctly awaited, the WhatsApp and email notifications are fire-and-forget. To improve observability, consider wrapping all non-critical side effects (including `db.insert(notifications)`) in a `Promise.all` with error handling to log failures collectively, ensuring no silent drops (`src/app/api/booking/route.ts`, notification block).
