# Booking API

The `/api/booking` route serves as the central hub for managing appointment bookings within the Odys platform. It exposes both a `GET` endpoint for clients to discover available time slots for a professional and a `POST` endpoint for submitting new appointment requests. This API is critical for the core functionality of Odys, connecting clients with professionals and orchestrating the complex logic of scheduling, availability, and notifications.

## Overview

The booking API interacts with several key database tables to fulfill its responsibilities. The `professionals` table provides details about the service providers, including their session duration and auto-confirmation preferences. `availability` defines the recurring time slots a professional offers. `appointments` stores the actual scheduled meetings, while `clients` manages client information. Finally, `notifications` are generated to inform both professionals and clients about booking events.

The `GET` request to `/api/booking` allows an unauthenticated user to query a professional's schedule for a specific day, returning their general availability rules and any existing appointments that might block slots. The `POST` request, on the other hand, handles the creation of a new appointment, incorporating validation, rate limiting, plan-based restrictions, and a sophisticated transaction mechanism to prevent scheduling conflicts.

## GET Endpoint: Retrieving Availability

The `GET` method for `/api/booking` is designed to provide potential clients with the necessary information to choose an available time slot for a professional. It expects two query parameters: `slug`, which uniquely identifies the professional, and `date`, representing the day for which availability is being queried.

Upon receiving a request, the endpoint first validates these parameters. If `slug` or `date` parameters are missing, a `400 Bad Request` error with message `Parâmetros inválidos` is returned. It then queries the `professionals` table, specifically selecting only the `id` and `sessionDuration` columns for the professional identified by the `slug`. This selective retrieval is a deliberate security measure, preventing the accidental exposure of sensitive professional data like `email`, `phone`, or `stripeCustomerId` to unauthenticated users. If no professional is found, a `404` error is returned.

Next, the endpoint concurrently fetches two sets of data:
1.  **Availability Rules**: It retrieves all entries from the `availability` table associated with the professional's `id`. These rules define the professional's general working hours for each `dayOfWeek`, specified by `startTime` and `endTime`.
2.  **Existing Appointments**: It queries the `appointments` table for all appointments belonging to the professional within the specified `date` range. Crucially, it filters out appointments with a `status` of `rejected` or `cancelled`, as these slots are considered available.

The response combines these two sets of data, allowing the client-side application to calculate and display the truly available booking slots by overlaying the existing appointments onto the general availability rules.

## POST Endpoint: Creating a Booking

The `POST` method to `/api/booking` is responsible for processing new appointment requests. This endpoint is significantly more complex due to the need for data validation, concurrency control, and a multi-channel notification system.

The process begins with a rate limit check using `getBookingLimiter` and the client's IP address. This helps prevent abuse and ensures fair access to the booking system. If the rate limit is exceeded, a `429 Too Many Requests` response is returned.

A critical security measure immediately following the rate limit is the implementation of a response-time floor. The `respondAfterFloor` function ensures that all responses, regardless of whether a client record existed or was created, take at least 250 milliseconds. This mitigates a "timing oracle" vulnerability, where an attacker could infer the existence of a client record by observing subtle differences in response times between database `SELECT` and `INSERT` operations.

The incoming request body undergoes validation using a Zod schema, `bookingSchema`. If validation fails, a `400 Bad Request` error with message `Dados inválidos` and `details` of field errors is returned. This schema enforces requirements for `slug`, `startsAt`, `clientName`, `clientPhone`, and optionally `clientEmail`. It also includes a "honeypot" field, `website`. If this hidden field is populated by a bot, the request is silently dropped, returning a fake success response to deter further bot activity without engaging the database.

After validation, the system retrieves the professional's full details from the `professionals` table using the provided `slug`. If the professional is not found, a `404` error is returned.

### Plan Limit Enforcement

Before proceeding with the booking, the system checks the professional's current plan using `effectivePlan` and `PLANS`. Odys offers different subscription tiers (`free`, `basic`, `pro`, `premium`), each with potential limits on the number of clients and appointments per month.

*   **Client Limit**: If a client limit is active, the system counts existing clients for the professional. If the count exceeds the limit, it checks if the current booking request is from an existing client (identified by `clientPhone`). Only new clients are blocked if the limit is reached. If the client limit is reached for a new client, a `403 Forbidden` error with message `Limite de clientes atingido para este profissional.` is returned.
*   **Monthly Appointment Limit**: If a monthly appointment limit is active, the system counts appointments for the current month. If the limit is exceeded, the booking is rejected. If the monthly appointment limit is exceeded, a `403 Forbidden` error with message `Limite mensal de agendamentos atingido.` is returned.

These checks ensure that professionals operate within the bounds of their chosen subscription plan.

### Atomic Booking Transaction

The core of the `POST` endpoint is an atomic database transaction, executed with a `serializable` isolation level. This is a crucial design choice to guarantee data integrity and prevent race conditions, especially when multiple clients might attempt to book the same time slot simultaneously.

Within this transaction:
1.  **Conflict Check**: The system first checks for any existing appointments that would overlap with the proposed `startsAt` and `endsAt` times for the professional. It specifically excludes `rejected` and `cancelled` appointments from this conflict check. If an overlap is detected, the transaction is aborted with a custom `SLOT_TAKEN` error, which is then translated into a `409 Conflict` response for the client. If an overlap is detected, a `409 Conflict` error with message `Este horário acabou de ser preenchido por outra pessoa. Por favor, escolha outro horário.` is returned. The `serializable` isolation level is vital here, as it prevents "phantom reads" where a concurrent transaction might insert a conflicting row *after* the initial conflict check but *before* the new appointment is inserted.
2.  **Client Upsert**: The system attempts to find an existing client record for the professional based on the provided `clientPhone` or `clientEmail`. If a client is found, their record is updated (specifically, the `email` is added if it was previously `null`). If no client is found, a new entry is inserted into the `clients` table.
3.  **Appointment Insertion**: Finally, the new appointment is inserted into the `appointments` table. The `status` of the appointment is set to `confirmed` if `professional.autoConfirm` is `true`, otherwise it defaults to `pending_confirmation`. The `paymentStatus` is initialized to `none`.

### Notifications and Analytics

After a successful booking (or request), a series of notifications are triggered:
*   **Database Notification**: A new entry is inserted into the `notifications` table for the professional, informing them of a `booking_confirmed` or `booking_request`.
*   **WhatsApp Messages**: Depending on whether the appointment is `confirmed` or `pending_confirmation`, different WhatsApp messages are sent. If confirmed, the client receives a `msgBookingConfirmed` and the professional receives a `msgBookingConfirmedReceipt`. If pending, the professional receives a `msgBookingRequest` to prompt their action. These messages use templates like `msgBookingRequest`, `msgBookingConfirmed`, and `msgBookingConfirmedReceipt`.
*   **Email Notification**: If the professional has an `email` configured, an email notification is sent via `sendBookingRequestEmailToProfessional`.

Finally, analytics events are captured using `capture` and `setOnce`. A `booking_received_server` event is recorded, attributed to the professional's `id`, along with properties like `slug`, `plan`, and `auto_confirmed`. A `first_booking_at` property is set once for each professional to track their initial booking time.

The `POST` endpoint concludes by returning an equalized `success: true` response, again to avoid leaking information through varying response bodies, consistent with the timing oracle mitigation.

## Design Decisions

*   **Minimal Data Exposure in GET**: The `GET` endpoint for `/api/booking` explicitly selects only `id` and `sessionDuration` from the `professionals` table. This was a conscious decision to prevent the accidental leakage of sensitive professional data (like `email`, `phone`, `userId`, `plan`, `trialEndsAt`, `stripeCustomerId`, `stripeSubscriptionId`) to unauthenticated users, which was a vulnerability in a previous iteration of the code.
*   **Rate Limiting**: The `getBookingLimiter` is applied to the `POST` endpoint to protect against brute-force attacks and ensure fair resource usage. This is a standard practice for public-facing APIs that involve resource creation.
*   **Timing Oracle Mitigation**: The `respondAfterFloor` function introduces an artificial delay to equalize response times for the `POST` endpoint. This design choice directly addresses a timing oracle vulnerability, making it harder for attackers to distinguish between existing and non-existing client records based on database query performance.
*   **Honeypot Field**: The `website` field in `bookingSchema` acts as a honeypot. By returning a fake success response for submissions that fill this hidden field, the system effectively thwarts bot submissions without revealing the trap, thus reducing spam and unnecessary database operations.
*   **Atomic Transaction with Serializable Isolation**: The use of `db.transaction` with `serializable` isolation level for the `POST` request is a critical architectural decision. It ensures that the conflict check, client upsert, and appointment insertion operations are treated as a single, indivisible unit. This prevents race conditions, such as two users simultaneously booking the same slot, by forcing the database to detect and reject one of the conflicting transactions (SQLSTATE 40001).
*   **Multi-channel Notifications**: The system uses a combination of database notifications, WhatsApp messages, and email to inform users about booking events. This multi-channel approach increases the likelihood that important information reaches both clients and professionals, catering to different communication preferences and ensuring timely updates. The branching logic based on `professional.autoConfirm` ensures the message content is appropriate for the appointment's status.
*   **Equalized POST Response Body**: Similar to the timing oracle mitigation, the `POST` endpoint returns a generic `{ success: true }` body. This prevents an attacker from gaining information about the internal state (e.g., whether a client was new or existing, or the exact `clientId`) by analyzing the response payload.

## Potential Improvements

1.  **Robust Error Handling for External Communications**: The `sendWhatsApp` and `sendBookingRequestEmailToProfessional` calls currently use `.catch((err) => console.warn(...))`. While this prevents the main booking transaction from failing due to a notification issue, it only logs a warning. A more robust solution could involve a dedicated retry mechanism, pushing failed notifications to a queue for asynchronous processing, or integrating with an observability platform to alert on persistent communication failures. This would ensure that critical notifications eventually reach their recipients even if external services experience transient issues.
2.  **Refined Client Email Update Logic**: In the `POST` endpoint's client upsert logic, the `client.email` is only updated if `!client.email && clientEmail`. This means if a client already has an email address on file, and a new booking request comes in with a *different* email for the same phone number, the existing email will not be overwritten. This might be an intentional design choice, but it could lead to stale or incorrect email information if clients update their email addresses. Consider whether the intent is to always update the email if a new one is provided, or to only fill it if missing.
3.  **Explicit Timezone Handling**: The booking logic relies on `Date` objects and `date-fns` functions like `startOfDay`, `endOfDay`, `addMinutes`, and `toLocaleDateString`/`toLocaleTimeString`. While Drizzle's `timestamp` type typically implies UTC storage, the interpretation and display of these dates can be ambiguous without explicit timezone management. For example, `startOfDay(date)` might behave differently depending on the server's local timezone versus the professional's or client's expected timezone. Introducing a consistent timezone strategy (e.g., always converting to UTC for storage and calculations, and converting to the user's preferred timezone for display) would prevent potential off-by-one day or hour errors in scheduling.
