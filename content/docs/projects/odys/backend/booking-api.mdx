# Booking API Internals

The `/api/booking` endpoint powers the core scheduling workflow in odys, enabling clients to view availability and book appointments with professionals. This document explores the implementation of both the `GET` (fetch availability) and `POST` (create booking) methods, highlighting security, performance, and consistency decisions baked into the code.

## Overview

The booking system relies on five key tables: `professionals`, `availability`, `appointments`, `clients`, and `notifications`. The `GET` route returns time rules and existing bookings for a given professional (identified by `slug`) and date. The `POST` route validates input, checks plan limits, prevents double-booking via serializable transactions, and triggers notifications. Rate limiting is enforced using `getBookingLimiter`, and honeypot fields help filter bot submissions.

## Design decisions

The `GET` handler avoids data leakage by selecting only `id` and `sessionDuration` from `professionals`, unlike earlier versions that used `SELECT *` and exposed sensitive fields like `email`, `userId`, and Stripe identifiers. The `POST` handler uses a serializable transaction to prevent race conditions when checking appointment conflicts — a necessity given that two concurrent requests could otherwise pass a non-atomic check. A 250ms response-time floor masks timing differences between known and unknown clients, mitigating timing oracle attacks. The honeypot `website` field returns a fake success to bots without hitting the database, reducing load and noise.

## Potential improvements

The `POST` handler performs a client upsert using phone or email, but the condition `or(...clientConditions)` could lead to inconsistent merging if a client changes email or phone. Consider enforcing phone as the primary identifier and treating email as secondary. The plan limit check for `clients` does not short-circuit when the client already exists, leading to an unnecessary count query — cache or pre-fetch the client before the limit check in `src/app/api/booking/route.ts`. Finally, the `sendWhatsApp` calls are fire-and-forget with `.catch()`, but failures aren’t retried or logged to a durable queue; consider using a task queue like `cron`-driven retries in `/api/cron/whatsapp-watchdog` for reliability.
