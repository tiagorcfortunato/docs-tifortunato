# Backend Overview: API Route Tree, Shared Helpers, Auth Pattern

The Odys backend powers a professional booking and client management platform, balancing scalability, security, and developer ergonomics. This document maps the structural foundation of the API—its routes, shared logic, and authentication model—providing context for maintainability and future growth.

## Overview

Odys operates on a 10-table Drizzle schema, centered around `professionals` as the core entity, with related data for `appointments`, `clients`, `messages`, and `recurringSchedules`. The API exposes 21 endpoints across functional domains: booking, messaging, payments, notifications, and cron jobs. Authenticated routes rely on Supabase JWTs, with rate limiting enforced via IP hashing and Redis-backed counters. Critical workflows—like Stripe webhooks and WhatsApp integrations—are isolated under `/api/webhook` and `/api/cron`, ensuring separation of concerns.

## API Structure and Routing

The route tree is organized by feature rather than resource, favoring action-oriented paths. For example, `/api/booking` handles both creation and availability checks, while `/api/recurring` manages recurring schedule mutations. Administrative tasks like `/api/cron/reminders` (scheduled at `0 8 * * *`) run via cron, reducing load on interactive routes.

Authentication is abstracted through shared helpers in `src/lib/api.ts`. Every protected route uses `getUser()` to extract the Supabase user, then `getProfessional()` to resolve the corresponding row in the `professionals` table via `userId`. This two-step flow decouples auth from business logic and ensures consistent access control.

Rate limiting is applied contextually: `/api/booking` allows 5 requests per 10 minutes, while general API usage is capped at 60 per minute. Limits are enforced using the `getIp()` utility, which extracts the `x-forwarded-for` header for distributed rate tracking.

## Design Decisions

The backend favors simplicity and composability. Response helpers like `ok()`, `badRequest()`, and `serverError()` standardize JSON output and prevent accidental leakage of internal errors. The use of Drizzle ORM enables type-safe queries while remaining close to SQL, reducing impedance mismatch.

Auth is intentionally minimal: no role-based access control beyond professional membership, and no session tokens stored in the database. This reduces attack surface and aligns with Supabase’s native auth model. The `userId` field in `professionals` and `clients` serves as the foreign key to Supabase auth, avoiding duplication.

Cron jobs are exposed as API routes for operational simplicity—deployed on Vercel, they’re triggered via external schedulers. While not RESTful, this pattern leverages existing infrastructure without requiring a separate worker layer.

## Potential Improvements

1. **Extract rate limit configuration into a centralized registry**  
   Currently, rate limits are hardcoded in middleware or route handlers. Moving them to a shared config (e.g., `rateLimits` object) would improve consistency and enable dynamic tuning. See `src/lib/api.ts` for scattered usage of `tooManyRequests`.

2. **Add input validation schema for POST routes**  
   Routes like `/api/booking` and `/api/messages` accept raw JSON without validation. Introducing Zod schemas would prevent invalid data from reaching the database and improve error clarity. No validation layer is currently present in `src/lib/api.ts`.

3. **Refactor `getProfessional` to include authorization checks**  
   The helper returns the professional row but doesn’t enforce active status or plan limits. Enhancing it to reject inactive or expired accounts early (e.g., checking `trialEndsAt`) would reduce redundant checks in individual routes. See `getProfessional()` in `src/lib/api.ts`.
