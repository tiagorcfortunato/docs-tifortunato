# Backend Overview

This page covers the backend overview, including the API route tree, shared helpers, and auth pattern.

## API Route Tree
The API route tree is structured around the App Router, with no `pages/` directory. Auth pages live in the `(auth)` route group, which shares the same URL space but has its own layout and no `/auth` prefix. The dashboard and client portal each have their own layouts with server-side auth guards. The public booking page (`/p/[slug]`) is an async Server Component that performs DB reads server-side before handing off to `BookingWidget` for interactivity.

## Shared Helpers
Shared helpers for API routes are defined in `src/lib/api.ts`. These helpers include response helpers (e.g., `ok`, `unauthorized`, `forbidden`), auth helpers (e.g., `getUser`, `getProfessional`), and utility helpers (e.g., `getIp` for rate limiting). Every route uses these building blocks to standardize JSON responses, authenticate users, and extract client IPs.

## Auth Pattern
The auth pattern relies on Supabase for end-to-end authentication. `src/lib/supabase/server.ts` wraps the Supabase server client with a Next-cookie adapter, while `src/lib/supabase/client.ts` handles browser-side sign-out and signup. The register route uses the anon client in production to send verification emails. Smart-linking is gated inside `/auth/callback` to ensure it only runs after verifying the user's inbox.

## Known Gaps
One known gap in the current implementation is the lack of a centralized middleware for authentication. Instead, every route that requires authentication must call `supabase.auth.getUser()` itself via `getUser()` and `getProfessional(userId)` in `src/lib/api.ts`. This approach is explicit but puts the burden on every new route author, and the security audit flags this as a potential issue.

## Why This Shape
The backend is structured around the App Router to take advantage of its server-side rendering and authentication capabilities. The shared helpers in `src/lib/api.ts` provide a standardized way of handling responses, authentication, and utility functions across all routes. The auth pattern relies on Supabase to handle end-to-end authentication, with a focus on verifying user inboxes and smart-linking. This shape allows for a flexible and scalable backend architecture that can adapt to changing requirements.
