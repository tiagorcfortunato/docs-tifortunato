# Backend Overview

This document provides a structural understanding of the backend system powering Odys, focusing on the API architecture, shared utilities, and authentication flow. Understanding this layer is essential for extending features, debugging issues, and maintaining consistency across endpoints.

## Overview

The backend consists of **23 API routes** organized under `/api`, including public, authenticated, and cron-triggered endpoints. These routes interact with a **10-table Drizzle ORM schema** that models professionals, clients, appointments, messages, and subscriptions. Key subsystems include Stripe integration for payments, Supabase for auth, and WhatsApp messaging via webhook handlers. Rate limiting is enforced on critical paths like booking and registration using Redis-backed limits, and two cron jobs run daily for reminders and system health checks.

## API Structure and Patterns

The API surface is logically grouped: user-facing actions like `POST /api/booking` and `GET /api/messages`, administrative tasks like `PATCH /api/settings`, and third-party integrations such as `POST /api/stripe/webhook` and `POST /api/whatsapp/webhook`. Specialized routes under `/api/cron` are protected by `CRON_SECRET` and execute time-based jobs like sending 24-hour appointment reminders.

Authentication relies on Supabase JWTs. The `getUser()` helper extracts the authenticated user from the session, while `getProfessional(userId)` links that user to their profile in the `professionals` table via the `userId` text column. This pattern avoids duplicating auth logic across endpoints and ensures consistent access control.

Shared response helpers like `ok()`, `unauthorized()`, `forbidden()`, `notFound()`, `badRequest()`, `conflict()`, `tooManyRequests()`, and `serverError()` standardize JSON output and error handling. Notably, `serverError()` logs internally but returns a generic message to prevent information leakage—a deliberate security choice.

## Design Decisions

The use of **Drizzle ORM** enables type-safe database access across the stack, especially valuable given the 24 columns in the `professionals` table. Foreign key constraints (e.g., `appointments.professionalId` referencing `professionals.id`) ensure referential integrity with cascading deletes, simplifying cleanup.

Rate limiting is applied selectively: `booking` allows 5 attempts per 10 minutes, while general API access permits 60 requests per minute. These are implemented using prefixed Redis keys like `rl:booking`, allowing isolation and monitoring per endpoint.

Cron jobs use a dual-header authorization model (`Authorization: Bearer <secret>` or `x-cron-secret`) to support both Vercel’s native scheduler and manual testing. The `safeEqual()` function prevents timing attacks during secret comparison—an important detail when defending against side-channel exploits.

## Potential Improvements

1. **Improve error context in `serverError()`** — in `src/lib/api.ts`, the function logs raw errors. While it accepts a `context` string, its current usage might not fully leverage structured context (e.g., request path, user ID). Enhancing the `context` parameter's usage would aid debugging without exposing internals.

2. **Centralize rate limit configuration** — currently hardcoded in middleware or route handlers, limits like `5/10m` for booking should be defined once and reused, reducing drift risk.

3. **Refactor `getProfessional()` to handle errors explicitly** — it returns `null` on not-found but doesn’t distinguish between DB errors and missing records. Wrapping in a try/catch would make failure modes clearer.
