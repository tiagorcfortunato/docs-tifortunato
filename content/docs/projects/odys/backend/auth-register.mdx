# Auth + Register Endpoints, Smart-Linking, C1 Fix

This document covers the core authentication and user registration flow in Odys, a system designed for professionals managing client appointments. It explains how users sign up, how identity is verified via Supabase, and how the system intelligently links authenticated users to existing client records—ensuring data continuity without manual intervention.

## Overview

Odys operates on a 10-table schema where identity and access are managed through Supabase Auth, while user roles (`professional`, `client`) are tracked via metadata and linked records. The `/api/auth/register` endpoint handles new signups with rate limiting (`rl:register`, 5 attempts/hour), while the `/auth/callback/route.ts` handles post-verification logic, including a critical *smart-linking* step that binds verified emails to unclaimed client profiles.

Registration behavior differs between environments: in production, `signUp()` triggers email confirmation; in development, the service role client creates pre-confirmed users for faster iteration. This duality is managed via `SUPABASE_SERVICE_ROLE_KEY` and `NODE_ENV`, ensuring realistic flows in production while accelerating local testing.

## Smart-Linking: Bridging Anonymous and Authenticated States

A key challenge in Odys is onboarding clients who may have been created manually (e.g., via admin UI) before having an account. The smart-linking mechanism solves this: when a client verifies their email, the system checks for unlinked `clients` records with matching emails and updates the `userId` field.

This occurs in `src/app/auth/callback/route.ts` after `exchangeCodeForSession`. Using Drizzle ORM, it runs:
```ts
await db.update(clients)
  .set({ userId: user.id })
  .where(and(isNull(clients.userId), eq(clients.email, user.email)))
```
This ensures that once a user verifies their email, they immediately gain access to their historical data—appointments, notes, messages—without manual association.

## Design Decisions

The use of `user_metadata.type` (`professional` | `client`) allows role-based routing post-login. However, the actual authorization is enforced at the API layer by checking ownership via `professionalId` or `clientId` in queries.

The decision to skip email verification in development using `admin.createUser({ email_confirm: true })` trades security for velocity. It avoids the need for test inboxes or manual link clicking, but relies on strict environment separation.

Rate limiting via `getRegisterLimiter()` (5/hour) prevents abuse while allowing normal usage. This is part of a broader rate-limiting strategy (`rl:booking`, `rl:api`, etc.) using Redis-backed tokens.

## Potential Improvements

1. **Move type assignment to registration metadata**  
   In `src/app/api/auth/register/route.ts`, the `type` is passed in request body but only conditionally applied in callback. Instead, always store `type` during signup via `options.data`, eliminating reliance on query param persistence.

2. **Add telemetry on smart-link success**  
   In `src/app/auth/callback/route.ts`, the `db.update(clients)` call lacks logging. Adding a debug log when a client is linked would help diagnose connection issues during onboarding.

3. **Validate `type` in register POST handler**  
   Currently, any string can be passed as `type` in `/api/auth/register`. Enforce allowed values (`"professional"`, `"client"`) early in the request lifecycle to prevent inconsistent metadata.
