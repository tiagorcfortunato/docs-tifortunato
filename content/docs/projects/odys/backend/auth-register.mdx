# Auth + Register

This document covers the authentication and registration flow in Odys, focusing on how users sign up, verify their identity, and are securely linked to existing records. It explains the interplay between Supabase auth, metadata handling, and database consistency—particularly for clients who register after previously being added by a professional.

## Overview

Odys uses Supabase Auth for identity management, with a split strategy between development and production for user registration. The system supports three user types—`professional`, `client`, and `recovery`—enforced via a closed set to prevent injection into user metadata. There are 23 API routes in total, with `/api/auth/register` and `/api/auth/callback` playing central roles in account creation and post-verification logic. Rate limiting is applied to registration (5 attempts/hour), and smart-linking ensures clients are connected to their `user.id` after email confirmation.

## Registration Flow

The `POST /api/auth/register` endpoint accepts `email`, `password`, `name`, and `type`. The `name` and `type` parameters are stored in Supabase's `user_metadata` during the initial registration call (see `options.data` for `signUp` and `user_metadata` for `admin.createUser` in `src/app/api/auth/register/route.ts`). In production, it uses Supabase’s `signUp()` with anon keys so that Supabase automatically sends a verification email. The response indicates whether email confirmation is required by checking if a session was created. In development, the service role client creates confirmed users directly, bypassing email verification for faster iteration.

Upon verification, Supabase redirects to `/auth/callback` with a one-time code. This endpoint then exchanges the code for a session and processes the `type` parameter. Redirects from `/auth/callback` construct their `origin` using `x-forwarded-host` and `x-forwarded-proto` headers, falling back to `req.nextUrl` properties, to ensure correct routing in proxy environments like Vercel (`src/app/auth/callback/route.ts:23`). Only values in the `VALID_TYPES` set (`professional`, `client`, `recovery`) are accepted, mitigating metadata injection risks. Specifically, if the `type` is `recovery`, the user is redirected to `/reset-password` after the session exchange (`src/app/auth/callback/route.ts:40`).

## Smart-Linking Clients

A key feature in `/auth/callback/route.ts` is **smart-linking**: when a client verifies their email, the system checks if a `clients` record exists with that email and no `userId`. If found, it updates the record with the authenticated `user.id`. This allows professionals to pre-register clients before they sign up. The update uses Drizzle ORM with `and(isNull(clients.userId), eq(clients.email, user.email))`, ensuring only unlinked clients are matched.

## Design decisions

The dual registration strategy (anon vs. service role) balances security and developer experience: production enforces email verification, while dev skips it for speed. The `VALID_TYPES` guard prevents arbitrary data in user metadata, addressing a prior security finding. Smart-linking is deferred to the callback because only then is the email proven valid—linking earlier would risk associating records with unverified emails.

## Potential improvements

1. **Move type validation into Supabase RLS policies** — While `VALID_TYPES` guards metadata updates in `/auth/callback/route.ts`, row-level policies could enforce `clients.type IN ('professional', 'client')` at the database level for stronger consistency.

2. **Add telemetry to failed register attempts** — `getRegisterLimiter().limit()` in `/api/auth/register/route.ts` returns 429 but logs only on internal error. Adding a debug log on rate limit hit would help monitor abuse patterns.

3. **Use transaction for smart-linking** — The `db.update(clients)` call in `/auth/callback/route.ts` runs outside a transaction. Wrapping it with `await db.transaction(...)` would ensure atomicity if future logic adds related updates.
