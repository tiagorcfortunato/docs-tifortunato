# WhatsApp Integration

WhatsApp serves as a critical communication channel within Odys, facilitating interactions between professionals and their clients. This document explores the architecture behind Odys's WhatsApp integration, focusing on the use of the Evolution API, the structured approach to message templating, and the underlying data models that support these communications.

## Overview

Odys leverages the **Evolution API** to manage its WhatsApp communications. This third-party service acts as the gateway, allowing the application to send and receive messages through a corporate WhatsApp number. The integration is designed to handle a variety of transactional and conversational messages, ensuring that both professionals and clients receive timely and relevant updates.

The core of this system relies on a set of 19 predefined message templates, such as `msgBookingConfirmed` and `msgReminder24h`, which standardize communication for common scenarios. These templates are centrally defined in `src/lib/whatsapp.ts`, promoting consistency and maintainability.

For local development and testing, the Evolution API is orchestrated via `docker-compose.evolution.yml`, which sets up a dedicated PostgreSQL database (`evolution-postgres`) and a Redis instance (`evolution-redis`) to support the API's operations. This local setup mirrors the production environment's dependencies, ensuring a consistent development experience.

Communication flow involves:
*   Outbound messages initiated by Odys, using the `sendWhatsApp` function and various message templates.
*   Inbound messages received by the Evolution API, which are then forwarded to Odys via the `/api/whatsapp/webhook` route.
*   Scheduled tasks, like the `/api/cron/whatsapp-watchdog` cron job, which runs daily at 9 AM, likely for monitoring the API's health or processing deferred messages.

The system also incorporates rate limiting, with a specific `rateLimits.whatsapp` configuration allowing 10 messages per minute, preventing abuse and ensuring service stability.

## Evolution API Configuration and Communication

The `src/lib/whatsapp.ts` file is the primary interface for interacting with the Evolution API. It abstracts the underlying HTTP requests, providing a clean API for sending messages. The API's base URL, authentication key, and instance identifier are loaded from environment variables (`EVOLUTION_API_URL`, `EVOLUTION_API_KEY`, `EVOLUTION_INSTANCE`), which are checked by the `isConfigured()` function before any message is sent. This ensures that the WhatsApp integration only attempts to operate when properly set up.

For local development, the `docker-compose.evolution.yml` file defines the `evolution-api` service, which uses the `evoapicloud/evolution-api:v2.3.7` Docker image. This container is configured to connect to a local `postgres` database and `redis` cache, both defined within the same Docker Compose setup. Key environment variables for the local Evolution API instance include `AUTHENTICATION_API_KEY` set to "odys-local-key-2026" and `SERVER_URL` pointing to "http://localhost:8080". This setup allows developers to run and test the WhatsApp integration without relying on external Evolution API services.

The `sendWhatsApp` function is the single entry point for sending text messages. It takes a `phone` number, the `text` content, and an optional `context` object. Before sending, the `normalizeBrazilianPhone` function converts Brazilian local formats to the `55...` international format, while preserving other international numbers as-is (e.g., `+49...` becomes `49...`). This normalization is crucial for reliable delivery across different regions. The actual API call to Evolution API includes a `delay` of 1200 milliseconds, which can help prevent messages from being flagged as spam by WhatsApp.

## Structured Messaging with Templates

A core design principle of Odys's WhatsApp integration is the use of predefined message templates. All 19 message types, from `msgBookingRequest` to `msgNewMessageToPro`, are defined as exported functions within `src/lib/whatsapp.ts`. This approach centralizes message content, making it easier to manage, update, and ensure consistent branding and tone.

Each client-facing transactional message (e.g., reminders, confirmations) is visually structured with a `transactionalHeader` and `TRANSACTIONAL_FOOTER`. The header includes an emoji, a label, and the professional's name, framed by `HEADER_RULE` lines. This visual distinction helps users differentiate automated notifications from conversational AI replies within the same WhatsApp chat thread, which is particularly important when a single Odys number handles both types of interactions.

For example, when a booking is confirmed, the `msgBookingConfirmed` function constructs a message that includes the professional's name, client's name, date, and time, all wrapped in the standardized header and footer. Similarly, `msgReminder24h` and `msgReminder1h` provide timely notifications to clients about upcoming appointments.

## Conversation Context and Routing

A significant challenge in using a single corporate WhatsApp number for multiple professionals is correctly routing client replies. Odys addresses this through the `context` parameter in the `sendWhatsApp` function. When a transactional message is successfully sent to a client, if a `context` object containing `professionalId` and optionally `appointmentId` is provided, this information is stored in Redis using `setOutboundContext`. This context is associated with the client's phone number and is intended to be valid for a limited duration (implicitly "next 2h" as per the comment).

When a client replies to a message, the `/api/whatsapp/webhook` endpoint receives the inbound message. By retrieving the stored context from Redis using the client's phone number, Odys can intelligently route the reply to the correct professional or associate it with a specific appointment. This mechanism is vital for maintaining coherent conversations and ensuring that professionals receive relevant client communications.

The `messages` table in the database (`messages`) stores a history of these interactions, linking each message to a `professionalId` and `clientId`. This table also tracks the `sender` (professional or client), `type` of message, `content`, and a `readAt` timestamp, providing a comprehensive log of all WhatsApp communications.

## Design decisions

1.  **Centralized Message Templating**: The decision to define all WhatsApp messages as functions in `src/lib/whatsapp.ts` was made to centralize message content. This prevents message strings from being scattered across various API routes or services, making them easier to maintain, update, and ensure consistent branding and tone. It also simplifies the process of applying common elements like the `transactionalHeader` and `TRANSACTIONAL_FOOTER`.
2.  **Evolution API Integration**: The choice of the Evolution API, as indicated by the `BASE_URL` and `API_KEY` environment variables and the `docker-compose.evolution.yml` setup, likely stems from its feature set and ease of integration for WhatsApp messaging. The explicit version `v2.3.7` in the Docker image suggests a commitment to a specific, tested API version.
3.  **Outbound Context for Reply Routing**: The `context` parameter in `sendWhatsApp` and its interaction with `setOutboundContext` in Redis is a deliberate design choice to solve the "corporate number" problem. By temporarily associating a client's phone number with a `professionalId` (and `appointmentId`), Odys can correctly route incoming replies to the relevant professional, even when multiple professionals share the same WhatsApp business number. This avoids ambiguity and improves the user experience.
4.  **Defensive `sendWhatsApp` Implementation**: The `sendWhatsApp` function is designed to be resilient, returning `true` or `false` to indicate success or failure rather than throwing exceptions. This allows calling code to handle communication failures gracefully without crashing, and errors are logged using `logger.error("whatsapp_send_failed")` or `logger.error("whatsapp_connect_failed")`.
5.  **Phone Number Normalization**: The `normalizeBrazilianPhone` function explicitly handles various Brazilian phone number formats and international numbers. This ensures that messages are sent to correctly formatted numbers, reducing delivery failures due to malformed recipient identifiers.

## Potential improvements

1.  **Template Localization**: All 19 message templates in `src/lib/whatsapp.ts` are currently hardcoded in Portuguese. For future internationalization, these strings would need to be extracted into a separate localization system. This would involve modifying each `msg*` function to accept a locale parameter or retrieve translated strings from a dedicated resource.
2.  **Rich Media Message Support**: The `sendWhatsApp` function currently only supports sending plain text messages. However, the `messages` table includes a `fileUrl` column, suggesting that rich media (images, documents) could be part of the conversation. Extending `sendWhatsApp` to support different message types (e.g., `sendImage`, `sendFile`) and utilizing the `fileUrl` would enable richer communication experiences for professionals and clients.
3.  **Explicit Context Expiration Management**: The comment for `setOutboundContext` mentions routing replies for "próximas 2h" (next 2 hours), but the actual expiration logic for the Redis key is not visible in the provided code. Making this expiration explicit and configurable within `setOutboundContext` would improve clarity and allow for dynamic adjustment of the conversation window.
4.  **Dynamic Template Variables and Structure**: While the current templates use string interpolation, for more complex messages or conditional content, a dedicated templating engine (e.g., Handlebars, Nunjucks) could offer greater flexibility. This would allow for more sophisticated message generation without cluttering the `msg*` functions with complex conditional logic.
5.  **Evolution API Client Abstraction**: The `sendWhatsApp` function directly uses `fetch` to interact with the Evolution API. As the API evolves (e.g., from v1.8 to v2.3+ mentioned in comments), direct `fetch` calls might require frequent updates. Introducing a dedicated Evolution API client library or a more abstract wrapper could encapsulate API versioning and changes, making the integration more resilient to upstream modifications.
