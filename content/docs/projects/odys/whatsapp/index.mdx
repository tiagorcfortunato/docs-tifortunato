# WhatsApp Integration

WhatsApp serves as a pivotal communication channel within Odys, enabling professionals to connect with their clients for appointment management, reminders, and direct messaging. This document explores the architecture and implementation behind Odys's WhatsApp integration, detailing how messages are composed, sent, and managed, and how the system maintains conversational context.

## Overview

Odys integrates with WhatsApp through the Evolution API, an external service that handles the complexities of the WhatsApp Business Platform. This integration is central to the application's functionality, facilitating automated transactional messagesŌĆösuch as appointment confirmations, reminders, and cancellationsŌĆöas well as direct client-professional communication.

The system defines a total of 19 distinct message templates, all centralized within `src/lib/whatsapp.ts`, ensuring consistency and ease of management. These templates cover various scenarios, from initial booking requests to post-appointment follow-ups. For local development, the Evolution API, along with its PostgreSQL and Redis dependencies, is orchestrated via `docker-compose.evolution.yml`, providing a self-contained environment for testing and development.

Communication flows are supported by several database tables. The `professionals` and `clients` tables store the core user information, including phone numbers essential for messaging. The `appointments` table provides the context for many transactional messages, while the `messages` table logs all communications between clients and professionals. Incoming WhatsApp messages are processed through the `/api/whatsapp/webhook` route, which acts as the entry point for the Evolution API's callbacks. Furthermore, the system incorporates AI capabilities, evidenced by the `agentMetrics` table and the `/api/ai/chat` route, suggesting that some WhatsApp interactions might be handled by an AI agent, subject to a `aichat` rate limit of 20 requests per hour.

## Evolution API Integration

The core of Odys's WhatsApp functionality resides in `src/lib/whatsapp.ts`, which acts as a client for the Evolution API. This file abstracts away the direct API calls, providing a clean interface for sending messages. The API's base URL, key, and instance identifier are configured via environment variables (`EVOLUTION_API_URL`, `EVOLUTION_API_KEY`, `EVOLUTION_INSTANCE`), which are checked by the `isConfigured` function before any message is dispatched. This design ensures that the WhatsApp integration can be enabled or disabled simply by configuring these variables, preventing message attempts if the service is not properly set up.

The `sendWhatsApp` function is the primary mechanism for dispatching messages. It accepts a `phone` number, the `text` content, and an optional `context` object. Before sending, phone numbers are normalized using `normalizeBrazilianPhone` to ensure they conform to an international format, specifically handling Brazilian numbers by prepending "55" if missing. This function is crucial for reliable message delivery across different input formats. The actual API call is a `POST` request to the Evolution API's `/message/sendText/{INSTANCE}` endpoint, including a small `delay` of 1200 milliseconds, likely to prevent rate limiting or ensure messages are processed in order by the recipient's device. Error handling within `sendWhatsApp` logs failures but gracefully returns `false`, preventing application crashes due to external API issues.

A notable feature of `sendWhatsApp` is its `context` parameter. When provided, this contextŌĆöcontaining `professionalId` and optionally `appointmentId`ŌĆöis stored in Redis via `setOutboundContext`. This mechanism is designed to associate subsequent client replies with the correct professional and appointment, allowing the `/api/whatsapp/webhook` to intelligently route incoming messages. This is particularly important for transactional messages like reminders, where a client's reply needs to be understood in the context of a specific booking.

## Message Templating

All client-facing and internal WhatsApp messages are generated using dedicated functions within `src/lib/whatsapp.ts`, rather than being hardcoded directly into application logic. This approach centralizes message content, making it easier to manage, update, and ensure consistency across all communications. Odys currently defines 19 such templates, including `msgBookingRequest`, `msgBookingConfirmed`, `msgReminder24h`, and `msgNewMessageToClient`.

For transactional messages sent to clients, a consistent visual structure is applied using `transactionalHeader` and `TRANSACTIONAL_FOOTER`. These functions add rule lines and branding information (e.g., "Odys ┬Ę agenda inteligente ┬Ę odys.com.br") to the message text. This visual distinction helps clients differentiate automated notifications from direct conversational AI replies, which might share the same WhatsApp thread. For instance, `msgBookingConfirmed` uses this header and footer to clearly mark it as an official communication from the professional.

The `msgRegistrationInvite` function is a clever addition, dynamically generating a registration link with pre-filled `email`, `phone`, and `name` parameters. This streamlines the onboarding process for clients who interact via WhatsApp but haven't yet created an Odys account.

## Local Development Environment

The `docker-compose.evolution.yml` file defines the local development setup for the Evolution API. It orchestrates three services: `postgres`, `redis`, and `evolution-api`.

*   The `postgres` service uses `postgres:15` and is configured with a dedicated user (`evolution`), password (`odys-pg-2026`), and database (`evolution`). It includes a health check to ensure the database is ready before the API attempts to connect.
*   The `redis` service uses `redis:7-alpine` and provides caching and session management for the Evolution API.
*   The `evolution-api` service runs the `evoapicloud/evolution-api:v2.3.7` image. It depends on both `postgres` and `redis` being healthy and started. Key environment variables configure its behavior:
    *   `AUTHENTICATION_API_KEY: "odys-local-key-2026"` sets the API key for local access.
    *   `DATABASE_PROVIDER: "postgresql"` and `DATABASE_CONNECTION_URI` specify the PostgreSQL connection details.
    *   `CACHE_REDIS_ENABLED: "true"` and `CACHE_REDIS_URI` enable and configure Redis caching.
    *   `WEBHOOK_GLOBAL_ENABLED: "false"` indicates that webhooks are configured on a per-instance basis, rather than globally.
    *   `TELEMETRY_ENABLED: "false"` disables data collection, which is appropriate for a local development setup.
    *   `DATABASE_SAVE_DATA_NEW_MESSAGE: "true"` ensures that new messages are persisted in the Evolution API database.
    *   `DATABASE_SAVE_IS_ON_WHATSAPP_DAYS: "7"` controls how long contact presence on WhatsApp is retained.
    *   `SERVER_URL: "http://localhost:8080"` sets the base URL for the local Evolution API instance.
    *   `LOG_LEVEL: "ERROR,WARN,INFO"` configures the logging verbosity to capture errors, warnings, and informational messages.

This Docker Compose setup provides a complete, isolated environment for developing and testing WhatsApp integrations without affecting production systems.

## Design Decisions

The design choices for Odys's WhatsApp integration reflect a balance between functionality, maintainability, and user experience.

*   **External API for WhatsApp**: Opting for the Evolution API rather than directly integrating with the WhatsApp Business API simplifies development and maintenance. The Evolution API handles the underlying infrastructure, message queuing, and compliance, allowing Odys developers to focus on application-specific logic. This trade-off means relying on a third-party service but reduces operational overhead.
*   **Centralized Message Templates**: Defining all 19 message templates as functions in `src/lib/whatsapp.ts` promotes consistency and simplifies content updates. Instead of scattering message strings throughout the codebase, developers can find and modify all WhatsApp messages in a single, well-defined location. This also makes it easier to apply global formatting or branding changes.
*   **Visual Distinction for Transactional Messages**: The use of `transactionalHeader` and `TRANSACTIONAL_FOOTER` with rule lines (`ŌöüŌöüŌöüŌöüŌöüŌöüŌöüŌöüŌöüŌöüŌöüŌöüŌöüŌöüŌöüŌöüŌöü`) is a deliberate choice to visually separate automated system messages from conversational AI or direct professional replies. This improves clarity for the end-user, helping them quickly understand the nature of the message, especially when a single WhatsApp number serves multiple communication purposes.
*   **Outbound Context for Reply Routing**: The `context` parameter in `sendWhatsApp` and its storage in Redis via `setOutboundContext` addresses a fundamental challenge in conversational systems: maintaining state. By associating an outbound message with a `professionalId` and `appointmentId` for a limited time (comment suggests 2 hours), Odys can intelligently route incoming client replies to the correct professional or appointment context, enabling more natural and efficient interactions without requiring the client to explicitly state the context in every message.
*   **Robust Phone Number Normalization**: The `normalizeBrazilianPhone` function is a pragmatic solution to handle the variability of phone number inputs. By ensuring all numbers are in a consistent international format before being sent to the Evolution API, it minimizes delivery failures due to malformed numbers, improving the reliability of the messaging system.

## Potential Improvements

1.  **Internationalization of Message Templates**: The current message templates in `src/lib/whatsapp.ts`, such as `msgBookingConfirmed` and `msgReminder24h`, are hardcoded in Portuguese. To support a broader user base, these templates could be externalized into a dedicated i18n system (e.g., JSON files or a database table). This would allow for easy translation and dynamic loading of messages based on the client's or professional's preferred language, without requiring code changes for each new language.
2.  **Support for Media Messages**: The `sendWhatsApp` function currently only supports sending `text` content. However, the `messages` table includes a `fileUrl` column, indicating a potential future need for sending media (images, documents, etc.). Extending `sendWhatsApp` to handle different message types, such as `sendImage` or `sendFile`, would allow professionals to share richer content with clients, enhancing communication capabilities. This would involve updating the Evolution API client to use appropriate endpoints for media.
3.  **Configurable Outbound Context Expiration**: The comment in `sendWhatsApp` mentions that the `last_outbound` context in Redis is used for routing client replies "nas pr├│ximas 2h". While a fixed duration is a reasonable starting point, making this expiration period configurable (e.g., via an environment variable or a professional-specific setting) could offer more flexibility. Some interactions might require a longer context window, while others might benefit from a shorter one to reduce stale data in Redis.
4.  **Granular WhatsApp Rate Limiting**: The system currently applies a global `whatsapp` rate limit of 10 messages per minute. While this prevents general abuse, it might not be optimal for all scenarios. Implementing more granular rate limits, perhaps per `professionalId` or `clientId`, could prevent a single professional from inadvertently spamming clients or protect against a single client overwhelming the system, while allowing other users to communicate freely.
