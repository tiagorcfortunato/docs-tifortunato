# WhatsApp Integration

This page covers the WhatsApp integration architecture, including the corporate number setup via the Evolution API, webhook configuration, and the implementation of transactional message templates in the application.

## Corporate Number Architecture

The WhatsApp integration uses the Evolution API as the backend service for sending and receiving messages through a corporate WhatsApp number. The API runs as a Docker container defined in `docker-compose.evolution.yml`, using the image `atendai/evolution-api:v1.82`.

The local instance is configured with API key authentication (`AUTHENTICATION_TYPE: "apikey"`) and a predefined key (`AUTHENTICATION_API_KEY: "odys-local-key-2026"`). Session metadata identifies the client as "Odys" (`CONFIG_SESSION_PHONE_CLIENT: "Odys"`).

Global webhooks are disabled via `WEBHOOK_GLOBAL_ENABLED: "false"`. Instead, webhooks are configured per instance using an API call to `/webhook/set/{INSTANCE}` with the event `MESSAGES_UPSERT` routed to `{APP_URL}/api/whatsapp/webhook`. This allows precise control over inbound message handling.

## Evolution API Integration

All interactions with the Evolution API are managed in `src/lib/whatsapp.ts`. The module reads configuration from environment variables:

- `EVOLUTION_API_URL`: Base URL for the API
- `EVOLUTION_API_KEY`: Authentication key
- `EVOLUTION_INSTANCE`: Target instance name

The `isConfigured()` function checks that all three values are set before allowing message operations.

The primary function, `sendWhatsApp()`, sends text messages via a `POST` request to `/message/sendText/{INSTANCE}`. It includes:
- Phone number normalization via `normalizeBrazilianPhone()`
- A fixed `delay` of `1200` ms in the payload
- Authentication using the `apikey` header
- Optional Redis context storage via `setOutboundContext()` when a `context` object (containing `professionalId` and optionally `appointmentId`) is provided

The function returns `true` on success and `false` on failure, logging errors to the console without throwing exceptions.

## Message Templating

All outbound WhatsApp messages are defined as exported functions in `src/lib/whatsapp.ts`, ensuring centralized and consistent messaging.

A visual templating system distinguishes automated notifications from conversational AI within the same thread:

- `HEADER_RULE`: A line of dashes (`━━━━━━━━━━━━━━━━━`) used to frame message headers
- `transactionalHeader()`: Generates a formatted header with an emoji, label, and professional name
- `TRANSACTIONAL_FOOTER`: A standardized footer identifying "Odys" and the app URL

The following message templates are implemented:

| Template | Recipient | Purpose |

| `msgBookingRequest` | Professional | New booking request notification |
| `msgBookingConfirmed` | Client | Booking confirmation |
| `msgBookingRejected` | Client | Rejection of booking request |
| `msgBookingCancelled` | Client | Cancellation by professional |
| `msgReminder24h` | Client | 24-hour appointment reminder |
| `msgReminder1h` | Client | 1-hour appointment reminder |
| `msgCancelledByClientToPro` | Professional | Cancellation by client |
| `msgCancelledByProToClient` | Client | Cancellation by professional |
| `msgBookingConfirmedReceipt` | Professional | Confirmation receipt |
| `msgBookingRejectedReceipt` | Professional | Rejection receipt |

## Known Gaps

*   **Error Handling in `sendWhatsApp`**: The function in `src/lib/whatsapp.ts` returns `false` on failure instead of throwing. This avoids blocking critical paths but requires every call site to explicitly handle failures (e.g. via `.catch(console.warn)`). Silent failures are possible if monitoring is not in place.
*   **Template Count vs. Documentation**: The deep-dive states "20+ templates" are used, but `src/lib/whatsapp.ts` defines 16 distinct message functions. This suggests either the count is approximate, some templates are generated dynamically, or additional templates exist outside this file (e.g. in AI-generated responses).

## Why this shape

The WhatsApp integration is designed for operational clarity and resilience:

*   **Centralized Templates**: All message content lives in `src/lib/whatsapp.ts`, preventing duplication and enabling consistent tone and branding.
*   **Visual Hierarchy**: The use of `transactionalHeader` and `TRANSACTIONAL_FOOTER` creates a clear visual distinction between system notifications and conversational messages in the same WhatsApp thread — essential for a shared corporate number.
*   **Non-Blocking Delivery**: `sendWhatsApp` returning `false` on failure ensures that transient WhatsApp API issues do not disrupt core application flows, aligning with the principle that notifications are best-effort.
*   **Contextual Routing**: The `context` parameter enables intelligent reply routing by storing `professionalId` and `appointmentId` in Redis, allowing inbound messages to be associated with the correct entity even in a shared-number architecture.
