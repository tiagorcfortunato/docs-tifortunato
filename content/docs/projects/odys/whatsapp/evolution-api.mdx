# Evolution API

The Evolution API serves as the foundational layer for Odys's WhatsApp-first communication strategy, enabling both automated transactional messages and conversational AI interactions. This document delves into the design choices behind its integration, how it's deployed, and the mechanisms that ensure a cohesive user experience across different communication channels.

## Overview

Odys leverages the Evolution API to manage all WhatsApp interactions, centralizing communication through a single, dedicated corporate WhatsApp number. This approach allows Odys to send automated reminders, confirmations, and operational alerts, as well as facilitate client-professional messaging and AI-driven scheduling, all without requiring professionals to integrate their personal WhatsApp accounts. The `src/lib/whatsapp.ts` file encapsulates the entire client-side logic for interacting with the Evolution API, defining 19 distinct message templates and the core `sendWhatsApp` function. For local development, the Evolution API runs as a Docker container, configured via `docker-compose.evolution.yml`, while in production, it's self-hosted on Railway.

## Evolution API Client

The `src/lib/whatsapp.ts` file is the heart of Odys's WhatsApp integration. It provides a robust, abstracted interface for sending messages, ensuring consistency and maintainability across the application.

At its core, the client relies on three environment variables: `EVOLUTION_API_URL`, `EVOLUTION_API_KEY`, and `EVOLUTION_INSTANCE`. These are crucial for establishing a connection to the Evolution API instance, with the `isConfigured` function acting as a guard to prevent message attempts when the API is not properly set up.

### Phone Number Normalization

A critical utility within `src/lib/whatsapp.ts` is the `normalizeBrazilianPhone` function. This function addresses the common challenge of varying phone number formats by converting them into a standardized international format without the leading `+`. It intelligently handles Brazilian numbers, prepending `55` if the country code is missing, while preserving other international numbers that already include their respective country codes. This normalization ensures that messages are always directed to the correct recipient, regardless of how the phone number was initially captured.

### Sending Messages

The `sendWhatsApp` function is the primary entry point for all outbound WhatsApp messages. It takes a `phone` number (which is first normalized), the `text` content, and an optional `context` object. The `context` parameter is a key design choice: when provided, it instructs the system to record the `professionalId` and `appointmentId` in Redis under a `last_outbound:{phone}` key. This mechanism is vital for routing client replies back to the correct professional and appointment, especially in a multi-professional platform where a single Odys WhatsApp number serves many. This ensures that a client's response to a reminder, for instance, is correctly associated with the ongoing conversation.

The function handles network requests to the Evolution API, including `POST` requests to `${BASE_URL}/message/sendText/${INSTANCE}`. It includes basic error handling, logging failures to the console rather than throwing exceptions, which allows the calling code to continue execution gracefully. A fixed `delay` of `1200` milliseconds is included in the message payload, likely to mimic human-like typing or to prevent rate limiting issues with the WhatsApp platform itself.

### Visual Templating

To enhance clarity and user experience, `src/lib/whatsapp.ts` implements a visual templating system for client-facing transactional messages. The `transactionalHeader` function, utilizing `HEADER_RULE` and `TRANSACTIONAL_FOOTER`, wraps messages with distinct rule lines and includes the professional's name and an emoji. This visual distinction is crucial for clients, helping them differentiate automated notifications from conversational AI replies within the same WhatsApp thread, especially since the same Odys number handles both channels.

### Message Templates

The file defines 19 pre-formatted message templates, categorized for different scenarios:
*   **Client-facing transactional**: These include confirmations, rejections, cancellations, 24-hour and 1-hour reminders, payment confirmations, and appointment completion notifications. They all leverage the `transactionalHeader` and `TRANSACTIONAL_FOOTER` for consistent branding and context.
*   **Professional-facing operational**: These messages, such as new booking requests, client cancellations, and payment receipts, are sent directly to the professional's phone from the corporate Odys number, providing critical operational alerts.
*   **Special templates**: `msgRegistrationInvite` is a unique template appended to confirmation messages for clients without an Odys account, encouraging them to register and manage their appointments.

## Evolution API Deployment

For local development, the Evolution API is configured using `docker-compose.evolution.yml`. This setup ensures that developers can run a fully functional WhatsApp integration stack without complex manual configurations.

The `docker-compose.evolution.yml` specifies the `atendai/evolution-api:v1.8.2` Docker image. Key environment variables are set to configure the local instance:
*   `SERVER_URL: "http://localhost:8080"`: Defines the local access point for the API.
*   `AUTHENTICATION_TYPE: "apikey"` and `AUTHENTICATION_API_KEY: "odys-local-key-2026"`: Establish a simple API key-based authentication for local testing.
*   `DATABASE_ENABLED: "false"` and `REDIS_ENABLED: "false"`: These settings are crucial for simplifying local development. By disabling external database and Redis connections, the Evolution API stores its instance data in local files within the container, reducing external dependencies.
*   `WEBHOOK_GLOBAL_ENABLED: "false"`: Webhooks are globally disabled in this local configuration. The `README.md` clarifies that webhooks are enabled per instance via an API call (e.g., `POST {EVOLUTION_API_URL}/webhook/set/{INSTANCE}`), allowing for fine-grained control over event subscriptions like `MESSAGES_UPSERT`.
*   A Docker volume named `evolution_data` is used to persist instance data, ensuring that WhatsApp sessions are not lost when the container restarts.

In production, the `README.md` indicates that Odys self-hosts Evolution API v2 on Railway, using a custom Docker image `tiagorcfortunato/evolution-api-odys`. This setup likely involves a more robust database (a separate Supabase project `odys-evolution`) and potentially a more advanced Redis configuration for scalability and reliability.

## Integration with Odys

The Evolution API is deeply integrated into Odys's operational flows:

*   **Outbound Messaging**: All automated communications, including the 19 message templates defined in `src/lib/whatsapp.ts`, are dispatched through the `sendWhatsApp` function. This includes cron jobs like `GET /api/cron/reminders` (scheduled daily at 8 AM) for sending 24-hour and 1-hour reminders, and various alerts triggered by booking confirmations, cancellations, and payment updates.
*   **Inbound Messaging**: The Evolution API is configured to post all incoming messages to Odys's `POST /api/whatsapp/webhook` endpoint. This webhook is responsible for processing client replies, routing them to the AI intake agent (`src/lib/ai-intake.ts`), and leveraging Groq LLM tool-calling for tasks like availability lookups and appointment creation.
*   **Reply Routing**: The `setOutboundContext` function, called after a successful `sendWhatsApp` operation with a `context` object, stores relevant `professionalId` and `appointmentId` information in Redis. When a client replies, the `whatsapp/webhook` uses this `last_outbound:{phone}` context to correctly route the message and seed the AI's understanding, preventing the need for clients to re-specify the professional in ongoing conversations.
*   **Conversation-Aware Delivery**: The system includes logic to prevent interrupting active client-AI conversations. The `GET /api/cron/reminders` cron job will skip sending a reminder if a client is currently engaged in a conversation (defined as less than 15 minutes idle), deferring the reminder to the next cycle. This ensures a smoother, less intrusive user experience.

## Design Decisions

The choice and implementation of the Evolution API reflect several key design decisions:

*   **WhatsApp-First Strategy**: The core decision to make Odys "WhatsApp-first" necessitated a robust, programmatic way to interact with WhatsApp. The Evolution API, acting as a WhatsApp Web shim, provided a viable path to achieve this without the immediate complexities and costs of Meta's official WhatsApp Business Cloud API.
*   **Abstraction for Future Scalability**: By centralizing all WhatsApp message sending logic within a single `sendWhatsApp` function and defining all message content as named functions in `src/lib/whatsapp.ts`, the system is designed for future flexibility. This abstraction means that if Odys decides to migrate to Meta's WhatsApp Business Cloud API for greater scale and official support, the underlying messaging engine can be swapped out with minimal changes to the application's call sites.
*   **Simplified Local Development**: Setting `DATABASE_ENABLED: "false"` and `REDIS_ENABLED: "false"` in `docker-compose.evolution.yml` for the local Evolution API instance was a deliberate choice to streamline the development environment. It allows developers to run the WhatsApp integration locally without needing to set up external database or Redis services, relying instead on file-based persistence within the Docker container.
*   **Enhanced User Experience through Context**: The implementation of the `context` parameter in `sendWhatsApp` and the subsequent use of Redis for `last_outbound:{phone}` is a crucial design choice for maintaining conversational continuity. It addresses the challenge of routing replies in a multi-professional environment, ensuring that client interactions are always relevant and personalized.
*   **Visual Disambiguation**: The `transactionalHeader` and `TRANSACTIONAL_FOOTER` are not merely aesthetic. They serve a functional purpose by visually distinguishing automated system messages from conversational AI responses within the same WhatsApp thread, preventing confusion for the end-user.

## Potential Improvements

1.  **Evolution API Version Alignment**: The `docker-compose.evolution.yml` file specifies `atendai/evolution-api:v1.8.2` for local development, while the `README.md` states that Evolution API v2 is used in production on Railway. This version discrepancy could lead to behavioral differences or unexpected issues between development and production environments. It would be beneficial to align the local Docker image version with the production version or clearly document any known differences and their implications.
2.  **Automated Webhook Configuration**: The `docker-compose.evolution.yml` comments indicate that `WEBHOOK_GLOBAL_ENABLED` is `false` and webhooks are enabled per instance via a `POST` API call. While this offers flexibility, it implies a manual step or an external script to configure webhooks after the Evolution API instance is deployed or restarted. Integrating this webhook setup directly into the Odys application's startup process or a dedicated management endpoint could enhance deployment reliability and reduce operational overhead.
3.  **Configurable Message Delay**: The `sendWhatsApp` function in `src/lib/whatsapp.ts` uses a hardcoded `delay: 1200` milliseconds in its `JSON.stringify` body. While this delay might be suitable for current needs, making it configurable (e.g., via an environment variable or a parameter in `SendTextParams`) could provide more control over message pacing. This would allow for adjustments based on Evolution API's rate limits, WhatsApp's own policies, or specific user experience requirements without modifying the core `sendWhatsApp` implementation.
