# Evolution API

Odys uses Evolution API as its WhatsApp gateway, enabling automated messaging and AI-driven client intake through a single managed phone number. This integration is central to the product’s WhatsApp-first experience in Brazil, where professionals and clients rely on WhatsApp for scheduling. This page explains the architecture, deployment, and design decisions behind our Evolution API usage.

## Overview

Evolution API acts as a WhatsApp Web wrapper, allowing Odys to send and receive messages programmatically without requiring professionals to connect their personal devices. The platform hosts a single Evolution instance on Railway, connected to a corporate WhatsApp number. All outbound messages—reminders, confirmations, AI replies—originate from this number. Inbound messages are routed via webhook to the AI intake agent or to the correct professional using Redis-based context tracking.

The system uses **19 message templates** defined in `src/lib/whatsapp.ts`, split into client-facing transactional messages (with headers and footers) and professional-facing operational alerts. These templates ensure consistency and brand clarity across all communications.

Two cron jobs automate key workflows:
- `GET /api/cron/reminders` runs daily at 8 AM to send 24h and 1h reminders.
- `GET /api/cron/whatsapp-watchdog` runs at 9 AM to check Evolution API health.

All API routes that trigger WhatsApp messages use the `sendWhatsApp()` function, which abstracts the Evolution API v2 payload format and handles Redis context storage for reply routing.

## Design decisions

The shift from **Evolution API v1.8.2 (local)** to **v2.3.7 (Railway)** was driven by stability, scalability, and schema requirements. Version 2 mandates PostgreSQL and Redis, which aligns with Odys’ use of Supabase and Upstash, whereas v1 used MongoDB. The Docker setup in `docker-compose.evolution.yml` isolates the Evolution stack with dedicated Postgres and Redis containers, ensuring clean separation from the main app database.

Evolution API is configured via `docker-compose.evolution.yml` to save instance data, messages, contacts, and chat information to its dedicated PostgreSQL database. It uses Redis for caching, configured to save instance data and disable local caching, as shown in `docker-compose.evolution.yml`.

The `sendWhatsApp()` function includes a `delay: 1200` (1.2 seconds) in the Evolution API payload to manage message throughput. Evolution API's logging is configured in `docker-compose.evolution.yml` to output ERROR, WARN, and INFO levels, with Baileys logs restricted to ERROR. Telemetry is explicitly disabled in `docker-compose.evolution.yml` via `TELEMETRY_ENABLED: "false"`.

Using a single shared WhatsApp number eliminates onboarding friction—professionals don’t need to connect their phones. The `sendWhatsApp()` function includes an optional `context` parameter that stores `{ professionalId, appointmentId }` in Redis under `last_outbound:{phone}`, enabling accurate reply routing when clients respond to automated messages.

Client-facing messages use a visual header (`━━━━━━━━━━━━━━━━━`) and footer to distinguish automated notifications from conversational AI replies, preventing confusion when multiple professionals share the same communication channel.

## Potential improvements

1. **Add retry logic with backoff in `sendWhatsApp()`** — Currently, failed sends are logged but not retried. Adding a retry mechanism (e.g., 3 attempts with exponential backoff) would improve delivery reliability, especially for time-sensitive reminders.

2. **Validate message templates at build time** — With 19 templates in `src/lib/whatsapp.ts`, there’s risk of broken links or missing variables. Introducing a test that renders all templates with mock data would catch interpolation errors early.

3. **Move Evolution API config to a dedicated module** — The `BASE_URL`, `API_KEY`, and `INSTANCE` values are read directly from env vars in `whatsapp.ts`. Extracting this into a config module would simplify testing and support multi-instance setups in the future.
