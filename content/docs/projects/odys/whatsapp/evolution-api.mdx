# Evolution API

The Evolution API serves as Odys's critical gateway to WhatsApp, enabling the platform to communicate with both professionals and clients. This document explores the design and implementation of this integration, detailing its role in Odys's "WhatsApp-first" strategy, its local deployment via Docker, and the evolution of its API usage.

## Overview

Odys relies on the Evolution API to manage all WhatsApp interactions, from automated appointment reminders and confirmations to direct messaging between professionals and clients. This approach allows Odys to operate a single, managed WhatsApp number that handles diverse communication needs without requiring individual professionals to connect their personal WhatsApp accounts. The core logic for interacting with the Evolution API resides in `src/lib/whatsapp.ts`, which centralizes message sending and formatting. For local development, the Evolution API is deployed using Docker Compose, as defined in `docker-compose.evolution.yml`, while the production environment runs on Railway.

The `README.md` highlights that Odys's key differentiator is its ability to serve both conversational AI intake and automated reminders through one managed line. This design choice aims to simplify the user experience by consolidating all communications within a single WhatsApp thread, clearly distinguishing automated notifications from AI-driven conversations through visual cues. The system also implements conversation-aware delivery, where the reminder cron skips pushes if a client is mid-conversation with the AI agent (<15 min idle). Automated reminders are sent via cron jobs: `GET /api/cron/reminders` daily at 8 AM and `GET /api/cron/whatsapp-watchdog` daily at 9 AM.

## Evolution API Client (`src/lib/whatsapp.ts`)

The `src/lib/whatsapp.ts` file is the single source of truth for all WhatsApp message interactions within Odys. It encapsulates the logic for connecting to the Evolution API using `EVOLUTION_API_URL`, `EVOLUTION_API_KEY`, and `EVOLUTION_INSTANCE` from environment variables, formatting messages, and handling phone number normalization.

At its heart is the `sendWhatsApp` function, which acts as the primary interface for sending text messages. This function accepts a `phone` number (which is then normalized), the `text` content, and an optional `context` object. The `context` is particularly important for transactional messages sent to clients, as it allows Odys to store `professionalId` and `appointmentId` in Redis. This stored context enables the `POST /api/whatsapp/webhook` endpoint to correctly route client replies back to the relevant professional or appointment, maintaining conversational continuity. The `sendWhatsApp` function is designed to be resilient, returning `true` on success and `false` on failure, logging errors internally without propagating exceptions to calling code.

The file also includes `normalizeBrazilianPhone`, a utility function that converts various phone number formats into a standardized international format, specifically handling Brazilian numbers by prepending "55" if the country code is missing. This ensures consistent delivery through the Evolution API.

To enhance user experience and distinguish automated messages from conversational AI, `src/lib/whatsapp.ts` implements a visual templating system. Functions like `transactionalHeader` (which uses `━━━━━━━━━━━━━━━━━`) and `TRANSACTIONAL_FOOTER` add distinct rule lines and branding to client-facing transactional messages (e.g., `msgBookingConfirmed`, `msgReminder24h`). This visual separation is crucial when the same Odys WhatsApp number handles both automated alerts and AI-driven chat, preventing confusion for the client.

The file defines a total of 19 message templates, such as `msgBookingRequest`, `msgBookingConfirmed`, and `msgNewMessageToClient`. These templates are categorized into client-facing transactional messages (which use the header and footer formatting) and professional-facing operational messages. This centralization ensures consistency in messaging and simplifies updates to communication flows.

The code explicitly notes the Evolution API's payload shape evolution, stating that `v2.3+` uses a flat `{number, text, delay?}` structure, unlike the nested `textMessage` object found in `v1.8`. This detail in the comments helps developers understand the API's progression and the current expected payload.

## Local Deployment with Docker

For local development, the Evolution API is set up using `docker-compose.evolution.yml`. This configuration orchestrates three services:

1.  **`postgres`**: A PostgreSQL 15 database container, named `evolution-postgres`, which serves as the persistent storage for the Evolution API's internal data. It uses a named volume `evolution_pg` to ensure data persistence across container restarts.
2.  **`redis`**: A Redis 7 Alpine container, named `evolution-redis`, used for caching and session management by the Evolution API. It also uses a named volume `evolution_redis` for data persistence.
3.  **`evolution-api`**: The core Evolution API service, running the `evoapicloud/evolution-api:v2.3.7` Docker image. This container is configured to depend on both `postgres` and `redis`, ensuring they are healthy and started before the API attempts to connect. It exposes port `8080` and is configured with various environment variables to control its behavior, including `AUTHENTICATION_API_KEY` for securing API access, `DATABASE_PROVIDER` set to `postgresql`, `CACHE_REDIS_ENABLED` to enable Redis caching, and `TELEMETRY_ENABLED` set to `false`. Notably, `WEBHOOK_GLOBAL_ENABLED` is set to `false`, indicating that webhooks are configured per-instance via the API rather than globally.

This Docker Compose setup provides a self-contained, reproducible environment for running the Evolution API locally, mirroring the production setup's dependencies on PostgreSQL and Redis.

## Production Deployment on Railway

In production, the Evolution API is self-hosted on Railway, as mentioned in the `README.md`. This deployment uses a Docker image, `tiagorcfortunato/evolution-api-odys`, and connects to a separate Supabase project (`odys-evolution`) for its database, utilizing a `evolution_api` schema. The connection to Supabase is facilitated via PgBouncer on port `6543`. This separation of concerns ensures that the Evolution API's operational data is isolated from Odys's primary application database.

## Design Decisions

### Centralized WhatsApp Messaging

The decision to centralize all WhatsApp message templates and sending logic within `src/lib/whatsapp.ts` was made to enforce consistency and simplify maintenance. By abstracting the Evolution API client, any changes to the underlying WhatsApp provider (e.g., migrating to Meta's WhatsApp Business Cloud API) can be contained within this single file, minimizing impact across the codebase. This design also prevents message strings from being scattered throughout API routes, making it easier to review and update communication content.

### Visual Disambiguation in Messages

The implementation of `transactionalHeader` and `TRANSACTIONAL_FOOTER` for client-facing messages addresses a critical user experience challenge. When a single WhatsApp number handles both automated notifications and conversational AI, clients can become confused about the source and nature of messages. The distinct visual formatting helps clients immediately identify automated, transactional alerts from a specific professional, preventing them from mistaking them for AI chat responses or general platform messages. This trade-off prioritizes clarity and user trust over a completely uniform message appearance.

### Context for Stateful Conversations

The `context` parameter in `sendWhatsApp` and its subsequent storage in Redis via `setOutboundContext` is a pragmatic solution for managing state in an otherwise stateless messaging environment. WhatsApp webhooks typically provide only the incoming message and sender. Without this context, Odys's AI agent would struggle to determine which professional or appointment a client's reply pertains to, especially for clients interacting with multiple professionals. The trade-off is the introduction of a short-lived state in Redis, which adds a dependency but significantly improves the conversational flow and reduces friction for multi-professional clients.

### Evolution API Selection

The choice of Evolution API was driven by the need to send WhatsApp messages from a dedicated corporate number without the complexities and costs associated with Meta's official WhatsApp Business API at the project's inception. Evolution API, by acting as a WhatsApp Web shim, offered a more accessible entry point for integrating WhatsApp functionality. The `README.md` explicitly outlines a "scale path" to Meta's API, indicating that the current Evolution API integration is a strategic stepping stone, acknowledging its limitations (e.g., not an officially sanctioned channel, potential ban risk) while providing immediate functionality.

### Resilient Message Sending

The `sendWhatsApp` function's design to return a boolean (`true`/`false`) rather than throwing exceptions reflects a decision to handle communication failures gracefully. This approach prevents application crashes due to transient network issues or API errors, allowing the calling code to react to a failed send without needing extensive `try-catch` blocks. The trade-off is that callers receive less detailed error information, relying on server-side logs for debugging.

## Potential Improvements

1.  **Enhanced Message Templating**: The current message templates in `src/lib/whatsapp.ts` use string concatenation, which can become cumbersome for complex messages with many variables or conditional logic. Consider adopting a more structured templating approach, perhaps using a library like Handlebars or even a JSX-like syntax if the messages were rendered in a React context, to improve readability, maintainability, and prevent potential formatting errors. For example, `msgBookingConfirmed` could benefit from a more declarative structure.

2.  **Granular Error Reporting in `sendWhatsApp`**: While `sendWhatsApp` currently returns `false` on failure, this boolean provides minimal information to the caller. For critical operations, knowing *why* a message failed (e.g., invalid phone number, API key error, network issue) could enable more specific retry logic or user feedback. Modifying `sendWhatsApp` to return a more detailed result object (e.g., `{ success: boolean, error?: string }`) would allow calling functions to make more informed decisions, such as distinguishing between a temporary API outage and a permanent client data issue.

3.  **Configurable Message Delay**: The `delay: 1200` (milliseconds) in the `sendWhatsApp` function is a hardcoded magic number. This delay is likely intended to mimic human typing speed or prevent rate limiting, but its fixed value might not be optimal for all scenarios or future Evolution API versions. Externalizing this value into an environment variable or a configuration constant would allow for easier tuning and adaptation without code changes, improving flexibility.

4.  **International Phone Number Handling**: The `normalizeBrazilianPhone` function is specifically tailored for Brazilian phone numbers, with a fallback for other international numbers if they already include a country code. If Odys plans to expand beyond Brazil, this function would need to be generalized to handle a wider range of international formats more explicitly, or replaced with a dedicated international phone number parsing library to prevent issues with non-Brazilian clients.
