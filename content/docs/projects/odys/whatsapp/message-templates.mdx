# WhatsApp Message Templates

The `src/lib/whatsapp.ts` file serves as the central hub for all WhatsApp messaging within Odys. It defines the interface for interacting with the Evolution API, a client for sending WhatsApp messages, and crucially, encapsulates all outbound message content. By centralizing message definitions here, the system ensures consistency in tone, branding, and formatting across all automated communications, preventing message strings from being scattered throughout various application routes.

## Overview

The `whatsapp` module is responsible for sending transactional and conversational messages to both clients and professionals. It integrates with the Evolution API, whose configuration (`BASE_URL`, `API_KEY`, `INSTANCE`) is loaded from environment variables. A critical `isConfigured()` check prevents message attempts if these variables are missing, logging a `whatsapp_not_configured` warning instead.

At its core, the `sendWhatsApp` function handles the actual API call. It takes a `phone` number (which is normalized by `normalizeBrazilianPhone`), the `text` content, and an optional `context` object. The `normalizeBrazilianPhone` function is designed to convert various phone number formats into a consistent international format without the leading `+`, specifically handling Brazilian numbers by prepending `55` if the country code is missing.

Odys defines a total of 19 distinct message templates, each represented by an exported function prefixed with `msg`. These include:
`msgBookingRequest`, `msgBookingConfirmed`, `msgBookingRejected`, `msgBookingCancelled`, `msgReminder24h`, `msgReminder1h`, `msgCancelledByClientToPro`, `msgCancelledByProToClient`, `msgBookingConfirmedReceipt`, `msgBookingRejectedReceipt`, `msgRegistrationInvite`, `msgPaymentConfirmedToClient`, `msgPaymentConfirmedToPro`, `msgAppointmentCompletedToClient`, `msgAppointmentCompletedToPro`, `msgNoShowToClient`, `msgNoShowToPro`, `msgNewMessageToClient`, and `msgNewMessageToPro`.

Each `msg*` function accepts an `opts` object, allowing dynamic content like client names, dates, and times to be injected into the predefined message structure.

## Message Formatting Strategy

A key design choice in `src/lib/whatsapp.ts` is the visual templating applied to client-facing transactional messages. These messages, such as reminders, confirmations, and cancellations, are designed to stand out from regular conversational AI replies within the same WhatsApp thread. This distinction is achieved through the use of a `transactionalHeader` and `TRANSACTIONAL_FOOTER`.

The `transactionalHeader` function prepends a consistent block to messages, featuring a `HEADER_RULE` (a line of `━━━━━━━━━━━━━━━━━` characters), an emoji, a label (e.g., "Consulta confirmada"), and the professional's name. This header immediately signals to the recipient that the message is an automated notification from Odys. Similarly, the `TRANSACTIONAL_FOOTER` appends a consistent branding line: `_Odys · agenda inteligente · odys.com.br_`.

Messages directed to professionals, or internal receipts, generally omit this elaborate header and footer, opting for a more concise format. This differentiation helps manage user expectations and clarifies the nature of the communication.

The `sendWhatsApp` function also supports an optional `context` parameter. When provided for client-facing messages, this context (containing `professionalId` and optionally `appointmentId`) is stored in Redis via `setOutboundContext`. This mechanism is crucial for routing subsequent client replies back to the correct professional or appointment, enabling a seamless conversational flow even after an automated message.

## Design Decisions

The architecture of `src/lib/whatsapp.ts` reflects several deliberate design choices:

*   **Centralized Message Definitions**: All 19 message templates are defined in a single file. This centralization simplifies maintenance, ensures consistent language and branding, and makes it easier to audit or update all outbound communications. Instead of embedding message strings directly within application logic or API routes, developers are encouraged to `import` and `call` these functions, promoting reusability and reducing the risk of inconsistencies.
*   **Visual Distinction for Transactional Messages**: The decision to implement `transactionalHeader` and `TRANSACTIONAL_FOOTER` with `HEADER_RULE` is a direct response to the need for clarity in a mixed-channel environment. When the same WhatsApp number handles both automated notifications and human-like AI conversations, visual cues become essential to help users quickly understand the message's origin and purpose. The rule lines provide a clear visual break, distinguishing system-generated alerts from ongoing dialogue.
*   **Robust Phone Number Normalization**: The `normalizeBrazilianPhone` function addresses the common challenge of varied phone number inputs. By automatically converting numbers to an international format, it reduces errors in API calls and ensures messages reach their intended recipients, regardless of how the number was initially entered. The logic specifically handles existing Brazilian country codes (`55`) and other international codes, preventing unintended modifications.
*   **Non-Throwing `sendWhatsApp`**: The `sendWhatsApp` function is designed to return a `boolean` (`true` for success, `false` for failure) rather than throwing exceptions. This choice simplifies error handling for callers, allowing them to react to a `false` return value without needing `try/catch` blocks, making the integration with various parts of the application more straightforward. Failures are logged using `logger.error` for observability.

## Potential Improvements

While the current implementation is effective, several areas could be enhanced:

1.  **Type Safety for Template Options**: The `opts` parameters for all `msg*` functions are currently explicitly typed with inline object literal types. Introducing explicit TypeScript interfaces for each template's options (e.g., `interface MsgBookingConfirmedOpts { professionalName: string; clientName: string; date: string; time: string; }`) would significantly improve type safety, provide better autocompletion, and reduce potential runtime errors when calling these functions.
2.  **Centralized URL Construction**: Several message templates, such as `msgCancelledByProToClient` and `msgRegistrationInvite`, construct URLs by concatenating `APP_URL` with paths and `URLSearchParams`. Abstracting this into a dedicated helper function (e.g., `buildAppUrl(path: string, params?: Record<string, string>)`) could prevent duplication, ensure consistent URL formatting, and make future changes to the application's URL structure easier to manage across all templates.
3.  **Configurable API Delay**: The `sendWhatsApp` function includes a hardcoded `delay: 1200` milliseconds in its API request body. While this might be a specific requirement for the Evolution API, making this value a configurable constant (e.g., `WHATSAPP_SEND_DELAY_MS`) or even an optional parameter to `sendWhatsApp` would allow for easier adjustments to message delivery timing without modifying the core logic of the `sendWhatsApp` function itself.
