# WhatsApp Message Templates

This page details the implementation of WhatsApp message sending and the standardized message templates used across the application. All WhatsApp-related logic, including message formatting and dispatch, is encapsulated within `src/lib/whatsapp.ts`.

## Overview

The `src/lib/whatsapp.ts` module serves as the client for the Evolution API, handling all outbound WhatsApp messages. It centralizes message definitions, ensuring consistency and preventing direct string concatenation in API routes or other parts of the application. Environment variables `EVOLUTION_API_URL`, `EVOLUTION_API_KEY`, and `EVOLUTION_INSTANCE` are required for the API to be configured and operational.

## Message Formatting Strategy

Client-facing transactional messages, such as appointment reminders, confirmations, and cancellations, adhere to a specific visual templating strategy. This is implemented by the `transactionalHeader` and `TRANSACTIONAL_FOOTER` functions, which wrap the message content with rule lines (`HEADER_RULE`) and include the professional's name and an Odys branding footer.

This visual distinction is critical for differentiating automated notifications from conversational AI replies within the same WhatsApp thread, especially when the same Odys number manages both channels.

```typescript
const HEADER_RULE = "━━━━━━━━━━━━━━━━━"

function transactionalHeader(emoji: string, label: string, professionalName: string): string {
  return `${HEADER_RULE}\n${emoji} *${label}* · ${professionalName}\n${HEADER_RULE}\n\n`
}

const TRANSACTIONAL_FOOTER = `\n\n_Odys · agenda inteligente · odys.com.br_`
```

## Sending Messages

The core function for dispatching messages is `sendWhatsApp({ phone, text, context })` in `src/lib/whatsapp.ts`. This asynchronous function handles the API call to the Evolution API.

Before sending, phone numbers are normalized using `normalizeBrazilianPhone` to ensure they conform to an international format without the `+` prefix. This function intelligently handles Brazilian numbers (adding `55` if missing) and preserves other international formats.

A key feature of `sendWhatsApp` is the optional `context` parameter. When provided and the message is successfully sent, it records `{ professionalId, appointmentId }` in Redis via `setOutboundContext` (from `src/lib/conversation`). This mechanism allows subsequent client replies within a 2-hour window to be automatically routed by the webhook to the correct professional and appointment, facilitating conversational continuity. This context is intended for transactional messages sent to clients; messages to professionals or AI agent replies should omit this field.

The `sendWhatsApp` function is designed to return `true` on success and `false` on failure, rather than throwing an exception. This design choice means that WhatsApp delivery is never load-bearing for the API response, and call sites are expected to handle potential failures gracefully, typically via `console.warn`.

## Pre-formatted Message Templates

`src/lib/whatsapp.ts` defines a suite of approximately 20 `msg*` functions, each responsible for generating a specific message type. These functions encapsulate the full message text, including the transactional header and footer where applicable, and accept an `opts` object for dynamic content.

Examples include:
-   `msgBookingRequest`: Notifies a professional of a new booking request.
-   `msgBookingConfirmed`: Informs a client that their booking has been confirmed.
-   `msgBookingRejected`: Notifies a client that their booking could not be confirmed.
-   `msgBookingCancelled`: Confirms a booking cancellation to a client.
-   `msgReminder24h`: Sends a 24-hour appointment reminder to a client.
-   `msgReminder1h`: Sends a 1-hour appointment reminder to a client.

This approach ensures consistent messaging, simplifies message generation across the application, and centralizes all WhatsApp message content for easier management and localization.

## Known Gaps

The `sendWhatsApp` function in `src/lib/whatsapp.ts` returns `false` on failure rather than throwing an exception. While this prevents WhatsApp delivery issues from blocking API responses, it places the burden on every call site to explicitly check the return value and handle potential delivery failures (e.g., via `console.warn`). This design choice prioritizes system resilience over strict error propagation for non-critical notifications.

## Why this shape

The decision to centralize all WhatsApp message templates and sending logic in `src/lib/whatsapp.ts` serves several purposes:

1.  **Consistency**: Ensures all outbound messages adhere to a consistent tone, branding, and formatting, especially the distinction between transactional and conversational messages.
2.  **Maintainability**: Simplifies updates to message content or the underlying WhatsApp API integration. All changes are localized to a single module.
3.  **Reliability**: The non-throwing `sendWhatsApp` function ensures that the core application flow is not interrupted by transient WhatsApp API issues, treating message delivery as a best-effort notification rather than a critical path operation.
4.  **Contextual Routing**: The `context` mechanism in `sendWhatsApp` is crucial for enabling intelligent routing of client replies back to the relevant professional and appointment, supporting a more seamless conversational experience.
