# WhatsApp Message Templates

This page details the implementation of WhatsApp message templates and the formatting strategy used across the application. All message definitions and the sending logic are centralized in `src/lib/whatsapp.ts`.

## Overview

The `src/lib/whatsapp.ts` module acts as the client for the Evolution API, managing all outbound WhatsApp messages. It exports approximately 20 `msg*` functions that generate pre-formatted messages, ensuring consistency and preventing ad-hoc string concatenation in other parts of the codebase. The module relies on environment variables `EVOLUTION_API_URL`, `EVOLUTION_API_KEY`, and `EVOLUTION_INSTANCE` for API configuration.

## Message Formatting Strategy

Client-facing transactional messages use a standardized visual template to distinguish automated notifications from conversational AI replies, especially when both share the same WhatsApp thread.

The formatting is implemented via:
- `HEADER_RULE`: A Unicode rule line (`"━━━━━━━━━━━━━━━━━"`) used as a visual divider.
- `transactionalHeader(emoji, label, professionalName)`: Generates a header block with emoji, bold label, professional name, and rule lines.
- `TRANSACTIONAL_FOOTER`: A consistent footer appended to client messages: `_Odys · agenda inteligente · odys.com.br_`.

This structure ensures brand consistency and clarity in automated communications.

```typescript
const HEADER_RULE = "━━━━━━━━━━━━━━━━━"

function transactionalHeader(emoji: string, label: string, professionalName: string): string {
  return `${HEADER_RULE}\n${emoji} *${label}* · ${professionalName}\n${HEADER_RULE}\n\n`
}

const TRANSACTIONAL_FOOTER = `\n\n_Odys · agenda inteligente · odys.com.br_`
```

## Sending Messages

The core function `sendWhatsApp({ phone, text, context })` handles message dispatch to the Evolution API. It normalizes phone numbers using `normalizeBrazilianPhone`, which:
- Removes non-digit characters.
- Preserves international numbers with non-55 country codes.
- Prefixes Brazilian numbers without a country code with `55`.

The optional `context` parameter, when provided, records `{ professionalId, appointmentId }` in Redis via `setOutboundContext(number, context)` upon successful send. This enables reply routing within a 2-hour window and is intended only for client-facing transactional messages.

`sendWhatsApp` returns `true` on success and `false` on failure, never throwing an exception. This design ensures WhatsApp delivery issues do not block API responses.

## Pre-formatted Message Templates

The module exports 20+ `msg*` functions, each generating a specific message type with consistent formatting. These are grouped by use case:

### Booking Lifecycle

|--------|-----------|-------|
| `msgBookingRequest` | Professional | New booking request notification |
| `msgBookingConfirmed` | Client | Confirmation with header/footer |

| `msgBookingCancelled` | Client | Cancellation confirmation |

## Known Gaps

The `sendWhatsApp` function in `src/lib/whatsapp.ts` returns `false` on failure rather than throwing an exception. While this prevents transient WhatsApp API issues from blocking core application flows, it places the burden on every call site to explicitly check the return value and handle failures (e.g., via `console.warn`). This design treats message delivery as best-effort, which is appropriate for non-critical notifications but may lead to silent failures if call sites neglect error handling.

## Why this shape

Centralizing all WhatsApp message templates and sending logic in `src/lib/whatsapp.ts` ensures:
1. **Consistency**: Uniform formatting, tone, and branding across all messages.
2. **Maintainability**: All message content and structure changes are localized to one file.
3. **Reliability**: Non-throwing `sendWhatsApp` decouples core business logic from external API stability.
4. **Contextual Routing**: The `context` parameter enables intelligent reply routing by storing outbound message context in Redis, critical for maintaining conversational state across asynchronous interactions.
