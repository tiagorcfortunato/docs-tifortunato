# WhatsApp Message Templates

Odys uses WhatsApp as a primary channel for transactional notifications and conversational engagement between professionals and clients. All outbound messages are centralized in `src/lib/whatsapp.ts` as pure functions, ensuring consistency, auditability, and localization-readiness. This design prevents message logic from scattering across routes or components, making it easier to maintain tone, branding, and compliance.

## Overview

There are exactly 19 message template functions, all prefixed with `msg`, covering booking lifecycle events (requests, confirmations, cancellations), reminders, payments, no-shows, completions, and messaging notifications. These are split by recipient: some target clients, others professionals, and some serve as receipts for actions taken in the system. Each function returns a formatted string consumed by the `sendWhatsApp` utility, which handles delivery via Evolution API (v2.3+), documented at `https://doc.evolution-api.com`, including a 1200ms delay for messages.

## Structural Patterns

Messages fall into three visual categories. Client-facing transactional messages — like confirmations or reminders — use a standardized header and footer. The header includes a rule line (`━━━━━━━━━━━━━━━━━`), an emoji, a bold label, and the professional’s name, creating a branded, automated look. The footer credits Odys with a subtle italicized tagline. This visual scaffolding distinguishes system messages from conversational AI replies sharing the same WhatsApp thread.

In contrast, messages to professionals are more concise and action-oriented, often starting with an emoji and bold summary (e.g., `✅ *Sucesso!*`). They omit the full header/footer to reduce noise for internal users. Some templates, like `msgNewMessageToPro`, include message previews and direct links to the dashboard, prioritizing utility over formality.

## Design decisions

The use of pure functions ensures messages are predictable and testable. Parameters are explicitly typed, reducing runtime errors. The `transactionalHeader` and `TRANSACTIONAL_FOOTER` abstractions enforce consistency while allowing dynamic content. Separating client and professional messages reflects their distinct needs: clients require reassurance and guidance; professionals need clarity and speed. The `context` parameter in `sendWhatsApp` enables conversation routing by storing routing metadata in Redis via `setOutboundContext`.

## Potential improvements

The `msgCancelledByProToClient` and `msgNoShowToClient` functions both include a rescheduling link using `opts.slug`, but this field is missing from other relevant templates like `msgBookingRejected`. Adding `slug` universally would enable consistent self-service recovery paths.

The `msgRegistrationInvite` function appends registration instructions to confirmation messages, but it’s not composed within `msgBookingConfirmed`. This manual composition risks omission. Consider embedding it conditionally inside the confirmation template when a client lacks an account.

The `normalizeBrazilianPhone` function assumes 55 as the default country code, but its logic for detecting non-Brazilian numbers (`digits.length >= 12`) may misfire on malformed inputs. A more robust approach would validate country code prefixes against a known set rather than relying solely on length.
