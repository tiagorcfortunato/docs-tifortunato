# Architecture: System Overview, Runtimes & Data Residency

Odys is a scheduling platform built for independent professionals in Brazil, deeply integrated with WhatsApp and Stripe. This document outlines the high-level architecture, runtime responsibilities, supporting services, and data residency model that enable a seamless, automated booking and communication experience.

## Overview

Odys operates on a serverless architecture hosted primarily on Vercel, with a 10-table PostgreSQL schema managed via Drizzle ORM and hosted on Supabase. The system is structured around two core runtimes: the Next.js application (handling API routes and SSR) and cron-triggered maintenance jobs. All client and professional data resides in Supabase, ensuring strong data isolation and compliance with Brazilian privacy expectations.

The platform exposes 21 API routes, including authenticated endpoints for dashboard operations (`/api/settings`, `/api/messages`) and webhook handlers for Stripe and WhatsApp. Two cron jobs—`/api/cron/reminders` (daily at 8 AM) and `/api/cron/whatsapp-watchdog` (daily at 9 AM)—run on Vercel to send automated reminders and monitor the Evolution API health. These are defined in `vercel.json` and executed in the same runtime as the main app.

Data flows through a clear hierarchy: professionals (`professionals` table) own clients (`clients`), appointments (`appointments`), and messages (`messages`), with referential integrity enforced via foreign keys and `ON DELETE CASCADE`. Sensitive identifiers like `userId` are stored as opaque text fields, while financial data (e.g., `sessionPrice` in cents) avoids direct exposure. WhatsApp communication is abstracted through a unified `sendWhatsApp()` interface, with 19 predefined message templates covering booking, reminders, and notifications.

## Design decisions

The architecture prioritizes operational simplicity and developer velocity. Next.js App Router consolidates pages and API routes, eliminating the need for a separate backend service. Drizzle ORM provides fully typed database access, reducing runtime errors and improving refactoring safety. Supabase is chosen for its integrated auth, real-time capabilities, and managed PostgreSQL, allowing focus on product logic over infrastructure.

The single WhatsApp number—managed via Evolution API—enables a scalable, multi-tenant communication model without requiring professionals to expose personal numbers. This is critical for user adoption and privacy. Rate limiting via Upstash Redis (`rl:booking`, `rl:api`) protects against abuse, particularly on high-risk endpoints like `/api/booking` and `/api/auth/register`.

Stripe webhooks (`checkout.session.completed`, `customer.subscription.updated`) are used to synchronize subscription state, ensuring plan limits (Free, Basic, Pro, Premium) are enforced accurately. The MCP tools (`get_appointments_today`, `update_appointment_status`) expose internal state to AI agents, enabling conversational scheduling.

## Potential improvements

1. **Centralize cron logic**: The cron handlers in `/api/cron/reminders` and `/api/cron/whatsapp-watchdog` could benefit from shared utilities in `lib/cron.ts` to reduce duplication in scheduling and error handling.

2. **Enforce plan limits at schema level**: While `plan-guard.ts` enforces feature access, adding a `CHECK` constraint on `professionals.plan` to restrict values to `free|basic|pro|premium` would prevent invalid states at the database level.

3. **Index `appointments.status` and `startsAt` together**: The query for today’s appointments likely filters on both `professionalId`, `status`, and `startsAt`. A composite index on `(professional_id, status, starts_at)` in `src/lib/db/schema.ts` would improve dashboard load performance.
