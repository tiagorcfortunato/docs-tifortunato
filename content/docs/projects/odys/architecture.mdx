# System Architecture

This page describes the high-level architecture of Odys, including the primary runtime, supporting services, data storage layers, and data residency considerations. The system is built as a single Next.js 16 App Router application deployed on Vercel, with Supabase Postgres as the primary database, and integrates with several managed and self-hosted services.

## Where the data lives

Data is distributed across multiple systems:

*   **Browser (client)**: Minimal state. Supabase issues an HttpOnly session cookie. React Server Components handle initial hydration, with a few `"use client"` components adding interactivity.
*   **Vercel edge + Node runtime**: Terminates TLS, runs the Next.js router, and executes Server Components and API routes. Sentry beacons are rewritten through the `/monitoring` route, configured in `next.config.ts`, to bypass ad-blockers.
*   **Persisted storage**:
    *   Supabase Postgres: The application's 10-table schema, defined in `src/lib/db/schema.ts`.
    *   Supabase Storage: Stores uploaded avatars.
    *   Upstash Redis: Used for rate limiting via `@upstash/ratelimit` and WhatsApp reply routing (`last_outbound:{phone}`).
    *   Railway Postgres: A separate database instance for the Evolution API.
    *   Stripe: Manages subscription state and payment history.

## Runtimes in use

Odys operates across three distinct runtimes:

*   **Node (Vercel)**: The primary runtime for most API routes and Server Components, forming the core application.
*   **Edge (Vercel)**: Currently used only for Sentry's edge bootstrap, as configured in `next.config.ts`. No application routes explicitly use the edge runtime.
*   **Local stdio (developer laptop)**: The MCP server runs as a local process, not deployed to production.

There are no Node workers, background job queues, or separate microservices. Periodic tasks are handled by **Vercel Cron**, as defined in `vercel.json`. Two daily schedules are configured: `/api/cron/reminders` at 08:00 UTC and `/api/cron/whatsapp-watchdog` at 09:00 UTC.

## MCP server — a note on architecture scope

The MCP server is a **separate process, not a separate service**. It shares the same `DATABASE_URL` and the same Drizzle schema, imported via a relative path from `src/lib/db/schema.ts`. Communication with AI agents occurs via JSON-RPC over stdio. It runs exclusively on the developer's laptop, without an HTTP endpoint or authentication layer, as there is no remote caller. This setup enables AI agents to query the database directly (e.g., "how many appointments does Dr. Ana have today?") without requiring direct Drizzle Studio access.

The future deployment path for an in-product AI assistant would involve either exposing the MCP server as an authenticated HTTP service or collapsing its functionality into direct Groq tool-calls that interact with existing API routes.

## Data residency

*   **Supabase (database, auth, storage)**: Configured for `sa-east-1` (São Paulo). This ensures that primary Brazilian user data remains at rest within Brazil.
*   **Sentry**: Ingests data via `*.ingest.us.sentry.io` (US). This layer captures error traces only, not business-critical data.
*   **PostHog**: Collects product analytics events via `https://us.i.posthog.com` (US).
*   **Vercel**: The edge layer defaults to `iad1` (US East).

For buyers with strict LGPD requirements, the primary data store is already in Brazil. The remaining US-hosted layers (error tracking, analytics, HTTP edge) can be migrated to EU or São Paulo alternatives through configuration changes, without requiring a rewrite of the application logic.

## Known gaps

The current configuration for Sentry, PostHog, and Vercel's edge layer defaults to US regions. While the primary Supabase database is in `sa-east-1` (Brazil), the presence of these US-hosted services means that some operational data (error traces, analytics events, and HTTP request metadata) is processed and stored outside of Brazil. For an LGPD-strict buyer, this would require reconfiguring these services to EU or Brazilian regions.

## Why this shape

This architecture prioritizes rapid development and maintainability for a single founder, rather than infrastructure complexity. It is intentionally:

*   **Not a microservice architecture**: A single Next.js application, one primary database, and ancillary managed services.
*   **Not using Kubernetes**: Leverages Vercel's serverless runtime and Railway's container runtime for Evolution API.
*   **No separate backend**: API routes and Server Components within Next.js serve as the backend.
*   **No GraphQL or gRPC**: Relies on plain RESTful APIs with `zod` validation.
*   **No multi-region replication**: Utilizes a single-region Postgres instance.

This approach focuses on delivering a production-grade SaaS product efficiently.
