# Observability Overview

This page covers the observability stack for error tracking, performance monitoring, and product analytics using Sentry and PostHog. It details runtime-specific Sentry configurations, client-side PostHog initialization with cookie consent, and current logging practices.

## Sentry Error Tracking and Performance Monitoring

Sentry is configured across three Next.js runtimes—Node.js, Edge, and Browser—using separate initialization files. The `src/instrumentation.ts` file serves as the entry point for server-side Sentry setup, conditionally importing `sentry.server.config.ts` for the Node.js runtime and `sentry.edge.config.ts` for the Edge runtime based on `process.env.NEXT_RUNTIME`.

Both server configurations initialize Sentry with `tracesSampleRate: 1`, `enableLogs: true`, and `sendDefaultPii: true`. The `onRequestError` export from `src/instrumentation.ts` captures request-level errors via `Sentry.captureRequestError`.

Client-side instrumentation is defined in `src/instrumentation-client.ts`, which initializes the browser SDK with:
- `Sentry.replayIntegration()` for session replay
- `replaysSessionSampleRate: 0.1` (10% of sessions sampled)
- `replaysOnErrorSampleRate: 1.0` (100% of error-triggered replays captured)
- `tracesSampleRate: 1`, `enableLogs: true`, and `sendDefaultPii: true`

A `beforeSend` hook filters out known noise:
- Events containing "Object Not Found Matching" (browser extension errors)
- Events containing "Non-Error promise rejection captured" (empty rejections without stack traces)

The `onRouterTransitionStart` export captures App Router navigation starts. Source maps are uploaded during CI via `withSentryConfig` in `next.config.ts` (not provided), configured with `widenClientFileUpload: true` and routed under the `odys-hx` organization. Browser SDK traffic is tunneled through `/monitoring` to avoid ad-blocker interference.

Global error handling is implemented in `src/app/global-error.tsx` (not provided), which uses `Sentry.captureException(error)` in a `useEffect` to report unhandled exceptions.

## PostHog Product Analytics

PostHog is initialized client-side only, gated by cookie consent. The `src/components/posthog-provider.tsx` component manages initialization:
- Checks `localStorage.getItem("odys_cookie_consent") === "accepted"` for prior consent
- Listens for the `odys:cookie-accepted` event dispatched by the `CookieBanner` component
- Calls `posthog.init()` only after consent is given

PostHog is configured with:
- `capture_pageview: false` to prevent double-firing due to App Router behavior
- `capture_pageleave: true` to track exits
- `persistence: "localStorage+cookie"`

Pageviews are manually tracked by the nested `PageViewTracker` component, which calls `posthog.capture("$pageview")` on changes to `pathname` or `searchParams` using `usePathname` and `useSearchParams`.

User identification occurs in the authenticated dashboard layout via `posthog.identify(userId)` (implementation in `posthog-identify.tsx` not provided). While the provider supports feature flags, no `posthog.isFeatureEnabled(...)` calls exist in the current codebase. The booking widget directly imports `posthog-js` and calls `posthog.capture(...)` at key interaction points.

## Other Logging

The application uses `console.error` and `console.warn` with tagged prefixes (e.g., `[BOOKING POST]`, `[STRIPE WEBHOOK]`) for general logging, visible in Vercel runtime logs. The `serverError(context, err)` function, exported from `src/lib/api.ts` (not provided), logs internal server errors and returns a generic 500 response to prevent leakage of sensitive details.

No structured logging library (e.g., Pino, Winston) is currently implemented; all non-Sentry logs are unstructured console output.

## Known Gaps

*   **Sentry Tracing Sample Rate**: `tracesSampleRate: 1` (100%) is set in `src/instrumentation-client.ts` and assumed in server configs. This is acceptable pre-launch but will require reduction in production to control data volume and costs.
*   **Structured Logging**: The absence of a structured logger limits log querying and analysis capabilities. All non-Sentry logs are plain console output.
*   **PostHog Feature Flags Unused**: Despite being configured, PostHog feature flags are not actively used in the codebase—no `posthog.isFeatureEnabled(...)` calls are present.

## Why this shape

The observability architecture separates concerns: Sentry handles system health (errors, performance, replays) across all runtimes, while PostHog focuses exclusively on client-side user behavior and event tracking. Cookie-gated initialization ensures compliance with privacy expectations. The dual-tool approach avoids feature overlap and leverages each platform’s strengths—Sentry for debugging and PostHog for product insights. Manual pageview tracking compensates for App Router quirks, and noise filtering in Sentry improves signal quality.
