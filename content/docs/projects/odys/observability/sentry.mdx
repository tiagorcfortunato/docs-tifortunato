# Sentry Observability

Observability is critical for maintaining reliability and diagnosing issues in production. This page covers how Sentry is integrated across the Odys stack—server, edge, and browser—with source map support and a tunneling strategy to bypass ad blockers. It explains the current setup, design rationale, and opportunities for refinement.

## Overview

Sentry is configured across three runtime environments: Node.js (server), Edge (edge functions), and browser (client). The setup ensures comprehensive error tracking, performance tracing, and session replay. Source maps are uploaded during deployment to enable readable stack traces, and a `/monitoring` tunnel route ensures client-side events reach Sentry even when ad blockers interfere. The configuration leverages `@sentry/nextjs`, with environment-specific initialization and security-conscious headers.

## Client-Side Initialization

The client-side Sentry configuration in `src/instrumentation-client.ts` initializes the SDK with several key features. The DSN is hardcoded, and integrations include session replay for debugging user flows. Error filtering is applied via `beforeSend` to suppress known noise—such as "Object Not Found Matching" and non-Error promise rejections—reducing alert fatigue. Tracing is fully enabled (`tracesSampleRate: 1`), and logs are forwarded to Sentry. Replay session sampling is set to 10% (`replaysSessionSampleRate: 0.1`), while replay sampling on error is set to 100% (`replaysOnErrorSampleRate: 1.0`). The session sample rate could be increased during incident investigations or tied to user opt-in for better data control.

User PII is sent via `sendDefaultPii: true`, which enables richer context but requires careful compliance handling. The `onRouterTransitionStart` export captures navigation performance, aiding in diagnosing routing delays.

### App Router Error Boundary

`src/app/global-error.tsx` implements a global error boundary for the App Router. It uses `useEffect` to call `Sentry.captureException(error)` whenever an error is caught, ensuring that all unhandled client-side exceptions in the App Router are reported to Sentry. This complements the automatic error capture from `instrumentation-client.ts` and guarantees visibility into rendering and component-level failures.

## Server and Edge Configuration

Server and edge initialization is handled in `src/instrumentation.ts`, which conditionally imports `sentry.server.config` or `sentry.edge.config` based on `NEXT_RUNTIME`. This ensures minimal bundle impact and correct environment targeting. The `onRequestError` hook captures server-side request errors, providing parity with client-side observability.

## Build and Security Integration

`next.config.ts` wraps the Next.js config with `withSentryConfig`, enabling source map upload and Vercel Cron Monitor integration. Source maps are widened via `widenClientFileUpload: true` for better stack trace fidelity. Additionally, Sentry webpack plugin logs, including those for source map uploads, are configured to be silent outside of CI environments (`silent: !process.env.CI`). The `tunnelRoute: "/monitoring"` rewrites client SDK traffic through a first-party endpoint, evading ad blockers that target known Sentry domains.

Security headers are defined to harden the app, though CSP is deferred due to complexity. Notably, `Permissions-Policy` restricts camera, microphone, and geolocation, while allowing Stripe.js payments. HSTS is set for two years with preload, and referrer policy is `strict-origin-when-cross-origin`.

## Design decisions

The current Sentry setup prioritizes visibility and resilience. Tunneling through `/monitoring` ensures reliable event delivery despite ad blockers, a common issue in client-side monitoring. Conditional imports in `instrumentation.ts` prevent edge/server code from bloating the client bundle. The `beforeSend` filter reduces noise from third-party scripts and known edge cases. Enabling `sendDefaultPii` trades privacy complexity for debugging efficiency—justified in a B2B context where user consent can be managed.

## Potential improvements

1.  **Externalize DSN**: The DSN in `src/instrumentation-client.ts` is hardcoded. It should be moved to an environment variable (e.g., `NEXT_PUBLIC_SENTRY_DSN`) to allow per-environment configuration and avoid exposing production keys in dev builds.

2.  **Refine replay sampling**: `replaysSessionSampleRate: 0.1` may miss critical issues. Consider increasing it during incident investigations or tying it to user opt-in for better data control.

3.  **Implement dynamic CSP**: The comment in `next.config.ts` notes CSP is deferred. A gradual rollout—starting with a report-only header—could allow safe allowlisting of `js.stripe.com`, Supabase domains (including realtime), `sentry.io` (for the tunnel route), PostHog, and Next.js inline scripts without risking app breakage.
