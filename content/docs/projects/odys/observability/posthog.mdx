# PostHog: Observability and Analytics

In Odys, understanding user behavior is crucial for refining the platform and enhancing the professional-client experience. PostHog serves as the primary analytics tool, providing insights into how users interact with the application. This document explores the architectural choices behind Odys's PostHog integration, focusing on how user consent is managed, pageviews are tracked, and users are identified within the PostHog dashboard.

## Overview

The PostHog integration in Odys is carefully structured to balance data collection with user privacy, particularly concerning cookie consent. At its core, the system defers PostHog initialization until explicit user consent is given. Once initialized, it meticulously tracks page navigation and identifies authenticated users, ensuring that all collected data is meaningful and attributable.

The main components orchestrating this behavior are:
*   The `PostHogProvider` component, found in `src/components/posthog-provider.tsx`, which manages the lifecycle of the PostHog client.
*   The `CookieBanner` component, located in `src/components/cookie-banner.tsx`, responsible for obtaining user consent.
*   The `PageViewTracker` component, nested within the `PostHogProvider`, which handles manual pageview capturing.
*   The `PostHogIdentify` component, in `src/components/posthog-identify.tsx`, dedicated to associating user actions with specific user identities.

This modular approach ensures that analytics are only enabled when appropriate and that tracking logic is cleanly separated.

## Cookie-Gated Initialization

A cornerstone of Odys's PostHog integration is its commitment to user privacy, specifically through cookie-gated initialization. The `PostHogProvider` in `src/components/posthog-provider.tsx` does not immediately initialize PostHog upon loading. Instead, it checks for prior user consent stored in `localStorage` under the key `odys_cookie_consent`.

If a user has previously accepted cookies, PostHog is initialized right away. If not, the system waits for an explicit signal. This signal comes from the `CookieBanner` component, which displays a prompt to the user. When a user clicks "Aceitar" (Accept) on the `CookieBanner`, it dispatches a custom browser event named `odys:cookie-accepted`. The `PostHogProvider` listens for this event and, upon receiving it, proceeds to initialize PostHog using the `initPostHog` function. This function ensures that PostHog is initialized only once, preventing redundant setups.

The `posthog.init` call is configured with `capture_pageview: false` and `capture_pageleave: true`. Disabling automatic pageview capture is a deliberate choice, allowing for more granular control over what constitutes a "pageview" event, which is then handled separately. The `persistence: "localStorage+cookie"` setting ensures that user session data is maintained across browser sessions and visits.

## Manual Pageview Tracking

Given that automatic pageview capture is disabled during PostHog initialization, Odys employs a dedicated `PageViewTracker` component within `src/components/posthog-provider.tsx` to manually record page navigation. This component leverages Next.js's `usePathname` and `useSearchParams` hooks to detect changes in the URL.

Whenever the URL's pathname or search parameters change, indicating a navigation event, the `PageViewTracker` explicitly calls `ph.capture("$pageview", { $current_url: window.location.href })`. This ensures that every significant page transition is recorded in PostHog, providing a clear picture of user flow through the application. Wrapping `PageViewTracker` in a `Suspense` boundary is a standard practice when using hooks like `useSearchParams` in client components, ensuring the component can render even if the search parameters are not immediately available.

## User Identification

To connect anonymous user behavior with known user accounts, Odys uses the `PostHogIdentify` component, located in `src/components/posthog-identify.tsx`. This component takes a `userId` as a prop. When this `userId` is provided or changes, the component calls `posthog.identify(userId)`.

This action tells PostHog that the current anonymous session should now be associated with the specified `userId`. This is crucial for tracking user journeys across multiple sessions and devices, allowing Odys to build a comprehensive profile of each professional or client. For instance, a professional's `userId` from the `professionals` table, which is a `text` type and unique, could be passed here to link their actions to their professional profile.

## Design decisions

The current PostHog integration reflects several intentional design decisions:

*   **Privacy-First Initialization**: The choice to gate PostHog initialization behind explicit cookie consent, managed by the `CookieBanner` and the `odys:cookie-accepted` event, prioritizes user privacy and compliance with data protection regulations. This ensures that no analytics data is collected from users who have not consented, building trust and adhering to legal requirements. The trade-off is that initial user behavior before consent is not captured, potentially leading to a slightly incomplete picture for some users.

*   **Manual Pageview Control**: Disabling `capture_pageview` in `posthog.init` and implementing a custom `PageViewTracker` provides fine-grained control over what constitutes a pageview. This allows Odys to define page transitions more precisely, potentially filtering out internal routing events that might not represent a true "page" change from an analytics perspective, or adding custom properties to pageview events that are relevant to Odys's specific context.

*   **Dedicated User Identification Component**: Encapsulating user identification logic within `PostHogIdentify` makes it reusable and ensures that `posthog.identify` is called only when a `userId` is available and changes. This prevents unnecessary re-identifications and keeps the analytics code clean and focused. It also allows for easy integration into different parts of the application where user authentication state might change.

## Potential improvements

While the current PostHog integration is functional and privacy-conscious, there are several areas where it could be enhanced:

1.  **Enrich User Identification with Properties**: When `posthog.identify(userId)` is called in `src/components/posthog-identify.tsx`, Odys could also pass additional user properties. For example, when a professional logs in, properties like their `name`, `email`, `profession`, and `plan` (from the `professionals` table) could be sent alongside the `userId`. This would enrich user profiles in PostHog, allowing for more detailed segmentation and analysis without requiring additional lookups.

2.  **Centralized Consent Management**: The current cookie consent mechanism relies on a custom `odys:cookie-accepted` event and `localStorage` checks in `src/components/posthog-provider.tsx` and `src/components/cookie-banner.tsx`. While effective, integrating a more robust, dedicated consent management platform (CMP) or library could streamline compliance, offer more granular consent options (e.g., for different cookie categories), and provide a more standardized approach to managing user preferences.

3.  **Server-Side Event Capture for Critical Actions**: For highly critical or sensitive user actions (e.g., subscription changes, payment events), consider implementing server-side event capture using PostHog's API. This would involve sending events directly from Odys's backend (e.g., from API routes like `/api/stripe/webhook` or `/api/booking`) rather than relying solely on client-side JavaScript. Server-side capture is more reliable as it bypasses potential client-side issues like ad blockers or network interruptions, ensuring that crucial data is never missed.

4.  **Consistent Event Naming and Taxonomy**: As Odys grows, establishing a clear and consistent event naming convention and taxonomy for all PostHog events will be vital. Currently, `$pageview` is used, but for custom events, a structured approach (e.g., `feature_action_object`) would improve data clarity and analysis. This could involve creating a shared utility or a documentation standard for all developers contributing to analytics tracking.
