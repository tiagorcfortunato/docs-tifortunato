# App Router Implementation
This page details the implementation of the Next.js App Router in the application, focusing on server components, route groups, and the absence of `middleware.ts`.

## Server Components
The application uses server components for server-side rendering and data fetching. The public booking page at `src/app/p/[slug]/page.tsx` is an `async` Server Component that directly queries the database using Drizzle ORM (e.g., `db.select().from(professionals)`). It fetches professional details, reviews, related professionals, and client data (if the user is authenticated) before rendering. The `BookingWidget` component, which handles interactivity, is passed pre-fetched data as props and is rendered as a client component.

## Route Groups
Route groups organize related routes and apply isolated layouts without affecting URL paths. The `(auth)` route group is used for authentication flows, though `src/app/(auth)/layout.tsx` is not present in the current codebase. The root layout in `src/app/layout.tsx` defines global structure, including fonts (`Plus_Jakarta_Sans`, `Montserrat`), metadata, and providers such as `PostHogProvider` and `Toaster`. It also includes a theme initialization script to prevent flash of unstyled content.

## No Middleware
The application does not use a `middleware.ts` file. Instead, authentication checks are performed directly within individual route handlers. For example, `src/app/p/[slug]/page.tsx` calls `await createClient().auth.getUser()` to determine if the current visitor is authenticated and whether they are the owner of the professional page. This explicit pattern avoids centralized middleware but requires each protected route to implement its own auth logic.

## Known Gaps
The lack of `middleware.ts` means there is no centralized enforcement point for authentication. This decentralized approach, while explicit, increases the risk of inconsistent auth checks across routes. The security audit has flagged this pattern due to the developer burden of manually implementing auth on every new protected route, which could lead to accidental exposure of sensitive pages.

## Why this Shape
The App Router implementation prioritizes server-side data fetching and layout organization through route groups. Server components enable efficient, direct database access without client-side hydration overhead. Route groups provide logical separation of concerns in the routing structure. The decision to omit `middleware.ts` was made to keep authentication logic co-located with route code, favoring transparency over abstraction. However, this trade-off sacrifices consistency and scalability in auth enforcement, a concern acknowledged for future refinement.
