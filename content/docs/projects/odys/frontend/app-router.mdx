# App Router

The Odys frontend is built with Next.js, leveraging its App Router to structure the application and manage data fetching. This architecture allows for a highly performant and maintainable codebase by embracing server-first rendering and co-locating data logic with UI components.

## Overview

The App Router organizes the application around the `src/app` directory, where each folder typically represents a route segment. This structure enables features like server components, dynamic routes, and route groups, which are central to how Odys delivers its user experience.

The global layout and essential services are defined in `src/app/layout.tsx`. This root layout wraps the entire application, providing a consistent shell for all pages. It also defines static global metadata for the application, including title, description, Open Graph, and Twitter card. It's responsible for importing global CSS, setting up fonts like `Plus_Jakarta_Sans` and `Montserrat`, and integrating third-party analytics and monitoring tools such as `PostHogProvider`, `Analytics` from Vercel, and `SpeedInsights`. Additionally, it includes an inline script in the `<head>` to prevent a flash of unstyled content (FOUC) by applying the saved theme before the page renders. The `Toaster` component for notifications and a `CookieBanner` are also rendered here, ensuring they are available across the application.

Dynamic content, such as individual professional profiles, is handled through dynamic routes. A prime example is `src/app/p/[slug]/page.tsx`, which serves as the public booking page for professionals. The `[slug]` segment indicates that this part of the URL is a dynamic parameter, allowing each professional to have a unique, SEO-friendly URL like `/p/john-doe`.

While the project structure includes a route group `(auth)`, the corresponding `src/app/(auth)/layout.tsx` file is not currently present. This route group would typically encapsulate authentication-related pages, allowing them to share a specific layout or set of functionalities distinct from the main application.

Notably, the Odys application does not utilize Next.js middleware. Instead, authentication and authorization checks are integrated directly into server components or API routes, providing fine-grained control over data access at the point of consumption.

## Server Components and Data Fetching

The App Router's server components are a cornerstone of Odys's frontend architecture. Pages like `src/app/p/[slug]/page.tsx` are server components by default, meaning they execute on the server before being sent to the client. This design choice offers significant advantages, particularly for data-intensive pages.

In `src/app/p/[slug]/page.tsx`, the `BookingPage` component directly interacts with the database to fetch all necessary information for a professional's public profile. It uses Drizzle ORM to query the `professionals`, `clients`, and `reviews` tables. For instance, it retrieves the professional's details using `db.select().from(professionals).where(eq(professionals.slug, slug)).limit(1)`. This direct database access from the server component eliminates the need for client-side API calls for initial page loads, reducing latency and simplifying data flow.

The page also fetches related professionals based on the current professional's `profession`, and then calculates their average ratings and review counts using `getProStats`. This data is then passed down to client components like `ProfessionalCard` for rendering.

Authentication status is also determined server-side using `createClient()` and `supabase.auth.getUser()`. This allows the page to detect if the visitor is the professional themselves (`isOwner`) or a logged-in client, enabling conditional rendering of UI elements like the "My dashboard" link or the `FollowButton`. If a client is logged in, their data is fetched from the `clients` table to pre-fill the `BookingWidget` form, enhancing user experience.

## Dynamic Metadata

The App Router's `generateMetadata` function is used in `src/app/p/[slug]/page.tsx` to create dynamic SEO metadata for each professional's page. This function runs on the server and allows the page's title, description, and Open Graph images to be tailored based on the specific professional's data fetched from the database.

For example, the `title` is constructed using the professional's `name` and `profession` (e.g., "John Doe — Therapist | Odys"), and the `description` uses the professional's `bio` or a default message. If an `avatarUrl` is available, it's included in the Open Graph images, ensuring rich previews when shared on social media. This dynamic approach is crucial for discoverability and provides a more engaging experience for users encountering Odys links.

## Absence of Middleware

Odys does not implement Next.js middleware. Instead, authentication and authorization logic are handled directly within the server components and API routes. For instance, in `src/app/p/[slug]/page.tsx`, the `supabase.auth.getUser()` call is made within the `BookingPage` component itself to determine the user's session. Similarly, API routes like `/api/booking` or `/api/follows` would contain their own checks to ensure requests are authorized.

This approach provides granular control over access at the component or route level. While middleware can offer a centralized point for global checks, the current strategy allows for specific authorization logic to be co-located with the data fetching and mutation operations it protects.

## Design Decisions

The architectural choices in Odys reflect a strong preference for server-side rendering and data fetching, primarily driven by performance and developer experience.

The decision to use **server components** for pages like `src/app/p/[slug]/page.tsx` is rooted in the desire to minimize client-side JavaScript and improve initial page load times. By fetching data directly from the database using Drizzle ORM on the server, the application avoids the "waterfall" effect of client-side data fetching, where the client first downloads the page, then makes an API call, and then renders the data. This also simplifies the data flow, as server components can directly access backend resources without exposing database credentials to the client.

**Dynamic metadata generation** via `generateMetadata` is a deliberate choice to optimize for search engine visibility and social media sharing. For a platform like Odys, where individual professional profiles are key landing pages, having unique and descriptive metadata for each profile is paramount for organic discovery.

The current approach of **not using Next.js middleware** suggests a preference for handling authentication and authorization closer to the data access layer. By performing `supabase.auth.getUser()` checks directly within server components or API routes, developers can ensure that data is only fetched or modified if the user is authorized for that specific operation, without the overhead of a global middleware layer that might not be necessary for every route. This can lead to simpler debugging and more explicit authorization logic for each endpoint.

## Potential Improvements

1.  **Implement `(auth)/layout.tsx` for Authentication Flow**: The `(auth)` route group is present in the file structure, but `src/app/(auth)/layout.tsx` is missing. Implementing this layout would allow for a consistent UI and shared logic (e.g., authentication checks, form wrappers) across all authentication-related pages (like login, register, password reset). This would centralize common elements and reduce duplication across individual auth pages.
2.  **Centralize User Authentication Logic**: The `supabase.auth.getUser()` call is made directly within `src/app/p/[slug]/page.tsx`. While effective, if similar user checks are needed across many server components, this logic could be abstracted into a reusable server utility function or a custom hook. This would promote DRY principles and make it easier to update authentication logic consistently across the application.
3.  **Optimize Related Professionals Query**: In `src/app/p/[slug]/page.tsx`, the `allRelated` query fetches all professionals with the same `profession` and then filters them in memory (`allRelated.filter(p => p.slug !== slug && p.id !== prof.id).slice(0, 4)`). While `slice(0, 4)` limits the final output, the initial query might fetch more data than strictly necessary if there are many professionals in the same category. Consider adding `limit` and `offset` to the Drizzle query itself, potentially combined with `not(eq(professionals.slug, slug))` to perform filtering at the database level, which could be more efficient for very large datasets.
