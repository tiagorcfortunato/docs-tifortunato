# App Router Usage

The App Router in Odys structures the entire frontend experience, leveraging React Server Components to deliver fast, data-driven pages while organizing routes logically. This document explains how routing, data loading, and layout are implemented across the application.

## Overview

Odys uses Next.js App Router with a clear separation of concerns: public pages like the booking interface are statically optimized, while authenticated areas like the dashboard are protected and personalized. The router contains **23 API routes** handling actions from booking to Stripe webhooks, and 11 database tables backing features like appointments, clients, and reviews. There is no middleware; authentication and redirects are handled at the component level using `getUser()` and Supabase. Route groups organize concerns—`(auth)` likely wraps login/signup, though its layout is missing—while dynamic routes like `/p/[slug]` power public booking pages.

## Public Booking Page: Server-Driven Rendering

The `/p/[slug]` page is a server component that fetches a professional’s data, reviews, and related profiles in a single server round-trip, with availability handled by a client component. It uses `generateMetadata` to create SEO-friendly titles and descriptions, pulling from the `professionals` table. The page conditionally renders a `FollowButton` and pre-fills client data if the user is logged in, using Supabase auth to identify the visitor. Related professionals are fetched by matching the same `profession`, enabling cross-promotion without client-side queries.

## Layout and Theming

The root `layout.tsx` sets global fonts, injects analytics (PostHog, Vercel), and applies a dark mode class from localStorage before first paint—avoiding theme flicker. It wraps children in `PostHogProvider` and includes a `CookieBanner`. No layout is defined for `(auth)`, suggesting auth pages may inherit the root layout or be unstyled. The booking page uses a custom header with navigation shortcuts and a theme toggle, ensuring consistency across the app.

## Design decisions

The absence of middleware simplifies deployment and reduces cold starts, pushing auth checks into components. Server components are used extensively to minimize client-side JavaScript—critical for performance on mobile networks common among Brazilian users. Route groups like `(auth)` and `(dashboard)` are implied by the codebase structure, but only the root layout is present; the `(auth)` layout is missing, and the booking page uses the root layout. The use of `notFound()` in `page.tsx` ensures SEO compliance by returning 404s early.

## Potential improvements

1. **Missing `(auth)/layout.tsx`** — The absence of this file may lead to inconsistent styling or missing providers on login pages. It should be created to wrap auth flows in the same analytics and theme context.
2. **Repeated client data fetching** — The pattern of fetching `clients` by `userId` appears in multiple routes. This logic should be abstracted into a `getCurrentClient()` helper in `lib/db` to reduce duplication.
3. **Over-fetching in related professionals** — The query for related professionals in `/p/[slug]/page.tsx` explicitly selects a subset of columns from `professionals`.
