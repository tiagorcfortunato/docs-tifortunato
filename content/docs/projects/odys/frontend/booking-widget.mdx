# Booking Widget Internals

The booking widget is the core client-facing component for scheduling appointments. It orchestrates user interaction, form validation, time slot generation, and backend communication to create a seamless booking flow. Understanding its internals reveals how Odys balances usability, performance, and spam resistance.

## Overview

The widget operates as a controlled 3-step state machine (`pick` Ôćĺ `form` Ôćĺ `success`) within a React Client Component. It fetches availability rules and existing appointments via `/api/booking`, computes valid time slots using `generateSlots`, and submits bookings through the same endpoint. The form leverages `react-hook-form` with Zod validation, while honeypot techniques deter bots. Data flows from `page.tsx`, which hydrates props like `sessionPrice` (in cents), `paymentType`, `sessionDuration`, `paymentPercentage`, `pixKeyType`, `pixKey`, `defaultName`, `defaultPhone`, `defaultEmail`, and `isLoggedIn` from the `professionals` and `clients` tables (one of 11 schema tables).

## State Machine and User Flow

The widgetÔÇÖs state is managed via `useState<"pick" | "form" | "success">`, guiding users through selection, data entry, and confirmation. On date selection, `useEffect` triggers a fetch to `/api/booking?slug=...&date=...`, retrieving `{ rules, existingAppointments }`. These feed into `generateSlots`, which returns a flat list of available times in `HH:mm` format. The raw slots are then grouped into morning, afternoon, and evening by `groupSlots` in `booking-widget.tsx` for display. Selecting a slot enables the ÔÇťContinuarÔÇŁ button, advancing to the form step. This separation prevents premature data submission and reduces cognitive load.

## Form Handling and Validation

The form uses `react-hook-form` with a Zod schema requiring `name` (min 2 chars) and `phone` (min 10 chars), while making `email` optional. A hidden `website` field acts as a honeypotÔÇöany value here is rejected server-side as spam. Default values are pre-filled if the user is logged in, sourced from the `clients` table. On submission, `onSubmit` constructs a `startsAt` timestamp from the selected date and slot, then POSTs to `/api/booking`. Success transitions to the `success` step, displaying a confirmation and, if applicable, a Pix QR code via the `PixQR` component.

## Design Decisions

The 3-step flow minimizes errors by isolating time selection from data entry. Using `react-hook-form` enables efficient re-renders and instant validation feedback. The honeypot (`website` field) is a lightweight anti-bot measure that doesnÔÇÖt burden users. Time slot grouping improves scannability on mobile. Fetching availability per date (not upfront) reduces initial payload, though it may cause per-date latency. The widget assumes `sessionDuration` and `paymentType` are static per professional, stored in the `professionals` table with defaults.

## Potential Improvements

1. **Avoid Re-fetching on Date Change**: In `booking-widget.tsx`, the `useEffect` re-fetches availability even when navigating between adjacent dates. Caching responses by date string could reduce redundant calls to `/api/booking`.

2. **Improve Slot Accessibility**: The `SlotButton` component uses visual styling for selection but lacks `aria-pressed` or `role="radio"` attributes. Adding these would clarify state for screen readers.

3. **Centralize Date Formatting**: The widget uses `date-fns` with `ptBR` locale in multiple places (e.g., success message, calendar). Extracting format strings into constants would ensure consistency and simplify i18n.
