# Professional Dashboard

The professional dashboard is the central hub for service providers in the Odys platform, offering a real-time view of appointments, client interactions, and business metrics. It combines data from 5 core tables—including `appointments`, `clients`, `reviews`, `professionals`, and `availability`—to deliver a personalized, actionable interface that adapts to the user’s subscription plan and trial status.

## Overview

The dashboard is structured around a persistent sidebar navigation and a dynamic main content area. Built with Next.js App Router and Drizzle ORM, it fetches data server-side using Supabase for authentication and PostgreSQL for storage. The layout is wrapped in `DashboardLayout`, which ensures the user is authenticated and has completed onboarding before rendering. The sidebar, a client component, conditionally locks Pro-only features like "Mensagens", "Relatórios", and "Assistente IA" based on the user’s effective plan, determined by `effectivePlan` and `trialDaysLeft` utilities.

## Core Functionality

The dashboard displays today’s and upcoming appointments, pulling from the `appointments` and `clients` tables with joins on `professionalId`. Status labels—such as "Aguardando confirmação"—are mapped via `STATUS_LABEL` and rendered with appropriate styling. The `AppointmentActions` component enables status transitions (confirm/reject) for pending or confirmed appointments, leveraging the `/api/appointments/[id]` PATCH endpoint.

Key metrics like client count, monthly appointments, average rating, and daily revenue are computed using Drizzle’s `count`, SQL aggregation, and `professional.sessionPrice`. The `reviews` table is joined with `clients` to display recent feedback, while `availability` count informs profile completeness via `findProfileGaps`. Trial and plan banners dynamically appear based on `trialEndsAt` and plan limits, guiding users toward upgrades.

## Design decisions

The UI prioritizes immediate actions: pending appointments are highlighted, and plan limits trigger prominent warnings. The sidebar uses iconography and color-coded plan badges (e.g., violet for Pro) for quick recognition. Mobile responsiveness is achieved with a drawer pattern, avoiding clutter on small screens. Server-side data fetching ensures security and performance, while client-side interactivity (e.g., notifications) is isolated to necessary components.

## Potential improvements

1. **Hardcoded limits in `PLANS`**: The `PLANS` object (referenced in `page.tsx`) defines `limits` for `clients` and `appointmentsPerMonth`, which are used to enforce plan limits.

2. **Redundant queries**: `totalClients` and `monthlyAppointments` are fetched separately but could be combined into a single query using conditional aggregation, reducing database round-trips.

3. **Overfetching in `recurringSchedules`**: The `recurring_schedules` table has a `frequency` column but no index on `professionalId` and `active`, which could slow down queries as data grows. Consider adding a composite index for common filter patterns.
