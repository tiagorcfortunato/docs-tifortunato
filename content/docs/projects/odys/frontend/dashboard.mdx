# Professional Dashboard

The professional dashboard serves as the central control panel for users managing their practice within Odys. It provides a comprehensive overview of daily operations, upcoming events, client interactions, and business performance. This page delves into the architecture and implementation of the dashboard, covering its layout, interactive sidebar, and the main content area that dynamically displays critical information and actions.

## Overview

The dashboard is structured as a Next.js application, leveraging server components for efficient data fetching and client components for interactive elements. The core structure is defined by `src/app/dashboard/layout.tsx`, which handles user authentication and fetches the primary professional data. This layout then renders `src/app/dashboard/sidebar.tsx` for navigation and the main content, which for the root dashboard view is `src/app/dashboard/page.tsx`.

The system interacts with several database tables to populate the dashboard:
*   `professionals`: Stores the core profile information for the logged-in user.
*   `appointments`: Manages all scheduled sessions, including status and payment details.
*   `clients`: Contains information about the professional's clients.
*   `reviews`: Holds client feedback and ratings for the professional.
*   `availability`: Defines the professional's working hours.

The dashboard also integrates with external services like Stripe for plan management and PostHog for analytics, ensuring a holistic view of the professional's business.

## Dashboard Layout (`src/app/dashboard/layout.tsx`)

The `DashboardLayout` component acts as the entry point for all routes under `/dashboard`. As an `async` server component, its primary responsibilities include:

1.  **Authentication**: It uses `createClient()` from `@/lib/supabase/server` to interact with Supabase for server-side authentication. The `supabase.auth.getUser()` call verifies the user's session. If no user is found, it immediately redirects to `/login`, ensuring that only authenticated users can access the dashboard.
2.  **Professional Profile Retrieval**: Once authenticated, the layout queries the `professionals` table using Drizzle ORM to fetch the current user's professional profile. This is a critical step, as most dashboard functionalities depend on the professional's `id` and other profile details. If a professional profile does not exist for the authenticated user, it redirects them to `/onboarding`, guiding new users through the initial setup process.
3.  **Global Components**: It renders the `PostHogIdentify` component, which is crucial for tracking user activity and associating it with the correct user ID in PostHog. This component is placed at the layout level to ensure it's active across all dashboard sub-routes.
4.  **Structure**: The layout establishes the two-column structure of the dashboard: a fixed `Sidebar` on the left and a scrollable main content area (`children`) on the right. The `Sidebar` receives essential professional data (`professionalName`, `professionalSlug`, `plan`, `trialEndsAt`) as props, allowing it to display user-specific information and adapt navigation based on the professional's subscription plan.

## Dashboard Sidebar (`src/app/dashboard/sidebar.tsx`)

The `Sidebar` component provides the primary navigation for the professional dashboard. It is a client component, indicated by the `"use client"` directive, which allows it to manage its own state (`mobileOpen`) and interact with client-side hooks like `usePathname` and `useRouter`.

The sidebar displays:
*   **Branding and Utilities**: The `OdysLogo`, `NotificationBell`, and `ThemeToggle` are integrated at the top, offering quick access to branding, notifications, and theme switching.
*   **Trial Information**: If the professional is on a trial, a prominent banner displays the `daysLeft` using the `trialDaysLeft` helper from `@/lib/plan-guard`, encouraging conversion to a paid plan.
*   **Navigation Links**: The `NAV` array defines the dashboard's main navigation items, each with an `href`, `label`, and an SVG `icon`. Crucially, some links are marked `proOnly`, meaning they are locked for users on the "free" plan. The `isActive` function dynamically highlights the current route.
*   **Plan-gated Features**: For `proOnly` features, the sidebar displays a lock icon and redirects to the `/dashboard/plans` page if a free user attempts to access them, clearly communicating the value proposition of paid plans.
*   **Public Profile and Client Mode**: Links to the professional's public booking page (`/p/[slug]`) and a "Client Mode" (`/c`) are provided, offering quick access to external views.
*   **Professional Profile Summary**: At the bottom, it shows the professional's initials, full name, and their current plan (e.g., "Free", "Basic", "Pro", "Premium"), dynamically colored based on the PLAN_COLOR mapping.
*   **Logout Functionality**: A dedicated "Sair" (Logout) button uses `createClient()` from `@/lib/supabase/client` to sign out the user and redirect them to the `/login` page.
*   **Responsiveness**: The sidebar implements a responsive design, appearing as a fixed sidebar on desktop (`md:flex`) and a collapsible drawer on mobile (`md:hidden`), controlled by the `mobileOpen` state.

## Dashboard Page (`src/app/dashboard/page.tsx`)

The `DashboardPage` component is the main content area for the `/dashboard` route, providing a comprehensive overview for the professional. Like the layout, it is an `async` server component, enabling direct database access for data fetching.

The page performs extensive data retrieval and processing:
*   **Authentication and Redirection**: It re-authenticates the user and fetches the `professional` profile, similar to the layout. This ensures that the page has the most up-to-date professional data and handles redirects if the user is unauthenticated or lacks a professional profile.
*   **Appointment Data**: It fetches `todayAppointments` and `upcomingAppointments` by joining the `appointments` and `clients` tables. These queries filter appointments based on the `professionalId` and date ranges (`gte`, `lte`, `gt` from `drizzle-orm`), ordering them by `startsAt`.
*   **Aggregate Statistics**: Using `Promise.all` for concurrent execution, it fetches several key metrics:
    *   `totalClients`: The total number of clients associated with the professional.
    *   `monthlyAppointments`: The count of appointments scheduled for the current month.
    *   `reviewStats`: The average rating and total count of reviews from the `reviews` table.
    *   `recentReviews`: The three most recent reviews, including client names, by joining `reviews` with `clients`.
    *   `availabilityCount`: The number of availability slots defined by the professional.
*   **Derived State and Business Logic**:
    *   `trialDaysLeft` and `effectivePlan` from `@/lib/plan-guard` are used to determine the professional's current plan status and remaining trial days.
    *   `findProfileGaps` from `@/lib/profile-completeness` identifies missing profile information, prompting the professional to complete their setup.
    *   It calculates `pendingCount` for appointments awaiting confirmation and revenue from today's appointments with captured payments.
*   **Conditional UI Elements**: The page dynamically renders various banners and cards based on the fetched data and derived state:
    *   A `WelcomeCard` is shown for new professionals with zero clients, guiding them to share their public booking link.
    *   Banners alert the professional about `plan.limits` being `isNearClientLimit`, `isAtClientLimit`, `isNearMonthlyLimit`, or `isAtMonthlyLimit`, prompting them to upgrade.
    *   A `TrialBanner` or a "Trial Pro ativo" banner is displayed based on the `trialEndsAt` status.
    *   A "Profile completeness" banner appears if `profileGaps` exist, encouraging the professional to update their settings.
*   **Dashboard Sections**:
    *   A personalized greeting.
    *   A "Stats" section displaying today's appointments, pending confirmations, and total clients.
    *   A "Reputation" section (only if reviewCount > 0) showing the avgRating and recentReviews, with recent comments displayed only if available.
    *   "Agenda de hoje" (Today's appointments) and "Próximos agendamentos" (Upcoming appointments) lists, each appointment displaying its status using `STATUS_LABEL` and offering `AppointmentActions` for pending or confirmed appointments.
    *   A "Receita de hoje" (Today's revenue) banner if there's any captured revenue.
*   **Helper Functions**: `getToday`, `formatTime`, and `formatCurrency` are utility functions to format dates, times, and monetary values for display.

## Design decisions

The dashboard's architecture reflects several deliberate design choices aimed at performance, maintainability, and user experience:

*   **Server Components for Data Fetching**: Both `src/app/dashboard/layout.tsx` and `src/app/dashboard/page.tsx` are implemented as `async` server components. This allows direct, secure database access using Drizzle ORM (`db`) without exposing database credentials to the client. Fetching data on the server reduces client-side JavaScript bundle size, improves initial page load performance, and simplifies data management by keeping data fetching logic co-located with the components that render it.
*   **Client Component for Interactivity**: `src/app/dashboard/sidebar.tsx` is explicitly marked as a client component. This choice is necessary because it utilizes client-side hooks like `useState` for managing mobile menu visibility, `usePathname` for active link highlighting, and `useRouter` for navigation after logout. It also integrates client-side Supabase for the `handleLogout` function. This separation ensures that only components requiring client-side interactivity are hydrated, minimizing the client-side footprint.
*   **Drizzle ORM for Database Interactions**: The application uses Drizzle ORM for all database operations. This provides a type-safe and expressive way to construct SQL queries, as seen in the complex `select` statements with `innerJoin`, `eq`, `gte`, `lte`, `gt`, `count`, and `sql` aggregations in `src/app/dashboard/page.tsx`. Drizzle's schema definition (`@/lib/db/schema`) ensures consistency and helps prevent common SQL injection vulnerabilities.
*   **Centralized Plan and Trial Logic**: The `effectivePlan` and `trialDaysLeft` functions are abstracted into `@/lib/plan-guard`. This centralizes business logic related to subscription plans and trial periods, ensuring consistent application across different parts of the dashboard (e.g., `page.tsx` for banners, `sidebar.tsx` for navigation locks). This approach reduces duplication and makes it easier to modify plan-related rules in the future.
*   **Modular UI Components**: Key UI elements like `AppointmentActions`, `TrialBanner`, `WelcomeCard`, `ThemeToggle`, `NotificationBell`, and `OdysLogo` are extracted into separate components. This promotes reusability, improves code organization, and makes the dashboard easier to understand and maintain. For instance, `AppointmentActions` encapsulates the logic for confirming or rejecting appointments, keeping `page.tsx` focused on data display.
*   **Conditional Rendering for User Guidance**: The dashboard makes extensive use of conditional rendering to provide context-sensitive information and guidance. Examples include the `WelcomeCard` for new users, various banners for plan limits or trial status, and profile completeness nudges. This design choice aims to reduce cognitive load by only showing relevant information at the appropriate time, improving the user's onboarding and ongoing experience.

## Potential improvements

1.  **Optimize Professional Data Fetching**: The `professional` data is fetched twice: once in `src/app/dashboard/layout.tsx` and again in `src/app/dashboard/page.tsx`. While both are server components, this is redundant. The `professional` object fetched in the layout could be passed down as a prop to its `children` (the `DashboardPage`), eliminating the need for the second database query. This would reduce database load and improve the overall efficiency of the dashboard's initial render.
2.  **Refine Plan Limit Queries**: In `src/app/dashboard/page.tsx`, the queries for `totalClients` and `monthlyAppointments` are executed unconditionally. However, if the professional's current `plan.limits.clients` or `plan.limits.appointmentsPerMonth` are `Infinity` (as defined in `PLANS`), these counts are not strictly necessary for displaying limit warnings. The queries could be conditionally executed only if the respective plan limit is finite, further optimizing database calls for users on higher-tier plans.
3.  **Internationalization of UI Strings**: The `STATUS_LABEL`, `PLAN_LABEL`, and `PLAN_COLOR` objects in `src/app/dashboard/page.tsx` and `src/app/dashboard/sidebar.tsx` contain hardcoded Portuguese strings. For a global application, these should be moved to an internationalization (i18n) system. This would allow the dashboard to be easily translated into multiple languages without modifying the core component logic, improving accessibility for a broader user base.
