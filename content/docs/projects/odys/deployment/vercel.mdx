# Vercel Deployment

Deploying `odys` on Vercel ensures a seamless, scalable production experience with built-in automation for cron jobs, security, and observability. This page covers how the application is configured for Vercel, including deployment triggers, environment management, and regional considerations.

## Overview

The `odys` application leverages Vercel’s full suite of deployment capabilities. With 23 API routes, 10 database tables, and 4 MCP tools, the system handles booking, messaging, payments via Stripe, WhatsApp integration, and AI chat. Two cron jobs—`/api/cron/reminders` and `/api/cron/whatsapp-watchdog`—are scheduled daily at 8 AM and 9 AM UTC respectively, ensuring timely appointment reminders and system health checks. The app uses Sentry for monitoring, with automatic Vercel cron instrumentation enabled, and enforces strict security headers across all responses.

## Auto-Deploy on Main Branch

Vercel is configured to automatically deploy every push to the `main` branch. This ensures that changes are rapidly reflected in production, supporting a continuous delivery workflow. While not explicitly defined in `vercel.json`, this behavior is standard for Vercel projects linked to GitHub with default settings. The presence of cron definitions confirms that Vercel is actively managing background tasks, implying full integration with the Git workflow.

## Environment Variables and Secrets

The application relies on several environment variables, particularly for Stripe integration. The pricing tiers—`basic`, `pro`, and `premium`—reference environment variables `STRIPE_BASIC_PRICE_ID`, `STRIPE_PRO_PRICE_ID`, and `STRIPE_PREMIUM_PRICE_ID`, which must be set in Vercel’s environment configuration (as referenced in `src/config/plans.ts`). Additionally, Stripe webhook events (`checkout.session.completed`, `customer.subscription.updated`, etc.) require the `STRIPE_WEBHOOK_SECRET` to validate payloads (as referenced in `src/api/webhooks/stripe.ts`). These secrets are not exposed in code and must be securely injected at deploy time.

## Regions and Performance

While the current configuration does not specify deployment regions in `vercel.json`, Vercel defaults to global CDN distribution with serverless functions deployed in multiple regions. Given the use of `Strict-Transport-Security` with `max-age=63072000; includeSubDomains; preload`, the app is optimized for secure, fast global access. The `remotePatterns` in `next.config.ts` allow image loading from Supabase and Google, ensuring media assets are efficiently served worldwide.

## Design decisions

The `securityHeaders` in `next.config.ts` prioritize defense-in-depth: `X-Frame-Options: DENY` prevents clickjacking, `X-Content-Type-Options: nosniff` blocks MIME sniffing, and `Permissions-Policy` restricts unused browser APIs, with an exception for the `payment` API for Stripe. The `tunnelRoute: "/monitoring"` in Sentry configuration routes error reports through a Next.js rewrite to bypass ad blockers. `automaticVercelMonitors: true` enables automatic tracking of cron jobs, despite the known limitation with App Router route handlers. Additional headers include `Referrer-Policy: strict-origin-when-cross-origin` to enhance privacy by limiting referrer data on cross-origin requests, and `X-DNS-Prefetch-Control: on` to opt out of FLoC / Topics and interest-based tracking cohorts. These choices reflect a balance between security, observability, and compatibility.

## Potential improvements

1. **Add region constraint in `vercel.json`** — To reduce latency for a geographically focused user base, consider pinning deployments to a specific region (e.g., `us-east-1`) using the `regions` field, currently absent.

2. **Move cron jobs to App Router-compatible setup** — The Sentry config notes that `automaticVercelMonitors` does not yet work with App Router route handlers. Migrating `/api/cron/*` endpoints to App Router with manual monitor wrapping would improve observability.

3. **Document required environment variables** — The `plans` object in the manifest shows three Stripe price ID env vars, but there’s no `.env.example` or Vercel dashboard documentation. Adding a `vercel.json` `env` section or runtime checks would prevent misconfiguration.
