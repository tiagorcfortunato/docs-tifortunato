# Deployment

Understanding how Odys is deployed reveals the operational backbone of a modern, WhatsApp-first SaaS platform. This page covers the infrastructure layout, automation pipelines, and key architectural choices that enable reliable, secure, and scalable delivery — from code commit to production.

## Overview

Odys is split across two primary hosting environments: the **Next.js application** runs on **Vercel**, while the **WhatsApp messaging layer (Evolution API)** is hosted separately on **Railway**. This separation ensures that high-availability messaging remains resilient even if the main app experiences downtime.

The Vercel deployment is fully automated via GitHub Actions. Every push to `main` triggers a CI pipeline that performs type-checking, linting, and a production build. Environment variables — including Supabase, Stripe, Redis, and Evolution API credentials — are injected during the build phase, ensuring secure configuration without hardcoding secrets.

Two **cron jobs** are defined in `vercel.json` and executed daily:
- `GET /api/cron/reminders` at 8 AM: sends 24-hour and 1-hour appointment reminders, skipping clients actively engaged in AI conversations.
- `GET /api/cron/whatsapp-watchdog` at 9 AM: verifies the health of the Evolution API instance.

These routes are protected by `CRON_SECRET`, ensuring only authorized triggers can invoke them. The cron system leverages Vercel’s built-in scheduling, eliminating the need for external tools.

Security is enforced at the edge via `next.config.ts`, which applies strict HTTP headers: `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-DNS-Prefetch-Control: on`, and `Strict-Transport-Security` with a two-year max age. Permissions are tightly scoped, disabling camera, microphone, and geolocation access except for Stripe.js.

## Design decisions

The deployment strategy prioritizes **isolation**, **observability**, and **developer velocity**. By hosting WhatsApp messaging separately on Railway, the platform avoids tying mission-critical communication to the app’s uptime. Evolution API runs in a Docker container with its own Supabase project (flag: specific project name `odys-evolution` not verifiable from provided code), minimizing blast radius.

Vercel was chosen not only for its seamless CI/CD integration but also for its support of **App Router**, **serverless functions**, and **cron monikers**. The `tunnelRoute` in Sentry configuration routes error reports through `/monitoring`, bypassing ad blockers — a pragmatic trade-off between reliability and slight server cost increase.

The use of **Drizzle ORM** enables zero-downtime schema pushes via `npm run db:push`, critical for iterative development. However, this approach skips formal migration files, relying instead on direct schema synchronization — acceptable in early-stage SaaS but risky at scale.

## Potential improvements

1. **Add formal migration pipeline**: Currently, schema changes use `drizzle-kit push` directly, risking accidental data loss. Introduce `drizzle-kit generate` to create versioned migrations in `drizzle/`, ensuring auditability and rollback capability.

2. **Strengthen rate limiting context**: The `ratelimit.ts` module uses Upstash Redis. (Flag: The claim that it does not differentiate between client IP and authenticated user ID could not be verified as `src/lib/ratelimit.ts` was not provided for audit.) For endpoints like `/api/booking`, this could allow circumvention via multiple accounts. Enforce per-phone-number limits using `req.body.phone` in booking routes.

3. **Improve cron observability**: While Vercel cron is convenient, it lacks detailed monitoring. Although `automaticVercelMonitors` is enabled in `next.config.ts:90`, it currently does not support App Router route handlers. Therefore, consider manually wrapping `/api/cron/reminders` with `withSentryConfig` to capture execution metrics and errors, especially since it handles time-sensitive reminders.
