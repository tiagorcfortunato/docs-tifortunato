# Deployment

Understanding how Odys is deployed is crucial for grasping its operational resilience and the choices made to ensure its availability and performance. This document walks through the infrastructure and processes that bring Odys to life, from continuous integration to hosting the application and its critical WhatsApp integration.

## Overview

Odys leverages a modern, cloud-native deployment strategy, segmenting its core application from its specialized WhatsApp integration. The main application, built with Next.js, finds its home on Vercel, benefiting from its tight integration with the framework and serverless capabilities. The dedicated Evolution API, which powers Odys's WhatsApp-first approach, runs as a containerized service on Railway, providing the necessary environment for its long-running processes. Orchestrating this entire process is a robust CI/CD pipeline powered by GitHub Actions, ensuring code quality and automated deployments. Additionally, critical background tasks, such as sending reminders and monitoring the WhatsApp API, are managed through Vercel's cron job capabilities.

## Application Deployment on Vercel

The primary Odys application, encompassing both the user-facing interface and its 21 API routes, is deployed on Vercel. This choice was a natural fit given Odys's foundation on Next.js 16 (App Router). Vercel provides a streamlined deployment experience, automatically building and deploying the application upon every push to the `main` branch, as configured in the `.github/workflows/ci.yml` workflow.

Vercel's serverless functions handle all API routes, such as `/api/booking` for managing appointments or `/api/stripe/webhook` for processing payment events. This architecture allows for efficient scaling of individual endpoints based on demand.

Error monitoring is integrated directly into the deployment process through Sentry. The `next.config.ts` file, specifically via `withSentryConfig`, ensures that client-side and server-side errors are tracked. It's configured to upload source maps and route browser requests through a `tunnelRoute` at `/monitoring` to circumvent potential ad-blockers, a common practice to ensure comprehensive error reporting. Furthermore, Sentry's `automaticVercelMonitors` feature is enabled within the webpack configuration, providing automatic instrumentation for Vercel Cron Monitors, which is vital for the reliability of scheduled tasks.

## Evolution API Deployment on Railway

The heart of Odys's WhatsApp integration, the Evolution API, operates independently on Railway. This separation is a deliberate architectural choice. While the main application thrives in Vercel's serverless environment, the Evolution API, being a self-hosted WhatsApp Web shim, often requires a more persistent, container-based hosting solution. Railway provides this by running the `tiagorcfortunato/evolution-api-odys` Docker image.

This Evolution API instance connects to its own dedicated Supabase project, `odys-evolution`, utilizing a separate `evolution_api` schema. This isolation ensures that the WhatsApp integration's operational data does not intermingle with the core Odys application data, providing clearer separation of concerns and potentially different scaling or backup strategies. The connection to this database is made via PgBouncer, typically on port `6543`, which helps manage database connections efficiently.

## Continuous Integration and Deployment (CI/CD)

Odys employs GitHub Actions for its CI/CD pipeline, defined in `.github/workflows/ci.yml`. This workflow is triggered on every `push` to the `main` branch and for all `pull_request` events targeting `main`.

The pipeline consists of two main jobs:

1.  **`check`**: This job focuses on code quality and correctness. It first checks out the code, sets up Node.js 20, and installs dependencies. Then, it performs a TypeScript type-check using `npx tsc --noEmit` to catch any type-related errors early in the development cycle. Following this, `npm run lint` executes ESLint to enforce coding standards and identify potential issues.
2.  **`build`**: Once the `check` job passes, the `build` job proceeds to create a production-ready build of the application using `npm run build`. This step is crucial for verifying that the application can be successfully compiled for deployment. All necessary environment variables, such as `DATABASE_URL`, `STRIPE_SECRET_KEY`, and `EVOLUTION_API_KEY`, are securely injected into the build process from GitHub Secrets, ensuring that sensitive credentials are never exposed in the repository.

This CI/CD setup ensures that only type-checked, linted, and successfully built code reaches the deployment stage, maintaining a high standard of code quality and reducing the risk of introducing regressions into production.

## Automated Cron Jobs

Odys relies on scheduled tasks for critical background operations, configured directly within `vercel.json` and executed by Vercel's cron job infrastructure.

Two primary cron jobs are defined:

1.  **`GET /api/cron/reminders`**: This endpoint runs daily at `0 8 * * *` (8 AM UTC). Its purpose is to send out automated 24-hour and 1-hour reminders for upcoming appointments. It also handles sending trial expiration emails to professionals. A clever design choice here, as noted in the `README.md`, is that it skips sending reminders if a client is currently engaged in a conversation with the AI agent, preventing message overload and ensuring a smoother user experience.
2.  **`GET /api/cron/whatsapp-watchdog`**: Scheduled daily at `0 9 * * *` (9 AM UTC), this cron job performs a health check on the Evolution API. This proactive monitoring helps ensure that the critical WhatsApp integration remains operational, allowing for early detection and resolution of any issues.

These cron jobs are essential for the automated communication features of Odys, significantly reducing the manual overhead for professionals and improving client engagement.

## Design Decisions

The deployment architecture of Odys reflects several deliberate design choices:

*   **Vercel for Next.js App**: The decision to host the main Odys application on Vercel is primarily driven by its deep integration with Next.js. This provides automatic serverless function deployment for API routes, optimized asset delivery, and a seamless developer experience, allowing Tiago to focus on application logic rather than infrastructure. The `next.config.ts` Sentry integration further solidifies this by providing robust error monitoring out-of-the-box for the Vercel environment.
*   **Railway for Evolution API**: Deploying the Evolution API on Railway, separate from the main application, addresses the specific needs of this component. The Evolution API, being a wrapper around WhatsApp Web, often requires a long-running process or a containerized environment that might not fit perfectly within Vercel's serverless function model, especially for maintaining persistent WhatsApp sessions. Railway offers the flexibility and control needed for such a service, including Docker image deployment and dedicated database connections.
*   **GitHub Actions for CI/CD**: Utilizing GitHub Actions for CI/CD is a standard and effective choice. It integrates directly with the source code repository, providing immediate feedback on code quality (type-checking, linting) and build success. This ensures that the `main` branch always contains deployable code, minimizing deployment risks.
*   **Separate Database for Evolution API**: The choice to use a separate Supabase project (`odys-evolution`) for the Evolution API's database, distinct from the main Odys database, promotes isolation. This means that the operational data of the WhatsApp integration (e.g., message queues, session states) is kept separate from core application data (e.g., `professionals`, `appointments`). This separation can simplify backups, scaling, and security considerations for each component.

## Potential Improvements

While the current deployment strategy is well-structured and functional, here are a few concrete areas for improvement that could enhance robustness, maintainability, and operational visibility:

1.  **Automated Database Migrations**: Currently, the `README.md` mentions `npm run db:push` for schema updates. While convenient for development, in a production environment, a robust migration system is critical. Implementing Drizzle Kit's migration capabilities would allow for version-controlled, incremental database schema changes, ensuring data integrity and providing a clear rollback path. This would involve generating migration files from `src/lib/db/schema.ts` and applying them as part of the deployment pipeline, rather than a direct push.
2.  **Comprehensive Service Health Checks**: The `GET /api/cron/whatsapp-watchdog` is a great start for monitoring the Evolution API. However, expanding this concept to other critical external services like Supabase, Stripe, and Resend would provide a more holistic view of Odys's operational health. A new `/api/health` endpoint could be introduced, perhaps leveraging helpers from `src/lib/api.ts`, to check the connectivity and basic functionality of all integrated services. This would allow for quicker diagnosis of issues beyond just the WhatsApp integration.
3.  **Staging Environment for Pre-Production Testing**: The current `ci.yml` deploys directly to Vercel on `main` branch pushes. Introducing a staging environment, perhaps by deploying a separate Vercel project from a `develop` or `staging` branch, would allow for thorough testing of new features and changes in a production-like setting before they impact live users. This would reduce the risk of introducing bugs into the production environment and provide a safe space for quality assurance.
