# Cron Jobs

This document explains the cron infrastructure powering time-based automation in the odys application. These jobs run in production via Vercel’s cron system and are secured to prevent unauthorized invocation. They handle critical background tasks suchs as appointment reminders and system health checks.

## Overview

The application defines **two cron jobs** in `vercel.json`, both routed through the Next.js API under `/api/cron`. These are:

- `GET /api/cron/reminders` — runs daily at 8:00 AM (`0 8 * * *`) to send 24-hour and 1-hour WhatsApp reminders for confirmed appointments, and email alerts for professionals nearing trial expiration.
- `GET /api/cron/whatsapp-watchdog` — runs daily at 9:00 AM (`0 9 * * *`). This endpoint is currently a placeholder; its implementation for monitoring WhatsApp message delivery and re-queueing failed messages is not yet present.

Both endpoints are protected by a shared secret mechanism, ensuring only Vercel’s cron or authorized internal calls can trigger them. The system uses header-based authentication, accepting either `x-cron-secret` or `Authorization: Bearer <secret>`.

There are 21 API routes in total, with these two cron endpoints being the only ones restricted to `GET` and secured via the `CRON_SECRET` environment variable.

## Reminder System: `/api/cron/reminders`

The reminders job performs three key operations:

1.  **24-hour reminders**: Queries for confirmed appointments starting within the next 22–26 hours (using `REMINDER_24H_MIN_MS` and `REMINDER_24H_MAX_MS` from `src/lib/constants.ts`), checks feature eligibility via `canUseFeature`, and skips clients currently in an AI conversation (`isClientActive`). Messages are sent via `sendWhatsApp` using the `msgReminder24h` template, with appointment date and time formatted using `formatDate` and `formatTime` (e.g., `sexta-feira, 1 de janeiro` and `10:00`).

2.  **1-hour reminders**: Similar logic, but for appointments within the next 45–75 minutes (using `REMINDER_1H_MIN_MS` and `REMINDER_1H_MAX_MS` from `src/lib/constants.ts`), using `msgReminder1h`.

3.  **Trial expiration emails**: Scans for professionals with active trials and sends email warnings at 3 and 1 day(s) before expiry using `sendTrialExpiringEmail`.

Sent reminders update the `reminderSent24h` and `reminderSent1h` flags in the `appointments` table to prevent duplicates.
The endpoint returns a JSON object with an `ok: true` status, detailing the number of 24-hour reminders sent (`reminders24h`) and deferred (`deferred24h`), 1-hour reminders sent (`reminders1h`) and deferred (`deferred1h`), and trial expiration emails sent (`trialRemindersSent`).

## Design decisions

The dual-header authentication (`x-cron-secret` and `Bearer`) supports both manual testing and Vercel’s cron, which uses `Authorization`. This avoids coupling to a single provider’s format. The use of time windows (e.g., ±1 hour around 24h) accommodates cron scheduling imprecision and ensures no appointment is missed due to timing drift.

The `isClientActive` check before sending messages prevents interrupting ongoing conversations, improving UX. Deferring rather than failing maintains delivery reliability.

Separating concerns into distinct cron jobs (reminders vs. watchdog) allows independent scheduling and failure isolation.

## Potential improvements

1.  The watchdog endpoint exists in `vercel.json` but its implementation is missing, as `src/app/api/cron/whatsapp-watchdog` is currently an empty directory. It should log its execution and define clear recovery actions, such as retrying failed WhatsApp messages tracked in a `failedMessages` table.

2.  **Avoid repeated DB queries in loops** — In `/api/cron/reminders`, `isClientActive` is called in a loop. For high-volume systems, batching client status checks via `inArray(clients.phone, [...])` would reduce round-trips.

3.  **Centralize cron configuration** — The cron schedules are defined in `vercel.json` but not type-checked or reusable in code. Moving them to a shared config (e.g., `cron.config.ts`) would improve maintainability and enable testing.
