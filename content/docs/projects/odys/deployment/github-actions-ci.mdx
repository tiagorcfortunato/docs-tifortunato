# GitHub Actions CI

The continuous integration (CI) pipeline defined in `.github/workflows/ci.yml` is a foundational element of the Odys development process. It ensures that every code change, whether a direct push to `main` or a pull request, adheres to established quality standards before it can be integrated or deployed. This workflow acts as an automated gatekeeper, performing essential checks like type validation, code style enforcement, and build verification, which are critical for maintaining a stable and reliable codebase.

## Overview

The CI workflow is structured into two distinct jobs: `check` and `build`. This separation allows for a clear progression, where foundational code quality and correctness are verified first, followed by the successful compilation of the application.

The workflow is triggered automatically on `push` events to the `main` branch and for all `pull_request` events targeting the `main` branch. This ensures that all proposed changes undergo scrutiny, preventing regressions and maintaining code health.

### The `check` Job

The `check` job is designed to rapidly identify common development errors and enforce code quality standards. It runs on an `ubuntu-latest` runner, providing a consistent environment for execution.

1.  **Checkout Code**: The `actions/checkout@v4` action fetches the repository's code, making it available for subsequent steps.
2.  **Node.js Setup**: The `actions/setup-node@v4` action configures Node.js version 20. It also leverages `cache: npm` to cache `node_modules`, significantly speeding up dependency installation in subsequent runs by reusing previously downloaded packages.
3.  **Install Dependencies**: `npm ci` is executed to perform a clean installation of project dependencies, ensuring that the exact versions specified in `package-lock.json` are used, which is crucial for reproducible builds.
4.  **Typecheck**: `npx tsc --noEmit` runs the TypeScript compiler to perform a type check across the entire codebase. This step is vital for catching type-related errors early, ensuring the strong typing benefits of TypeScript are fully realized.
5.  **Lint**: `npm run lint` executes the project's linting configuration. This step enforces code style, identifies potential bugs, and ensures consistency, which is particularly important for a project with a diverse set of API routes and database interactions.
6.  **Unit Tests**: `npm test` runs the project's unit tests. This step verifies the correctness of individual components and functions, providing confidence that changes haven't introduced regressions.

### The `build` Job

The `build` job is responsible for compiling the application into its deployable artifacts. It is configured with `needs: check`, meaning it will only execute if the `check` job completes successfully. This dependency ensures that no resources are wasted attempting to build code that has failed basic quality or correctness checks.

Like the `check` job, it runs on `ubuntu-latest` and includes steps for checking out the code, setting up Node.js 20 with npm caching, and installing dependencies via `npm ci`.

Crucially, the `build` job requires a comprehensive set of environment variables, which are securely injected from GitHub secrets. These include sensitive keys and URLs for various services that Odys integrates with, such as `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `DATABASE_URL`, `STRIPE_SECRET_KEY`, `STRIPE_BASIC_PRICE_ID`, `STRIPE_WEBHOOK_SECRET`, `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY`, `EVOLUTION_API_URL`, `EVOLUTION_API_KEY`, `EVOLUTION_INSTANCE`, `CRON_SECRET`, `RESEND_API_KEY`, `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN`, and `SENTRY_AUTH_TOKEN`. The build process needs access to these configurations to correctly compile the application.

Finally, `npm run build` is executed to compile the application. This step generates the optimized production-ready assets.

## Design Decisions

The design of this CI pipeline reflects a deliberate choice to prioritize early feedback and maintain a high standard of code quality.

1.  **Separation of Concerns (`check` vs. `build`)**: By splitting the workflow into `check` and `build` jobs, the pipeline provides faster feedback on code quality issues. Developers don't have to wait for a full build to complete to know if their changes introduced a type error or linting violation. The `needs: check` dependency ensures that the more resource-intensive build process only runs on code that has already passed initial quality gates.
2.  **Comprehensive Pre-Build Checks**: The `check` job includes typechecking, linting, and unit tests. This combination covers a broad spectrum of potential issues, from static type errors to runtime logic bugs, before the code is even considered for deployment.
3.  **Consistent Environment**: Using `ubuntu-latest` and explicitly setting `node-version: 20` ensures that the CI environment is consistent across all runs, minimizing "works on my machine" issues.
4.  **Dependency Caching**: The `cache: npm` configuration for `actions/setup-node` is a pragmatic choice to optimize build times. While `npm ci` ensures exact dependency versions, caching prevents repeated downloads of the same packages, making the CI process more efficient.
5.  **Secure Environment Variables**: Relying on GitHub secrets for all sensitive environment variables in the `build` job is a standard security practice. It prevents hardcoding credentials and ensures that secrets are not exposed in the repository.

## Potential Improvements

While the current CI pipeline is robust, several opportunities exist to enhance its capabilities and efficiency:

1.  **Code Formatting Enforcement**: The `check` job currently includes linting, but it does not explicitly enforce code formatting. Adding a step like `npm run format:check` (assuming a corresponding script is added to `package.json`) after linting in the `check` job would ensure consistent code style across the project. This helps prevent merge conflicts related to formatting and improves readability, especially in a project with multiple contributors.
2.  **Code Coverage Reporting**: The `check` job runs unit tests, but it doesn't report on code coverage. Integrating a step to generate and upload code coverage reports (e.g., using a tool like Codecov or Coveralls) after `npm test` would provide valuable insights into the thoroughness of the test suite.
3.  **Dependency Vulnerability Scanning**: The pipeline installs dependencies but does not include a step to scan for known security vulnerabilities within those dependencies. Incorporating a step using `npm audit` or a dedicated vulnerability scanner (e.g., Snyk, Dependabot) in the `check` job would proactively identify and flag security risks, which is essential for the long-term security posture of the `odys` application.
