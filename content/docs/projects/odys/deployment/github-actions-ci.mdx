# CI Pipeline

The Continuous Integration (CI) pipeline for the `odys` project, defined in `.github/workflows/ci.yml`, serves as the initial gatekeeper for code quality and build integrity. Its primary purpose is to automatically validate changes pushed to the `main` branch or submitted as pull requests, ensuring that the codebase remains healthy, type-safe, and capable of being built successfully. This automated process is fundamental to maintaining a stable development environment and catching potential issues early, before they can impact deployment or other developers.

## Overview

The CI workflow is structured around two distinct, yet interdependent, jobs: `check` and `build`. This separation allows for a logical progression of validation steps, where static analysis precedes the more resource-intensive build process. The workflow is triggered automatically on every `push` to the `main` branch and for every `pull_request` targeting `main`, ensuring comprehensive coverage for all significant code changes.

The `check` job focuses on static code analysis, specifically typechecking and linting. Upon successful completion of these checks, the `build` job is initiated, which then attempts to compile the entire application. This dependency, explicitly declared with `needs: check`, ensures that no build resources are expended on code that already fails basic quality and correctness checks.

## The `check` Job: Typecheck & Lint

The `check` job is designed to provide rapid feedback on code quality and correctness without needing to fully compile the application. It runs on an `ubuntu-latest` environment, leveraging standard tooling to analyze the codebase.

The first step involves checking out the repository's code using `actions/checkout@v4`. Following this, a Node.js environment is set up using `actions/setup-node@v4`, specifically targeting `node-version: 20`. This choice aligns with common long-term support (LTS) versions and ensures compatibility with the project's dependencies, including `next` version `16.2.4` and `react` version `19.2.4`. The `cache: npm` option is employed here to speed up dependency installation by caching `node_modules`, a common optimization in CI environments.

After the environment is prepared, `npm ci` is executed. This command is preferred over `npm install` in CI contexts because it guarantees a clean and reproducible installation of dependencies based on the `package-lock.json` file, preventing discrepancies that might arise from differing dependency versions.

With dependencies in place, the job proceeds to its core validation steps:

-   **Typecheck**: The command `npx tsc --noEmit` is run. This is a critical step for a TypeScript project like `odys`. It invokes the TypeScript compiler to perform a full type analysis of the entire codebase, which includes 10 Drizzle tables and 21 API routes, without actually emitting any JavaScript files. This ensures that all types are correctly defined and used, catching potential bugs related to data structures and function signatures early in the development cycle.
-   **Lint**: Immediately after typechecking, `npm run lint` is executed. This step enforces coding standards and identifies stylistic issues or potential runtime problems that might not be caught by the type checker. It helps maintain code consistency and readability across the project.

## The `build` Job: Application Build

Once the `check` job successfully completes, the `build` job takes over. Its purpose is to verify that the application can be compiled into a deployable artifact. Like the `check` job, it also runs on `ubuntu-latest`.

A key aspect of the `build` job is the extensive set of environment variables it requires. The `odys` application relies on various external services and configurations, and these secrets are injected into the build environment from GitHub's secrets store. These include:

-   Supabase credentials (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`) for database interactions and authentication.
-   Stripe API keys (`STRIPE_SECRET_KEY`, `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY`, `STRIPE_WEBHOOK_SECRET`, `STRIPE_BASIC_PRICE_ID`) which are essential for handling payments and subscriptions, including the `basic` plan priced at 39.
-   Evolution API details (`EVOLUTION_API_URL`, `EVOLUTION_API_KEY`, `EVOLUTION_INSTANCE`) likely for WhatsApp integration, given the 19 defined WhatsApp message templates.
-   Other service keys like `DATABASE_URL`, `CRON_SECRET` (for the two cron jobs at `/api/cron/reminders` and `/api/cron/whatsapp-watchdog`), `RESEND_API_KEY` (for email services), `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN` (for rate limiting and caching, as evidenced by the 5 defined rate limits), and `SENTRY_AUTH_TOKEN` for error monitoring.

The steps within the `build` job mirror the initial setup of the `check` job: checking out the code, setting up Node.js 20 with `npm` caching, and running `npm ci` for reproducible dependency installation.

The final and most crucial step is `npm run build`. This command triggers the Next.js build process, compiling the entire application, including all client-side and server-side components, API routes, and static assets. A successful build here confirms that the application's code is syntactically correct, all modules can be resolved, and the final output is ready for deployment.

## Design Decisions

The current CI pipeline reflects several deliberate design choices aimed at efficiency, reliability, and maintainability:

-   **Separation of `check` and `build` Jobs**: By splitting static analysis (`check`) from the full application build (`build`), the pipeline provides faster feedback. If a developer introduces a type error or linting violation, the `check` job fails quickly, allowing for immediate correction without waiting for the potentially longer build process to complete. This optimizes developer iteration time.
-   **Explicit Job Dependency (`needs: check`)**: The `build` job's explicit dependency on `check` is a critical decision. It prevents the CI system from wasting computational resources on building an application that is already known to have static analysis issues. This ensures that only code that passes basic quality gates proceeds to the more resource-intensive build phase.
-   **Consistent Node.js Environment**: Specifying `node-version: 20` across both jobs ensures that the build and check processes run in a consistent environment, matching the expected runtime environment and preventing "works on my machine" scenarios due to differing Node.js versions.
-   **`npm ci` for Reproducibility**: The use of `npm ci` instead of `npm install` is a standard best practice for CI. It guarantees that the exact dependency versions specified in `package-lock.json` are installed, leading to more reliable and reproducible builds. This minimizes the risk of CI failures caused by unexpected dependency updates.
-   **Comprehensive Environment Variable Injection**: Passing a wide array of secrets to the `build` job is necessary because Next.js applications can incorporate environment variables at build time, especially for static generation or client-side bundles. While some of these might only be used at runtime, including them during the build ensures that the build process has access to all necessary configurations, preventing potential build failures related to missing environment-dependent values.

## Potential Improvements

While the current CI pipeline provides a solid foundation for code quality and build verification, several areas could be enhanced to further improve reliability, efficiency, and developer experience:

1.  **Add a Dedicated `test` Job**: The current pipeline focuses on static analysis and building but lacks an explicit step for running automated tests (unit, integration, or end-to-end). Introducing a new job, perhaps named `test`, that runs `npm test` (or a similar command) after the `check` job would significantly increase confidence in the functional correctness of the codebase. This would catch logical errors and regressions that typechecking and linting cannot detect.
2.  **Implement a `deploy` Job for Continuous Delivery**: For the `main` branch, after the `build` job successfully completes, an automated deployment step could be added. This would transform the CI pipeline into a Continuous Delivery (CD) pipeline, automatically deploying verified code to a staging or production environment. This could be a new job, `deploy`, that `needs: build` and is triggered only on `push` to `main`.
3.  **Refine Environment Variable Exposure**: The `build` job currently receives a large number of secrets, such as `STRIPE_WEBHOOK_SECRET`, `EVOLUTION_API_URL`, and `EVOLUTION_API_KEY`. Many of these are likely only required at runtime by the deployed application, not during the static build process itself. Reviewing which environment variables are truly necessary at build time versus runtime could allow for a more granular and potentially more secure approach to secret management, reducing the attack surface by exposing fewer secrets during the build phase.
