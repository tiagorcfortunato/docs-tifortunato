# GitHub Actions CI Pipeline

This page details the Continuous Integration (CI) pipeline implemented using GitHub Actions, covering the automated typechecking, linting, and build processes for the application.

The CI pipeline is defined in `.github/workflows/ci.yml` and is triggered on every `push` to the `main` branch and for every `pull_request` targeting `main`. It consists of two sequential jobs: `check` and `build`.

## Typecheck & Lint Job

The `check` job is responsible for validating the codebase's type integrity and adherence to linting rules. It runs on `ubuntu-latest` and performs the following steps:

1.  **Checkout Code**: Uses `actions/checkout@v4` to retrieve the repository content.
2.  **Setup Node.js**: Configures Node.js version 20 and caches `npm` dependencies using `actions/setup-node@v4`.
3.  **Install Dependencies**: Executes `npm ci` to install project dependencies.
4.  **Typecheck**: Runs `npx tsc --noEmit` to perform a TypeScript type check without emitting any JavaScript files.
5.  **Lint**: Executes `npm run lint` to enforce code style and quality standards.

This job ensures that all proposed changes meet the project's baseline quality and type safety requirements before proceeding to the build stage.

## Build Job

The `build` job executes the full application build process. It depends on the successful completion of the `check` job (`needs: check`) and also runs on `ubuntu-latest`.

This job requires a comprehensive set of environment variables, which are securely injected from GitHub Secrets. These include credentials for Supabase, Stripe, Evolution API, Resend, Upstash Redis, and Sentry, as well as the `CRON_SECRET`.

The steps involved are:

1.  **Checkout Code**: Uses `actions/checkout@v4`.
2.  **Setup Node.js**: Configures Node.js version 20 and caches `npm` dependencies.
3.  **Install Dependencies**: Executes `npm ci`.
4.  **Build Application**: Runs `npm run build` to compile the Next.js application for production.

The full list of environment variables passed to the build job is defined in `.github/workflows/ci.yml`:

```yaml
env:
  NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.NEXT_PUBLIC_SUPABASE_URL }}
  NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.NEXT_PUBLIC_SUPABASE_ANON_KEY }}
  SUPABASE_SERVICE_ROLE_KEY: ${{ secrets.SUPABASE_SERVICE_ROLE_KEY }}
  DATABASE_URL: ${{ secrets.DATABASE_URL }}
  STRIPE_SECRET_KEY: ${{ secrets.STRIPE_SECRET_KEY }}
  STRIPE_BASIC_PRICE_ID: ${{ secrets.STRIPE_BASIC_PRICE_ID }}
  STRIPE_WEBHOOK_SECRET: ${{ secrets.STRIPE_WEBHOOK_SECRET }}
```

## Known Gaps

The current CI pipeline, as defined in `.github/workflows/ci.yml`, does not include any automated testing for the main application. While the MCP server has its own Vitest tests, these are not integrated into this workflow and are assumed to run only locally by developers. This means that functional or integration regressions in the main application would not be caught by CI.

## Why this shape

The separation of `check` and `build` into distinct jobs allows for faster feedback on code quality issues. If typechecking or linting fails, the `build` job is not initiated, saving build time and resources. This sequential dependency ensures that only code that passes static analysis proceeds to the more resource-intensive build step.
