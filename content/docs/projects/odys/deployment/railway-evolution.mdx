# Railway for Evolution API

This page details the deployment configuration for Evolution API, the WhatsApp integration layer in Odys. It runs as a standalone service on Railway, decoupled from the main Next.js application, ensuring reliable message delivery and isolation of real-time communication concerns.

## Overview

Evolution API is hosted on Railway using a multi-container setup defined in `docker-compose.evolution.yml`. It consists of three services: a PostgreSQL database, Redis for caching, and the Evolution API itself. The Postgres instance (`evolution-postgres`) uses a dedicated volume `evolution_pg`, storing data for the `evolution` database under the `evolution` user. Redis (`evolution-redis`) provides session and instance caching with its own persistent volume. The main API container (`evolution-api`) runs the Docker image `evoapicloud/evolution-api:v2.3.7`, exposing port 8080 and connecting to both data stores via a shared bridge network `evolution-net`. Internally, the API container is configured with `SERVER_URL: "http://localhost:8080"`. Telemetry is explicitly disabled via `TELEMETRY_ENABLED: "false"`. The global webhook is disabled by default with `WEBHOOK_GLOBAL_ENABLED: "false"`, requiring per-instance configuration via the API. Health checks ensure Postgres is ready before the API starts, and Redis is available via a `service_started` condition before full API initialization.

## Design decisions

The separation of Evolution API from the main Odys app reflects a clear boundary between business logic and real-time messaging. Using a managed Docker image (`evoapicloud/evolution-api:v2.3.7`) reduces maintenance overhead while allowing full control over configuration through environment variables. The use of a dedicated Postgres instance — isolated from the main Supabase database — ensures that WhatsApp session data, message logs, and instance state do not interfere with core application data. Redis is used not just for caching but to maintain session state and track outbound message context, which is critical for routing client replies to the correct professional. The health check on Postgres (`pg_isready`) and dependency ordering (`depends_on`) ensure reliable startup sequences in a containerized environment.

## Potential improvements

The current Redis volume configuration in `docker-compose.evolution.yml` does not set a persistence mode explicitly; adding `command: redis-server --appendonly yes` would ensure AOF persistence and prevent data loss on restart. The `LOG_LEVEL` is set to include `INFO`, which may generate excessive logs in production — tightening it to `WARN` or `ERROR` unless debugging would reduce noise. The `AUTHENTICATION_API_KEY` and `POSTGRES_PASSWORD` are hardcoded in the compose file; these should be injected via Railway’s secret management system instead to avoid exposure in logs or version control.
