# Four Tools

This page details the four tools exposed by the Model Context Protocol (MCP) server, including their input schemas and the implementation of their handler logic as defined in `mcp-server/src/tools.ts`. Each tool enables AI agents to perform specific queries or mutations within the Odys system.

The tool definitions are declared in `mcp-server/src/tools.ts` within the `TOOL_DEFINITIONS` array. All handlers are implemented in the same file and use Zod for input validation, Drizzle ORM for database access, and structured error handling.

## `get_appointments_today`

Retrieves a professional's appointments for the current civil day, resolved in a specified IANA timezone.

### Input Schema

Defined by `GetAppointmentsTodayInput` in `mcp-server/src/tools.ts`:

```typescript
export const GetAppointmentsTodayInput = z.object({
  professionalId: z.string().uuid(),
  timezone: z.string().optional().default("America/Sao_Paulo"),
})
```

### Handler Logic

The `getAppointmentsToday` function:
1. Parses input using `GetAppointmentsTodayInput.parse()`.
2. Uses `Intl.DateTimeFormat("en-CA", { timeZone })` to determine the civil date string for "today" in the given timezone.
3. Constructs naive UTC start and end times (`00:00:00Z` and `23:59:59.999Z`) from that date string.
4. Adjusts these times to true UTC using the `tzOffsetMs` helper, which computes the offset via `Intl.DateTimeFormat` to respect historical DST rules.
5. Queries the database using Drizzle ORM, joining `appointments`, `clients`, and `professionals` tables.
6. Filters by `professionalId` and the UTC `startsAt` range using `and`, `eq`, `gte`, and `lte`.
7. Orders results by `startsAt` and maps them to a standardized output format.
8. Wraps the query in `try/catch`, returning an `error` field on failure.

## `get_customer_by_phone`

Looks up customer records by phone number, normalizing Brazilian formats and returning recent appointment history.

### Input Schema

Defined by `GetCustomerByPhoneInput` in `mcp-server/src/tools.ts`:

```typescript
export const GetCustomerByPhoneInput = z.object({
  phone: z.string().min(4),
})
```

### Handler Logic

The `getCustomerByPhone` function:
1. Parses input using `GetCustomerByPhoneInput.parse()`.
2. Normalizes the input phone using `normalizeBrazilianPhone` from `mcp-server/src/phone.ts`, which strips non-digits and ensures a `55` country code prefix.
3. Generates query variants using `getPhoneVariants`, which returns both E.164 (`55...`) and 11-digit (`11...`) forms.
4. Queries `clients` table using `inArray(clients.phone, variants)`.
5. For each matching client, retrieves their last 5 appointments ordered by `startsAt`.
6. Maps results into a structured format including client details and appointment history.
7. Handles errors via `try/catch`, returning an `error` field if the database operation fails.

## `get_revenue_summary`

Provides a rolling revenue summary for a professional over the last week, month, or year.

### Input Schema

Defined by `GetRevenueSummaryInput` in `mcp-server/src/tools.ts`:

```typescript
export const GetRevenueSummaryInput = z.object({
  professionalId: z.string().uuid(),
  period: z.enum(["week", "month", "year"]),
})
```

### Handler Logic

The `getRevenueSummary` function:
1. Parses input using `GetRevenueSummaryInput.parse()`.
2. Maps `period` to a number of days: 7, 30, or 365.
3. Calculates the `start` timestamp as `now - days * 24 * 60 * 60 * 1000`.
4. Queries the `professionals` table to retrieve `sessionPrice` by `professionalId`.
5. If no professional is found, returns zero values and an `"Professional not found"` error.
6. Otherwise, aggregates appointment count where `paymentStatus = 'captured'` and `startsAt` is within the window.
7. Computes `totalRevenueCents` as `appointmentCount * sessionPrice` and `averageTicketCents` as the rounded average.
8. Returns structured summary with `currency: "BRL"`.
9. Wraps logic in `try/catch`, returning database errors appropriately.

## `update_appointment_status`

Updates an appointment's status with ownership verification and idempotency.

### Input Schema

Defined by `UpdateAppointmentStatusInput` in `mcp-server/src/tools.ts`:

```typescript
export const APPOINTMENT_STATUSES = [
  "pending_confirmation",
  "confirmed",
  "rejected",
  "completed",
  "cancelled",
  "no_show",
] as const

export const UpdateAppointmentStatusInput = z.object({
  appointmentId: z.string().uuid(),
  professionalId: z.string().uuid(),
  status: z.enum(APPOINTMENT_STATUSES),
  reason: z.string().optional(),
})
```

### Handler Logic

The `updateAppointmentStatus` function:
1. Parses input using `UpdateAppointmentStatusInput.parse()`.
2. Loads the existing appointment by `appointmentId`.
3. Returns `{ success: false, error: "Appointment not found" }` if missing.
4. Checks that `existing.professionalId === professionalId`, returning an unauthorized error if not.
5. If the current `status` matches the target `status`, returns `{ success: true, noChange: true }` (idempotent).
6. Otherwise, performs an `UPDATE` on `appointments`, setting `status` and optionally `notes` if `reason` is provided.
7. Uses `.returning()` to get the updated row and maps it via `serializeAppointment`.
8. Handles errors via `try/catch`, returning a database error message on failure.

## Known Gaps

The `get_customer_by_phone` tool queries both E.164 and 11-digit phone number formats using `getPhoneVariants` in `mcp-server/src/phone.ts`. This is a workaround for inconsistent phone number storage (11-digit) versus preferred format (E.164). The long-term fix requires a database migration and shared phone utility, but this is not implemented in the current code.

## Why this shape

The four tools are designed to expose safe, well-scoped operations to AI agents via MCP's JSON-RPC over stdio interface. Each tool encapsulates input validation, business logic, ownership checks (e.g., `professionalId` matching), and error handling, ensuring agents operate within defined boundaries without direct database access. The use of Zod schemas, Drizzle ORM, and structured responses enables reliable integration while isolating complexity within the MCP server.
