# MCP Server: Four Tools

This page details the four tools exposed by the Model Context Protocol (MCP) server, including their input schemas and the implementation of their handler logic where available in the provided code. These tools allow AI agents to interact with the Odys system for common queries and mutations.

The tool definitions, including their names, descriptions, and JSON schema for inputs, are declared in `mcp-server/src/tools.ts` within the `TOOL_DEFINITIONS` array.

## `get_appointments_today`

This tool retrieves a professional's appointments for the current civil day, resolving "today" based on a specified IANA timezone.

### Input Schema

The input for `get_appointments_today` is defined by `GetAppointmentsTodayInput` in `mcp-server/src/tools.ts`:

```typescript
export const GetAppointmentsTodayInput = z.object({
  professionalId: z.string().uuid(),
  timezone: z.string().optional().default("America/Sao_Paulo"),
})
```

The `professionalId` is a UUID identifying the professional, and `timezone` is an optional IANA timezone string, defaulting to `America/Sao_Paulo`.

### Handler Logic

The `getAppointmentsToday` function in `mcp-server/src/tools.ts` implements the handler. It performs the following steps:

1.  Parses and validates the input using `GetAppointmentsTodayInput.parse()`.
2.  Determines the civil date for "today" in the specified `timezone` using `Intl.DateTimeFormat("en-CA", { timeZone })`.
3.  Calculates the UTC start and end timestamps for this civil day by applying the `tzOffsetMs` helper (also in `mcp-server/src/tools.ts`). This helper uses `Intl.DateTimeFormat` to correctly account for historical DST rules.
4.  Executes a Drizzle query to select appointments within this UTC window, joining `appointments`, `clients`, and `professionals` tables.
5.  Filters results by `professionalId` and the calculated `startsAt` range.
6.  Maps the database rows to a standardized output format, including `appointment_id`, `time`, `customer_name`, `status`, and `service_name`.
7.  Includes `try/catch` error handling for database operations, returning an `error` field on failure.

## `get_customer_by_phone`

This tool looks up customer records based on a phone number, handling various Brazilian phone number formats.

### Input Schema

The input for `get_customer_by_phone` is defined by `GetCustomerByPhoneInput` in `mcp-server/src/tools.ts`:

```typescript
export const GetCustomerByPhoneInput = z.object({
  phone: z.string().min(4),
})
```

The `phone` field expects a string of at least 4 characters.

### Handler Logic

While the full handler code is not provided, the `TOOL_DEFINITIONS` in `mcp-server/src/tools.ts` indicates that this tool "Look[s] up customer records matching a phone number (any format — will be normalized to Brazilian E.164 digits). Returns customer details plus their last 5 appointments."

The `mcp-server/src/phone.ts` file contains the `normalizeBrazilianPhone` and `getPhoneVariants` functions, which are used to process the input phone number. `normalizeBrazilianPhone` converts various formats into a consistent Brazilian E.164 or 11-digit format. `getPhoneVariants` generates both E.164 and 11-digit variants to query the `clients.phone` column using `inArray`.

## `get_revenue_summary`

This tool provides a summary of revenue for a professional over rolling periods (week, month, or year).

### Input Schema

The input for `get_revenue_summary` is defined by `GetRevenueSummaryInput` in `mcp-server/src/tools.ts`:

```typescript
export const GetRevenueSummaryInput = z.object({
  professionalId: z.string().uuid(),
  period: z.enum(["week", "month", "year"]),
})
```

It requires a `professionalId` and a `period` which can be `week` (last 7 days), `month` (last 30 days), or `year` (last 365 days).

### Handler Logic

The full handler code is not provided. However, the `TOOL_DEFINITIONS` in `mcp-server/src/tools.ts` states that this tool "Summarize[s] revenue for a professional over a rolling period... Counts only appointments with `paymentStatus='captured'`. Returns `totalRevenueCents`, `appointmentCount`, `averageTicketCents`." The deep-dive further clarifies that revenue is calculated as `count * professional.sessionPrice`.

## `update_appointment_status`

This tool allows updating the status of an appointment.

### Input Schema

The input for `update_appointment_status` is defined by `UpdateAppointmentStatusInput` in `mcp-server/src/tools.ts`:

```typescript
export const APPOINTMENT_STATUSES = [
  "pending_confirmation",
  "confirmed",
  "rejected",
  "completed",
  "cancelled",
  "no_show",
] as const

export const UpdateAppointmentStatusInput = z.object({
  appointmentId: z.string().uuid(),
  professionalId: z.string().uuid(),
  status: z.enum(APPOINTMENT_STATUSES),
  reason: z.string().optional(),
})
```

It requires an `appointmentId`, `professionalId` for ownership verification, and a `status` from the `APPOINTMENT_STATUSES` enum. An optional `reason` string can be provided to update the appointment's notes.

### Handler Logic

The full handler code is not provided. The `TOOL_DEFINITIONS` in `mcp-server/src/tools.ts` describes the handler's behavior: "Updates appointment status. Caller must provide `professionalId` to verify ownership — the appointment's `professionalId` must match, otherwise the update is rejected. Idempotent: if the appointment already has the target status, no write is performed. If `reason` is provided, it overwrites the appointment's `notes` column."

## Known Gaps

The `get_customer_by_phone` tool currently relies on `normalizeBrazilianPhone` and `getPhoneVariants` in `mcp-server/src/phone.ts` to query for both E.164 and 11-digit phone number formats. This is a pragmatic workaround for the fact that the application's database stores 11-digit phones while the MCP and intake agents prefer E.164. The long-term solution, involving a shared `src/lib/phone.ts` utility and a database migration to standardize phone number storage, is tracked separately.

## Why this shape

The MCP server and its tools are designed to provide a clean, JSON-RPC over stdio interface for AI agents. This abstraction allows agents, such as Claude Code sessions or the in-product assistant, to query and mutate Odys data without needing direct database access or understanding the underlying Drizzle ORM schema. By encapsulating business logic and ownership checks within the tool handlers, the system ensures that agents operate within defined boundaries and do not require sensitive credentials like `DATABASE_URL`.
