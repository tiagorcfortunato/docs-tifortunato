# MCP Server Overview

Tiago, let's dive into the Model Context Protocol (MCP) server within Odys. This component is designed to expose specific business logic and data queries to external models or AI agents in a structured, discoverable way. While the `odys-mcp-server` is currently configured but not actively deployed as a standalone service, understanding its architecture and the capabilities it offers is crucial for future integrations and for grasping how Odys intends to interact with intelligent systems.

## Overview

The `odys-mcp-server` acts as an interface, translating requests from an MCP client (likely an AI model) into concrete actions within the Odys application's backend. It adheres to the Model Context Protocol, which provides a standardized way for models to discover and invoke "tools" or functions exposed by a service.

Your `package.json` in `mcp-server` defines this as `odys-mcp-server` with version `0.1.0`. The server is configured to run via Node.js, as specified in the `.mcp.json` file, which points to `node ./mcp-server/dist/server.js` as its execution command. This setup allows an MCP client to discover and interact with the server by executing this command and communicating over standard I/O.

At its core, the server exposes a set of four distinct tools, each designed to perform a specific operation related to professional and client management within Odys. These tools are: `get_appointments_today`, `get_customer_by_phone`, `get_revenue_summary`, and `update_appointment_status`.

## Architecture

The MCP server's architecture is cleanly separated into two primary files: `mcp-server/src/server.ts` and `mcp-server/src/tools.ts`.

### `mcp-server/src/server.ts`

This file is the entry point for the MCP server. It initializes an instance of the `Server` class from the `@modelcontextprotocol/sdk` and sets up the communication transport using `StdioServerTransport`. This means the server communicates with its client over standard input/output streams, which is a common pattern for local or containerized tool execution.

The `server.ts` file defines two main request handlers:

1.  **`ListToolsRequestSchema`**: When an MCP client asks what tools are available, this handler responds by providing the `TOOL_DEFINITIONS` array, which is imported from `mcp-server/src/tools.ts`. This allows the client to understand the capabilities of the `odys` server, including each tool's name, description, and expected input schema.
2.  **`CallToolRequestSchema`**: This is the core handler for executing tools. When a client requests to call a specific tool, this handler receives the tool's `name` and `arguments`. It then uses a `switch` statement to dispatch the request to the appropriate function (e.g., `getAppointmentsToday`, `getCustomerByPhone`), also imported from `mcp-server/src/tools.ts`. The result of the tool execution is then formatted and returned to the MCP client. Error handling is robust, catching any exceptions during tool execution and returning a structured error message to the client.

### `mcp-server/src/tools.ts`

This file contains the actual business logic for each of the four exposed tools. It leverages `zod` for input validation and `drizzle-orm` for interacting with the Odys database.

Each tool function (e.g., `getAppointmentsToday`) takes a raw input, parses it against a `zod` schema (like `GetAppointmentsTodayInput`), and then performs the necessary database operations.

Let's look at the specific tools:

*   **`get_appointments_today`**: This tool retrieves all appointments scheduled for a given `professionalId` on the current day, considering a specified `timezone` (defaulting to `America/Sao_Paulo`). It queries the `appointments` table, joining with `clients` and `professionals` to enrich the output with customer names and the professional's service type. The `tzOffsetMs` helper function is used to correctly determine the start and end of "today" in the specified timezone.
*   **`get_customer_by_phone`**: This tool allows looking up client information using a phone number. It normalizes the input phone number to a Brazilian E.164 format and queries the `clients` table. If a client is found, it also fetches their last 5 appointments from the `appointments` table, providing a historical context. The `clients` table has 8 columns, including `name`, `phone`, and `email`.
*   **`get_revenue_summary`**: This tool calculates revenue metrics for a `professionalId` over a specified `period` (week, month, or year). It queries the `professionals` table (which has 24 columns, including `sessionPrice`) to get the professional's standard session price and then counts `appointments` with a `paymentStatus` of 'captured' within the defined period. It returns the total revenue, appointment count, and average ticket value.
*   **`update_appointment_status`**: This tool modifies the `status` of an existing appointment. It requires both `appointmentId` and `professionalId` to ensure that only the owning professional can update an appointment. The `APPOINTMENT_STATUSES` constant defines the allowed statuses: `pending_confirmation`, `confirmed`, `rejected`, `completed`, `cancelled`, `no_show`. If a `reason` is provided, it updates the `notes` column of the `appointments` table. The `appointments` table has 13 columns, including `status` and `notes`.

All these tools interact with the Odys database, which consists of 10 tables. Notably, many tables like `availability`, `clients`, `clientNotes`, `recurringSchedules`, `appointments`, `messages`, `follows`, and `reviews` have foreign key relationships to `professionalId` or `clientId` with `onDelete: cascade` behavior, ensuring that related records are automatically removed when a professional or client is deleted.

## Design decisions

The design of the `odys-mcp-server` reflects several deliberate choices:

*   **AI/Model Integration**: The primary motivation for using the Model Context Protocol is to enable AI models to programmatically interact with Odys's backend. By exposing specific, well-defined tools, you're creating a controlled and auditable interface for AI agents to query data and perform actions, rather than giving them direct database access or complex API endpoints. This simplifies the AI's task and enhances system security.
*   **Clear Separation of Concerns**: The division between `mcp-server/src/server.ts` and `mcp-server/src/tools.ts` is a good practice. `server.ts` focuses solely on the MCP communication protocol and dispatching requests, while `tools.ts` encapsulates the application-specific business logic and database interactions. This makes each part easier to understand, test, and maintain independently.
*   **Input Validation with Zod**: The use of `zod` for defining input schemas (e.g., `GetAppointmentsTodayInput`) is a strong choice for type safety and runtime validation. It ensures that incoming tool arguments conform to expected structures and types before any business logic is executed, preventing common errors and improving the robustness of the tools.
*   **Declarative Tool Definitions**: The `TOOL_DEFINITIONS` array provides a declarative way to describe each tool's capabilities, including its `name`, `description`, and `inputSchema`. This JSON Schema representation is essential for MCP clients to dynamically discover and understand how to use the tools without prior knowledge of the Odys codebase.
*   **Drizzle ORM for Database Access**: Leveraging `drizzle-orm` provides a type-safe and expressive way to interact with the database. This reduces the likelihood of SQL injection vulnerabilities and makes database queries more readable and maintainable compared to raw SQL.

## Potential improvements

1.  **Consistent Error Response Structure**: In `mcp-server/src/tools.ts`, the error responses from tool functions are not entirely consistent. For instance, `getAppointmentsToday` and `getRevenueSummary` return an object with an `error` field alongside other data, while `updateAppointmentStatus` returns `{ success: false, error: "..." }`. Standardizing these error responses (e.g., always returning an object with `success: boolean` and an optional `error: string` or `errorMessage: string`) would make it easier for MCP clients to parse and handle errors uniformly across all tools.
2.  **Robust Timezone Handling**: The `tzOffsetMs` function in `mcp-server/src/tools.ts` is a custom implementation for calculating timezone offsets. While it works for basic cases, timezone calculations can be complex due to Daylight Saving Time (DST) changes, historical data, and various edge cases. Consider replacing this custom logic with a more robust, well-tested library like `date-fns-tz` or `Luxon` to ensure accuracy and reduce potential bugs related to time zone conversions, especially for critical features like appointment scheduling.
3.  **Dynamic Revenue Calculation**: The `getRevenueSummary` tool in `mcp-server/src/tools.ts` calculates `totalRevenueCents` as `appointmentCount * professional.sessionPrice`. This assumes a fixed `sessionPrice` for all appointments of a professional. If `sessionPrice` can vary per appointment (e.g., due to discounts, different service types, or price changes over time), this calculation might not be accurate. A more robust approach would be to store the actual price charged for each appointment in the `appointments` table and sum those values, or to join with a `services` table that defines prices.
4.  **Synchronization of Zod and JSON Schemas**: The `mcp-server/src/tools.ts` file defines both `zod` schemas (e.g., `GetAppointmentsTodayInput`) for runtime validation and JSON schemas within `TOOL_DEFINITIONS` for MCP client discovery. It's crucial that these two representations remain synchronized. You could explore generating the JSON schemas directly from the Zod schemas to eliminate duplication and prevent inconsistencies, perhaps using a library like `zod-to-json-schema`. This would ensure that the validation logic and the advertised tool interface are always aligned.
