# Shared Schema

The shared Drizzle schema is the central source of truth for database structure across both the main application and the MCP server. By exposing the same `schema.ts` file from `src/lib/db/`, both services interact with the database using identical type definitions and table configurations, ensuring consistency and reducing duplication.

## Overview

The schema defines **11 tables**: `professionals`, `clients`, `appointments`, `availability`, `messages`, `reviews`, `clientNotes`, `recurringSchedules`, `follows`, `notifications`, and `agentMetrics`. These tables are used by both the Next.js frontend API routes and the MCP server to query and mutate data. The MCP server imports the schema via a relative path (`../../src/lib/db/schema.js`) and references it in its `tsconfig.json`'s `include` array, allowing TypeScript to resolve the types without duplication. This setup enables the MCP tools—such as `get_appointments_today` and `update_appointment_status`—to safely interact with the same database structure as the main app.

## Design decisions

The schema uses `uuid` primary keys with `defaultRandom()` for distributed safety, and leverages PostgreSQL-specific features like `ON DELETE CASCADE` on foreign keys to maintain referential integrity. Indexes are strategically placed on high-query fields like `professionalId`, `startsAt`, and `status`. The `appointments` table includes a partial unique index to prevent double-booking by rejecting overlapping slots with non-terminal statuses. Timestamps use `defaultNow()` for consistency, and nullable fields are used for truly optional data, such as `email`, `bio`, conditional foreign keys, and telemetry fields. The `agentMetrics` table captures telemetry from AI agent interactions, using `numeric` types for cost precision and arrays for `providersUsed`.

## Potential improvements

1. **Move schema import to a package alias** — Instead of using a relative path (`../src/lib/db/schema.ts`), configure a path alias (e.g., `@odys/db`) in both `tsconfig.json` files. This would decouple the MCP server from the physical location of the schema and reduce refactoring risk.

2. **Add foreign key validation in `recurringSchedules`** — The `recurringSchedules` table references `clients` and `professionals`, but does not validate that the `clientId` belongs to the associated `professionalId`. Adding a composite foreign key or application-level check would prevent inconsistent data.

3. **Enforce `dayOfWeek` constraints in `availability`** — The `dayOfWeek` column accepts integers 0–6, but there is no database-level constraint to enforce this range. Adding a `check` constraint (e.g., `check(sql`${t.dayOfWeek} >= 0 and ${t.dayOfWeek} <= 6`)`) would prevent invalid values at the schema level.
