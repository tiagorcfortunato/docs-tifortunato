# Shared Schema

The Odys application employs a centralized Drizzle ORM schema, defined in `src/lib/db/schema.ts`, which serves as the single source of truth for the database structure. This design choice is fundamental for maintaining consistency and reducing redundancy across various services, including the `mcp-server`. This document explores the structure of this shared schema, the rationale behind its sharing mechanism, and opportunities for refinement.

## Overview

The core database schema for Odys is meticulously defined using Drizzle ORM, encompassing a total of 10 tables. These tables model the essential entities and relationships within the platform, ranging from `professionals` and `clients` to `appointments`, `messages`, and `notifications`. By consolidating the schema definition into a single file, `src/lib/db/schema.ts`, the project ensures that all services interacting with the database operate against an identical and consistent data model.

The `mcp-server` component, for instance, accesses this shared schema through a relative import in its `mcp-server/src/db.ts` file. This setup allows the `mcp-server` to initialize its Drizzle client with the exact same schema definitions used elsewhere in the application, facilitating seamless data operations and type safety. The TypeScript configuration for `mcp-server` explicitly includes the shared schema file, `../src/lib/db/schema.ts`, ensuring that type checking and compilation correctly incorporate these definitions.

## Database Schema Details

The schema defines a comprehensive set of tables to manage the Odys platform's operations:

*   **`professionals`**: This central table, with 32 columns, stores detailed information about the service providers. It includes fields like `userId` (a unique identifier from Supabase auth), `name`, `slug` for public profile URLs, contact information (`phone`, `email`), professional details (`profession`, `bio`), and service configuration (`sessionDuration`, `sessionPrice`). It also tracks payment preferences (`plan`, `paymentType`, `paymentPercentage`, `pixKeyType`, `pixKey`, `stripeCustomerId`, `stripeSubscriptionId`), trial status (`trialEndsAt`), and physical location details (`addressLine`, `city`, `state`, `latitude`, `longitude`). The table includes indices on `profession` and `city` to optimize common search queries.
*   **`availability`**: Linked to `professionals` via `professionalId` with a cascade delete, this table records the time slots when a professional is available. It specifies `dayOfWeek` (0-6 for Sunday-Saturday) and `startTime`/`endTime` as text.
*   **`clients`**: This table, with 8 columns, stores information about the clients. It includes an optional `userId` (allowing clients to book without a full account), `professionalId` (linking to their service provider), `name`, `phone`, `email`, and `avatarUrl`.
*   **`clientNotes`**: A private record for professionals, this table stores `content` notes about specific `clients`, linked by `professionalId` and `clientId`.
*   **`recurringSchedules`**: Manages recurring appointments between a professional and a client, specifying `frequency` (e.g., "weekly", "biweekly"), `dayOfWeek`, `secondDayOfWeek`, and `time`.
*   **`appointments`**: This critical table tracks individual bookings, linking `professionalId` and `clientId`. It includes `startsAt` and `endsAt` timestamps, `status` (e.g., "pending_confirmation", "confirmed"), `paymentStatus`, and flags for `reminderSent24h` and `reminderSent1h`. It also allows for an optional `recurringScheduleId` to link back to recurring bookings.
*   **`messages`**: Facilitates communication between professionals and clients, recording `sender`, `type` (e.g., "text", "pdf"), `content`, and `fileUrl`.
*   **`follows`**: Records when a `userId` (client) follows a `professionalId`.
*   **`reviews`**: Stores client feedback, including a `rating` (1-5) and `comment` for a specific `appointmentId`.
*   **`notifications`**: A generic notification system, storing `recipientId`, `recipientType` (e.g., "professional", "client"), `type`, `title`, `body`, and `read` status.

Each table leverages Drizzle's `pgTable` function, defining columns with specific PostgreSQL types like `uuid`, `text`, `timestamp`, `boolean`, `integer`, and `doublePrecision`. Many columns include `defaultRandom()` for UUIDs, `defaultNow()` for timestamps, or other sensible default values, streamlining data insertion. Foreign key relationships are established using `references()` with `onDelete: "cascade"` where appropriate, ensuring data integrity.

## Design Decisions

The decision to define the entire Drizzle schema in a single file, `src/lib/db/schema.ts`, and share it across different application components like `mcp-server` via relative imports, is driven by several key principles:

1.  **Single Source of Truth**: Centralizing the schema definition eliminates duplication and ensures that all parts of the application interact with an identical database structure. This prevents inconsistencies that could arise from separate, potentially divergent schema definitions.
2.  **Type Safety and Developer Experience**: By importing the Drizzle schema directly, services like `mcp-server` gain immediate access to the generated TypeScript types. This provides strong type checking for database operations, catching errors at compile-time rather than runtime, and significantly improving developer experience with auto-completion and type inference.
3.  **Simplified Migrations**: With a unified schema, database migrations become more straightforward. Changes to the schema only need to be applied in one place, and all dependent services will automatically reflect these changes upon recompilation, assuming the Drizzle migration tools are configured to use this central schema.
4.  **Monorepo Structure**: The project's monorepo-like structure naturally lends itself to sharing code via relative paths. The `mcp-server` is a distinct application but resides within the same repository, making `../../src/lib/db/schema.js` a practical way to access the shared module. The `tsconfig.json` in `mcp-server` explicitly includes `../src/lib/db/schema.ts` to ensure TypeScript correctly processes the types from the source file, even though the runtime import points to the compiled `.js` output.

The trade-off for this approach is a tighter coupling between the `mcp-server` and the main application's `src` directory. While beneficial for consistency, it means that changes to the schema file directly impact `mcp-server`, requiring careful coordination during development and deployment.

## Potential Improvements

While the current shared schema setup is functional and provides consistency, several areas could be considered for enhancement:

1.  **Refine `paymentPercentage` Nullability and Default**: The `professionals` table defines `paymentPercentage` as `integer("payment_percentage").default(100)`, making it implicitly non-nullable.
2.  **Stronger Typing for Time and Day-of-Week Fields**: Columns like `startTime` and `endTime` in `availability`, and `time` in `recurringSchedules`, are defined as `text`. Similarly, `dayOfWeek` and `secondDayOfWeek` are `integer`. While functional, using more specific types or custom Drizzle types could enhance data integrity. For instance, `time` could be a custom Drizzle type that enforces a specific time format (e.g., "HH:MM"), and `dayOfWeek` could be an enum-like type to ensure values are strictly within 0-6. This would shift validation from application logic to the schema definition.
3.  **Abstract Shared Schema into a Dedicated Package/Alias**: The current relative import `../../src/lib/db/schema.js` works but tightly couples `mcp-server` to the exact file structure of the main application. For larger monorepos or more complex build systems, abstracting the schema into a dedicated internal package or using TypeScript path aliases (e.g., `@odys/db-schema`) could provide a cleaner, more maintainable import path. This would decouple the `mcp-server`'s knowledge of the schema's physical location from its logical dependency, making refactoring easier.
