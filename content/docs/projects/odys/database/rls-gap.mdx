# RLS Gap

The odys application's authorization pattern, as demonstrated in `src/lib/api.ts`, relies on application-layer checks rather than database-enforced Row-Level Security (RLS) policies. This architectural choice means that data access control is primarily managed at the application layer, rather than being enforced directly by the database. Understanding this pattern is crucial for comprehending how user permissions and data isolation are maintained throughout the system.

## Overview

The core of odys's authorization strategy resides within the application's API routes, specifically leveraging the shared helpers defined in `src/lib/api.ts`. This file acts as a central hub for authentication, authorization, and utility functions, but its usage across all API routes is not explicitly shown in the provided code.

When a request arrives at an API endpoint, the application first authenticates the user. This is achieved through the `getUser()` helper, which interacts with Supabase to retrieve the authenticated user's details. Once a user is identified, the system then needs to establish their professional context. This is where the `getProfessional(userId: string)` function becomes vital. It queries the `professionals` table using the authenticated `userId` to fetch the corresponding professional profile. The `professionals` table's `userId` column is designed to ensure a one-to-one mapping between a Supabase user and their professional record.

The `id` column of the `professionals` table is designed to serve as the primary key and is expected to be referenced by many other tables in the schema. For instance, tables like `availability`, `clients`, `clientNotes`, `recurringSchedules`, `appointments`, `messages`, `follows`, and `reviews` are expected to include a `professionalId` column. This `professionalId` acts as the implicit ownership identifier, linking specific data records to a particular professional. The absence of RLS means that any query involving these tables must explicitly include a `WHERE professionalId = <current_professional_id>` clause at the application level to ensure data isolation. The application also uses a `getIp` function to extract the client IP from the request headers for rate limiting purposes.

## Design Decisions

The decision to implement authorization primarily at the application layer, rather than relying on database-level RLS, comes with several considerations and trade-offs:

1.  **Flexibility and Control**: Implementing authorization logic in application code, as seen with the `unauthorized()`, `forbidden()`, and `badRequest()` response helpers in `src/lib/api.ts`, offers a high degree of flexibility. Complex business rules, such as plan-based limits (e.g., for "basic", "pro", or "premium" plans) or dynamic permissions, can be more easily expressed and managed within the application's codebase. This approach gives developers granular control over how and when access checks are performed, often integrating seamlessly with the existing programming language and frameworks.

2.  **Developer Familiarity**: Many developers are more accustomed to writing authorization logic directly in their application code. This can lead to a faster development cycle and easier debugging, as the entire authorization flow is visible and debuggable within the application's execution path. The use of Drizzle ORM for database interactions, as demonstrated by the `db.select().from(professionals).where(eq(professionals.userId, userId)).limit(1)` query, fits naturally into this paradigm.

3.  **Performance Considerations**: While RLS can be highly optimized by the database, some developers might perceive application-layer filtering as having more predictable performance characteristics, especially for simpler access patterns. By fetching the `professionalId` once and then applying it to subsequent queries, the application avoids the potential overhead of evaluating RLS policies on every database operation, though this benefit is often negligible for well-indexed RLS.

However, this approach introduces significant trade-offs:

*   **Increased Risk of Data Exposure**: Without RLS, the responsibility for enforcing data isolation falls entirely on the application. A single missed `WHERE` clause in any of the application's API routes could inadvertently expose sensitive data belonging to one professional to another, or even to an unauthenticated user. This requires rigorous testing and code reviews for every data access path.
*   **Duplication and Inconsistency**: The logic to filter data by `professionalId` must be manually applied in every relevant query across the application. This can lead to duplicated code and a higher risk of inconsistencies if the filtering logic is not applied uniformly.
*   **Maintenance Burden**: As the application grows, maintaining consistent authorization logic across numerous API endpoints can become challenging. Any change to how data ownership is determined or enforced requires modifications across multiple parts of the codebase.

## Potential Improvements

The current application-layer authorization, while functional, presents several opportunities for enhancement to improve security, maintainability, and consistency:

1.  **Implement Row-Level Security (RLS) Policies**: The most impactful improvement would be to introduce RLS policies directly within the database. Given the application's data model, where tables such as `availability`, `clients`, `clientNotes`, `recurringSchedules`, `appointments`, `messages`, `follows`, and `reviews` are expected to contain a `professionalId` column, RLS could be configured to automatically filter records based on the authenticated user's associated `professionalId`. This would provide a robust, database-enforced security layer, significantly reducing the risk of accidental data leaks due to application-layer bugs. The `professionals` table's `userId` column could be used in conjunction with Supabase's `auth.uid()` function to map the authenticated user to their `professionalId` for RLS context.

2.  **Centralize Authorization Middleware**: While `src/lib/api.ts` centralizes helper functions like `getUser()` and `getProfessional()`, the actual *enforcement* of authorization still needs to be explicitly called in each API route. A more robust pattern would involve creating a dedicated middleware or a higher-order function that wraps API route handlers. This middleware would automatically perform the `getUser()` and `getProfessional()` checks and, if successful, inject the `professionalId` into the request context or pass it as an argument to the handler. This would ensure that every protected route automatically benefits from these checks, reducing boilerplate and the chance of missed authorization.

3.  **Introduce a Request-Scoped Professional Context**: To make data access more explicit and less error-prone, consider establishing a request-scoped `ProfessionalContext` object. After `getProfessional(userId: string)` successfully retrieves the professional's data, this context could store the `professionalId` and other relevant professional details. Subsequent Drizzle queries within that request's lifecycle could then explicitly draw the `professionalId` from this context, rather than requiring it to be passed around or re-derived. This pattern would clearly delineate which professional's data is being operated on, improving code readability and reducing the likelihood of using an incorrect ID.
