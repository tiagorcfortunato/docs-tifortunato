# RLS Gap: No Row-Level Security Policies

Odys operates a 10-table schema where user data isolation is critical—professionals manage clients, appointments, messages, and payments. Despite this, the system currently has no Row-Level Security (RLS) policies in place. Access control is instead enforced entirely at the application layer, a decision with significant implications for security, maintainability, and long-term scalability.

## Overview

The database schema includes core entities such as `professionals`, `clients`, `appointments`, `messages`, and `recurringSchedules`, all linked via `professionalId` or `userId`. There are 21 API routes handling operations from booking to notifications, and every authenticated action depends on verifying ownership or relationship before allowing access. For example, fetching messages or updating appointments requires checking whether the requesting professional owns the associated data. This logic is centralized in `src/lib/api.ts` through helpers like `getProfessional`, but it is not backed by database-level enforcement.

Without RLS, any direct database access—whether through a compromised admin interface, misconfigured API, or future migration—could bypass these checks. The current model assumes perfect application logic, which becomes riskier as the codebase grows.

## Design decisions

The app-layer authorization pattern was likely chosen for its simplicity and tight integration with Next.js server components and Supabase auth. Functions like `getProfessional` abstract identity resolution and are reused across API routes, promoting consistency. This approach also avoids the complexity of managing Postgres policies alongside Drizzle ORM, especially during early development.

However, this trade-off centralizes trust in the application code. If a route mistakenly omits an ownership check—such as failing to validate `professionalId` on a `DELETE /api/recurring` call—the database will still allow the operation. Rate limiting (e.g., `rl:booking` at 5/10m) helps mitigate abuse but does nothing to prevent unauthorized data access.

Additionally, the use of `userId` as a text field (rather than UUID) in tables like `professionals` and `follows` suggests alignment with Supabase auth IDs, but without RLS, this linkage remains logically enforced, not structurally guaranteed.

## Potential improvements

1. **Introduce RLS policies for `appointments` and `messages`**  
   These tables contain sensitive user interactions and scheduling data. Adding RLS policies that restrict access based on `professionalId` would provide defense-in-depth. The logic in `src/lib/api.ts` already extracts the user and resolves the professional, so enabling RLS would align the database with existing app assumptions. See `getProfessional` in `src/lib/api.ts`.

2. **Enforce ownership in `client_notes` via policy**  
   Client notes are highly sensitive. While the app checks ownership today, a direct DB query could expose them. A policy ensuring `auth.uid() = professionals.userId` via a join would close this gap. This strengthens trust without changing app logic.

3. **Add RLS to `recurringSchedules` and `reviews`**  
   These tables are tied to both `professionalId` and `clientId`. Without policies, a malicious actor could forge requests to view or delete others' recurring bookings or reviews. The current `eq(professionals.userId, userId)` pattern in `getProfessional` should be mirrored in SQL policies to ensure consistency.
