# RLS Gap

This page examines a critical security and architecture decision in the application: the absence of Row-Level Security (RLS) policies in the database, and the resulting reliance on application-layer authorization. Understanding this gap is essential for assessing data safety, scalability, and long-term maintainability.

## Overview

The database schema consists of **10 tables**, including core entities like `professionals`, `clients`, `appointments`, and `messages`. Despite handling sensitive user data and supporting authenticated interactions, **no RLS policies are defined** in the schema. Instead, access control is entirely managed in the application layer—primarily through helper functions in `src/lib/api.ts`. This pattern centralizes authorization logic but shifts the burden of enforcement from the database to the application code, creating a potential single point of failure if not consistently applied.

## Authorization Through Application Logic

Access control is implemented via a set of reusable helpers in `api.ts`. The `getProfessional(userId)` function, for example, retrieves a professional’s record using the authenticated Supabase user ID. This function is used across endpoints like `/api/messages` and `/api/appointments/[id]` to ensure users only access data tied to their account. Similarly, `getUser()` extracts the authenticated Supabase session, forming the basis of identity resolution.

IP extraction for rate limiting is provided by `getIp(req)`. The enforcement of rate limits, such as '5 booking attempts per 10 minutes', is handled by other parts of the application logic. Cron jobs, like `/api/cron/reminders`, use `isCronAuthorized()`—a constant-time secret comparison—to restrict execution, illustrating a more robust pattern for sensitive operations.

## Design decisions

The lack of RLS suggests a deliberate choice to keep the database "dumb" and delegate security to the application. This simplifies initial development and allows fine-grained control over logic, such as handling soft deletes (`active` flags) or multi-tenant patterns via `professionalId` foreign keys. Using Drizzle ORM, queries are built programmatically, making it easier to inject filters (e.g., `.where(eq(professionals.userId, userId))`) than to manage SQL-level policies. However, this assumes every query correctly applies these filters—an implicit contract that grows riskier as the codebase scales.

## Potential improvements

1. **Introduce RLS policies for `professionals` and related tables**  
   Currently, a direct database query could expose all professionals’ data. Adding RLS on `professionals` (e.g., `USING (userId = auth.uid())`) would enforce isolation at the database level, providing defense in depth.

2. **Refactor `getProfessional` to include soft-delete awareness**  
   The function in `src/lib/api.ts` does not filter by `active = true`. This could allow access to deactivated accounts. Adding this condition would align with business logic seen in other parts of the app.

3. **Centralize authorization checks in a middleware or decorator pattern**  
   Endpoints like `/api/settings` and `/api/reviews` manually invoke auth helpers. A reusable middleware could reduce duplication and prevent accidental omissions, especially as new routes are added.
