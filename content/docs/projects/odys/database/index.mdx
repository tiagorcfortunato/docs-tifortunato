# Database Overview

This page provides a structural and operational overview of the Odys application’s database, built with Drizzle ORM on Supabase Postgres. It covers the schema design, key relationships, and how data models support core features like booking, messaging, and AI agent telemetry.

## Overview

The database consists of **11 tables** that model professionals, clients, appointments, and interactions. At its core, `professionals` acts as the central entity, with tables like `appointments`, `clients`, and `availability` referencing it via foreign keys with `ON DELETE CASCADE`. This ensures data integrity when a professional account is removed. The schema supports both online and in-person services, with geolocation fields and address components stored directly on `professionals`. Financial workflows are handled through Stripe and Pix, with fields like `stripeCustomerId`, `paymentType`, and `pixKey` enabling flexible payment routing. Recurring appointments, client notes, reviews, and notifications are all modeled as first-class entities, reflecting the app’s focus on long-term client relationships.

The `agentMetrics` table stands out as a telemetry sink for the WhatsApp AI intake agent, capturing cost, latency, and outcome per conversation turn. This enables observability into AI performance without blocking user interactions. Supabase auth is integrated via `userId` fields in `professionals`, `clients`, and `follows`, linking app data to authenticated users.

## Design decisions

The schema uses **Drizzle ORM** for type safety and maintainable migrations. All primary keys are `uuid` with `defaultRandom()`, avoiding predictable IDs. Timestamps use `defaultNow()` for consistency. Indexes are strategically placed on foreign keys (`professionalId`, `clientId`) and query-heavy fields like `appointments.startsAt` and `status`. A partial unique index on `appointments(professionalId, startsAt)` prevents double-booking by rejecting overlapping active appointments at the database level, even under high concurrency.

The `recurringSchedules` table decouples recurrence logic from `appointments`, allowing dynamic generation of instances while preserving historical data. The `messages` table enforces a one-message-at-a-time constraint via application logic, not schema, giving flexibility in UX design. Notifications are generic, with `type` and `recipientType` enabling broad use across professionals and clients.

## Potential improvements

1. **Add `updatedAt` trigger for `appointments`** — The `appointments` table lacks an `updatedAt` column, unlike most others. Adding it and using a trigger (or Drizzle middleware) would help track status changes, useful for audit and analytics. See `src/lib/db/schema.ts`.

2. **Normalize `paymentType` and `status` enums** — Currently, values like `"upfront"` or `"pending_confirmation"` are magic strings. Defining a `payment_type` or `appointment_status` enum type in Postgres would prevent invalid states and improve query clarity. See `professionals.paymentType` and `appointments.status`.

3. **Index `clients(phone)` for WhatsApp lookups** — The app frequently resolves clients by phone (e.g., WhatsApp messages). Adding an index on `clients.phone` would speed up `get_customer_by_phone`. Currently, only `professional_id_idx` and `user_id_idx` exist. See `src/lib/db/schema.ts`.
