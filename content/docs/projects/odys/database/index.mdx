# Database: Drizzle on Supabase Postgres

The Odys platform relies on a robust, type-safe database layer built with **Drizzle ORM** over **Supabase Postgres**, enabling efficient, scalable interactions across professional scheduling, client management, payments, and real-time communication. This page explores the schema design, integration patterns, and architectural rationale behind the persistence layer that powers core features like booking, messaging, and subscription management.

## Overview

Odys’s database is a 10-table schema structured to support a two-sided marketplace between professionals (e.g., therapists, trainers) and their clients. Each table maps cleanly to a business domain: `professionals`, `clients`, `appointments`, `messages`, and `recurringSchedules` form the core workflow, while auxiliary tables like `availability`, `clientNotes`, and `reviews` enrich functionality. Relationships are strictly enforced via foreign keys with cascade deletes, ensuring referential integrity—especially critical when a professional deletes their account. The schema is fully reflected in Drizzle’s type system, enabling compile-time safety across API routes and cron jobs.

## Schema in Practice

The `professionals` table acts as the central identity for service providers, storing not only profile data (`name`, `slug`, `bio`) but also business logic fields like `sessionPrice` (in cents), `paymentType`, and Stripe integration keys. It supports multiple monetization models—upfront payments, revenue sharing (`paymentPercentage`), or post-session billing—making it adaptable across regions and professions.

Scheduling is powered by `availability` (weekly time slots) and `appointments` (instantiated sessions), with `recurringSchedules` enabling automated weekly or biweekly bookings. Appointments track both business state (`status`, `paymentStatus`) and engagement signals (`reminderSent24h`, `reminderSent1h`), which feed into cron-triggered WhatsApp reminders.

Communication is handled via the `messages` table, where `sender` distinguishes between client and professional, and `readAt` enables read receipts. The `notifications` table delivers in-app alerts for booking updates or messages, while `follows` tracks client interest in professionals—potentially for newsletter or recommendation features.

## Design decisions

Drizzle was chosen over raw SQL or heavier ORMs for its **type inference**, **schema-as-code** approach, and seamless Next.js compatibility. The use of `defaultRandom()` for UUIDs ensures global uniqueness, while `defaultNow()` standardizes timestamp handling. Indexes are strategically placed on foreign keys (`professionalId`, `clientId`) and query-heavy fields like `appointments.startsAt`, optimizing read performance for calendar views and dashboards.

Supabase Postgres provides immediate scalability, JSON support (not yet used), and Row Level Security (RLS)—though RLS policies are not yet defined in the current schema. The decision to store `startTime` and `endTime` as text (e.g., "09:00") in `availability` simplifies frontend parsing but sacrifices time arithmetic in SQL—likely a trade-off favoring application-level logic.

## Potential improvements

1. **Add RLS policies in Supabase** — Currently, row-level security is not enforced in the schema. For example, any user could query another professional’s `clientNotes` if authentication is misconfigured. Implementing RLS in Supabase (via SQL policies) would harden data isolation. See `src/lib/db/schema.ts`.

2. **Normalize `paymentType` and `status` fields** — String literals like `"upfront"` or `"pending_confirmation"` are prone to typos. Introducing a `payment_types` or `statuses` lookup table—or using PostgreSQL enums—would ensure consistency. See `professionals.paymentType` and `appointments.status`.

3. **Index `appointments.startsAt` for time-range queries** — While indexed, the current `appointments_starts_at_idx` is a basic B-tree. For time-based queries (e.g., “today’s appointments”), a BRIN index might be more efficient given the append-heavy write pattern. Consider tuning in `src/lib/db/schema.ts`.
