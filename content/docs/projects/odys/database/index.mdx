# Database

The Odys application relies on a PostgreSQL database hosted by Supabase, with its schema defined and managed using Drizzle ORM. This setup provides a robust, type-safe, and developer-friendly interface for interacting with the underlying data store, ensuring data integrity and simplifying complex queries.

## Overview

Odys' database schema is composed of 10 distinct tables, meticulously designed to support the core functionalities of connecting professionals with their clients, managing appointments, and facilitating communication. Drizzle ORM acts as the bridge between the TypeScript application code and the PostgreSQL database, allowing developers to define the schema using familiar TypeScript syntax and generate SQL migrations. The `src/lib/db/index.ts` file initializes the Drizzle client, connecting to the Supabase-hosted PostgreSQL instance using the `postgres-js` driver. The schema itself is comprehensively defined in `src/lib/db/schema.ts`.

## Drizzle ORM Configuration

The `drizzle.config.ts` file serves as the central configuration for Drizzle Kit, the command-line tool used for generating and applying database migrations. It specifies that the schema definitions are located in `./src/lib/db/schema.ts` and that generated migrations should be output to the `./drizzle` directory. The configuration explicitly sets the dialect to `postgresql` and retrieves the database connection URL from the `DATABASE_URL` environment variable, which is loaded via `dotenv` from `.env.local`. This separation of concerns ensures that database credentials are not hardcoded and can be easily managed across different environments.

## Schema Definition

The heart of the database interaction lies within `src/lib/db/schema.ts`, where all 10 tables are defined using Drizzle's `pgTable` function. Each table is structured with specific columns, types, constraints, and indexes to optimize data storage and retrieval.

### Professionals

The `professionals` table, with its 32 columns, is arguably the most central entity, representing the service providers on the platform. It stores a wealth of information, from basic contact details like `name`, `phone`, and `email`, to professional specifics such as `profession`, `bio`, and `avatarUrl`. Crucially, it links to the Supabase authentication system via `userId`, which is marked as `notNull` and `unique`, ensuring each professional account is tied to a single user. The `slug` column, also `notNull` and `unique`, is designed to create human-readable URLs for public profiles, like `odys.com.br/p/dr-ana`.

Payment preferences are detailed through `paymentType` (defaulting to "upfront") and `paymentPercentage` (defaulting to 100), `pixKeyType`, `pixKey`, and integration identifiers for Stripe (`stripeCustomerId`, `stripeSubscriptionId`). The table also tracks `trialEndsAt`, `welcomeMessage`, and `autoConfirm` settings.

For professionals offering in-person services, location details like `addressLine`, `neighborhood`, `city`, `state`, `postalCode`, `latitude`, and `longitude` are included. The `serviceType` column defaults to "both", indicating flexibility in service delivery. Timestamps `createdAt` and `updatedAt` are automatically managed with `defaultNow()`. The `professionals` table includes an `active` boolean column, defaulting to `true`, indicating profile visibility. To enhance query performance, indexes are defined on `profession` and `city`, allowing for efficient filtering of professionals by their specialty or location.

### Availability

The `availability` table captures the working hours of professionals. It includes `professionalId`, which is a foreign key referencing `professionals.id` with an `onDelete: "cascade"` action. This means that if a professional record is deleted, all their associated availability slots will also be automatically removed, maintaining data consistency. `dayOfWeek` (0 for Sunday, 6 for Saturday), `startTime`, and `endTime` define the specific time blocks. An index on `professionalId` facilitates quick lookups of a professional's schedule.

### Clients

The `clients` table stores information about the individuals who book services. It includes a nullable `userId` for clients who might have a Supabase account, but also allows for clients to be created without one, identified primarily by `name` and `phone`. Each client is associated with a `professionalId`, again with an `onDelete: "cascade"` reference to the `professionals` table. The `clients` table also includes nullable `email` and `avatarUrl` columns. Indexes on `professionalId` and `userId` support efficient client management.

### Client Notes

`clientNotes` provides a private space for professionals to record observations about their clients. It links to both `professionalId` and `clientId` with `onDelete: "cascade"` rules, ensuring notes are removed if either the professional or client is deleted. The `content` column stores the actual note, and `createdAt`/`updatedAt` track its lifecycle.

### Recurring Schedules

For ongoing client relationships, the `recurringSchedules` table defines repeating appointments. It links to `professionalId` and `clientId` with `onDelete: "cascade"`. Key columns include `frequency` (e.g., "weekly", "biweekly", "twice_weekly"), `dayOfWeek`, `secondDayOfWeek` (for twice-weekly schedules), and `time`. The `active` boolean, defaulting to `true`, allows for easy deactivation of a recurring schedule.

### Appointments

The `appointments` table records individual bookings. It references `professionalId`, `clientId` (both with `onDelete: "cascade"`), and optionally `recurringScheduleId`. The `startsAt` and `endsAt` timestamps define the appointment's duration. `status` tracks the appointment's lifecycle (e.g., "pending_confirmation", "confirmed", "rejected", "completed", "cancelled", "no_show"), defaulting to "pending_confirmation". `paymentStatus` (e.g., "none", "authorized", "captured", "refunded") and `stripePaymentIntentId` manage payment states. Boolean flags `reminderSent24h` and `reminderSent1h` track reminder notifications. The `appointments` table also includes a nullable `notes` column for private remarks. Indexes are provided for `professionalId`, `clientId`, `startsAt`, and `status` to support common queries like fetching a professional's upcoming appointments or filtering by status.

### Messages

The `messages` table facilitates communication between professionals and clients. It links to `professionalId` and `clientId` with `onDelete: "cascade"`. `sender` identifies who sent the message ("professional" or "client"), `type` (defaulting to "text") indicates the message format, and `content` holds the message body. `fileUrl` is available for attachments like PDFs, and `readAt` tracks when a message was viewed. A compound index on `professionalId` and `clientId` optimizes fetching message threads.

### Follows

The `follows` table tracks when a user (identified by `userId`) follows a professional (`professionalId`). The `professionalId` has an `onDelete: "cascade"` reference. Indexes on both `professionalId` and `userId` allow for efficient querying of followers or followed professionals.

### Reviews

After an appointment, clients can leave a review in the `reviews` table. It links to `professionalId`, `clientId`, and `appointmentId`, all with `onDelete: "cascade"`. The `appointmentId` is also `unique`, ensuring only one review per appointment. `rating` (1-5 integer) and `comment` capture the feedback. An index on `professionalId` helps aggregate reviews for a given professional.

### Notifications

The `notifications` table stores alerts for users. It includes `recipientId` (a Supabase user ID) and `recipientType` ("professional" or "client"). `type` categorizes the notification (e.g., "booking_request", "new_message"), while `title` and `body` provide the content. The `read` boolean, defaulting to `false`, tracks whether a notification has been seen. An index on `recipientId` allows for quick retrieval of a user's notifications.

## Design decisions

The choice of Drizzle ORM with `postgres-js` for Supabase Postgres was a deliberate one, prioritizing type safety and developer experience. Drizzle allows the entire database schema to be defined in TypeScript, providing compile-time checks and autocompletion that significantly reduce common SQL-related errors. This approach also keeps the schema definition close to the application code, making it easier to understand and evolve.

Using `postgres-js` as the underlying driver offers a direct, performant connection to the PostgreSQL database, avoiding the overhead of an additional query builder layer that some ORMs introduce. The `prepare: false` option in the `postgres` client initialization is a performance optimization, disabling prepared statements by default. This can be beneficial in environments where queries are highly dynamic or connection pooling is handled externally, though it might slightly increase parsing overhead for repeated identical queries.

The extensive use of `uuid` for primary keys across all tables provides globally unique identifiers, simplifying data merging and distributed system design, even if not immediately apparent in the current architecture. The `defaultRandom()` function for `uuid` columns ensures that the database automatically generates these unique IDs upon insertion.

Many columns, such as `profession`, `plan`, `paymentType`, `serviceType`, `frequency`, `status`, `paymentStatus`, `sender`, `type` (messages), `recipientType`, and `type` (notifications), are defined as `text` with default values. This offers flexibility, allowing for easy addition of new categories or states without requiring schema migrations. However, it relies on application-level validation to enforce valid values.

The widespread use of `onDelete: "cascade"` for foreign key relationships, particularly from `professionalId` to `professionals.id`, reflects a strong design choice for data integrity. When a professional's record is deleted, all associated data—their availability, clients, notes, recurring schedules, appointments, messages, follows, and reviews—are automatically removed. This prevents orphaned records and simplifies cleanup logic, but it also means that deleting a professional is a highly destructive operation.

## Potential improvements

1.  **Introduce Drizzle Enums for Categorical Data**: Many `text` columns like `profession`, `plan`, `paymentType`, `serviceType` in `professionals`, `frequency` in `recurringSchedules`, `status` and `paymentStatus` in `appointments`, `sender` and `type` in `messages`, and `recipientType` and `type` in `notifications` currently rely on string literals. While flexible, this approach lacks strict type enforcement at the database level. Defining these as Drizzle `pgEnum` types would create corresponding `ENUM` types in PostgreSQL, providing database-level validation and improving type safety within the application. This would make it impossible to insert an invalid `status` value, for example, without a database error.

2.  **Optimize Indexing for Common Queries**: The `appointments` table has indexes on `professionalId`, `clientId`, `startsAt`, and `status`. For queries that frequently filter appointments by a professional and then order them by start time (e.g., "get all upcoming appointments for professional X"), a compound index on `(professional_id, starts_at)` would be more efficient than separate indexes. Similarly, for `clients`, a unique compound index on `(professional_id, phone)` could prevent a professional from having multiple client records for the same phone number, even if the `userId` is null.

3.  **Refine Time and Date Storage**: The `startTime` and `endTime` columns in `availability`, and `time` in `recurringSchedules`, are stored as `text`. While this works for simple display, it can complicate time-based calculations, comparisons, and timezone handling. Consider using PostgreSQL's `time` type (if Drizzle supports it directly or via custom type) or storing these as `integer` representing minutes from midnight, which would simplify arithmetic operations and ensure consistent formatting.
