# Database Schema

The Odys application relies on a well-defined database schema to manage its core functionalities, from professional profiles and client appointments to messaging and reviews. This document provides a deep dive into the 10 tables that form the backbone of Odys, explaining the purpose of each, the rationale behind their design, and how they interact to support the platform's operations. Understanding this schema is fundamental to comprehending how data flows through the system and how various features are implemented.

## Overview

The Odys database schema is designed to facilitate the interaction between professionals and their clients. At its heart, the schema revolves around the `professionals` table, which stores comprehensive details about service providers. Clients, represented in the `clients` table, can then interact with these professionals by scheduling `appointments`, setting up `recurringSchedules`, and exchanging `messages`. The system also tracks `availability` for professionals, allows for `clientNotes` to be kept privately, and enables `reviews` after appointments. Social features like `follows` and system-wide `notifications` round out the schema, ensuring a connected and informed user experience.

The schema uses UUIDs as primary keys for most tables, generated randomly by default, which helps distribute writes and simplifies merging data from different sources if needed. Foreign key relationships, often with `onDelete: "cascade"`, ensure data integrity by automatically cleaning up dependent records when a parent record is removed.

## Professionals

The `professionals` table is arguably the most central entity, containing 32 columns that define a service provider on the Odys platform. Each professional is uniquely identified by a `uuid` `id` and linked to an external authentication system via a `userId` (a `text` field, which is `notNull` and `unique`). A human-readable `slug` (also `text`, `notNull`, and `unique`) is provided for public profile URLs, allowing for clean, memorable links like `odys.com.br/p/dr-ana`.

Beyond basic contact information like `name`, `phone`, and `email`, this table captures critical business logic:
*   `profession`: A `text` field indicating the professional's specialty (e.g., "psychologist", "personal_trainer").
*   `bio`: A `text` field for the professional's biography.
*   `avatarUrl`: A `text` field storing a link to the professional's profile picture.
*   `sessionDuration` and `sessionPrice`: `integer` fields with default values of 50 minutes and 0 cents respectively, defining the standard session parameters.
*   `plan`: A `text` field defaulting to "free", indicating the professional's subscription tier.
*   `paymentType` and `paymentPercentage`: These `text` and `integer` fields, defaulting to "upfront" and 100%, describe how professionals prefer to receive payments from clients.
*   `pixKeyType` and `pixKey`: `text` fields for handling Pix payments, a common payment method in Brazil.
*   `stripeCustomerId` and `stripeSubscriptionId`: `text` fields to integrate with Stripe for payment processing and subscription management.
*   `trialEndsAt`: A `timestamp` indicating the end of a trial period.
*   `welcomeMessage`: A `text` field for an automated message sent to new clients.
*   `autoConfirm`: A `boolean` defaulting to `false`, allowing professionals to automatically confirm appointments.
*   `serviceType`: A `text` field defaulting to "both", indicating if services are "in_person", "online", or "both".
*   Location details: `addressLine`, `neighborhood`, `city`, `state`, `postalCode`, `latitude`, and `longitude` (a `doublePrecision` type) capture the physical location for in-person services. These are often populated via geocoding services.
*   `active`: A `boolean` defaulting to `true`, controlling the visibility and availability of the professional.
*   `createdAt` and `updatedAt`: `timestamp` fields with `defaultNow()` to track record creation and modification.

The table includes indexes on `profession` and `city` to optimize search and filtering operations, which are crucial for clients looking for specific types of professionals in certain locations.

## Availability

The `availability` table defines when a professional is generally available for appointments. It has 5 columns, including its own `uuid` `id` and a `professionalId` (a `uuid` foreign key referencing `professionals.id` with `onDelete: "cascade"`). This cascade ensures that if a professional's profile is deleted, all their associated availability slots are also removed.

Key columns include `dayOfWeek` (an `integer` where 0 is Sunday and 6 is Saturday) and `startTime` and `endTime` (both `text` fields, e.g., "09:00" and "18:00"). Storing times as `text` offers flexibility but requires careful parsing and validation in the application layer. An index on `professionalId` speeds up queries for a professional's schedule.

## Clients

The `clients` table stores information about individuals who book appointments. It has 8 columns, with a `uuid` `id` as its primary key. Each client is linked to a `professionalId` (a `uuid` foreign key referencing `professionals.id` with `onDelete: "cascade"`), indicating which professional manages this client's record. This design implies that a client record is scoped to a specific professional, rather than a global client entity.

A `userId` (a `text` field) is present but `nullable`, reflecting the design choice that clients can book appointments without necessarily having a full Odys user account. `name`, `phone`, and `email` capture contact details, while `avatarUrl` stores a link to their profile picture. `createdAt` tracks when the client record was created. Indexes on `professionalId` and `userId` facilitate efficient lookups.

## Client Notes

The `clientNotes` table provides a private space for professionals to record observations about their clients. It contains 6 columns, including its `uuid` `id`. It links to both `professionalId` and `clientId` (both `uuid` foreign keys with `onDelete: "cascade"`), ensuring that notes are deleted if either the professional or the client record is removed. The `content` column (a `text` field) holds the actual note. `createdAt` and `updatedAt` timestamps track the note's lifecycle.

## Recurring Schedules

For clients who have regular appointments, the `recurringSchedules` table manages these patterns. It has 9 columns, with a `uuid` `id` and foreign keys to `professionalId` and `clientId` (both `uuid` with `onDelete: "cascade"`).

The `frequency` column (a `text` field) defines the recurrence pattern (e.g., "weekly", "biweekly", "twice_weekly"). `dayOfWeek` and `secondDayOfWeek` (both `integer` and `nullable`) specify the days, accommodating different frequencies. The `time` column (a `text` field like "14:00") indicates the appointment time. An `active` boolean, defaulting to `true`, allows for pausing or resuming recurring schedules. `createdAt` tracks the creation date.

## Appointments

The `appointments` table is central to the scheduling functionality, detailing each individual booking. It has 13 columns, with a `uuid` `id` as its primary key. It links to `professionalId` and `clientId` (both `uuid` foreign keys with `onDelete: "cascade"`). It also includes a `nullable` `recurringScheduleId` (a `uuid` foreign key referencing `recurringSchedules.id`), allowing individual appointments to be associated with a recurring series without cascading deletion.

Crucial time-based columns are `startsAt` and `endsAt` (both `timestamp` fields). The `status` column (a `text` field defaulting to "pending_confirmation") tracks the appointment's lifecycle, moving through states like "confirmed", "rejected", "completed", "cancelled", or "no_show". Similarly, `paymentStatus` (a `text` field defaulting to "none") tracks payment states: "authorized", "captured", or "refunded". `stripePaymentIntentId` stores the ID for Stripe transactions.

The `reminderSent24h` and `reminderSent1h` `boolean` fields, both defaulting to `false`, are used to track whether automated reminders have been dispatched. A `notes` field (a `text` field) allows for specific details about the appointment. `createdAt` tracks the booking time. Indexes on `professionalId`, `clientId`, `startsAt`, and `status` are provided to optimize common queries for scheduling and reporting.

## Messages

The `messages` table facilitates communication between professionals and clients. It has 9 columns, with a `uuid` `id` and foreign keys to `professionalId` and `clientId` (both `uuid` with `onDelete: "cascade"`). Clients are limited to sending one message at a time; the ability to send another message is released after the professional reads or replies.

The `sender` column (a `text` field) indicates whether the "professional" or "client" sent the message. `type` (a `text` field defaulting to "text") allows for different message formats like "text", "link", or "pdf". The `content` column (a `text` field) holds the message body, and `fileUrl` (a `text` field) is used for attachments like PDFs. `readAt` (a `timestamp`) tracks when a message was viewed, and `createdAt` records its sending time. An index on `(professionalId, clientId)` optimizes message retrieval for specific conversations.

## Follows

The `follows` table implements a simple "follow" feature, allowing users to express interest in professionals. It has 4 columns, with a `uuid` `id`. It links a `userId` (a `text` field, `notNull`) to a `professionalId` (a `uuid` foreign key with `onDelete: "cascade"`). `createdAt` tracks when the follow occurred. Indexes on `professionalId` and `userId` support efficient querying of followers or followed professionals.

## Reviews

The `reviews` table stores client feedback after appointments. It has 7 columns, with a `uuid` `id`. It links to `professionalId` and `clientId` (both `uuid` foreign keys with `onDelete: "cascade"`). Crucially, it also links to `appointmentId` (a `uuid` foreign key referencing `appointments.id` with `onDelete: "cascade"`), and this `appointmentId` is `unique`, ensuring that only one review can be submitted per appointment.

The `rating` column (an `integer`) stores a 1-5 star rating, and `comment` (a `text` field) allows for qualitative feedback. `createdAt` tracks when the review was submitted. An index on `professionalId` helps aggregate reviews for a given professional.

## Notifications

The `notifications` table manages system alerts and messages for users. It has 8 columns, with a `uuid` `id`. It uses a generic `recipientId` (a `text` field) and `recipientType` (a `text` field, e.g., "professional" or "client") to allow notifications to be sent to different types of users without direct foreign key dependencies on `professionals` or `clients` tables. This design choice offers flexibility but relies on application-level logic to resolve the `recipientId` to the correct user entity.

The `type` column (a `text` field) categorizes the notification (e.g., "booking_request", "reminder_24h", "new_message"). `title` and `body` (both `text` fields) contain the notification content. A `read` boolean, defaulting to `false`, tracks whether the notification has been viewed. `createdAt` records the notification's generation time. An index on `recipientId` allows for quick retrieval of all notifications for a specific user.

## Design Decisions

Several key design decisions shape the Odys database schema:

*   **UUID Primary Keys**: The consistent use of `uuid` for primary keys, generated with `defaultRandom()`, provides several benefits. It avoids sequential ID generation bottlenecks in distributed environments, makes it harder to guess record IDs, and simplifies data merging if the system were to scale horizontally or integrate with other systems.
*   **External User IDs (`text` type)**: The `userId` columns in `professionals`, `clients`, `follows`, and `notifications` are defined as `text` rather than `uuid` or a foreign key to an internal `users` table. This strongly suggests an integration with an external authentication provider, such as Supabase, where user IDs are typically strings. This design choice delegates user management and authentication concerns to a specialized service, simplifying the application's internal user model.
*   **`onDelete: "cascade"` for Core Relationships**: Many foreign keys, particularly those linking to `professionals.id` and `clients.id`, employ `onDelete: "cascade"`. This is a deliberate choice to simplify data cleanup. For instance, if a professional deletes their account, all their associated availability, clients, appointments, messages, and reviews are automatically removed by the database. While convenient, this requires careful consideration to prevent accidental data loss.
*   **String-based Enums**: Many categorical fields like `profession`, `plan`, `paymentType`, `serviceType`, `status` (in `appointments`), `paymentStatus`, `sender`, `type` (in `messages` and `notifications`), and `recipientType` are defined as `text` fields. This offers flexibility, allowing new categories to be added without schema migrations. However, it shifts the responsibility for validation and consistency to the application layer, as the database itself doesn't enforce a predefined set of values.
*   **Denormalization for `professionalId` and `clientId`**: The `professionalId` and `clientId` foreign keys are present in almost every related table (`availability`, `clients`, `clientNotes`, `recurringSchedules`, `appointments`, `messages`, `follows`, `reviews`). This denormalization strategy allows for direct access to records scoped by a professional or client without requiring multiple joins, which can improve query performance for common access patterns.
*   **`createdAt` and `updatedAt` Timestamps**: The inclusion of `createdAt` and `updatedAt` columns with `defaultNow()` is a standard practice for auditing and tracking the lifecycle of records. While `createdAt` is correctly set on insertion, `updatedAt` typically requires a database trigger to automatically update on every modification, which is not explicitly defined in the Drizzle schema but would be a common addition in a production environment.

## Potential Improvements

1.  **Introduce `pgEnum` for Categorical Fields**: Many `text` columns like `professionals.profession`, `professionals.plan`, `appointments.status`, `messages.sender`, and `notifications.type` represent a finite set of values. Using Drizzle's `pgEnum` feature would allow these to be defined as actual PostgreSQL enum types. This would enforce data integrity at the database level, preventing invalid values from being inserted, and provide better type safety in the application code.
2.  **Add Unique Constraint to `follows` Table**: The `follows` table currently allows a `userId` to follow the same `professionalId` multiple times. Adding a unique constraint on the combination of `(userId, professionalId)` would prevent duplicate follow entries, ensuring that each user can only follow a professional once. This would be defined within the `follows` table's Drizzle schema definition.
3.  **Implement Database Triggers for `updatedAt`**: While `updatedAt` columns have `defaultNow()`, this only applies on initial record creation. For `updatedAt` to truly reflect the last modification time, a database trigger is typically required. This trigger would automatically update the `updatedAt` column to `NOW()` whenever a row is modified. This is a common pattern that enhances data auditing without requiring application-level code to manually update the timestamp on every write.
4.  **Consider a Dedicated `users` Table**: The current schema uses `userId` (a `text` field) in `professionals`, `clients`, `follows`, and `notifications` without an explicit foreign key to a central `users` table. While this works with external authentication, introducing an internal `users` table (even if it primarily stores the external `userId` and a few other core attributes) could centralize user management within the database. This would allow for explicit foreign key constraints on `userId` columns, improving referential integrity and making it easier to query user-related data across different entities.
