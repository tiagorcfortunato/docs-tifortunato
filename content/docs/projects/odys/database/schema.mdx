# Database Schema: The 10 tables in detail

Understanding the database schema is essential to grasping how Odys orchestrates professional-client interactions, scheduling, payments, and communication. This document dives into the 10 core tables that power the platform, revealing how data flows and interconnects across features.

## Overview

The Odys schema consists of **10 tables** that model professionals, their availability, client relationships, appointments, and auxiliary data like messages and reviews. Every table uses UUIDs for primary keys and leverages Drizzle ORM with PostgreSQL. Foreign keys enforce referential integrity, and cascading deletes ensure data consistency when a professional is removed. Timestamps with defaults (`createdAt`, `updatedAt`) are standard, and indexes optimize common query patterns—especially around `professionalId`, `clientId`, and temporal fields.

## Core Tables and Their Roles

The `professionals` table anchors the system, storing key profile data like `name`, `slug`, `sessionPrice`, and `plan`. It links to Stripe via `stripeCustomerId` and `stripeSubscriptionId`, supports Pix payments through `pixKeyType` and `pixKey`, and tracks trial status with `trialEndsAt`. The `userId` field ties each professional to Supabase auth.

Scheduling is managed through `availability`, which defines daily time slots per professional using `dayOfWeek`, `startTime`, and `endTime`. Appointments are stored in `appointments`, capturing `startsAt`, `endsAt`, `status`, and `paymentStatus`, with optional linkage to `recurringSchedules`. The latter supports weekly, biweekly, or twice-weekly patterns, enabling automated session planning.

Client management spans `clients`, `clientNotes`, and `follows`. Clients are tied to professionals and may have a `userId` if registered. Notes are private, indexed by professional and client. Follows track which users are interested in a professional’s availability.

Communication is handled by `messages`, allowing message exchange with `sender` and `readAt` tracking. `reviews` are tied to completed appointments, ensuring one review per session via the unique `appointmentId` constraint. Finally, `notifications` deliver alerts to `recipientId` (Supabase user ID) with a `read` flag for state tracking.

## Design decisions

The schema prioritizes data integrity and query efficiency. Cascading deletes (`onDelete: "cascade"`) prevent orphaned records—critical when a professional leaves. Default values (e.g., `autoConfirm`, `active`) reduce client-side logic. Text fields store enums like `status` and `paymentType`, favoring flexibility over rigid constraints. Indexes on foreign keys and frequently filtered columns (e.g., `appointments.startsAt`) support performance at scale. The use of `timestamp` with `defaultNow()` ensures consistent time tracking.

## Potential improvements

1. **Add `updatedAt` to `appointments`** — While most tables have `updatedAt`, `appointments` lacks it. This makes auditing state changes difficult. Adding it in `src/lib/db/schema.ts` would improve traceability of status or payment updates.

2. **Normalize `frequency` in `recurringSchedules`** — The `frequency` field uses text values like "weekly". Introducing a lookup table or constraint would prevent inconsistencies. This could be enforced in the same schema file.

3. **Index `notifications.read`** — With high notification volume, filtering unread items will become slow. Adding an index on `read` in `notifications` (in `schema.ts`) would optimize user-facing queries.
