# Database Schema

The Odys platform connects professionals and clients through a robust booking and communication system. This schema forms the backbone of that interaction, modeling everything from availability to payments. Understanding its structure is key to extending features safely and efficiently.

## Overview

The database consists of **11 tables**, each serving a distinct domain concern. At the center is the `professionals` table, which anchors relationships to `availability`, `clients`, `appointments`, and more. Most child tables cascade-delete on professional removal, ensuring data integrity when a user leaves the platform. Indexes are thoughtfully applied to foreign keys and query-heavy fields like `status` and `startsAt`, while unique constraints prevent double-booking and duplicate reviews.

## Core Entities

The `professionals` table stores all practitioner data — 32 columns including contact info, service preferences (`serviceType`, `sessionDuration`), and payment configuration (`paymentType`, `pixKey`). It uses `userId` as a non-primary link to Supabase auth, allowing external identity management. Geolocation fields (`latitude`, `longitude`) support in-person bookings, while `slug` enables public profile URLs. The `plan` column defaults to `free`. The `sessionDuration` defaults to 50 minutes. The `paymentType` defaults to `upfront`.

Linked to professionals, `availability` defines weekly time slots via `dayOfWeek`, `startTime`, and `endTime`. This granular approach allows flexible scheduling without time-zone complications. Meanwhile, `clients` tracks individuals managed by professionals, with optional `userId` support for unauthenticated clients.

## Relationships and Workflow

Appointments tie these together: `appointments` references both `professionalId` and `clientId`, with `startsAt`/`endsAt` defining the session window. A partial unique index on `(professionalId, startsAt)` where status is non-terminal prevents overlapping confirmed bookings — a critical guardrail implemented at the database level. The `status` defaults to `pending_confirmation`.

Recurring patterns are handled by `recurringSchedules`, which supports weekly, biweekly, or twice-weekly cadences. Appointments can link back via `recurringScheduleId`, enabling automated series management. Communication occurs through `messages`, where `sender` distinguishes between parties and `readAt` tracks engagement.

## Supporting Features

Auxiliary tables enhance the experience: `clientNotes` stores private practitioner observations; `reviews` captures post-session feedback with a unique constraint on `appointmentId` to prevent duplicates. `follows` enables client tracking of professionals, while `notifications` delivers timely updates across multiple types like reminders or payment confirmations.

Finally, `agentMetrics` logs telemetry from the WhatsApp intake agent — `totalLatencyMs`, `toolCallCount`, and `outcome` help measure performance and conversion, with `traceId` enabling traceability across system boundaries.

## Design decisions

The schema favors application-level logic with database safeguards: for example, appointment conflicts are checked in code but enforced via a partial unique index. This balances flexibility with correctness. Text fields are used extensively (e.g., `status`, `type`) for simplicity, avoiding enum tables that complicate migrations. Cascading deletes ensure orphaned data doesn’t persist when a professional is removed.

## Potential improvements

In `schema.ts`, the `appointments` table could benefit from a check constraint ensuring `endsAt > startsAt`, preventing invalid time ranges at the database level.

The `providersUsed` column in `agentMetrics` is defined as `text.array()` but the Drizzle definition uses `text("providers_used").array()` — this may not serialize correctly without a custom codec; consider using a JSON column instead.

The `notifications.recipientType` allows "professional | client" but lacks a constraint; adding a check or domain would prevent invalid values and improve query reliability.
