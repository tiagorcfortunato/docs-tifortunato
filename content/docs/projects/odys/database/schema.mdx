# Database Schema

This page details the ten core tables defined in `src/lib/db/schema.ts`, which constitute the Odys database schema. All tables use UUID primary keys generated via `defaultRandom()` and timestamps managed with `defaultNow()`. Foreign key relationships are configured with `onDelete: "cascade"`, ensuring that deletion of a `professionals` record cascades to all associated records.

## Professionals

The `professionals` table (`src/lib/db/schema.ts:10`) is the central entity, representing professional users.

- `userId`: `text`, not null, unique — stores the Supabase authentication ID.
- `slug`: `text`, not null, unique — used in public profile URLs.
- `sessionDuration`: `integer`, default `50` — session length in minutes.
- `sessionPrice`: `integer`, default `0` — price in cents.
- `plan`: `text`, default `"free"` — subscription tier (`free`, `pro`, `premium`).
- `trialEndsAt`: `timestamp` — end of Pro trial period.
- `stripeCustomerId`, `stripeSubscriptionId`: `text` — Stripe billing identifiers.
- `paymentType`: `text`, default `"upfront"` — payment method (`upfront`, `percentage`, `after`).
- `paymentPercentage`: `integer`, default `100` — percentage charged upfront if `paymentType` is `percentage`.
- `pixKeyType`, `pixKey`: `text` — PIX payment details.
- `autoConfirm`: `boolean`, default `false` — whether appointments are auto-confirmed.
- `active`: `boolean`, default `true` — active status.
- Indexes: none explicitly defined beyond primary key and unique constraints.

## Availability

The `availability` table (`src/lib/db/schema.ts:39`) defines recurring weekly availability for professionals.

- `professionalId`: `uuid`, not null, references `professionals.id` with `onDelete: "cascade"`.
- `dayOfWeek`: `integer`, not null — 0 (Sunday) to 6 (Saturday).
- `startTime`, `endTime`: `text`, not null — stored as `"HH:mm"`.
- Index: `availability_professional_id_idx` on `professionalId`.

## Clients

The `clients` table (`src/lib/db/schema.ts:51`) stores client data scoped to a professional.

- `userId`: `text`, nullable — Supabase ID, back-filled if client registers later.
- `professionalId`: `uuid`, not null, references `professionals.id` with `onDelete: "cascade"`.
- `name`, `phone`, `email`: `text`, not null.
- `avatarUrl`: `text` — optional.
- Indexes: `clients_professional_id_idx` on `professionalId`, `clients_user_id_idx` on `userId`.

## Client Notes

The `clientNotes` table (`src/lib/db/schema.ts:66`) stores private notes professionals keep on clients.

- `professionalId`: `uuid`, not null, references `professionals.id` with `onDelete: "cascade"`.
- `clientId`: `uuid`, not null, references `clients.id` with `onDelete: "cascade"`.
- `content`: `text`, not null.
- `createdAt`, `updatedAt`: `timestamp`, not null, defaults to `defaultNow()`.
- Indexes: none beyond primary key.

## Recurring Schedules

The `recurringSchedules` table (`src/lib/db/schema.ts:77`) defines recurring appointment patterns.

- `professionalId`: `uuid`, not null, references `professionals.id` with `onDelete: "cascade"`.
- `clientId`: `uuid`, not null, references `clients.id` with `onDelete: "cascade"`.
- `frequency`: `text`, not null — values like `weekly`, `biweekly`, `twice_weekly`.
- `dayOfWeek`: `integer` — day for weekly/biweekly schedules.
- `secondDayOfWeek`: `integer` — second day for `twice_weekly`.
- `time`: `text`, not null — time in `"HH:mm"`.
- `active`: `boolean`, default `true`.
- Indexes: none beyond primary key.

## Appointments

The `appointments` table (`src/lib/db/schema.ts:93`) stores individual booking instances.

- `professionalId`: `uuid`, not null, references `professionals.id` with `onDelete: "cascade"`.
- `clientId`: `uuid`, not null, references `clients.id` with `onDelete: "cascade"`.
- `recurringScheduleId`: `uuid`, nullable, references `recurringSchedules.id`.
- `startsAt`, `endsAt`: `timestamp`, not null.
- `status`: `text`, default `"pending_confirmation"` — lifecycle state.
- `paymentStatus`: `text`, default `"none"` — payment flow state.
- `stripePaymentIntentId`: `text` — present but unused (subscription-only billing).
- `reminderSent24h`, `reminderSent1h`: `boolean`, default `false` — prevent duplicate reminders.
- `notes`: `text` — optional.
- Indexes: `appointments_professional_id_idx`, `appointments_client_id_idx`, `appointments_starts_at_idx`, `appointments_status_idx`.

## Messages

The `messages` table (`src/lib/db/schema.ts:115`) stores in-product chat between professionals and clients.

- `professionalId`: `uuid`, not null, references `professionals.id` with `onDelete: "cascade"`.
- `clientId`: `uuid`, not null, references `clients.id` with `onDelete: "cascade"`.
- `sender`: `text`, not null — `"professional"` or `"client"`.
- `type`: `text`, default `"text"` — `"text"`, `"link"`, `"pdf"`.
- `content`: `text`, not null.
- `fileUrl`: `text` — for file attachments like PDFs.
- `readAt`: `timestamp`, nullable.
- Index: `messages_professional_client_idx` on `(professionalId, clientId)`.

## Follows

The `follows` table (`src/lib/db/schema.ts:135`) captures the social graph of clients following professionals.

- `userId`: `text`, not null — Supabase user ID of the follower.
- `professionalId`: `uuid`, not null, references `professionals.id` with `onDelete: "cascade"`.
- Indexes: `follows_professional_id_idx` on `professionalId`, `follows_user_id_idx` on `userId`.

## Reviews

The `reviews` table (`src/lib/db/schema.ts:147`) stores client feedback on completed appointments.

- `professionalId`: `uuid`, not null, references `professionals.id` with `onDelete: "cascade"`.
- `clientId`: `uuid`, not null, references `clients.id` with `onDelete: "cascade"`.
- `appointmentId`: `uuid`, not null, unique, references `appointments.id` with `onDelete: "cascade"` — enforces one review per appointment.
- `rating`: `integer`, not null — 1 to 5.
- `comment`: `text` — optional.
- Index: `reviews_professional_id_idx` on `professionalId`.

## Notifications

The `notifications` table (`src/lib/db/schema.ts:163`) powers the in-app notification feed.

- `recipientId`: `text`, not null — Supabase user ID.
- `recipientType`: `text`, not null — `"professional"` or `"client"`.
- `type`: `text`, not null — e.g., `booking_request`, `reminder_24h`, `new_message`.
- `title`, `body`: `text`, not null — notification content.
- `read`: `boolean`, default `false`.
- Index: `notifications_recipient_id_idx` on `recipientId`.

## Known Gaps

The schema does not use Postgres views, computed columns, triggers, or Row Level Security (RLS) policies. All derived data (e.g., revenue, next available slot) is computed in application code, primarily via Drizzle queries in API routes or server components. This centralizes logic in the application layer but requires developers to manually ensure consistency and performance for complex queries that could benefit from database-level optimizations.

## Why this shape

The schema prioritizes simplicity and application-centric logic. UUIDs with `defaultRandom()` ensure unguessable identifiers, while `onDelete: "cascade"` guarantees clean data deletion (e.g., GDPR compliance). By avoiding advanced database features, the design reduces operational complexity and keeps business logic in TypeScript, where most development occurs. This trade-off favors developer velocity and debugging clarity over potential database-level performance gains.
