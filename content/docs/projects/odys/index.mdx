# Odys Overview

This page provides a high-level overview of Odys, including its purpose, target users, core functionality, current state, and documentation structure.

Odys is a scheduling SaaS designed for **Brazilian independent professionals** in health, fitness, wellness, beauty, education, and creative services — such as psychologists, personal trainers, hairdressers, estheticians, nutritionists, and photographers. It replaces the common practice of managing appointments via chaotic WhatsApp threads.

## The Problem

Most Brazilian independent professionals rely on WhatsApp to manage their schedules. Clients message directly to book sessions, leading to fragmented communication, missed payments, manual reminders, and no centralized calendar or analytics. Existing tools like Calendly or Simples Agenda fail to address local needs:

- Overly complex for solo practitioners
- Lack native integration with WhatsApp, the primary communication channel
- Built for English-speaking corporate markets, not Brazilian realities like PIX payments

## What Odys Does

Odys provides a complete operational stack:

- **Public booking page** at `/p/[slug]` — clients select slots and submit details without friction
- **AI WhatsApp agent** — enables autonomous booking via chat using Groq LLM tool-calling
- **Professional dashboard** — centralized interface to manage appointments, clients, and messages
- **Transactional WhatsApp messages** — automated reminders, confirmations, and alerts sent from a single corporate number via Evolution API
- **Stripe subscription billing** — professionals pay for Basic, Pro, or Premium plans
- **PIX payment flow** — supports Brazilian-native off-Stripe client-to-professional payments

## Scope of Professions Supported

Odys supports 29 professions across 6 categories, defined in `src/lib/professions.ts`:

- **Health**: psychologist, physiotherapist, nutritionist, dentist, therapist, speech therapist
- **Fitness**: personal trainer, yoga instructor, pilates instructor, dance instructor
- **Wellness**: massage therapist, acupuncturist, holistic therapist, osteopath
- **Beauty**: hairdresser, barber, manicure, esthetician, makeup artist, tattoo artist, eyebrow designer, lash designer
- **Education**: tutor, music instructor, language instructor, coach
- **Creative**: photographer, personal stylist, personal organizer

The data model is strictly per-individual-professional — there is no `salons` table. A salon owner would onboard each professional separately.

## The Corporate-Number Architecture

Odys operates a **single corporate WhatsApp number** (via Evolution API) for all outbound and inbound interactions:

- **Outbound**: reminders, confirmations, AI replies, and professional alerts originate from this number
- **Inbound**: client and professional messages are routed through `/api/whatsapp/webhook` to the AI intake agent
- **Reply routing**: context is preserved using a Redis key `last_outbound:{phone}`, allowing the AI to identify which professional or appointment a reply concerns
- **Conversation-aware delivery**: the reminder cron skips pushes if a client is mid-conversation with the AI (\<15 min idle), preventing interleaved notifications

This architecture ensures professionals' personal numbers remain private, while enabling scalable, brand-consistent communication. The system is designed to transition to Meta's WhatsApp Business Cloud API when volume demands it.

## Current State (2026-04-23)

- **Deployed** at [odys.com.br](https://odys.com.br) — launch-ready
- **Zero paying customers** — intentionally pre-launch
- Core systems are production-hardened: Drizzle ORM with Supabase Postgres, Stripe + PIX payments, active security audit (15 findings in progress), three-runtime Sentry observability, and an MCP server exposing the database to AI agents

When documentation states "the system handles X," it means the feature is implemented and tested end-to-end — but not yet validated under real user load.

## Known Gaps

- **Data Residency**: Current infrastructure likely hosts data in the US (Sentry DSN points to `*.ingest.us.sentry.io`, PostHog defaults to `https://us.i.posthog.com`, Vercel defaults to `iad1`). For LGPD compliance, migrating Supabase to `sa-east-1` and Vercel to `gru1` is a future requirement. This is an honest flag regarding data sovereignty for the target market.
