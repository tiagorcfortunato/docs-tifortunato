# Security Fixes

This page details critical security and validation improvements across the Odys platform, addressing vulnerabilities identified in recent audits and hardening key user flows. These changes strengthen data integrity, prevent abuse, and ensure robust handling of user input across registration, onboarding, and appointment management.

## Overview

The platform consists of 11 core database tables, including `professionals`, `clients`, and `appointments`, backed by 23 API routes and governed by rate limits and Stripe webhooks. Recent fixes focus on securing high-risk endpoints—particularly `/api/auth/register`, `/api/onboarding`, and `/api/appointments/[id]`—by introducing strict Zod validation, slug reservation, and action whitelisting. These changes respond to findings like unvalidated OAuth `type` parameters and silent failure on unknown actions, ensuring the system fails securely and predictably.

## Input Validation with Zod

All major user-facing endpoints now use Zod for schema validation, replacing ad-hoc checks. In `/api/auth/register/route.ts`, the `registerSchema` enforces email format and 8-character passwords, aligning with Supabase’s server-side rules. Invalid payloads return 400 with structured error details via `parsed.error.flatten().fieldErrors`. Similarly, `/api/onboarding/route.ts` uses `onboardingSchema` to validate professional details, including time format (`^\d{2}:\d{2}$`) for availability. This shift prevents malformed data from entering the system and gives users clear feedback.

## Reserved Slug Protection

During onboarding, the `generateUniqueSlug` function in `/api/onboarding/route.ts` prevents URL collisions by checking against a `RESERVED_SLUGS` set containing 16 critical paths like `api`, `auth`, and `p`. If a professional’s name (e.g., "Admin") would produce a conflicting slug, the system appends a numeric suffix (e.g., "admin-2"). This ensures user profiles cannot shadow application routes, mitigating potential access confusion or spoofing.

## Action Whitelisting in Appointments

The `/api/appointments/[id]/route.ts` endpoint now enforces a strict `ALLOWED_ACTIONS` set (`confirm`, `reject`, `cancel`, etc.). Previously, typos like `action: "cancle"` would default to rejection due to loose conditionals. Now, an authorized user sending an unrecognized action triggers a 400 response, while unauthorized users receive a 403, avoiding information leakage. This change ensures only intended state transitions occur and makes API behavior explicit.

## Design decisions

The code prioritizes defense in depth: Zod validation runs early, rate limits apply per-IP, and database operations use parameterized queries via Drizzle. The `RESERVED_SLUGS` list is hardcoded for simplicity and speed, avoiding DB lookups during onboarding. Action whitelisting is placed after authentication checks to prevent distinguishing between invalid actions and unauthorized access. Using `Set` for `ALLOWED_ACTIONS` and `VALID_TYPES` ensures O(1) lookups, critical in high-frequency paths.

## Potential improvements

1. **Centralize Zod schemas** — The `onboardingSchema` in `/api/onboarding/route.ts` duplicates validation logic that could be shared with the frontend. Extracting it to `lib/zod/schemas.ts` would ensure consistency and reduce drift.

2. **Enhance slug uniqueness** — `generateUniqueSlug` in `/api/onboarding/route.ts` uses a linear suffix (`-2`, `-3`), which could be slow under high contention. Consider using a random suffix or UUID fragment for better performance.

3. **Log invalid actions** — In `/api/appointments/[id]/route.ts`, unknown actions return 400 but aren’t logged. Adding a `logger.warn("unknown_action", { action, appointmentId: id, userId: user.id })` would help detect abuse patterns.
