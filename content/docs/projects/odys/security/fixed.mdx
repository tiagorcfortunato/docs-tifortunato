# Security Fixes

This document details several recent security enhancements implemented across the Odys platform. These changes address specific vulnerabilities and improve the overall integrity and reliability of the application, particularly concerning user input, route handling, and API action processing. The fixes span critical areas, from user registration and onboarding flows to core API interactions, ensuring that the system behaves predictably and securely even when faced with malformed or malicious requests.

## Overview

The Odys platform relies on a database schema comprising 10 distinct tables, managing entities such as `professionals`, `clients`, `appointments`, and `notifications`. Interactions with this data are primarily facilitated through 23 API routes, which serve as the primary interface for client applications. To safeguard these interactions, the system employs various security measures, including rate limiting on critical endpoints like `/api/onboarding` and `/api/auth/register`, which are configured with specific limits and windows, such as 3 requests per hour for onboarding and 5 requests per hour for registration.

Recent updates have focused on hardening several key areas:
1.  **Input Validation**: Implementing Zod schemas for robust validation of incoming data, particularly in user registration and professional onboarding.
2.  **Reserved Slugs**: Preventing user-generated slugs from conflicting with core application routes.
3.  **Explicit Unknown Action Handling**: Ensuring API endpoints respond appropriately to unrecognized actions, rather than silently failing or misinterpreting requests.
4.  **Auth Callback Parameter Validation**: Strictly validating parameters passed during authentication callbacks to prevent arbitrary data injection.

These improvements collectively reduce the attack surface, enhance data integrity, and provide clearer error feedback to clients.

## Input Validation with Zod

A significant enhancement to input validation has been the adoption of Zod, a TypeScript-first schema declaration and validation library. This change is evident in critical user-facing endpoints, such as `src/app/api/auth/register/route.ts` and `src/app/api/onboarding/route.ts`.

In `src/app/api/auth/register/route.ts`, the `registerSchema` defines the expected structure and constraints for new user registrations. It mandates a valid `email` format, a `password` with a minimum length of 8 characters (matching Supabase's default, avoiding unnecessary divergence). Optional fields include `name`, which must be at least 2 characters long, and `type`, which must be one of 'client' or 'professional'. The use of `z.string().min(8)` for the password ensures basic complexity requirements are met before the request even reaches the authentication service. When a request body fails to conform to this schema, the API now explicitly returns a `400` status code along with detailed `fieldErrors` from `parsed.error.flatten()`, providing precise feedback on what went wrong. This prevents malformed data from proceeding further into the system, reducing potential attack vectors and improving the developer experience.

Similarly, `src/app/api/onboarding/route.ts` utilizes `onboardingSchema` to validate the data submitted by professionals during their initial setup. This schema is considerably more detailed, covering fields like `name`, `profession`, `phone`, `bio`, `sessionDuration`, `sessionPrice`, `serviceType`, `city`, `state`, and `availability`. For instance, `sessionDuration` is validated as an integer, positive, and capped at a maximum of 480 minutes, while `sessionPrice` must be a non-negative integer. The `availability` array, representing a professional's working hours, also undergoes rigorous validation, requiring a minimum of one entry and ensuring each entry has a valid `dayOfWeek` (0-6) and correctly formatted `startTime` and `endTime` strings (`^\d{2}:\d{2}$`). This comprehensive validation prevents the insertion of invalid or out-of-bounds data into the `professionals` and `availability` tables, maintaining data quality and preventing application logic errors down the line.

## Reserved Slugs

The `src/app/api/onboarding/route.ts` file also introduces a mechanism to prevent user-generated slugs from clashing with existing or future application routes. The `RESERVED_SLUGS` set explicitly lists paths like `"api"`, `"auth"`, `"dashboard"`, `"login"`, `"register"`, and others. This is a proactive measure to ensure that a professional's public profile URL, generated from their name, does not inadvertently shadow a critical application endpoint.

The `toSlug` function normalizes a professional's name into a URL-friendly format. The subsequent `generateUniqueSlug` function then checks if the generated slug is either empty (e.g., if a name consists only of diacritics that are stripped) or if it collides with any of the `RESERVED_SLUGS`. If a collision or an empty slug occurs, or if the slug already exists in the `professionals` table, the system automatically appends a numeric suffix (e.g., `user-2`, `john-doe-3`) to create a unique and non-conflicting identifier. This design prevents potential routing issues, ensures the integrity of application URLs, and mitigates risks associated with URL hijacking or content confusion.

## Explicit Unknown Action Handling

In `src/app/api/appointments/[id]/route.ts`, the `PATCH` endpoint for managing appointments now includes explicit handling for unknown actions. Previously, an unrecognized `action` parameter could lead to unexpected behavior, such as a typo like `"cancle"` silently falling through conditional checks and resulting in an unintended state change (e.g., rejecting an appointment).

The `ALLOWED_ACTIONS` set defines a strict whitelist of permissible actions: `"confirm"`, `"reject"`, `"cancel"`, `"paid"`, `"complete"`, and `"no_show"`. Before any state modification, the code now checks `if (!ALLOWED_ACTIONS.has(action))`. If the provided action is not in this set, the API immediately returns a `400` status code with an `"Unknown action"` error message. This "fail-safe" approach ensures that only explicitly defined and understood actions can modify appointment states, preventing logic errors and potential security vulnerabilities arising from unexpected inputs. The `403` for unauthorized users is returned when an unknown action is received; authorized users with an unknown action receive a `400`. This placement ensures that an unauthorized caller receives a `403` before a `400` might reveal the existence of an appointment, avoiding an "existence oracle".

## Auth Callback `type` Parameter Validation

The `src/app/auth/callback/route.ts` endpoint, responsible for handling Supabase authentication redirects, has been hardened to prevent arbitrary data injection via the `type` query parameter. Previously, the raw value of the `type` parameter from the URL could be written directly into the `user_metadata` of a Supabase session. This presented a vulnerability where an attacker could craft an OAuth callback URL to inject arbitrary JSON into a user's session metadata (e.g., `type=admin`, `type=<script>`, or `type=<big-string>`), potentially leading to privilege escalation or denial-of-service through excessive data storage.

To address this, a `VALID_TYPES` set has been introduced, containing only `"professional"`, `"client"`, and `"recovery"`. The `parseType` function now strictly filters the incoming `type` query parameter. If the raw value is not present in `VALID_TYPES`, it is ignored, and `null` is returned. This ensures that only a closed set of predefined and safe values can ever be written to `user_metadata`, effectively closing the injection vector and maintaining the integrity of user session data.

## Design decisions

The design decisions behind these fixes prioritize security, data integrity, and maintainability:

*   **Zod for API Input Validation**: Choosing Zod provides a declarative, type-safe way to define expected data shapes. This not only catches malformed requests early but also generates TypeScript types, improving developer productivity and reducing runtime errors. The trade-off is the introduction of an external dependency and a slight increase in boilerplate for schema definitions, but the benefits in terms of code clarity and error handling are substantial.
*   **Explicit Allow-listing for Actions**: The `ALLOWED_ACTIONS` set in the appointments API is a classic security pattern: "deny by default, allow by exception." This is inherently more secure than a "permit by default" approach with chained `if/else` statements, as it prevents new, unhandled actions from accidentally being processed. This design choice makes the API's behavior predictable and easier to audit.
*   **Reserved Slugs Strategy**: Proactively defining `RESERVED_SLUGS` prevents user-generated content from interfering with core application routing. This foresight avoids complex refactoring later if a user's chosen slug clashes with a new feature's URL. The `generateUniqueSlug` function handles collisions gracefully, ensuring a unique and valid slug is always created.
*   **Supabase Admin Client in Development**: The lazy initialization of `_adminClient` and its conditional use based on `NODE_ENV` is a pragmatic trade-off. In production, `supabase.auth.signUp()` is used to trigger email verification, which is a critical security step. In development, `admin.createUser` with `email_confirm: true` bypasses this for faster testing cycles. This acknowledges the different needs of development and production environments while maintaining security in the live system.

## Potential improvements

While these fixes significantly enhance the platform's security posture, several areas present opportunities for further refinement:

1.  **Centralized Input Validation Middleware**: Currently, Zod schemas are defined and applied within individual route handlers (e.g., `src/app/api/auth/register/route.ts`, `src/app/api/onboarding/route.ts`). As the application grows to 23 API routes, consider implementing a centralized validation middleware or utility that can apply Zod schemas to incoming requests before they reach the main handler logic. This would reduce duplication, enforce consistent validation practices across all endpoints, and make it easier to update or extend validation rules.
2.  **Dynamic Reserved Slugs**: The `RESERVED_SLUGS` set in `src/app/api/onboarding/route.ts` is hardcoded. While effective, it requires manual updates as new top-level routes are added. An improvement could involve dynamically generating this list from the application's routing configuration or maintaining it in a dedicated configuration file that is automatically checked during development or CI/CD to ensure it remains comprehensive. This would prevent future route conflicts more reliably.
3.  **Granular Action-Based Authorization**: In `src/app/api/appointments/[id]/route.ts`, the authorization logic for actions like `"cancel"`, `"paid"`, `"complete"`, and `"no_show"` is handled through a series of `if (!isProfessional)` or `if (!isClient)` checks. This can become complex as more actions and roles are introduced. A more structured approach could involve mapping actions to required roles (e.g., `actionPermissions = { "paid": ["professional"], "cancel": ["professional", "client"] }`) and then performing a single, clear authorization check based on the user's role and the requested action. This would improve readability, maintainability, and auditability of authorization logic.
