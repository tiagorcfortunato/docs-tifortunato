# Why Supabase

Supabase powers the core data, authentication, and file infrastructure of Odys. This page explains how and why Supabase was chosen, how it's structured, and the architectural trade-offs that shape its integration.

## Overview

Odys uses Supabase as its backend engine, leveraging its Postgres database, authentication system, and storage capabilities in a unified stack. The schema consists of **11 tables** that model professionals, clients, appointments, messages, and business logic like recurring schedules and AI agent metrics. Relationships are enforced via foreign keys with cascade deletes — for example, deleting a `professional` removes their `appointments`, `clients`, and `availability` entries.

Authentication is managed through Supabase Auth, with user sessions tied to professionals via the `userId` field in the `professionals` table. File uploads — such as avatars — are handled through Supabase Storage, accessed via the `/api/upload/avatar` endpoint. The API surface includes **23 routes**, many of which interact directly with Supabase using server- and client-side clients.

Supabase is also central to real-time operations: cron-triggered reminders (`/api/cron/reminders`) and WhatsApp sync (`/api/cron/whatsapp-watchdog`) rely on data stored and queried from the Supabase Postgres instance.

## Design decisions

The Supabase integration is split into two clients: a **server client** and a **browser client**, each tailored to Next.js rendering constraints.

The server client (`src/lib/supabase/server.ts`) uses `createServerClient` from `@supabase/ssr` and integrates with Next.js cookies. It enables server components and API routes to securely access Supabase with SSR-safe cookie handling.

The browser client (`src/lib/supabase/client.ts`) uses a `Proxy` to delay initialization until first access. This avoids build-time errors when environment variables like `NEXT_PUBLIC_SUPABASE_URL` are unset during CI/CD (e.g., Dependabot runs). Without this, `createBrowserClient` would throw during static generation of auth pages.

Supabase’s real-time capabilities are indirectly used: while no live subscriptions are visible in the current API routes, the schema supports reactive patterns (e.g., `notifications` table with `read` status), suggesting future use of Supabase’s real-time channels.

## Potential improvements

The proxy pattern in `createClient()` (`src/lib/supabase/client.ts`) is clever but could be simplified. Instead of a `Proxy`, a lazy-initialized singleton function would be more readable and debuggable while achieving the same goal.

The `agentMetrics` table includes `promptHash` and `promptVersion`, suggesting AI observability needs. However, no API route currently exposes this data. Adding a `/api/metrics` route would enable performance tracking and prompt tuning.

Rate limiting uses Upstash Redis but could be enhanced with Supabase’s own Row Level Security (RLS) policies for finer control. For example, the `booking` rate limit (`5 requests / 10 min`) is enforced in API middleware — moving part of this logic to RLS could reduce backend complexity.
