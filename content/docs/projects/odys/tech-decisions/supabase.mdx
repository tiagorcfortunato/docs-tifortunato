# Supabase

In the `odys` project, Supabase serves as a foundational backend-as-a-service, providing the core database, authentication, and file storage capabilities. This decision allows the development team to focus on application-specific logic rather than managing complex infrastructure, leveraging Supabase's robust, open-source stack built around PostgreSQL.

## Overview

Supabase is essentially a hosted PostgreSQL database augmented with a suite of tools for common backend needs. For `odys`, this means a single platform handles several critical aspects of the application's data layer and user management.

The application's data is structured across 10 distinct tables, all residing within the Supabase-managed PostgreSQL instance. These tables, defined using `drizzle-orm`, capture the essence of the `odys` domain:

*   The `professionals` table, with its 24 columns, stores comprehensive details about service providers, including their `userId` (a `text` field, likely linking to Supabase's authentication system), contact information, service pricing (`sessionDuration`, `sessionPrice`), payment details (`pixKeyType`, `pixKey`, `stripeCustomerId`), and operational settings (`autoConfirm`, `active`).
*   Related to professionals, the `availability` table tracks their working hours, while `clients` stores information about the individuals they serve.
*   The `appointments` table, a central entity, manages scheduled sessions, linking `professionalId` and `clientId`, and tracking `startsAt`, `endsAt`, `status`, and `paymentStatus`.
*   Communication is handled via the `messages` table, and client-specific notes are stored in `client_notes`.
*   User engagement is captured through `follows` and `reviews`, while `recurring_schedules` supports repeat bookings.
*   Finally, `notifications` provides a system for alerting users to important events.

Supabase's authentication service is implicitly used to manage user accounts. While the `professionals` and `clients` tables have their own `userId` columns, these are typically populated by the IDs generated by Supabase Auth, allowing for a unified user identity across the application. The `@supabase/supabase-js` and `@supabase/ssr` packages are integral to interacting with this service, providing client-side and server-side capabilities respectively.

For file storage, such as user avatars or attachments in messages, Supabase Storage is utilized. Columns like `avatarUrl` in `professionals` and `clients`, and `fileUrl` in `messages`, are designed to store references to files hosted within Supabase's object storage buckets. The `/api/upload/avatar` API route specifically handles the upload process, demonstrating this integration.

The application interacts with Supabase through two distinct client instances: `src/lib/supabase/server.ts` for server-side operations and `src/lib/supabase/client.ts` for browser-side interactions. The `createServerClient` function in the server-side utility is crucial for handling cookies, ensuring that user sessions are correctly managed across Next.js server components and API routes.

## Design Decisions

The choice of Supabase for `odys` was driven by a desire for rapid development and reduced operational overhead, particularly for common backend functionalities.

Firstly, opting for a managed service like Supabase means that the team doesn't have to provision, scale, or maintain a PostgreSQL database, an authentication system, or an object storage solution independently. This "batteries included" approach significantly accelerates the initial development phase, allowing engineers to focus on the unique business logic of `odys` rather than infrastructure concerns. The underlying PostgreSQL database offers a robust and feature-rich data store, which is a significant advantage over simpler NoSQL alternatives for an application with complex relational data like appointments, clients, and professionals.

Secondly, the decision to use `drizzle-orm` alongside Supabase's PostgreSQL database is a deliberate architectural choice. While Supabase provides its own client library for database interactions, Drizzle offers a type-safe, schema-first approach to database management. This allows for defining the entire database schema in TypeScript, which then generates SQL migrations (`drizzle-kit generate`, `drizzle-kit migrate`). This provides strong type inference for queries and results, catching potential database errors at compile time rather than runtime. It also offers a more idiomatic way for TypeScript developers to interact with the database, providing a familiar ORM pattern while still allowing for raw SQL when needed, which is facilitated by the `postgres` driver.

Finally, the separation of `createServerClient` and `createBrowserClient` in `src/lib/supabase/server.ts` and `src/lib/supabase/client.ts` respectively, is a direct consequence of building on Next.js with its server-side rendering (SSR) capabilities. The `createServerClient` is specifically designed to interact with cookies, which are essential for maintaining user sessions across server-rendered pages and API routes. This ensures that authentication state is correctly passed between the client and server, providing a seamless user experience while adhering to the security requirements of session management. The `createBrowserClient` is a simpler instance, used for client-side data fetching and interactions where direct cookie manipulation is not required or appropriate.

## Potential Improvements

1.  **Implement Row-Level Security (RLS)**: Currently, the schema defines relationships like `professionalId` and `clientId` across many tables (e.g., `availability`, `clients`, `client_notes`, `recurring_schedules`, `appointments`, `messages`, `follows`, `reviews`). While application-level checks are likely in place, Supabase's RLS feature allows for enforcing data access policies directly at the database level. This would provide a robust, server-agnostic layer of security, ensuring that users can only access data they are authorized to see, regardless of how they interact with the database. This is a critical security enhancement for multi-tenant applications.
2.  **Leverage Supabase Realtime for Notifications and Messages**: The `messages` and `notifications` tables are prime candidates for real-time updates. Supabase offers real-time subscriptions to database changes. Implementing this feature would allow clients to receive new messages or notifications instantly without needing to poll the server, significantly enhancing the user experience. This could be integrated into the existing API routes like `/api/messages` and `/api/notifications` to push updates to connected clients.
3.  **Standardize `userId` Type**: The `userId` columns in `professionals` and `clients` are defined as `text`. Supabase Auth typically generates UUIDs for user IDs. While `text` can store UUIDs, changing these columns to `uuid` type would provide better type consistency with Supabase's internal user management and could potentially allow for direct foreign key relationships to Supabase's `auth.users` table if such a direct link is desired in the future. This would improve data integrity and clarity.

## Complementary Backend Services

While Supabase provides core database, authentication, and storage, the `odys` project integrates several other backend services to fulfill a complete set of application requirements:

*   **Rate Limiting**: `@upstash/ratelimit` and `@upstash/redis` are used to implement robust rate limiting, protecting API endpoints from abuse and ensuring fair resource usage.
*   **AI/LLM Capabilities**: `groq-sdk` is integrated to provide AI and Large Language Model functionalities, enabling advanced features such as content generation or intelligent responses.
*   **Email Services**: `resend` is utilized for sending transactional and marketing emails, handling all outbound email communication for the application.
*   **Error Monitoring**: `@sentry/nextjs` provides comprehensive error tracking and performance monitoring, allowing the team to identify and resolve issues proactively across the Next.js application.
*   **Analytics**: `posthog-js` is integrated for product analytics, capturing user behavior and providing insights into application usage and feature adoption.
*   **Payment Processing**: `stripe` and `@stripe/stripe-js` are used for managing payment processing, including subscriptions, one-time payments, and handling `stripeCustomerId` for client billing.
