# Tech Decisions

This page outlines the foundational technical choices shaping Odys, a platform connecting professionals with clients through booking, messaging, and AI-driven automation. Understanding these decisions reveals how scalability, maintainability, and real-world constraints were balanced across the stack.

## Overview

Odys is built on a modern full-stack TypeScript architecture, centered around **Next.js 16.2.4** and **React 19.2.5**, providing server-side rendering and API route capabilities. The database schema consists of **11 tables** managed via **Drizzle ORM** (`^0.45.2`) using the `postgres` client, which offers type-safe SQL queries and schema migrations. Data persistence uses **Supabase** (`@supabase/supabase-js@^2.100.1`) for authentication and PostgreSQL-backed storage, with `@supabase/ssr` handling server-side rendering integration. **Stripe** (`^22.1.0`) handles server-side subscription logic, with `@stripe/stripe-js` (`^9.3.1`) managing client-side payment flows across four pricing tiers: free, basic (\$39), pro (\$79), and premium (\$149).

The system integrates AI through OpenAI and a custom agent framework, with **4 MCP tools** like `get_appointments_today` and `update_appointment_status` enabling automated workflows. WhatsApp interactions are coordinated using **19 message templates**, such as `msgBookingConfirmed` and `msgReminder24h`, triggered via webhooks, and email communications are handled via **Resend** (`^6.12.2`). Rate limiting is enforced using **Upstash Redis**, with distinct limits for critical paths like `/api/booking` (5 requests per 10 minutes) and `/api/ai/chat` (20 per hour).

## Core Architecture

The backend exposes **23 API routes**, including RESTful endpoints for appointments, clients, and settings, alongside webhook handlers for Stripe and WhatsApp. Cron-like tasksŌĆösuch as sending reminders (`/api/cron/reminders`) and monitoring WhatsApp health (`/api/cron/whatsapp-watchdog`)ŌĆörun on fixed schedules using Vercel Cron Jobs. Stripe webhooks listen to three events: `checkout.session.completed`, `customer.subscription.updated`, and `customer.subscription.deleted`, ensuring subscription state stays in sync.

Client state is managed using React Hook Form and Zod for validation, while the UI leverages **shadcn/ui** primitives built on **Radix UI**, along with components from `@base-ui/react`, all styled with Tailwind CSS. Analytics are tracked via PostHog and Vercel, with error monitoring through Sentry.

## Design decisions

The use of Drizzle ORM over Prisma reflects a preference for lightweight, SQL-first schema definitions that integrate tightly with Turso or Supabase. The `professionals` tableŌĆö32 columns wideŌĆöacts as the central profile store, combining business logic (e.g., `sessionPrice`, `plan`) and operational data (`latitude`, `addressLine`). This denormalization reduces joins but increases row size, a trade-off favoring read performance over strict normalization.

API rate limits (e.g., `rl:booking`, `rl:api`) are implemented to prevent abuse while allowing legitimate usage, especially on high-risk endpoints like registration and booking. The AI agentŌĆÖs metrics are logged in the `agentMetrics` table, capturing cost, latency, and outcomeŌĆöenabling observability without blocking primary flows.

## Potential improvements

1. The `/api/booking` route enforces a 5-request/10-minute limit via `@upstash/ratelimit`, but this is not documented in OpenAPI or exposed in the UI. Adding user-facing rate limit hints would improve UX.

2. The `recurringSchedules` table lacks a `timezone` column, yet schedules depend on time (`dayOfWeek`, `time`). Without timezone context, rescheduling across regions may failŌĆöadding `timezone` (text, nullable) would future-proof this.

3. The `agentMetrics` table logs `promptHash` and `promptVersion`, but no mechanism exists to retrieve or compare prompt versions. A `/api/agent/prompts` endpoint could serve versioned prompts for debugging and A/B testing.
