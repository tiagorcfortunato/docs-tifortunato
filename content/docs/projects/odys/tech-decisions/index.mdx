# Technical Decisions

The Odys application is built upon a carefully selected set of technologies and architectural patterns, designed to provide a robust and efficient platform for professionals and their clients. This document outlines the core technical decisions, explaining the rationale behind key structural choices, and identifying areas for future refinement.

## Overview

Odys operates as a Next.js application, leveraging its full-stack capabilities for both frontend rendering and backend API services. The data layer is managed by Drizzle ORM, interacting with a PostgreSQL database, likely hosted via Supabase, which also handles authentication. Payments are integrated through Stripe, while real-time communication and notifications are facilitated by WhatsApp and a custom notification system. For enhanced functionality, the application incorporates AI capabilities via OpenAI, and ensures system stability with rate limiting provided by Upstash Redis.

The database schema comprises 10 distinct tables, centralizing around the `professionals` table, which holds 32 columns detailing a professional's profile, contact information, service settings, and payment details. Each professional is uniquely identified by an `id` (UUID) and a `userId` (text), suggesting integration with an external authentication provider. Related tables like `availability`, `clients`, `clientNotes`, `recurringSchedules`, `appointments`, `messages`, `follows`, and `reviews` all establish a `professionalId` foreign key, often with `onDelete: cascade` to ensure data consistency. For instance, deleting a professional automatically removes all their associated availability slots, clients, and messages. The `appointments` table, with 13 columns, tracks individual sessions, linking professionals and clients, and managing status and payment information.

The application exposes 23 API routes, covering a wide range of functionalities from user account management (`/api/account`, `/api/onboarding`), booking and scheduling (`/api/booking`, `/api/recurring`, `/api/appointments/[id]`), client interactions (`/api/client-profile`, `/api/clients/notes`, `/api/messages`), to payment processing (`/api/stripe/checkout`, `/api/stripe/webhook`). Two critical cron jobs are defined: `/api/cron/reminders` scheduled daily at 8 AM, and `/api/cron/whatsapp-watchdog` at 9 AM, indicating automated tasks for client engagement and system health.

Payment plans are structured into four tiers: `free`, `basic`, `pro`, and `premium`, each with a defined price and a corresponding Stripe Price ID environment variable, enabling flexible subscription management. The AI capabilities are exposed through `/api/ai/chat` and utilize 4 specific tools (`get_appointments_today`, `get_customer_by_phone`, `get_revenue_summary`, `update_appointment_status`), suggesting an AI assistant designed to help professionals manage their practice. To prevent abuse and ensure fair resource allocation, 6 distinct rate limits are implemented using Upstash, covering critical endpoints like `booking`, `api` (general API calls), `onboarding`, `register`, `whatsapp`, and `aichat`.

Stripe webhook events (`checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`) are explicitly handled, indicating a robust system for reacting to changes in subscription status and payment events. Communication with clients is heavily reliant on WhatsApp, with 19 predefined message templates covering various scenarios such as booking requests, confirmations, reminders, cancellations, and payment notifications. The `package.json` file confirms the use of Next.js `16.2.4`, React `19.2.5`, Drizzle `0.45.2`, Supabase `2.100.1`, Stripe `22.1.0`, OpenAI `6.34.0`, `@upstash/redis` `1.37.0`, and `@upstash/ratelimit` `2.0.8`, alongside other utilities for UI, analytics, and error tracking.

## Design Decisions

The architectural choices in Odys reflect a pragmatic approach to building a modern web application, balancing developer experience, performance, and maintainability.

The selection of **Next.js** as the primary framework allows for a unified development experience, enabling server-side rendering (SSR) or static site generation (SSG) for performance and SEO, alongside API route handling for backend logic. This avoids the complexity of managing separate frontend and backend repositories and deployment pipelines.

**Drizzle ORM** was chosen for database interactions, providing a type-safe and SQL-like interface to the PostgreSQL database. This decision prioritizes developer productivity and reduces the likelihood of runtime errors compared to raw SQL queries. The use of `uuid` for primary keys across all tables, such as `professionals.id` and `appointments.id`, is a common practice for distributed systems, preventing sequential ID guessing and simplifying data merging across different environments. The widespread use of `onDelete: cascade` for `professionalId` foreign keys (e.g., in `availability`, `clients`, `appointments`) is a deliberate choice to simplify data management. When a professional's account is removed, all associated data, such as their schedule, clients, and appointments, are automatically cleaned up by the database, maintaining referential integrity without requiring explicit application-level deletion logic for dependent records. Notably, `appointments.recurringScheduleId` does *not* have `onDelete: cascade`, indicating a design choice to preserve individual appointment records even if the recurring schedule that generated them is deleted.

**Stripe** integration is central to the business model, handling subscriptions and one-time payments. The dedicated `/api/stripe/webhook` endpoint is crucial for processing asynchronous events from Stripe, such as `checkout.session.completed` or `customer.subscription.updated`. This webhook-driven approach ensures that the application's internal state (e.g., a professional's `plan` or `trialEndsAt` in the `professionals` table) is accurately updated in response to external payment events, rather than relying on potentially unreliable client-side confirmations.

The extensive use of **WhatsApp message templates** (19 in total) highlights a strong focus on direct, automated client communication. This decision likely stems from a desire to meet clients where they are, providing timely updates and reminders through a widely used messaging platform, thereby improving engagement and reducing no-shows.

The inclusion of **OpenAI** for AI chat functionality, coupled with specific tools like `get_appointments_today` and `update_appointment_status`, suggests an intention to empower professionals with an intelligent assistant. This allows for natural language interaction to perform common administrative tasks, aiming to reduce manual effort and improve efficiency.

**Upstash Redis** and its `@upstash/ratelimit` library are employed to implement rate limiting across various API endpoints. This is a critical security and stability measure, preventing abuse, denial-of-service attacks, and ensuring fair usage of resources, especially for computationally intensive or sensitive operations like `booking` and `aichat`.

For the user interface, the project uses **shadcn**, `tailwind-merge`, and `class-variance-authority`. This combination indicates a preference for a component-based UI system built on Tailwind CSS, allowing for highly customizable and consistent styling while maintaining a strong design system.

## Potential Improvements

1.  **Indexing for Query Performance**: Many tables, such as `appointments`, `clients`, `messages`, and `reviews`, frequently reference `professionalId` and `clientId`. While these are foreign keys, explicit database indexes on these columns would significantly accelerate read operations, especially for queries filtering or joining on these identifiers. Similarly, `appointments.startsAt` and `appointments.endsAt` are critical for time-based scheduling queries and would benefit greatly from indexing.
2.  **Consistent Time Representation**: The `availability` table stores `startTime` and `endTime` as `text`, whereas the `appointments` table uses `timestamp` for `startsAt` and `endsAt`. This inconsistency can lead to challenges in time zone management, comparisons, and range queries. Adopting a consistent data type, such as a dedicated `time` type (if supported by the database and ORM) or `timestamp` with a dummy date, across all time-related columns would simplify logic and reduce potential parsing errors.
3.  **`email` Uniqueness in `professionals`**: The `professionals.email` column is currently `nullable: true` and `unique: false`. If the intention is for each professional to have a distinct email address for communication or login purposes, enforcing `unique: true` at the database level would prevent data integrity issues. If multiple professionals can legitimately share an email (e.g., a shared administrative contact), the current design is acceptable, but the implications should be clearly understood.
