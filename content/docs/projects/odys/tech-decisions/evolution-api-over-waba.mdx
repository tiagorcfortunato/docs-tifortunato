# Evolution API

Odys relies heavily on WhatsApp for communicating with both professionals and clients, sending everything from appointment reminders to payment confirmations. Choosing the right platform for this critical communication channel was a foundational decision, leading the team to adopt the Evolution API rather than directly integrating with Meta's official WhatsApp Business API (WABA). This document explores the rationale behind this choice, the architectural implications, and areas for future refinement.

## Overview

The Evolution API acts as a intermediary layer, simplifying WhatsApp integration for Odys. Instead of navigating the complexities of Meta's direct API, Odys interacts with a more straightforward HTTP interface provided by Evolution API. This approach is evident in the `src/lib/whatsapp.ts` module, which encapsulates all WhatsApp messaging logic. This file defines a `sendWhatsApp` function and a collection of 19 pre-formatted message templates, such as `msgBookingConfirmed` and `msgReminder24h`, ensuring consistent communication across the platform.

For local development and testing, the `docker-compose.evolution.yml` file orchestrates the Evolution API service alongside its `postgres` database and `redis` cache. This setup allows developers to run a full WhatsApp integration stack locally, mirroring the production environment.

## Evolution API Architecture

The `docker-compose.evolution.yml` file reveals the core components of the Evolution API setup. The `evolution-api` service, based on the `evoapicloud/evolution-api:v2.3.7` Docker image, is configured to run as a standalone application. It depends on a `postgres:15` database, named `evolution-postgres`, for persistent storage and a `redis:7-alpine` instance, `evolution-redis`, for caching and session management.

Key environment variables in the `evolution-api` service define its behavior:
*   `AUTHENTICATION_API_KEY`: Set to `odys-local-key-2026` for local authentication, controlling access to the API.
*   `DATABASE_PROVIDER`: Explicitly set to `postgresql`, with `DATABASE_CONNECTION_URI` pointing to the `evolution-postgres` service. This configuration ensures that all instance data, messages, contacts, and chats are saved within the dedicated PostgreSQL database.
*   `DATABASE_CONNECTION_CLIENT_NAME`: Set to `odys`, identifying the client connecting to the database.
*   `CACHE_REDIS_ENABLED`: Set to `true`, directing the API to use the `evolution-redis` service for caching, improving performance and managing session state.
*   `WEBHOOK_GLOBAL_ENABLED`: Set to `false`, indicating that webhooks are not globally enabled but rather configured on a per-instance basis via API calls, offering more granular control over event notifications.
*   `TELEMETRY_ENABLED`: Set to `false`, disabling telemetry reporting for the Evolution API service.

This containerized setup provides a self-contained, reproducible environment for the WhatsApp integration, making it easier to deploy and manage.

## WhatsApp Messaging in Odys

The `src/lib/whatsapp.ts` file serves as Odys's primary interface with the Evolution API. It begins by checking if the necessary environment variables (`EVOLUTION_API_URL`, `EVOLUTION_API_KEY`, `EVOLUTION_INSTANCE`) are configured via the `isConfigured` function, preventing message attempts if the service is not ready.

A crucial utility function, `normalizeBrazilianPhone`, ensures that all phone numbers are in a consistent international format, specifically handling Brazilian numbers by prefixing them with `55` if needed, while preserving other international formats. This standardization is vital for reliable message delivery.

The core messaging functionality resides in the asynchronous `sendWhatsApp` function. It takes a `phone` number, the `text` content, and an optional `context` object. The `context` is particularly important for transactional messages sent to clients, as it allows Odys to record the `professionalId` and `appointmentId` in Redis via `setOutboundContext`. The mechanism for consuming this context to route client replies, such as a `whatsapp/webhook`, is not defined or configured in the provided code excerpts. The `sendWhatsApp` function uses `fetch` to make a `POST` request to the Evolution API's `/message/sendText/${INSTANCE}` endpoint, including the `number`, `text`, and a `delay` of 1200 milliseconds. It includes robust error handling, logging failures without throwing exceptions, which helps maintain application stability.

To enhance user experience and distinguish automated messages from human-like AI interactions, the module employs visual templating. Functions like `transactionalHeader` and the `TRANSACTIONAL_FOOTER` constant add rule lines and branding to messages, clearly signaling their automated nature. This design choice is critical for maintaining clarity in a channel that also supports conversational AI.

The module defines 19 distinct message templates, each tailored for specific scenarios. Examples include:
*   `msgBookingRequest`: Notifies a professional about a new appointment request.
*   `msgBookingConfirmed`: Confirms an appointment with a client.
*   `msgReminder24h` and `msgReminder1h`: Send timely reminders before an appointment.
*   `msgCancelledByClientToPro` and `msgCancelledByProToClient`: Handle cancellation notifications for both parties.
*   `msgPaymentConfirmedToClient`: Informs a client that their payment has been received.
*   `msgNewMessageToClient` and `msgNewMessageToPro`: Notify users of new messages, directing them to the appropriate interface within the Odys application.

These templates centralize message content, ensuring consistency and making updates more manageable.

## Design decisions

The decision to use Evolution API over Meta's WhatsApp Business API (WABA) was driven by several practical considerations, balancing development speed, cost, and flexibility against potential long-term trade-offs.

Primarily, the Evolution API offers a significantly simpler and faster integration path. WABA typically involves a more complex setup, including business verification, template pre-approvals, and a more intricate API structure. For a product like Odys, especially in its earlier stages, this reduced friction translates directly into quicker feature development and iteration. The ability to send dynamic text messages, as seen in the various `msg*` functions in `src/lib/whatsapp.ts`, is a key advantage. WABA often requires pre-approved templates for most outbound messages, which can introduce delays and restrict the flexibility of communication. Evolution API, by contrast, allows for more ad-hoc message generation, which is crucial for the varied and personalized interactions Odys facilitates.

The local development setup, clearly defined in `docker-compose.evolution.yml`, is another strong indicator of this choice. Running a full WhatsApp integration stack locally with `postgres` and `redis` makes development and testing much more efficient, removing dependencies on external Meta services during the development cycle. This ease of local setup is often a hallmark of third-party APIs that abstract away underlying complexities.

However, this choice comes with inherent trade-offs. Relying on a third-party API like Evolution API introduces a dependency on an external vendor whose stability, scalability, and compliance with WhatsApp's terms of service are outside Odys's direct control. There's a risk of breaking changes if WhatsApp updates its client or policies, potentially requiring the Evolution API to adapt, which could impact Odys. While the current implementation is functional, WABA generally offers higher reliability and official support directly from Meta, along with advanced features like interactive messages and richer media types that might not be fully supported or as seamlessly integrated through a third-party wrapper. The current message templates are primarily text-based, which aligns with the simpler capabilities often found in such APIs.

## Potential improvements

1.  **Centralize `APP_URL` usage**: The `APP_URL` constant is imported from `lib/constants` but then hardcoded as `https://odys.com.br/dashboard` in `msgBookingRequest` and `https://odys.com.br/c` in `msgAppointmentCompletedToClient`. For consistency and easier maintenance, all references to the application's base URL should use the imported `APP_URL` constant. This prevents discrepancies if the domain ever changes.
2.  **Enhance `setOutboundContext` error handling**: In `src/lib/whatsapp.ts`, the `sendWhatsApp` function includes a `try...catch` block for the `fetch` call to the Evolution API. However, the `await setOutboundContext(number, context)` call within the `try` block is not individually wrapped or explicitly handled for potential errors. If `setOutboundContext` fails (e.g., due to Redis connectivity issues), this failure would currently be silent, leading to incorrect context tracking for client replies. Adding specific error handling or logging around this call would improve observability.
3.  **Refine phone number normalization for internationalization**: The `normalizeBrazilianPhone` function in `src/lib/whatsapp.ts` currently handles Brazilian numbers and attempts to preserve other international formats. As Odys potentially expands, this logic could become a source of errors. Using a dedicated, battle-tested library for international phone number parsing and formatting (e.g., `libphonenumber-js`) would provide more robust validation and normalization, ensuring correct delivery to diverse international numbers beyond the current `55` prefix logic.
