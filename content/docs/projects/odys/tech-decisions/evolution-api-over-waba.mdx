# Evolution API vs WhatsApp Business API

Odys relies on WhatsApp for real-time, transactional communication with clients and professionals—appointment confirmations, reminders, payment receipts, and support messages. While Meta offers the WhatsApp Business API (WABA), Odys uses **Evolution API**, a third-party WhatsApp Web automation layer. This document explains why that choice was made, how it's implemented, and the trade-offs involved.

## Overview

Odys sends 19 distinct WhatsApp message types—from booking confirmations to no-show follow-ups—via a centralized `sendWhatsApp` function. These messages are templated in `src/lib/whatsapp.ts` and triggered by backend events like appointment updates or cron jobs. The system uses Evolution API’s self-hosted instance, running in a Docker stack with PostgreSQL and Redis, and authenticates via API key and instance name. All outbound messages are sent with a 1.2-second delay between calls to the Evolution API. Context for inbound replies is tracked in Redis using `last_outbound:{phone}`.

## Why Evolution API?

Meta’s WhatsApp Business API is a robust, enterprise-grade solution, but it requires business verification, phone number approval, and strict template review cycles—barriers for rapid iteration. Evolution API, in contrast, uses WhatsApp Web automation, allowing Odys to:

- **Launch faster**: No need for Meta business verification or template pre-approval.
- **Iterate freely**: Message templates (e.g., `msgReminder24h`) can be modified instantly without waiting for Meta’s review.
- **Self-host and control**: The Docker stack (`docker-compose.evolution.yml`) gives full control over uptime, logging, and instance state.
- **Use a single number**: Evolution API supports multi-tenant messaging from one number, critical for Odys’ shared `professionalId`-based routing.

This approach aligns with Odys’ need for agility during early growth, especially in Brazil, where WhatsApp is the dominant communication channel.

## Design decisions

The `sendWhatsApp` function abstracts Evolution API’s v2.3.7 `/sendText` endpoint with a simple `{number, text, delay}` payload. It normalizes Brazilian phone numbers (e.g., `(11) 99999-9999` → `5511999999999`) and logs failures without throwing, ensuring non-blocking behavior. Transactional messages use a visual header (`━━━━━━━━━━━━━━━━━`) and footer to distinguish automated alerts from AI replies in the same chat thread—a UX necessity when both channels share a number.

Context tracking via `setOutboundContext` enables reply routing: when a client responds within 2 hours, the webhook uses Redis to associate the message with the correct `professionalId` and `appointmentId`. This mimics WABA’s context retention but is implemented independently.

The Evolution API instance is configured with `SERVER_URL: "http://localhost:8080"`. The Evolution API's PostgreSQL connection uses `DATABASE_CONNECTION_CLIENT_NAME: "odys"`.

## Potential improvements

1. **Add message template versioning** — Currently, changes to functions like `msgBookingConfirmed` are untracked. Introduce a `version` field in template functions to support A/B testing or rollback (e.g., `msgBookingConfirmed_v2`).
   
2. **Improve error handling in `sendWhatsApp`** — The function returns `false` on failure but doesn’t retry. For critical messages (e.g., `msgPaymentConfirmedToClient`), implement exponential backoff using a task queue like BullMQ.

3. **Validate message length** — WhatsApp limits text messages to 4096 characters. Add a check in `sendWhatsApp` to truncate or split long messages, preventing silent failures.
