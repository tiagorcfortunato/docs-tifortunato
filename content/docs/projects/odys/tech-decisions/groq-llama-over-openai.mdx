# Groq + Llama Decision

This page details the rationale behind selecting Groq with Llama 3.3 for AI functionalities within Odys, specifically in contrast to using OpenAI models. It covers the primary drivers for this decision, the specific use cases, and the known trade-offs.

## Rationale for Groq + Llama

The decision to implement Groq with Llama 3.3 was driven by performance and cost efficiency for well-defined AI tasks. Groq's inference engine provides significantly faster response times compared to GPT-4-class models, which is critical for real-time interactions involving sequential tool calls. Cost estimates indicate a 10–20x reduction in inference expenses relative to OpenAI's offerings. This combination enables scalable, low-latency AI features without introducing prohibitive operational costs.

## Implementation

Groq is used in two core AI components:

1.  **AI WhatsApp Intake Agent**: Implemented in `src/lib/ai-intake.ts`, this agent processes inbound WhatsApp messages and uses Groq's LLM to manage appointment booking workflows. It follows a two-pass pattern: the first LLM call determines whether to invoke a tool, and the second generates the final response after incorporating tool results. The `TOOLS` array in `src/lib/ai-intake.ts` defines functions such as `get_available_slots` and `book_appointment`, which interface directly with the database via Drizzle ORM.

2.  **AI Assistant**: Located in `src/app/api/ai/chat/route.ts`, this assistant supports professionals by answering queries about appointments, client behavior, and revenue. It uses the same two-pass tool-calling pattern. The `SYSTEM_PROMPT` in `src/app/api/ai/chat/route.ts` enforces strict formatting rules for responses, particularly for no-show rate reporting and monthly summaries. The `TOOLS` array includes `get_stats`, `get_upcoming`, and `get_no_show_clients`, each designed to return structured data from the database.

Both implementations instantiate the Groq client using `new Groq({ apiKey: process.env.GROQ_API_KEY })` within the `getGroq()` function, ensuring consistent configuration across services.

## Known Gaps

*   **Instruction-following limitations**: Llama 3.3 on Groq is weaker than GPT-4 in complex instruction interpretation. This is evident in constrained prompt designs that avoid open-ended reasoning. The system mitigates this by limiting LLM responsibilities to narrow, tool-driven actions—such as fetching availability or calculating no-show rates—where output structure is tightly controlled by predefined functions.

## Why this shape

The architecture prioritizes speed, cost, and determinism for high-frequency AI interactions. By restricting the LLM’s role to tool selection and natural language response generation over pre-validated data, the system leverages Groq’s low-latency inference while minimizing exposure to Llama’s relative weaknesses in general reasoning. This design ensures that time-sensitive features like WhatsApp intake and professional analytics remain responsive and economically sustainable at scale.
