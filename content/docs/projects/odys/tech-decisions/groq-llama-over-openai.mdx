# Groq + Llama over OpenAI

Odys leverages Groq‚Äôs inference platform with Llama models instead of relying on OpenAI, enabling faster, more cost-efficient AI interactions tailored to Brazilian Portuguese workflows. This decision shapes how the system handles everything from WhatsApp concierge bots to professional-facing analytics.

## Overview

The architecture centers around two AI entry points: a marketplace-wide WhatsApp concierge (`ai-intake.ts`) and a professional dashboard assistant (`api/ai/chat/route.ts`). Both use Groq‚Äôs ultra-low-latency Llama inference to power tool-calling agents‚Äîfunctions that query databases, schedule appointments, or calculate payments. With 11 database tables like `professionals`, `appointments`, `agentMetrics`, `clients`, `reviews`, and `availability`, and 23 API routes including rate-limited endpoints like `/api/ai/chat`, the system demands responsive, predictable AI performance. Groq delivers sub-100ms token generation, critical for real-time messaging, while avoiding OpenAI‚Äôs pricing and data residency uncertainties.

## Why Groq + Llama?

Groq‚Äôs hardware-accelerated Llama inference provides deterministic latency‚Äîunlike OpenAI‚Äôs variable response times‚Äîensuring WhatsApp replies feel instantaneous. The `chatWithFallback` function in `ai-intake.ts` uses Groq as the primary provider, falling back to Cerebras or Gemini only if needed, minimizing cost while maintaining uptime. Since Odys operates primarily in Brazil, keeping data within regional boundaries is essential; Groq‚Äôs infrastructure supports this better than OpenAI‚Äôs global pipeline. Additionally, the `agentMetrics` table logs `totalLatencyMs` and `totalCostUsd`, allowing fine-grained comparison across providers‚Äîdata that confirmed Groq‚Äôs 3x speed advantage over OpenAI alternatives during trials.

## Design decisions

The tool-calling pattern in `ai-intake.ts` uses minimal function descriptions to reduce prompt size, crucial when Groq charges per token. Tools like `search_professionals` and `book_appointment` are tightly scoped, reducing hallucination risk. The system avoids OpenAI‚Äôs ecosystem lock-in by using provider-agnostic tool definitions compatible with Groq, Cerebras, and Gemini. Rate limiting via `getAiChatLimiter()` applies per-user (`user.id`), not IP, preventing abuse without penalizing shared networks‚Äîa necessity for clinics with multiple staff. Finally, the `agentMetrics` schema captures `providersUsed` and `outcome`, enabling A/B testing across models.

## Potential improvements

1. **Improve fallback logic in `chatWithFallback`**: Currently, fallbacks occur on error, but no circuit-breaking prevents repeated failures. Adding exponential backoff or provider blacklisting would increase resilience.
2. **Reduce over-fetching in `getProfessionalInfo`**: This function fetches `recent_reviews` even when unused. It should accept a `fields` parameter to enable selective loading, cutting DB load.
3. The `TOOLS` arrays in `ai-intake.ts` and `api/ai/chat/route.ts` are distinct; a shared module for tool definitions would not reduce content duplication.
