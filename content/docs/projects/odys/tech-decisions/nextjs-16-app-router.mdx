# Next.js 16 App Router

Odys leverages the App Router introduced in Next.js 13 and fully matured by version 16.2.4, the version currently in use. This decision shapes how routing, data fetching, and rendering are structured across the application. Understanding this choice is key to navigating the codebase, especially as it interacts with server actions, API routes, and edge-compatible patterns.

## Overview

The App Router is central to Odys’ architecture, enabling React Server Components, nested layouts, and simplified data flow. With 23 API routes defined — from `/api/booking` to `/api/stripe/webhook` — the backend surface is substantial, but cleanly separated. These routes handle everything from appointment scheduling to WhatsApp webhook ingestion, and they coexist with App Router server actions that mutate data like client notes or recurring schedules.

The database schema reflects a domain-rich professional booking system: 11 tables including `professionals`, `appointments`, `clients`, and `recurringSchedules`. Each table is accessed via Drizzle ORM (`drizzle-orm@0.45.2`), which integrates tightly with Next.js’s server context. The presence of cron-like API routes such as `/api/cron/reminders` (scheduled at `0 8 * * *`) shows how background logic is implemented using Vercel Cron Jobs — a pattern enabled by the App Router’s server-first mindset and configured via `automaticVercelMonitors: true` in `next.config.ts`.

## Why App Router Over Alternatives?

Next.js 16’s App Router was selected over the Pages Router or standalone frameworks like Remix or Express because it unifies full-stack concerns within a single, React-centric model. Server Components allow direct database queries in route handlers without leaking credentials, while client components use React Hook Form and Zod for validation — as seen in `/api/client-profile` and `/api/settings`.

The App Router also supports granular data fetching. For example, the `messages` table is queried in `/api/messages` to support real-time chat, while `reviews` and `notifications` are fetched on-demand in dashboard views. This flexibility avoids over-fetching and aligns with React 19’s streaming capabilities, which are unlocked by the App Router’s server rendering.

Additionally, Vercel’s deep integration with Next.js enables zero-config deployment of API routes, cron-like endpoints, and edge middleware. The `next.config.ts` file configures security headers and Sentry monitoring via `withSentryConfig`, a pattern optimized for App Router deployments.

## Design decisions

The use of `securityHeaders` in `next.config.ts` — including CSP, HSTS, and frame protection — reflects a security-first posture that App Router facilitates through middleware and headers. The CSP allows Stripe and Supabase domains explicitly, acknowledging third-party dependencies while minimizing XSS risk.

API routes are kept lean and focused: `/api/booking` handles only booking logic, while `/api/ai/chat` routes to an LLM agent. This modularity mirrors the App Router’s philosophy of colocation and separation of concerns. The presence of rate limits (e.g., `aichat: { limit: 20, window: "1 h" }`) shows how App Router server actions can be protected using Upstash Redis, a pattern that scales across routes.

## Potential improvements

The CSP in `next.config.ts` uses `'unsafe-inline'` for `script-src` due to Next.js’s current reliance on inline scripts. This weakens XSS protection; migrating to a nonce-based CSP would improve security, though it requires custom script loading.

The `/api/cron/reminders` route is hit via a cron job but lacks idempotency checks. If the job runs twice, duplicate reminders could be sent. Adding a lock using Redis (e.g., `@upstash/redis`) would prevent this.

The `agentMetrics` table logs AI agent performance but isn’t queried in any API route. Creating a dashboard route under `/app/agent-insights` would unlock observability, especially when paired with the `promptHash` and `totalLatencyMs` fields.
