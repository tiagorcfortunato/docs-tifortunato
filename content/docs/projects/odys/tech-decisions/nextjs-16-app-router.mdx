# Next.js 16 App Router

Odys is built upon Next.js `16.2.4` and React `19.2.5`, embracing the App Router paradigm as its foundational framework. This architectural choice shapes how the application's user interface, data fetching, and API interactions are structured, aiming for a modern, performant, and maintainable web experience. This document explores the rationale behind this decision, its implications for the codebase, and areas for future refinement.

## Overview

The App Router in Next.js provides a file-system based routing mechanism that supports both React Server Components and client components, allowing developers to choose the appropriate rendering environment for each part of the application. This hybrid approach is central to Odys, where server components handle data fetching and initial rendering, while client components manage interactive UI elements.

The application defines `23` distinct API routes, serving various functionalities from user account management (`/api/account`) and booking (`/api/booking`) to integrations with external services like Stripe (`/api/stripe/webhook`) and WhatsApp (`/api/whatsapp/webhook`). These routes are critical for handling client-side requests and server-to-server communications.

Beyond standard API endpoints, Odys also incorporates `2` dedicated cron jobs, `/api/cron/reminders` and `/api/cron/whatsapp-watchdog`, scheduled to run daily at `0 8 * * *` and `0 9 * * *` respectively. These background tasks are essential for automating time-sensitive operations, such as sending appointment reminders or monitoring WhatsApp message delivery.

The application's data layer interacts with a PostgreSQL database, managed through Drizzle ORM, which defines `10` tables including `professionals`, `clients`, `appointments`, and `messages`. This schema underpins all core functionalities, from scheduling to client communication.

## App Router Architecture

The App Router's directory-based structure, where folders define routes and special files like `page.tsx` and `route.ts` define UI and API endpoints, offers a clear and intuitive organization. This contrasts with the older Pages Router, which primarily focused on client-side rendering with API routes in a separate `pages/api` directory.

A key advantage of the App Router is its deep integration with React Server Components. This allows Odys to fetch data directly within components on the server, reducing the amount of JavaScript sent to the client and improving initial page load performance. For instance, a page displaying a professional's profile can fetch all necessary data from the `professionals` table directly in a server component, rendering the complete HTML before it reaches the browser. This approach minimizes client-side hydration and the associated performance overhead.

For interactive elements, client components are used. The framework automatically handles the boundary between server and client components, ensuring that only the necessary JavaScript is bundled and shipped to the browser. This separation of concerns helps in building a more efficient and responsive user interface.

## API Routes and Rate Limiting

Odys defines a comprehensive set of `23` API routes to manage its various operations. These routes are implemented as Next.js Route Handlers, residing within the `app/api` directory. Examples include `/api/booking` for managing appointments, `/api/messages` for handling client-professional communications, and `/api/stripe/checkout` for payment processing.

To safeguard against abuse and ensure service stability, Odys implements rate limiting using `@upstash/ratelimit` and `@upstash/redis`. Six distinct rate limiters are configured:
- `booking`: `5` requests per `10 m`
- `api`: `60` requests per `1 m`
- `onboarding`: `3` requests per `1 h`
- `register`: `5` requests per `1 h`
- `whatsapp`: `10` requests per `1 m`
- `aichat`: `20` requests per `1 h`

These limits are applied to specific API endpoints, such as `/api/booking` and `/api/auth/register`, to prevent excessive requests from individual users or malicious actors. The `next.config.ts` also defines `images.remotePatterns` to allow image loading from `*.supabase.co` and `lh3.googleusercontent.com`, which are used for user avatars and other media.

## Security Headers and Content Security Policy

The `next.config.ts` file plays a crucial role in defining application-wide security policies. A set of `securityHeaders` is applied to every response, including:
- `X-Frame-Options: DENY`: Prevents clickjacking by disallowing the application from being embedded in iframes.
- `X-Content-Type-Options: nosniff`: Mitigates MIME type sniffing vulnerabilities.
- `Referrer-Policy: strict-origin-when-cross-origin`: Controls how much referrer information is sent with requests.
- `Permissions-Policy`: Denies access to powerful browser APIs like `camera`, `microphone`, and `geolocation`, while explicitly allowing `payment` for Stripe.
- `Strict-Transport-Security`: Enforces HTTPS for a long duration (`max-age=63072000`) to prevent man-in-the-middle attacks.
- `X-DNS-Prefetch-Control: on`: Opts out of FLoC / Topics and interest-based tracking cohorts.

A `Content-Security-Policy-Report-Only` header is also configured. This policy is initially set to report-only mode, meaning violations are logged without blocking content, allowing for safe testing and monitoring. The `csp` string defines a detailed allowlist for various resource types, including `default-src`, `script-src`, `style-src`, `img-src`, `font-src`, `connect-src`, `frame-src`, `object-src`, `base-uri`, `form-action`, and `frame-ancestors`. This policy explicitly permits necessary third-party domains such as `js.stripe.com`, `*.supabase.co`, `us.i.posthog.com`, `fonts.googleapis.com`, and `accounts.google.com`. The `script-src` currently includes `'unsafe-inline'` and `'unsafe-eval'` due to Next.js's internal requirements, a known trade-off.

## Design decisions

The decision to adopt Next.js 16 with the App Router was driven by several key considerations:

1.  **Performance and User Experience:** The App Router's emphasis on React Server Components allows for significant performance gains by rendering pages on the server and sending minimal JavaScript to the client. This reduces initial load times and improves the perceived responsiveness of the application, which is critical for a service like Odys that handles professional-client interactions. The ability to fetch data directly in server components simplifies the data flow and reduces the need for complex client-side data fetching libraries.

2.  **Developer Experience and Maintainability:** The file-system based routing and co-location of components, styles, and tests within the same directory structure (`app/`) promotes a clear and organized codebase. This structure makes it easier for developers to understand the application's layout and locate relevant files, contributing to long-term maintainability.

3.  **Unified Backend and Frontend:** Next.js's ability to host both the frontend and API routes within a single project streamlines development and deployment. This unified approach simplifies dependency management and allows for direct, type-safe data access between server components and the Drizzle ORM, reducing context switching.

4.  **Security Posture:** The framework's support for custom `next.config.ts` headers was a significant factor in implementing a comprehensive set of security measures, including `X-Frame-Options` and `Content-Security-Policy-Report-Only`. Starting with a report-only CSP allows for careful monitoring of potential violations in a production environment without immediately breaking functionality, providing a controlled path to full enforcement. The trade-off here is the temporary use of `'unsafe-inline'` for `script-src` in the CSP, which is a known limitation with Next.js's current architecture.

## Potential improvements

While the current implementation provides a solid foundation, several areas offer opportunities for enhancement:

1.  **Stricter Content Security Policy:** The `next.config.ts` file explicitly notes that `'unsafe-inline'` for `script-src` is "unfortunately required by Next.js today for inline bootstrap chunks." A concrete improvement would be to transition to a nonce-based CSP for `script-src` if Next.js or the application's build process evolves to support it, eliminating the need for `unsafe-inline` and further hardening the application against XSS attacks.

2.  **Sentry Cron Monitor Integration for App Router:** The `next.config.ts` includes a `webpack.automaticVercelMonitors` option within the Sentry configuration, with a comment stating, "(Does not yet work with App Router route handlers.)". As the App Router matures, enabling automatic Sentry Cron Monitors for the `/api/cron/reminders` and `/api/cron/whatsapp-watchdog` route handlers would provide more comprehensive monitoring and alerting for these critical background tasks.

3.  **Granular Rate Limiting for Professional Actions:** The existing `rateLimits` provide general protection for `api` and `booking` endpoints. However, the `professionals` table has `32` columns, indicating a rich set of configurable options. Implementing more granular rate limits for specific professional-facing actions, such as updating `sessionPrice` or `availability` settings, could prevent rapid, unintended changes or abuse, enhancing the stability and integrity of professional profiles.
