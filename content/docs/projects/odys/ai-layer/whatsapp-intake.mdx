# WhatsApp AI Intake

The `ai-layer/whatsapp-intake` component is a core part of Odys's automated client interaction, designed to streamline appointment scheduling directly through WhatsApp. It acts as an intelligent agent, receiving inbound messages from clients, interpreting their intent using a Groq Large Language Model (LLM), and performing actions like checking availability or booking appointments by calling specific tools. This system significantly reduces the manual overhead for professionals by automating the initial client engagement and scheduling process.

## Overview

At its heart, the WhatsApp AI Intake system is an event-driven architecture. When a client sends a message to a professional's WhatsApp number, it triggers an inbound webhook, specifically the `/api/whatsapp/webhook` route. While the implementation details of this webhook are not provided, its primary responsibility is to receive the message payload and delegate the AI processing to the `handleIncomingMessage` function located in `src/lib/ai-intake.ts`.

The `handleIncomingMessage` function orchestrates the entire AI interaction. It first establishes or retrieves the ongoing conversation state for the client and professional from Redis, managed by `src/lib/conversation.ts`. This state includes message history and contextual information, crucial for the AI to maintain coherence.

The system then engages with the Groq LLM, utilizing the `llama-3.3-70b-versatile` model, in a two-pass pattern:

1.  **First Pass**: The LLM receives the conversation history and a set of predefined tools. It determines if any of these tools are necessary to fulfill the client's request.
2.  **Tool Execution**: If the LLM decides to use a tool (e.g., `get_available_slots` or `book_appointment`), the system executes the corresponding function. These tool implementations interact with the Drizzle ORM, querying and updating various tables such as `professionals`, `availability`, `appointments`, `clients`, and `notifications`.
3.  **Second Pass**: The LLM receives the original conversation, its own initial decision, and the results from any executed tools. It then generates a natural language response to the client, incorporating the tool's output.

Throughout this process, the conversation state is continuously updated and saved back to Redis, ensuring that the AI maintains context across multiple messages. The system also leverages a set of 19 predefined WhatsApp message templates for various transactional communications, such as `msgBookingRequest`.

## Inbound Message Handling

The `handleIncomingMessage` function in `src/lib/ai-intake.ts` is the central processing unit for every incoming WhatsApp message. It begins by fetching the relevant professional's details from the `professionals` table using their `professionalId`. This information is vital for personalizing the AI's responses and understanding the professional's specific settings, such as `sessionDuration` and `autoConfirm` status.

A critical aspect of maintaining a natural dialogue is conversation state. The system uses `getConversation` from `src/lib/conversation.ts` to retrieve the ongoing chat history and context for the specific client and professional. If no prior conversation exists, a new one is initialized. During this initialization, the system attempts to identify the client by their `senderPhone` against existing entries in the `clients` table, allowing the AI to address known clients by name.

Before interacting with the LLM, a comprehensive set of messages is constructed. This includes a `systemPrompt` that defines the AI's persona as the professional's scheduling assistant, built dynamically using the professional's `name` and `profession`. Crucially, if the client's inbound message is a reply to a recent transactional notification (like a reminder or confirmation), the system injects an `outboundContext` into the LLM's prompt. This context, retrieved via `getOutboundContext` from `src/lib/conversation.ts`, informs the AI about the specific appointment the client might be referring to, enabling intelligent responses to phrases like "preciso remarcar".

The conversation history, including the new user message, is then passed to the Groq LLM. The LLM's response is processed: if it directly generates a reply, that reply is sent back to the client. If, however, the LLM determines that a tool needs to be called, the system executes the specified tool function with the arguments provided by the LLM. The results of these tool calls are then fed back to the LLM in a second pass, allowing it to formulate a final, informed response to the client. Finally, the updated conversation state, including the AI's reply, is saved using `saveConversation`, which also calls `markClientActive` to indicate the client is currently engaged.

## Tool Definitions and Implementations

The AI's capabilities are defined by a set of `TOOLS` that the Groq LLM can invoke. These tools are declared with a `name`, `description`, and `parameters` schema, guiding the LLM on when and how to use them.

### `get_available_slots`

This tool is designed to query the professional's schedule for open appointment times on a given date. It first retrieves the professional's `sessionDuration` from the `professionals` table. It then determines the day of the week in the "America/Sao_Paulo" timezone and fetches the professional's `availability` rules from the `availability` table for that specific day. To prevent double-bookings, it queries the `appointments` table for existing appointments, excluding those with `rejected` or `cancelled` statuses, within the specified date range. The tool then generates a list of potential slots by iterating through the availability rules and `addMinutes` based on `sessionDuration`, filtering out any slots that conflict with existing appointments or are in the past.

### `book_appointment`

This is the most critical tool, responsible for creating new appointments. It takes `date`, `time`, `client_name`, and `client_phone` as parameters. The function first normalizes the client's phone number using `normalizeBrazilianPhone`. To ensure data integrity and prevent race conditions, the entire booking process—from conflict checking to client upsertion and appointment creation—is wrapped in a Drizzle `transaction` with a `serializable` isolation level. This means that if two concurrent requests attempt to book the same slot, one will fail with a `SLOT_TAKEN` error (or a `40001` SQLSTATE), ensuring that no slot is accidentally double-booked.

Within the transaction:
1.  It checks for any conflicting appointments in the `appointments` table.
2.  It either finds an existing client in the `clients` table or creates a new one if the `normalizedPhone` is not found.
3.  It inserts the new appointment into the `appointments` table, setting its `status` to `confirmed` or `pending_confirmation` based on the professional's `autoConfirm` setting.

After a successful booking, the system dispatches several notifications: an in-app notification to the professional (stored in the `notifications` table), a WhatsApp message to the professional using the `msgBookingRequest` template, and an email to the professional via `sendBookingRequestEmailToProfessional` if their `email` is available.

### `get_professional_info`

This tool provides the LLM with details about the professional, including their `name`, `profession`, `sessionDuration`, `sessionPrice`, their general `schedule` based on the `availability` table, and their public `slug`. This allows the AI to answer client questions about the professional's services and working hours.

## Conversation Management

The `src/lib/conversation.ts` file is dedicated to managing the transient state of client-AI interactions using Redis.

The `ConversationState` interface defines the structure for storing messages, client details, and contextual flags like `pendingDate` or `appointmentCreated`. Each conversation is uniquely identified by a key generated by `convKey`, which combines the `professionalId` and a `stripDigits`-normalized `clientPhone`. This normalization ensures consistent key generation regardless of the input phone format. Conversations are stored with a `TTL` of 30 minutes, balancing responsiveness with resource usage. To prevent context windows from growing indefinitely, `saveConversation` trims the message history to the `MAX_MESSAGES` (20) most recent entries.

Beyond conversation history, this module also manages client activity and outbound context:

*   **Client Activity**: The `markClientActive` function sets a `client_active` key in Redis with an `ACTIVE_TTL` of 15 minutes whenever a conversation is saved. The `isClientActive` function checks this key. This mechanism is crucial for the system's reminder cron job. If a client is actively conversing with the AI, reminders are deferred to avoid interrupting the ongoing dialogue with automated messages.
*   **Outbound Context**: The `setOutboundContext` and `getOutboundContext` functions store information about the last transactional message sent to a client (e.g., a booking confirmation or reminder), including the `professionalId` and `appointmentId`. This context is stored with an `OUTBOUND_TTL` of 2 hours. When an inbound message arrives, `handleIncomingMessage` retrieves this context, allowing the AI to understand if the client's reply is related to a specific prior notification, enhancing the relevance of the AI's response.

## Design Decisions

The architecture of the WhatsApp AI Intake agent reflects several deliberate design choices aimed at balancing responsiveness, accuracy, and data integrity:

*   **Two-Pass LLM Interaction**: The decision to use a two-pass approach for LLM interactions (first for tool selection, second for response generation) is a common pattern in AI agents. It provides greater control and reliability. The first pass focuses solely on understanding intent and identifying necessary actions, while the second pass can then synthesize a coherent, context-aware response using the concrete results of those actions. This separation helps prevent the LLM from "hallucinating" tool results or generating responses based on speculative outcomes.
*   **Serializable Transaction for `book_appointment`**: The `book_appointment` function employs a Drizzle `transaction` with a `serializable` isolation level. This is a critical choice for ensuring atomicity and preventing race conditions in a multi-user booking system. By guaranteeing that concurrent transactions appear to execute serially, it robustly handles scenarios where multiple clients might attempt to book the same slot simultaneously, ensuring only one succeeds and maintaining the integrity of the `appointments` table.
*   **Redis for Conversation State Management**: Using Redis for `ConversationState` is a pragmatic choice for storing transient, high-volume conversational data. Redis's in-memory nature provides low-latency access, which is essential for a responsive AI agent. The built-in TTL (30 minutes for conversations, 15 minutes for active status, 2 hours for outbound context) automatically handles cleanup of stale data, preventing unbounded memory growth and simplifying state management.
*   **Explicit Timezone Handling**: The code explicitly handles timezones, particularly for "America/Sao_Paulo", using functions like `saoPauloDate`, `formatSaoPauloTime`, and `formatSaoPauloDate`. This is crucial for an appointment booking system to ensure that dates and times are interpreted and displayed correctly, avoiding common pitfalls associated with timezone ambiguities and daylight saving changes (though Brazil abolished DST in 2019).
*   **Outbound Context for AI Grounding**: Storing `OutboundContext` in Redis and feeding it back to the LLM is a sophisticated design choice. It allows the AI to "remember" the last transactional message sent to a client. This significantly improves the AI's ability to understand follow-up messages, such as "preciso remarcar" or "confirma por favor", by grounding them to a specific `appointmentId` and `professionalId`, rather than treating every inbound message as a fresh, uncontextualized query.

## Potential Improvements

1.  **Robust Notification Error Handling**: In `bookAppointment`, the `sendWhatsApp` and `sendBookingRequestEmailToProfessional` calls use `.catch((err) => console.warn(...))`. While this prevents the main booking flow from failing, it offers limited resilience. A more robust approach could involve implementing a retry mechanism, pushing failed notifications to a dead-letter queue for later processing, or integrating with a dedicated notification service that handles delivery guarantees and error reporting. This would ensure that professionals reliably receive booking requests even if external communication services experience transient issues.
2.  **Dynamic Tool Descriptions and Context**: The `TOOLS` array in `src/lib/ai-intake.ts` contains static descriptions for the LLM. While functional, these could be enhanced. For instance, `get_available_slots` could dynamically include the professional's `profession` or `sessionDuration` in its description to guide the LLM more precisely (e.g., "Retorna horários disponíveis para uma sessão de 60 minutos de massagem"). Similarly, `book_appointment` could mention the professional's `name` or `profession` to reinforce context. This would make the LLM's tool-calling decisions more informed and potentially reduce errors.
3.  **Enhanced Client Name Inference**: Currently, if `conversation.clientName` is `null`, the system explicitly tells the LLM to ask for the client's name. The `bookAppointment` tool also defaults to "Cliente WhatsApp" if `client_name` is not provided. A potential improvement would be to implement a more sophisticated client name inference mechanism. This could involve using the LLM to extract a name from the initial user message, or performing a fuzzy match against existing `clients` records based on phone number and message content, reducing the need for explicit prompting.
4.  **Webhook Code Documentation**: The `src/app/api/whatsapp/webhook` file is noted as unreadable. Documenting its implementation, including how it validates incoming WhatsApp messages (e.g., signature verification), extracts the `senderPhone` and `message` content, and then calls `handleIncomingMessage`, would provide a complete picture of the system's entry point and security considerations.
