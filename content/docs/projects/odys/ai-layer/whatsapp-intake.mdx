# WhatsApp AI Intake

The `src/lib/ai-intake.ts` module orchestrates the Odys WhatsApp Concierge, a central AI persona designed to manage all inbound WhatsApp messages directed to the corporate Odys number. This agent serves as a comprehensive assistant for clients, guiding them through the process of finding a professional, browsing their schedules, and booking appointments across the entire Odys marketplace. Unlike a dedicated AI for a single professional, this concierge operates at a platform level, connecting clients with the right service provider.

## Overview

The core logic for the WhatsApp AI Concierge resides within `src/lib/ai-intake.ts`, which acts as the brain of the operation. Incoming WhatsApp messages are routed through the `/api/whatsapp/webhook` API endpoint, which then dispatches them to the AI intake agent for processing.

The agent employs a sophisticated conversation state machine to adapt its behavior:
*   **Discovery**: When no specific professional is selected, the AI guides the user to articulate their needs, often by asking about the desired profession or location. It then uses tools like `search_professionals` to present suitable matches.
*   **Focused**: Once a user expresses interest in a particular professional, the `selectedProfessionalId` is set, and the AI shifts its focus to fetching available slots and facilitating booking for that specific provider.
*   **Reset**: Users can explicitly request to search for "outro profissional" (another professional), which clears the current selection and returns the conversation to the Discovery state.

A key architectural choice is the "two-pass tool-calling" mechanism. In the first pass, the Large Language Model (LLM) may identify the need to call one or more tools based on the user's query. After executing these tools and receiving their results, the LLM performs a second pass, using the tool outputs as additional context to formulate a coherent and helpful final reply to the user. This iterative approach allows for more complex reasoning and action execution.

The system relies on a Drizzle ORM schema comprising 11 tables, with critical interactions involving `professionals`, `availability`, `appointments`, `clients`, `notifications`, `reviews`, and `agentMetrics`. Communication with clients and professionals is facilitated by 19 distinct WhatsApp message templates, such as `msgBookingRequest` and `msgBookingConfirmed`. To prevent abuse and ensure system stability, the `/api/whatsapp/webhook` endpoint is protected by a rate limit of 10 requests per minute.

## Conversation Management

Effective AI conversations require robust state management, which is handled by `src/lib/conversation.ts`. This module uses Redis to store `ConversationState` objects, each uniquely identified by a client's normalized phone number.

A `ConversationState` object captures essential details like `clientPhone`, `clientName`, the `selectedProfessionalId` (crucial for the multi-pro concierge model), and a history of `messages`. It also maintains a `context` object for transient information, such as `pendingDate`, `pendingSlot`, and `userCity`, which helps the AI maintain continuity across turns. Conversations are designed to be relatively short-lived, with a `TTL` of 30 minutes in Redis, reflecting the typical duration of an active booking dialogue. The `messages` array is capped at `MAX_MESSAGES` (12 entries) to keep the LLM context window manageable and cost-effective.

To ensure data integrity and prevent race conditions, `saveConversation` implements an optimistic concurrency control mechanism using a `version` field. Each successful save increments this version. If a concurrent write attempts to save an older version of the state, a `ConversationStaleError` is thrown, preventing silent data corruption. This is complemented by a per-phone message lock, `acquireMessageLock`, which attempts to serialize message processing for a given client, releasing it via `releaseMessageLock` once a reply is sent. This lock has a `LOCK_TTL` of 45 seconds, designed to cover even the slowest LLM round-trips.

Beyond conversation history, `src/lib/conversation.ts` also manages a `client_active` key, which is touched whenever a conversation is saved. This key, with an `ACTIVE_TTL` of 15 minutes, signals that a client is actively engaged with the AI. This is particularly useful for cron jobs, such as `/api/cron/reminders`, which can defer sending transactional reminders if a client is mid-conversation, avoiding disruptive interleaving of messages.

## Tool Definitions and Implementations

The AI's ability to interact with the Odys platform is encapsulated in a set of OpenAI-compatible tools defined in the `TOOLS` array within `src/lib/ai-intake.ts`. These tool descriptions are intentionally terse to minimize token usage, as the entire array is sent with every LLM request. The prompt itself is designed to guide the LLM on when and how to use each tool.

Key tools include:

*   **`search_professionals`**: This tool is fundamental for the Discovery phase. It allows the AI to search for professionals based on `profession`, `city`, `name`, and `service_type`. It intelligently uses `resolveProfessionAlias` to map Portuguese terms (e.g., "psicologa") to canonical English keys. The search results are enriched with statistics from `getProStats` (average rating, review count, completed appointments) and then ranked using `rankScore`. This ranking considers factors like `avgRating`, `reviewCount`, `completedCount`, `plan` tier, and, if a city centroid can be geocoded via `geocodeCity`, the `distanceKm` from that centroid. If no professionals are found in a specific city, it gracefully falls back to `findNearestProfessionals` to suggest options in nearby locations.
*   **`list_professions`**: To prevent dead-end searches, this tool dynamically lists only those professions that currently have at least one active professional registered on the platform.
*   **`get_professional_info`**: Once a professional is selected, this tool retrieves detailed information, including their `bio`, `session_duration_min`, `session_price_brl`, `service_type`, `address`, and `schedule` (derived from `availability` rules). It also fetches `avg_rating`, `review_count`, `completed_count`, and recent `reviews` to provide social proof. Payment policies are dynamically built using `buildPaymentPolicy`.
*   **`get_available_slots`**: This critical tool computes actual available appointment slots for a given `professionalId` and `date`. It first uses `computeSlotsForDate` to determine availability based on the professional's weekly `availability` rules and existing `appointments`. It also handles parsing Portuguese date expressions via `parsePtDate`. If no slots are found on the requested date, it proactively looks for and suggests alternatives on the next 7 days, presenting up to 3 alternative dates with available slots.
*   **`book_appointment`**: This is the core transaction tool. It takes `professionalId`, `date`, `time`, `client_name`, and `client_phone` as arguments. It normalizes the client's phone number using `normalizeBrazilianPhone` and performs several checks: ensuring the appointment is in the future, and checking for existing active appointments for the same client with the same professional to prevent duplicates. The booking itself occurs within a Drizzle transaction with `serializable` isolation, which includes checks for slot conflicts and either creates a new client entry or reuses an existing one. The appointment's initial `status` (`confirmed` or `pending_confirmation`) depends on the professional's `autoConfirm` setting. Post-booking, it triggers a series of notifications: an internal `notifications` entry, WhatsApp messages to both client and professional (using `msgBookingConfirmed` or `msgBookingRequest` templates), and an email to the professional via `sendBookingRequestEmailToProfessional`. For professionals with upfront or percentage payment policies, it also generates `paymentInstructions`, including a copy-pasteable PIX payload using `buildPixPayload` if a PIX key is configured.
*   Other tools like `select_professional`, `list_my_appointments`, `cancel_appointment`, and `reschedule_appointment` provide additional conversational capabilities.

## Data Persistence and Metrics

The AI intake agent interacts extensively with the Drizzle ORM to persist and retrieve data. Key tables involved are:
*   `professionals`: Stores details about service providers, including their availability, pricing, and contact information.
*   `availability`: Defines the weekly working hours for each professional.
*   `clients`: Manages client profiles, linked to specific professionals.
*   `appointments`: Records all scheduled appointments, their status, and payment details.
*   `Conversation history is stored in Redis within the ConversationState.messages object.`
*   `notifications`: Logs internal notifications for professionals.
*   `reviews`: Stores client feedback and ratings for professionals.
*   `agentMetrics`: This table is crucial for monitoring the AI's performance and cost. It records `traceId`, `conversationPhone`, `promptVersion`, `promptHash`, `totalCostUsd`, `totalCostBrl`, `totalLatencyMs`, `toolCallCount`, `providersUsed`, and `outcome` for each AI interaction, providing valuable insights into the agent's efficiency and effectiveness.

The LLM interactions themselves are managed by `chatWithFallback`, which intelligently switches between providers (Groq, Cerebras, Gemini) to ensure availability and potentially optimize cost. The `calculateCost` utility helps track the expenses associated with these LLM calls, which are then logged to `agentMetrics`.

## Design Decisions

The design of the WhatsApp AI Concierge reflects several deliberate choices aimed at optimizing user experience, system reliability, and operational efficiency:

*   **Two-Pass Tool Calling**: The decision to use a two-pass approach for tool execution allows the LLM to first identify the need for a tool, then execute it, and finally synthesize the tool's output into a natural language response. This prevents the LLM from "hallucinating" tool results and leads to more accurate and contextually relevant replies, especially for complex booking flows.
*   **Redis for Ephemeral Conversation State**: Using Redis for `ConversationState` provides a fast, in-memory store suitable for the transient nature of conversations. The `TTL` ensures that stale conversations are automatically purged, preventing memory bloat and maintaining data relevance. The `MAX_MESSAGES` limit further controls the context size, directly impacting LLM costs and latency.
*   **Optimistic Concurrency and Locking**: The `version` field in `ConversationState` and the `acquireMessageLock` mechanism are critical for handling concurrent messages from the same client. While the lock is the primary defense, optimistic concurrency provides a robust fallback, preventing data loss or corruption if the lock mechanism were to fail or if a race condition occurred in the brief window between reading and writing state. The choice to throw `ConversationStaleError` rather than silently overwrite ensures that potential issues are surfaced.
*   **Terse Tool Descriptions**: The tool descriptions are kept minimal to reduce the number of tokens sent with each LLM request. This is a direct cost-saving measure, as LLM pricing is often based on token usage. The complexity of guiding the LLM on tool usage is shifted to the main prompt, which is typically larger but sent less frequently.
*   **Brazil-Specific Localization**: The inclusion of functions like `saoPauloDate`, `normalizeBrazilianPhone`, `parsePtDate`, `resolveProfessionAlias`, and `buildPixPayload` demonstrates a strong commitment to the Brazilian market. These utilities ensure that the AI can correctly interpret local date/time formats, handle phone numbers, understand profession aliases, and facilitate common payment methods like PIX, providing a truly localized and effective experience.
*   **Proactive Slot Alternatives**: When `getAvailableSlots` finds no availability on a requested date, it automatically searches for and suggests slots on nearby dates. This design choice minimizes user frustration and reduces the number of turns required to find a suitable appointment, improving the overall efficiency of the booking process.
*   **Professional Ranking**: The `rankScore` function in `search_professionals` is a deliberate effort to present the most relevant and high-quality professionals first. By incorporating factors like ratings, review counts, completed appointments, and subscription plan tiers, the system prioritizes professionals who are active, well-regarded, and contribute more to the platform.

## Potential Improvements

1.  **Graceful `ConversationStaleError` Handling**: The `saveConversation` function currently throws a `ConversationStaleError` if an optimistic concurrency conflict is detected. While this prevents data corruption, the calling context (likely the webhook handler) might have already sent a reply to the user. A more sophisticated handling mechanism could involve logging the conflict with higher severity and potentially attempting a re-fetch and re-process of the message, or at least ensuring the user is informed if their last message's context was not fully saved.
2.  **Enhanced `acquireMessageLock` Reliability**: The `acquireMessageLock` function is designed to "fail open" on Redis errors, meaning it will return `true` and allow processing to continue if Redis is unavailable. While this prevents message drops, it increases the risk of concurrent processing and non-deterministic state mutations during Redis outages. Implementing a circuit breaker pattern or a more robust fallback mechanism (e.g., queuing messages for later processing) would improve resilience without sacrificing data integrity.
3.  **Refined Location-Based Search**: The `search_professionals` tool currently uses `geocodeCity` to find a city centroid for ranking purposes. While useful, it's a broad approximation. Introducing a `near` parameter to the `search_professionals` tool, allowing users to specify a neighborhood or a more precise location, would enable `findNearestProfessionals` to provide significantly more accurate and relevant results for in-person services. This would require updating the tool definition and its implementation to accept and utilize this additional geographical context.
4.  **Robust Notification Error Reporting**: The `bookAppointment` function includes `catch` blocks for `sendWhatsApp` and email notifications, logging warnings if they fail. For critical notifications like booking confirmations, a simple warning might not be sufficient. Implementing a dedicated retry mechanism for failed notifications or integrating with an external alerting system would ensure that these vital communications are delivered, or that failures are promptly addressed.
