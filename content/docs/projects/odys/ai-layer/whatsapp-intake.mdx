The `ai-layer/whatsapp-intake` module orchestrates the Odys WhatsApp AI Concierge, a central intelligence designed to manage all inbound messages to the corporate Odys number. This agent serves as a multi-professional discovery and booking system, guiding clients through finding a suitable professional, browsing their schedules, and ultimately booking appointments across the entire platform. It represents a critical touchpoint for client acquisition and engagement, automating initial interactions and streamlining the booking process.

## Overview

The core logic for the WhatsApp AI Concierge resides within `src/lib/ai-intake.ts`, which defines the AI's persona and its interaction capabilities. This system operates on a "two-pass tool-calling" mechanism: the Language Model (LLM) first determines if a tool is needed based on the user's query, and then, with the tool's results in context, generates a final, informed reply. This approach allows for complex, multi-step interactions, such as searching for professionals and then booking an appointment.

The conversation flow is managed by a state machine, transitioning between a "Discovery" phase where no specific professional is selected, and a "Focused" phase once a professional has been chosen. Users can "Reset" the conversation to clear their selection and start a new search. Conversation state, including messages and context like `selectedProfessionalId` and `userCity`, is persisted in Redis via `src/lib/conversation.ts` for a `TTL` of 30 minutes, ensuring continuity across interactions. A `MAX_MESSAGES` limit of 12 is applied to keep the context concise for each LLM call.

All inbound WhatsApp messages are processed through the `/api/whatsapp/webhook` API route, which is subject to a `whatsapp` rate limit of 10 requests per minute to prevent abuse and manage load. The system uses Groq as its primary LLM provider, leveraging the `chatWithFallback` utility for resilience.

The AI agent interacts extensively with the Odys database, utilizing Drizzle ORM to manage data across 10 tables, including `professionals`, `availability`, `appointments`, `clients`, `notifications`, and `reviews`. It also integrates with the WhatsApp Business API for sending structured messages using 19 predefined message templates, and with email services for professional notifications.

To prevent race conditions and ensure deterministic state updates when multiple messages arrive from the same client, a per-phone processing lock is implemented using Redis, managed by `acquireMessageLock` and `releaseMessageLock` in `src/lib/conversation.ts`. This lock has a `LOCK_TTL` of 45 seconds, designed to cover the slowest LLM round-trip. Additionally, a `client_active` key is maintained to defer transactional reminders if a client is actively conversing with the AI, preventing interruptions.

### Tool Definitions and Implementations

The `src/lib/ai-intake.ts` file defines a set of OpenAI-compatible tools that the LLM can invoke. These tool descriptions are intentionally terse to minimize token usage, as the entire `TOOLS` array is sent with every LLM request. The tools enable the AI to perform various actions:

*   **`search_professionals`**: This tool is fundamental for discovery. It allows the AI to find professionals based on `profession`, `city`, or `name`. It uses `resolveProfessionAlias` to handle Portuguese aliases for professions (e.g., "psicologa" to "psychologist") and `ilike` for flexible matching. Results are enriched with `getProStats` (average rating, review count, completed appointments) and ranked using a `rankScore` that considers rating, popularity, and the professional's `plan` tier. If no professionals are found in a specific `city`, it intelligently falls back to `findNearestProfessionals` to suggest options in nearby locations.
*   **`list_professions`**: This tool provides a dynamic list of professions that currently have active professionals on the platform. This prevents the AI from suggesting professions for which no services are available, improving user experience.
*   **`select_professional`**: The `select_professional` tool is designed to set the `selectedProfessionalId` in the conversation state, focusing subsequent interactions on that professional.
*   **`get_professional_info`**: Retrieves detailed information about a selected professional, including their `bio`, `address`, `schedule` (derived from `availability` rules), `sessionDuration`, `sessionPrice`, `serviceType`, and `payment_policy`. It also fetches social proof like `avg_rating`, `review_count`, `completed_count`, and recent `reviews` snippets.
*   **`get_available_slots`**: This critical tool computes real-time available appointment slots for a given professional and `date`. It considers the professional's `availability` rules and existing `appointments` to identify conflicts. It also handles `time_preference` (e.g., "manha", "tarde") and, if no slots are found on the requested date, proactively suggests `alternatives` from the next 7 days to minimize user friction. Brazilian-specific date and time formatting (`saoPauloDate`, `formatSaoPauloTime`, `formatSaoPauloDate`) are used throughout.
*   **`book_appointment`**: The `book_appointment` tool is designed to handle the creation of new appointments, including validation, conflict checks, client management, status assignment based on `autoConfirm`, and sending various notifications and payment instructions.
*   **`list_my_appointments`**: Allows clients to view their upcoming appointments with any professional by providing their phone number.
*   **`cancel_appointment`**: Enables clients to cancel their own appointments. It includes an authorization check to ensure only the client associated with the phone number can cancel their booking.
*   **`reschedule_appointment`**: This tool facilitates changing an existing appointment's date and time. The provided code snippet is truncated, but its purpose is to update an appointment after checking for availability and conflicts, similar to the booking process.

## Design Decisions

The architecture of the WhatsApp AI Concierge reflects several deliberate design choices aimed at efficiency, user experience, and system reliability:

*   **Two-Pass Tool-Calling**: The decision to use a two-pass approach for tool invocation (LLM decides tool, then LLM generates response with tool output) allows the AI to reason more effectively. It separates the "what to do" from the "what to say," leading to more accurate tool selections and more coherent, context-aware responses.
*   **Ephemeral Conversation State in Redis**: Storing conversation state in Redis with a relatively short `TTL` (30 minutes) is a trade-off. It provides fast access for ongoing dialogues, which is crucial for a responsive chat agent. The ephemerality means that long-dormant conversations are automatically cleaned up, reducing storage overhead. The `MAX_MESSAGES` limit further optimizes token usage for LLM calls.
*   **`selectedProfessionalId` for Focused Interactions**: Introducing `selectedProfessionalId` in the conversation state allows the AI to transition from a broad discovery mode to a focused interaction with a specific professional. This prevents the AI from repeatedly asking for professional details once a choice has been made, streamlining the booking flow.
*   **Transactional Booking with `autoConfirm`**: The `book_appointment` function uses a Drizzle transaction to ensure atomicity when creating appointments and clients. The `autoConfirm` flag on the `professionals` table provides flexibility, allowing professionals to choose whether bookings are instantly confirmed or require manual approval, catering to different business models.
*   **Per-Phone Message Lock**: The `acquireMessageLock` mechanism is a critical concurrency control measure. By preventing simultaneous processing of messages from the same client, it eliminates race conditions that could lead to inconsistent conversation states or duplicate actions (e.g., double-booking). The fail-open design on Redis errors prioritizes message processing over strict consistency in a cache outage scenario.
*   **Proactive Reminder Deferral**: The `isClientActive` check, which marks a client as active during an AI conversation, is used by the reminder cron job (`/api/cron/reminders`). This prevents transactional reminders from interrupting an active AI-driven scheduling dialogue, enhancing the user experience by avoiding message interleaving.
*   **Brazilian-Specific Date/Time/Phone Handling**: The explicit use of `saoPauloDate`, `parsePtDate`, `parsePtTime`, and `normalizeBrazilianPhone` demonstrates a commitment to localization. This ensures that the AI correctly understands and communicates dates, times, and phone numbers in the Brazilian context, which is essential for accurate scheduling and communication.
*   **Professional Ranking and Nearest Fallback**: The `searchProfessionalsTool` incorporates a composite `rankScore` and a "nearest professionals" fallback. This design prioritizes presenting the most relevant and highly-regarded professionals first, while also providing helpful alternatives when an exact match isn't found, improving the utility of the search feature.

## Potential Improvements

1.  **Incomplete `rescheduleAppointment` Logic**: The provided code for the `rescheduleAppointment` function in `src/lib/ai-intake.ts` is truncated. To ensure a complete and robust implementation, the full function should include comprehensive conflict checks (similar to `book_appointment`), appropriate notification logic for both client and professional, and clear return values for success or failure.
2.  **Centralized WhatsApp/Email Error Handling**: The `sendWhatsApp` and email notification calls within `book_appointment` and `cancel_appointment` in `src/lib/ai-intake.ts` currently handle errors with `.catch((err) => console.warn(...))`. While this logs issues, it lacks retry mechanisms or dead-letter queue integration. Implementing a more resilient messaging pattern, such as a dedicated queue for outbound messages with retries and failure handling, would prevent missed notifications due to transient network issues or API outages.
3.  **Dynamic Conversation Context Management**: The `MAX_MESSAGES` constant in `src/lib/conversation.ts` is fixed at 12. For more complex or extended conversations, this limit could lead to the AI losing context of earlier dialogue turns. A more dynamic approach, perhaps based on token count limits or a summarization step for older messages, could allow for longer, more coherent interactions without excessively increasing LLM call costs.
4.  **`listProfessionsTool` "Other" Exclusion**: The `listProfessionsTool` in `src/lib/ai-intake.ts` explicitly filters out the "other" profession key (`key !== "other"`). If the platform were to expand to include bookable professionals under an "other" category, this hardcoded exclusion would prevent them from being listed by the AI. A more flexible approach might involve a configuration flag or a database-driven list of "AI-discoverable" professions.
5.  **`searchProfessionalsTool` Result Limiting**: The `searchProfessionalsTool` in `src/lib/ai-intake.ts` initially fetches up to 20 rows from the database but then slices the `enriched` matches to the top 10. While ranking is applied, this hard limit might discard potentially relevant professionals if the initial 20 results are not diverse enough or if the ranking algorithm needs to consider a wider pool to find the absolute best matches.
