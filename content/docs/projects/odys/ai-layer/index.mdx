# AI Layer

The AI layer powers both client-facing concierge interactions and professional analytics through two distinct agents. Built on Groq for low-latency inference, it uses structured tool-calling to safely bridge LLMs with live database state—enabling natural language booking flows and performance insights without hallucination.

## Overview

Two separate agents serve different user roles: the **WhatsApp Concierge** handles inbound client messages on the corporate number, enabling multi-professional discovery and booking; the **Professional Assistant** gives logged-in professionals natural language access to their appointment stats, upcoming sessions, and no-show trends. Both use `chatWithFallback`—a provider chain (Groq → Cerebras → Gemini)—ensuring availability even under load. The system leverages 12 distinct tools across both agents, such as `book_appointment` and `get_stats`, with rate limits (`aichat`: 20/h) preventing abuse. Conversations are stateful via `getConversation`/`saveConversation`, and all AI actions are logged in `agentMetrics` for cost and latency tracking. The WhatsApp Concierge agent implements a state machine for conversations, transitioning between 'Discovery' (finding a pro), 'Focused' (interacting with a selected pro), and 'Reset' states. Profession labels are resolved using a `PROFESSIONS` constant, ensuring human-readable names are displayed and only active professions are offered to clients. The Concierge agent includes a profession alias resolver (`resolveProfessionAlias`) to map common Portuguese terms to canonical English profession keys, improving search accuracy.

## Design decisions

The dual-agent pattern isolates concerns: the concierge operates on public data (via `search_professionals`, `get_available_slots`) and drives bookings, while the professional assistant accesses private analytics (`get_stats`, `get_no_show_clients`). Tool definitions are kept minimal—parameter descriptions are terse because the prompt already guides usage, reducing token overhead per request. Timezone handling is hardcoded to UTC-3 (São Paulo) since the platform targets Brazil exclusively, avoiding DST complexity. The two-pass tool-calling flow ensures the LLM only replies after seeing real data, preventing fabricated responses. State persistence via `ConversationState` allows context retention across messages without relying on ephemeral sessions. The Professional Assistant's behavior is guided by a system prompt that includes specific formatting rules for responses, such as detailed breakdowns for no-show rates and monthly summaries.

## Potential improvements

1. **`src/lib/ai-intake.ts` – `searchProfessionalsTool`**: The fallback to `ilike(professionals.profession, ...)` when an alias isn’t found could return noisy results. Adding a Levenshtein distance check or using a dedicated fuzzy matching library would improve precision.

2. **`src/app/api/ai/chat/route.ts` – `getStats`**: The function computes no-show rates client-side after fetching raw statuses. Moving this aggregation to SQL with window functions would reduce memory usage and improve performance, especially for professionals with thousands of appointments.

3. **`src/lib/ai-intake.ts` – `bookAppointment`**: The transaction retries only on known error codes but doesn’t implement exponential backoff. Adding a retry loop with jitter would make the function more resilient to transient serialization failures under high concurrency.
