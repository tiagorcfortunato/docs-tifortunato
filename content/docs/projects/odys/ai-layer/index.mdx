# AI Layer

Odys incorporates an AI layer designed to enhance both client acquisition and professional management. This layer is built around two distinct AI agents, each serving a specific purpose: an external-facing WhatsApp concierge that guides clients through professional discovery and booking, and an internal professional assistant that provides business insights to subscribed professionals. Both agents utilize Groq as their primary large language model (LLM) provider, leveraging its speed for interactive experiences, and are designed with an OpenAI-compatible tool-calling interface for flexibility.

## Overview

The AI layer in Odys is structured around a clear separation of concerns, with two primary agents operating independently to address different user needs:

1.  **WhatsApp Concierge (`src/lib/ai-intake.ts`)**: This agent acts as the primary interface for new and existing clients interacting with Odys via WhatsApp. Its core function is to facilitate the discovery of professionals, provide detailed information, and manage the booking process across the entire marketplace. It's designed to handle a conversational flow, moving from general inquiries to specific bookings, and even managing self-service actions like listing, canceling, or rescheduling appointments.
2.  **Professional Assistant (`src/app/api/ai/chat/route.ts`)**: This agent is an internal tool available to professionals within their Odys dashboard. It helps professionals understand their business performance by providing insights into appointments, client behavior, and revenue. This assistant is designed to answer specific queries about their practice, such as no-show rates or upcoming schedules, by calling specialized data retrieval tools.

Both agents rely on a "two-pass tool-calling" mechanism, where the LLM first determines if a tool is needed, executes it, and then uses the tool's output to formulate a final, context-aware response. This approach allows the LLM to perform complex actions and retrieve specific data before generating a user-friendly reply.

## WhatsApp Concierge

The `src/lib/ai-intake.ts` file defines the logic for the Odys WhatsApp Concierge. This agent is a multi-professional discovery and booking system, designed to handle all inbound WhatsApp messages on the corporate Odys number. It guides clients through finding a professional by profession, city, or name, browsing their schedule, and booking appointments.

The concierge operates with a conversation state machine:
-   **Discovery**: When no professional is selected (`selectedProfessionalId` is null), the AI prompts the user for their needs, calls `search_professionals`, and presents matching professionals.
-   **Focused**: Once a `selectedProfessionalId` is set, the AI focuses on that professional, fetching available slots or proceeding with booking.
-   **Reset**: Users can explicitly ask for "outro profissional" to clear the current selection and return to discovery mode.

The concierge interacts with several database tables to fulfill its functions, including `professionals`, `availability`, `appointments`, `clients`, `notifications`, and `reviews`. It also integrates with external services for WhatsApp messaging (`sendWhatsApp`) and email notifications (`sendBookingRequestEmailToProfessional`, `sendBookingConfirmedEmailToClient`).

### Tools

The WhatsApp Concierge utilizes a suite of 9 OpenAI-compatible tools to perform its functions:

-   `search_professionals`: This tool is crucial for the discovery phase. It queries the `professionals` table, filtering by `profession`, `city`, and `name`. It also incorporates a `resolveProfessionAlias` utility to handle Portuguese aliases for professions and `findNearestProfessionals` as a fallback if no direct matches are found in a requested city. Results are enriched with `avg_rating`, `review_count`, and `completed_count` from `getProStats` and ranked using a `rankScore` that considers rating, popularity, and plan tier.
-   `list_professions`: This tool dynamically lists only the professions that have at least one active professional registered in the `professionals` table, preventing the AI from suggesting unavailable services.
-   `select_professional`: This tool updates the conversation state to focus on a specific professional, identified by their `professional_id`.
-   `get_professional_info`: Retrieves detailed information about a professional from the `professionals` table, including their `bio`, `address`, `schedule` (from `availability`), `avg_rating`, `review_count`, `completed_count`, `payment_policy`, and recent `reviews`.
-   `get_available_slots`: This tool is critical for booking. It calculates actual available time slots for a given `professional_id` and `date` by cross-referencing the `availability` rules and existing `appointments`. It also proactively suggests alternative dates if the requested date has no availability.
-   `book_appointment`: Handles the creation of new appointments. It first checks for conflicts in the `appointments` table and ensures the client exists in the `clients` table (creating a new entry if necessary). It then inserts the new appointment, setting its `status` to "confirmed" or "pending_confirmation" based on the professional's `autoConfirm` setting. It triggers WhatsApp and email notifications and generates PIX payment instructions if applicable.
-   `list_my_appointments`: Allows clients to view their future appointments by querying the `appointments` table, filtered by the client's phone number.
-   `cancel_appointment`: Enables clients to cancel their own appointments. It updates the `status` in the `appointments` table and notifies the professional via WhatsApp and notifications.
-   `reschedule_appointment`: Facilitates changing the date and time of an existing appointment.

## Professional Assistant

The `src/app/api/ai/chat/route.ts` file implements the API endpoint for the internal professional assistant. This assistant is designed to help professionals understand their business data. It operates within a `POST` API route, requiring user authentication (`getUser`) and professional identification (`getProfessional`).

A key design choice for this assistant is the implementation of rate limiting (`getAiChatLimiter`) per authenticated user, with a limit of 20 requests per hour. This prevents abuse and manages LLM costs. Additionally, access to this feature is gated by the professional's subscription plan (`canUseFeature`), making it available only to "Pro" and "Premium" subscribers.

The assistant is guided by a `SYSTEM_PROMPT` that instructs it to respond in Portuguese and use tools to retrieve data, emphasizing that it should never invent numbers.

### Tools

The Professional Assistant uses 3 tools to provide insights:

-   `get_stats`: This tool retrieves comprehensive statistics for the professional over the last 6 months. It queries the `appointments` table to calculate global and month-by-month summaries, including total appointments, completed sessions, no-shows, no-show rates, and estimated revenue. The `sessionPrice` from the `professionals` table is used for revenue calculations.
-   `get_upcoming`: This tool fetches appointments scheduled for the next 7 days, providing details like client name, date, time, and status. It queries the `appointments` and `clients` tables.
-   `get_no_show_clients`: This tool identifies clients with the highest number of no-shows over the past 6 months. It aggregates data from `appointments` and `clients` to rank clients by their no-show count and rate.

## Design Decisions

The decision to implement two distinct AI agents stems from a clear separation of concerns. The **WhatsApp Concierge** is optimized for external, public-facing interactions, focusing on discovery and transactional booking across the entire Odys marketplace. Its narrative flow and toolset are geared towards guiding new and existing clients. In contrast, the **Professional Assistant** is an internal, value-added feature for subscribed professionals, providing analytical insights into their specific practice. This separation allows for tailored system prompts, tool definitions, and access controls (like plan guarding and rate limiting) appropriate for each use case.

Both agents utilize an OpenAI-compatible tool definition format, which is a strategic choice to ensure **provider flexibility**. While Groq is the primary LLM, this compatibility means Odys can easily switch to or integrate other LLMs like Cerebras or Gemini without rewriting the tool definitions themselves. This reduces vendor lock-in and allows Odys to adapt to evolving LLM capabilities and cost structures.

The **two-pass tool-calling** mechanism is employed to enhance the quality and reliability of AI responses. In the first pass, the LLM analyzes the user's query and determines if any tools are necessary. If a tool is called, its results are then fed back into the LLM as context for a second pass. This ensures that the final response is not a speculative guess but is grounded in actual data retrieved from the database or external services. For instance, the `book_appointment` tool in the concierge can return a `duplicate_active_booking` reason, which the LLM can then use to suggest rescheduling instead of attempting a new booking.

Database interactions are managed through Drizzle ORM, which provides a type-safe and expressive way to query the PostgreSQL database. This choice contributes to developer productivity and reduces the likelihood of SQL injection vulnerabilities or runtime errors due to schema mismatches. The schema, comprising 6 tables like `professionals`, `appointments`, and `clients`, is central to both AI agents' ability to retrieve and manipulate business data.

## Potential Improvements

1.  **Dynamic Tool Loading for WhatsApp Concierge**: The `TOOLS` array in `src/lib/ai-intake.ts` is a static list of all 9 available tools. This entire array is sent with every LLM request, consuming tokens even if only a subset of tools is relevant to the current `ConversationState`. An improvement could involve dynamically filtering the `TOOLS` array based on the current state (e.g., if `selectedProfessionalId` is not set, only `search_professionals` and `list_professions` might be relevant). This would reduce token usage and potentially improve LLM inference speed.
2.  **Optimized Availability Rule Fetching**: In `src/lib/ai-intake.ts`, the `computeSlotsForDate` function fetches `availability` rules from the database for a given `professionalId` and `dayOfWeek` on each call. When `getAvailableSlots` checks for `alternatives` by iterating through the next 7 days, this can lead to redundant database queries for the same professional's weekly availability rules. A more efficient approach would be to fetch all `availability` rules for the `professionalId` once at the beginning of `getAvailableSlots` and then filter them in memory for each date.
3.  **Database-Level Aggregation for Professional Assistant Stats**: The `getStats` function in `src/app/api/ai/chat/route.ts` fetches *all* appointments for the last 6 months and then performs month-by-month filtering and aggregation in memory. For professionals with a very high volume of appointments, this could become inefficient. Drizzle ORM supports advanced SQL functions, so a more performant solution would be to perform the monthly aggregations directly within the database query using `GROUP BY` and `date_trunc` functions, returning pre-aggregated data.
4.  **Modular Payment Logic**: The `bookAppointment` function in `src/lib/ai-intake.ts` contains logic for `buildPaymentPolicy` and `buildPixPayload`. While functional, this embeds payment-specific details directly into the booking flow. As payment options or policies evolve, this could become harder to maintain. Extracting payment-related logic into a dedicated service or utility module would improve modularity, making the `bookAppointment` function cleaner and the payment logic more reusable and testable.
