# AI Layer

Odys integrates artificial intelligence to streamline interactions for both clients and professionals. This document explores the architectural choices behind the AI layer, focusing on the selection of Groq with Llama 3.3 70B and the implementation of a two-pass tool-calling pattern. These decisions aim to balance performance, cost-effectiveness, and the accuracy required for complex, stateful conversations and data retrieval.

## Overview

The AI layer in Odys serves two primary functions, each with its own dedicated context and set of capabilities:

1.  **WhatsApp Concierge**: Handled by `src/lib/ai-intake.ts`, this persona acts as a marketplace agent, assisting clients in discovering professionals, checking availability, and booking appointments across the entire Odys platform via WhatsApp. It manages a conversation state machine to guide users through discovery and booking flows.
2.  **Professional Assistant**: Implemented in `src/app/api/ai/chat/route.ts`, this AI helps professionals understand their business data, such as appointments, client behavior, and revenue. It provides insights by querying the professional's specific data.

Both applications rely on a shared underlying LLM strategy, `chatWithFallback`, which orchestrates interactions with language models. A core architectural pattern across both is the "two-pass tool-calling" mechanism, where the LLM first decides *if* and *which* tool to call, and then, in a second pass, generates a final response incorporating the tool's output.

## WhatsApp Concierge (`src/lib/ai-intake.ts`)

The WhatsApp Concierge is designed to be a comprehensive, multi-professional discovery and booking agent. It operates on the corporate Odys WhatsApp number, providing a unified entry point for clients.

The conversation flow is managed by a state machine:
*   **Discovery**: When no professional is selected, the AI guides the user to specify their needs (e.g., profession, city) and uses tools like `search_professionals` to present matches.
*   **Focused**: Once a `selectedProfessionalId` is set, the AI shifts its focus to that professional, enabling actions like fetching available slots via `get_available_slots` or booking an appointment with `book_appointment`.
*   **Reset**: Users can explicitly request "outro profissional" to clear the current selection and return to the discovery state.

The concierge is equipped with 9 distinct tools, each serving a specific purpose in the client journey:
*   `search_professionals`: The primary tool for finding professionals based on criteria like `profession`, `city`, or `name`. It leverages the `professionals` table and enriches results with `avg_rating`, `review_count`, and `completed_count` from `reviews` and `appointments` data, using `getProStats` and `rankScore` for intelligent sorting.
*   `list_professions`: Dynamically lists only those professions that have active professionals on the platform, preventing dead-end searches. This queries the `professionals` table.
*   `select_professional`: Explicitly sets the `selectedProfessionalId` in the conversation state.
*   `get_professional_info`: Retrieves detailed information about a selected professional, including their `bio`, `address`, `schedule` (from `availability`), `session_price_brl`, and `payment_policy`. It also fetches recent `reviews` from the `reviews` and `clients` tables.
*   `get_available_slots`: Crucial for booking, this tool computes real-time availability for a given professional and date, considering their `availability` rules and existing `appointments`. It also proactively suggests alternative dates if the requested date has no slots.
*   `book_appointment`: Orchestrates the booking process, creating new entries in the `appointments` table and, if necessary, the `clients` table. It handles `autoConfirm` logic and triggers notifications via `sendWhatsApp` (using templates like `msgBookingConfirmed` or `msgBookingRequest`) and `sendBookingRequestEmailToProfessional`. It also generates `pix_payload` for upfront payments using `buildPixPayload`.
*   `list_my_appointments`: Allows clients to view their future appointments by querying `appointments` and `clients` tables.
*   `cancel_appointment`: Enables clients to cancel their own appointments, updating the `appointments` table and notifying the professional via WhatsApp.
*   `reschedule_appointment`: Facilitates changing an existing appointment's date and time.

Localization is a key consideration, with functions like `saoPauloDate`, `formatSaoPauloTime`, `formatSaoPauloDate`, `parsePtDate`, and `parsePtTime` ensuring correct handling of Brazilian dates and times. Phone numbers are normalized using `normalizeBrazilianPhone`.

## Professional Assistant (`src/app/api/ai/chat/route.ts`)

The Professional Assistant is an internal tool, accessible via the `/api/ai/chat` route, designed to provide professionals with insights into their business performance. It operates under a distinct `SYSTEM_PROMPT` that guides it to act as an assistant for understanding appointments, clients, and revenue, always responding in Portuguese and formatting values in BRL.

This assistant is equipped with 3 specialized tools:
*   `get_stats`: Provides a comprehensive summary of the professional's activity over the last 6 months, including global and month-by-month breakdowns of total appointments, completed sessions, no-shows, no-show rates, and revenue. This tool queries the `appointments` table.
*   `get_upcoming`: Lists appointments scheduled for the next 7 days, pulling data from `appointments` and `clients`.
*   `get_no_show_clients`: Ranks clients by their no-show count over the last 6 months, providing individual no-show rates. This involves a grouped query on `appointments` and `clients`.

Access to this assistant is protected by `getUser` and `getProfessional` checks, ensuring only authenticated professionals can use it. Furthermore, `canUseFeature` enforces plan-based access, restricting the "assistant" feature to "Pro" and "Premium" plans. To prevent abuse and manage resource consumption, a rate limit of 20 requests per hour is applied per user via `getAiChatLimiter`.

## Design decisions

### Groq and Llama 3.3 70B

The choice of Groq as the primary inference provider, coupled with the Llama 3.3 70B model, is a deliberate trade-off favoring speed and cost-efficiency for tool-calling applications. Groq's LPU (Language Processing Unit) architecture offers significantly lower latency compared to traditional GPU-based inference, which is critical for interactive applications like the WhatsApp Concierge where quick responses enhance user experience. Llama 3.3 70B provides a strong balance of reasoning capabilities and cost-effectiveness for the specific tasks of understanding user intent, calling tools, and synthesizing responses. The `chatWithFallback` mechanism in `src/lib/llm.ts` provides a layer of resilience, allowing the system to gracefully degrade to other providers if Groq experiences issues, though the primary design optimizes for Groq's performance characteristics.

### Two-Pass Tool-Calling Pattern

Both AI applications employ a two-pass tool-calling pattern. In the first pass, the LLM receives the user's message and the available `TOOLS` definitions. Its primary task is to decide whether a tool needs to be called and, if so, which one and with what arguments. The output of this first pass is either a direct text response or a `tool_calls` object. If tool calls are suggested, the system executes these tools (e.g., `searchProfessionalsTool`, `getStats`). The results of these tool executions are then injected back into the conversation history as `role: "tool"` messages. In the second pass, the LLM receives the original conversation, its own suggested tool calls, and the actual `content` (results) from those tool calls. This second pass is then dedicated solely to synthesizing a natural language response that incorporates the tool results.

This separation of concerns offers several advantages:
*   **Improved Reliability**: It reduces the cognitive load on the LLM, making it more reliable in both identifying the correct tool and generating a coherent, factual response based on concrete data.
*   **Enhanced Control**: The application gains explicit control over tool execution, allowing for pre- and post-processing, error handling, and logging of tool interactions.
*   **Reduced Hallucinations**: By providing the LLM with actual tool outputs in the second pass, the risk of the model hallucinating facts or misinterpreting data is significantly reduced.

The trade-off for this pattern is increased latency due to two distinct LLM calls per interaction that requires a tool. However, with Groq's speed, this overhead is minimized, and the benefits in accuracy and control are considered paramount for critical functions like booking appointments or providing financial summaries.

### Separate AI Contexts and Tool Sets

The decision to maintain entirely separate system prompts and tool definitions for the WhatsApp Concierge (`src/lib/ai-intake.ts`) and the Professional Assistant (`src/app/api/ai/chat/route.ts`) is fundamental. Each AI persona has a distinct role, target user, and access to different data. The concierge needs tools for marketplace discovery and booking, interacting with `professionals`, `availability`, and `appointments` across the platform. The assistant, conversely, needs tools to analyze a *single* professional's data, focusing on `appointments` and `clients` specific to them. This modularity ensures that each AI is highly specialized and efficient for its intended purpose, preventing unnecessary tool definitions from being sent in prompts and maintaining clear boundaries for data access and functionality.

## Potential improvements

1.  **Structured Tool Output Validation**: The current system uses `JSON.stringify` to pass tool results back to the LLM. While effective, the LLM's interpretation of these JSON strings can sometimes be brittle. Implementing a more structured validation or parsing layer for tool outputs, perhaps using Zod schemas or similar, could make the LLM's consumption of tool results more robust and less prone to misinterpretation, especially for complex data structures returned by `searchProfessionalsTool` or `getStats`.
2.  **Asynchronous Tool Execution**: In `src/app/api/ai/chat/route.ts`, the tool calls are executed sequentially within a `for...of` loop. If the LLM were to request multiple independent tools in a single turn (e.g., `get_stats` and `get_upcoming`), executing them in parallel using `Promise.all` could reduce the overall latency of the tool execution phase, improving response times for complex queries.
3.  **Caching for Static/Infrequently Changing Data**: Functions like `listProfessionsTool` in `src/lib/ai-intake.ts` and `getProStats` (which is called by `searchProfessionalsTool` and `getProfessionalInfo`) query the database for data that changes relatively infrequently. Introducing a short-lived cache (e.g., Redis-backed) for these results could significantly reduce database load and further decrease response times for common queries, especially during peak usage.
