# Groq + Llama 3.3 70B

This page explains the AI architecture behind Odys, focusing on the integration of Groq’s inference engine with Llama 3.3 70B and the two-pass tool-calling pattern used across the platform. It covers how AI agents interact with the database, manage state, and deliver reliable responses in high-stakes domains like booking and financial summaries.

## Overview

Odys uses a multi-provider LLM chain (Groq → Cerebras → Gemini) to run Llama 3.3 70B for real-time WhatsApp and dashboard AI interactions. The system handles two distinct agent types: a marketplace-wide concierge for clients (`ai-intake.ts`) and a personal assistant for professionals (`/api/ai/chat`). Both rely on a two-pass tool-calling pattern—first to decide which data to fetch, then to generate a final response with results—ensuring accuracy and preventing hallucination. The backend schema includes 11 tables, with `professionals`, `appointments`, and `agentMetrics` playing central roles in AI decision-making and observability. The `ai-intake.ts` agent manages state via a conversation state machine with Discovery, Focused, and Reset states.

## Design decisions

The two-pass pattern is critical for reliability. In the first pass, the LLM analyzes the user’s message and may request a tool call—such as `search_professionals` or `get_available_slots`. The system executes the tool, fetching real data from the database via Drizzle ORM. In the second pass, the LLM receives the tool result as a `tool`-role message and generates a final reply. This separation prevents the model from fabricating availability or pricing, which is essential in a booking system. Groq’s low-latency inference makes this two-step process feel instantaneous to users. The `agentMetrics` table logs every interaction, capturing `totalLatencyMs`, `totalCostUsd`, and `outcome`, enabling cost and performance tracking. The `ai-intake.ts` agent uses 9 tools (e.g., `search_professionals`), and the `ai/chat` agent uses 3 tools (e.g., `get_stats`). The `ai/chat` agent's system prompt includes specific formatting rules for no-show rates and monthly summaries.

## Potential improvements

The `searchProfessionalsTool` function in `src/lib/ai-intake.ts` performs a substring match on `profession` when no alias is found, but this could return irrelevant results. Replacing `ilike(professionals.profession, ...)` with a semantic similarity check using embedded vectors would improve precision. The `searchProfessionalsTool` ranks professionals using a composite score based on rating, popularity, plan tier, and distance from city centroid. The `getAvailableSlots` function computes availability for only 7 future days; extending this window or caching recurring patterns in `recurringSchedules` could reduce database load. In `src/app/api/ai/chat/route.ts`, the `get_stats` function recalculates no-show rates in JavaScript—pushing this aggregation into SQL with window functions would be more efficient and consistent.
