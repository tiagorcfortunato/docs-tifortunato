# AI Tool Calling

The AI layer within Odys employs a sophisticated tool-calling pattern, enabling large language models (LLMs) to interact with the application's backend services and database. This mechanism is central to two distinct AI agents: the WhatsApp Concierge, which assists clients in discovering and booking professionals, and the Professional Assistant, which provides professionals with insights into their business data. This document delves into the structure, implementation, and rationale behind this tool-calling architecture.

## Overview

At its core, the AI tool-calling system in Odys operates on a "two-pass" principle. In the first pass, the LLM receives a user query and a list of available tools, then decides whether to call one or more tools. If a tool is called, its output is then fed back to the LLM in a second pass, allowing it to generate a final, context-aware response to the user. This iterative approach helps the LLM reason about complex requests and integrate real-world data into its replies.

Odys currently features two primary contexts for AI interaction, each with its own set of specialized tools:

1.  **WhatsApp Concierge (`src/lib/ai-intake.ts`)**: This agent handles inbound messages on the corporate WhatsApp number. Its tools are designed for client-facing tasks such as searching for professionals, checking availability, and booking appointments.
2.  **Professional Assistant (`src/app/api/ai/chat/route.ts`)**: Accessible via the `/api/ai/chat` route, this agent provides business intelligence to professionals. Its tools focus on retrieving data related to appointments, clients, and revenue.

Across both contexts, the system uses a consistent `OpenAI.Chat.Completions.ChatCompletionTool[]` schema for defining tools, ensuring compatibility with various LLM providers like Groq, Cerebras, and Gemini via the `chatWithFallback` utility.

## Tool Definitions

The `TOOLS` array in both `src/lib/ai-intake.ts` and `src/app/api/ai/chat/route.ts` defines the capabilities exposed to the LLM. Each tool is described with a `name`, a concise `description`, and a `parameters` schema adhering to JSON Schema standards.

For instance, the `ai-intake.ts` file defines 9 tools, including:

*   `search_professionals`: Allows the AI to find professionals based on `profession`, `city`, or `name`. The description "Busca pros. SEMPRE antes de listar." explicitly guides the LLM on its usage.
*   `book_appointment`: Facilitates scheduling, requiring `professional_id`, `date`, `time`, `client_name`, and `client_phone`. The description "Agenda. Confirme dados antes." emphasizes a pre-booking verification step.
*   `list_professions`: Provides a list of active professions, ensuring the AI only suggests services actually available on the platform.

Conversely, the `src/app/api/ai/chat/route.ts` for the Professional Assistant defines 3 tools tailored for business analytics:

*   `get_stats`: Retrieves a summary of appointments, no-shows, and revenue over the last 6 months.
*   `get_upcoming`: Lists appointments scheduled for the next 7 days.
*   `get_no_show_clients`: Identifies clients with the highest no-show rates.

The descriptions for these tools are intentionally terse. This design choice minimizes the token count sent with each LLM request, as the entire `TOOLS` array is included in the context. The LLM's prompt is responsible for providing more detailed instructions on when and how to use each tool effectively.

## Tool Implementations

Each tool defined in the `TOOLS` array corresponds to an asynchronous JavaScript function that encapsulates the necessary business logic and data access.

### WhatsApp Concierge Tools

In `src/lib/ai-intake.ts`, tools interact extensively with the Drizzle ORM-backed database schema, which includes 10 tables such as `professionals`, `availability`, `appointments`, and `clients`.

*   **`searchProfessionalsTool`**: This function queries the `professionals` table, applying filters for `profession`, `city`, and `name`. It uses `resolveProfessionAlias` to normalize profession names (e.g., "psicologa" to "psychologist") and `getProStats` to enrich results with `avg_rating`, `review_count`, and `completed_count`. Results are then ranked using `rankScore` based on these metrics and the professional's `plan` tier. A notable feature is its fallback to `findNearestProfessionals` if no matches are found in a specific city, allowing the AI to suggest alternatives.
*   **`getAvailableSlots`**: This tool calculates real-time availability for a given professional and date. It first fetches the professional's `sessionDuration` and then uses `computeSlotsForDate` to determine open slots by comparing `availability` rules against existing `appointments`. It also proactively looks for `alternatives` in the next 7 days if the requested date has no slots, reducing conversational turns.
*   **`bookAppointment`**: This is a critical tool that handles the end-to-end booking process. It performs several checks, including validating dates/times, preventing past bookings, and detecting `duplicate_active_booking` for the same client and professional. The booking logic is wrapped in a Drizzle `db.transaction` with `serializable` isolation to prevent race conditions, especially for `SLOT_TAKEN` scenarios. Upon successful booking, it dispatches notifications via WhatsApp (`sendWhatsApp` using templates like `msgBookingConfirmed` or `msgBookingRequest`), email (`sendBookingRequestEmailToProfessional`), and inserts records into the `notifications` table. It also computes `payment_instructions` and generates a `pix_payload` if the professional has upfront payment policies.

### Professional Assistant Tools

The tools in `src/app/api/ai/chat/route.ts` focus on aggregating and presenting data from the `appointments` and `clients` tables.

*   **`getStats`**: This function queries all relevant `appointments` for a given professional over the last 6 months. It then processes this data to provide a `global` summary (total appointments, completed, no-shows, revenue) and a `perMonth` breakdown, calculating metrics like `noShowRate`.
*   **`getUpcoming`**: Fetches `appointments` for the next 7 days, joining with the `clients` table to include client names.
*   **`getNoShowClients`**: Aggregates data to identify clients with the highest number of `no_show` appointments, calculating individual no-show rates.

## Tool Dispatch and Execution

The `POST` handler in `src/app/api/ai/chat/route.ts` exemplifies the tool dispatch and execution flow:

1.  **Authentication and Authorization**: It first authenticates the user via `getUser()` and retrieves the `professional` profile using `getProfessional()`. It then applies rate limiting using `getAiChatLimiter()` (with a limit of 20 requests per hour per user, prefix `rl:ai-chat`) and checks plan eligibility with `canUseFeature()`.
2.  **First LLM Call**: The user's messages, along with a `SYSTEM_PROMPT` and the `TOOLS` definitions, are sent to the LLM via `chatWithFallback()`. The `tool_choice: "auto"` setting allows the LLM to decide whether to call a tool.
3.  **Tool Execution**: If the LLM's response (`first.choices[0].message`) contains `tool_calls`, the handler iterates through them. For each `function` tool call, it dynamically invokes the corresponding JavaScript function (e.g., `getStats`, `getUpcoming`).
4.  **Second LLM Call**: The original messages, the LLM's tool call request, and the `JSON.stringify()`-ed results from the tool execution are all appended to the conversation history. This augmented history is then sent back to the LLM for a second `chatWithFallback()` call, allowing it to synthesize a final, informed response.

Error handling within tool implementations often involves returning structured error objects (e.g., `{ error: string }` or `{ success: false, reason: string }`), which the LLM can then interpret and communicate to the user. Broader API errors are handled by `serverError()`.

## Design Decisions

*   **Two-Pass Tool Calling**: This pattern was chosen to enhance the reliability and reasoning capabilities of the AI agents. By separating the tool-calling decision from the final response generation, the LLM gains an opportunity to review and incorporate the actual results of a tool call, leading to more accurate and contextually relevant answers. It also simplifies error handling, as tool execution failures can be reported back to the LLM for graceful recovery.
*   **Terse Tool Descriptions**: The decision to keep tool descriptions short in the `TOOLS` array is a direct optimization for token usage. Since the entire `TOOLS` array is sent with every LLM request, minimizing its size reduces costs and improves latency, especially with providers that charge per token. Detailed instructions are instead provided in the `SYSTEM_PROMPT`.
*   **Context-Specific Tools**: The separation of tools into `ai-intake.ts` (Concierge) and `src/app/api/ai/chat/route.ts` (Professional Assistant) reflects a clear architectural choice to maintain distinct AI personas and capabilities. This prevents the Concierge from accessing sensitive professional data and ensures the Professional Assistant remains focused on business analytics, improving security and reducing the cognitive load on each LLM instance.
*   **Drizzle ORM for Database Interactions**: The use of Drizzle ORM provides a type-safe and expressive way to interact with the PostgreSQL database. This choice simplifies complex queries, transactions, and schema definitions, contributing to the maintainability and reliability of the data access layer within tool implementations.
*   **São Paulo Timezone Handling**: Brazil abolished DST, making São Paulo consistently UTC-3. The explicit `saoPauloDate` and `formatSaoPauloTime`/`formatSaoPauloDate` functions ensure all date and time calculations and displays are consistent with the local timezone, preventing common off-by-one errors in scheduling.
*   **Authorization and Rate Limiting**: The `src/app/api/ai/chat/route.ts` handler includes explicit checks for user authentication (`getUser`), professional status (`getProfessional`), plan features (`canUseFeature`), and rate limits (`getAiChatLimiter`). This layered approach protects the AI service from unauthorized access, feature abuse, and excessive resource consumption.

## Potential Improvements

1.  **Centralize Date/Time Utilities**: The `ai-intake.ts` file contains several timezone-specific date and time utility functions (`saoPauloDate`, `formatSaoPauloTime`, `formatSaoPauloDate`, `addDaysToDateStr`). While functional, these could be extracted into a dedicated `src/lib/date-utils.ts` or similar module. This would reduce duplication, improve consistency across the codebase, and make it easier to manage timezone logic if requirements change.
2.  **Standardize Tool Error Responses**: Tool implementations currently return errors in varied formats, such as `{ error: string }` (e.g., `searchProfessionalsTool`, `getAvailableSlots`) or `{ success: false, reason: string }` (e.g., `bookAppointment`, `cancelAppointment`). Adopting a single, consistent error structure (e.g., always returning `{ error: { code: string, message: string } }`) would simplify the LLM's interpretation of tool failures and streamline the error handling logic in the dispatching code.
3.  **Dynamic Tool Registration**: The `TOOLS` arrays are currently hardcoded in each AI agent file. As the number of tools grows, managing these static arrays can become cumbersome. Exploring a more dynamic tool registration mechanism, perhaps by defining tools in a central registry and importing them, could reduce boilerplate and improve modularity. This would also facilitate conditional loading of tools based on context or user permissions.
