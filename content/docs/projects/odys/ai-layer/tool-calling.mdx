# Tool-calling pattern

The tool-calling pattern in Odys enables AI agents to interact with backend systems through structured function calls, allowing natural language queries to trigger database lookups, business logic, and side effects. This page explores how tool definitions are declared, dispatched, and handled across the AI concierge and professional assistant.

## Overview

Odys implements two distinct AI agents: a **WhatsApp concierge** for clients (`ai-intake.ts`) and a **professional assistant** (`api/ai/chat/route.ts`). Both use OpenAI-style tool definitions to expose backend capabilities, but serve different audiences and use cases. The WhatsApp concierge (`ai-intake.ts`) defines 9 tools, and the professional assistant (`api/ai/chat/route.ts`) defines 3 tools. The system also leverages 23 API routes, with rate limits applied at the endpoint level. Database access is abstracted through Drizzle ORM, operating on 11 tables including `professionals`, `appointments`, and `agentMetrics`.

## Tool schema and dispatch

Tool definitions are declared as plain objects conforming to `OpenAI.Chat.Completions.ChatCompletionTool`. In `ai-intake.ts`, 9 tools support client-facing workflows like `search_professionals`, `book_appointment`, and `get_available_slots`. Each tool includes a concise description and parameter schema, optimized for token efficiency. The professional assistant in `api/ai/chat/route.ts` defines 3 tools focused on analytics: `get_stats`, `get_upcoming`, and `get_no_show_clients`.

Tool dispatch follows a two-phase pattern. The LLM's first response may include `tool_calls`, which are then executed in sequence. Results are appended as `tool` role messages, and a second LLM call produces the final response. This approach ensures the model has full context before replying. Error handling is explicit: failed operations return structured `{ error }` objects, while business rules (e.g., duplicate bookings) surface as `{ success: false, reason }` payloads.

## Design decisions

The tool schema prioritizes clarity over completeness—descriptions are terse because the system prompt already defines usage context. This reduces token overhead on every request. The two-pass LLM call pattern allows the model to reason about tool results before responding, improving accuracy. Database operations use serializable transactions in `book_appointment` to prevent race conditions during booking, with Postgres-level constraints as a final safeguard. The separation between client and professional agents reflects their distinct data needs: discovery and booking vs. analytics and reporting.

## Potential improvements

1.  **Tool reuse across agents**: The `get_stats` logic in `api/ai/chat/route.ts` duplicates functionality available in the MCP layer. The `ai-intake.ts` agent does not define a tool named `get_revenue_summary`.

2.  **Error handling consistency**: In `ai-intake.ts`, some tool implementations return `{ error: string }` while others use `{ success: false, reason: string }`. Standardizing on one pattern would simplify client code and improve maintainability.

3.  **Tool result validation**: The `getAvailableSlots` function in `ai-intake.ts` returns raw database results without schema validation. Adding Zod parsing would prevent malformed data from propagating to the LLM, reducing hallucination risk.
