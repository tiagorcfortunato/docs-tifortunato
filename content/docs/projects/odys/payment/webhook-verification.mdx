# Webhook Verification

The `src/app/api/stripe/webhook/route.ts` file defines the API endpoint responsible for handling incoming webhook events from Stripe. This endpoint is a critical component of the payment system, as it allows odys to react to changes in subscription statuses and payment events initiated by Stripe, ensuring that the application's internal state remains synchronized with the payment provider.

## Overview

This API route, accessible at `/api/stripe/webhook`, is configured to accept `POST` requests. Its primary function is to receive event notifications from Stripe, verify their authenticity, and then process specific events to update the application's database. The system relies on the `professionals` table, which has 24 columns, to store and manage subscription-related information such as `plan`, `stripeCustomerId`, and `stripeSubscriptionId`.

The webhook is specifically designed to listen for three key Stripe events: `checkout.session.completed`, `customer.subscription.updated`, and `customer.subscription.deleted`. Each of these events triggers a distinct logic flow to modify a professional's subscription status within the odys platform, ensuring that their access and features align with their current Stripe subscription.

## Stripe Webhook Processing

The `POST` function in `src/app/api/stripe/webhook/route.ts` orchestrates the entire webhook handling process.

Upon receiving a request, the first crucial step is to extract the raw request body using `req.text()` and the `stripe-signature` header from `req.headers.get("stripe-signature")`. Stripe's security model mandates that the raw body, not a parsed JSON object, be used for signature verification. This ensures that no part of the payload has been altered during transit. If the signature is missing, the request is immediately rejected with a 400 status code.

The extracted raw body and signature are then passed to `stripe.webhooks.constructEvent`, along with the `STRIPE_WEBHOOK_SECRET` environment variable. This function performs a cryptographic verification to confirm that the event genuinely originated from Stripe and has not been tampered with. If the verification fails, an error is logged, and a 400 status code is returned, preventing malicious or invalid requests from affecting the system.

Once the event's authenticity is established, the system proceeds to dispatch logic based on the `event.type`:

*   **`checkout.session.completed`**: This event signifies that a customer has successfully completed a Stripe Checkout session, typically for a new subscription or a one-time payment. The handler extracts `professionalId` and `plan` from the `session.metadata`. These metadata fields are crucial for linking the Stripe event back to the corresponding professional in the odys database. The `professionals` table is then updated to reflect the new `plan`. The `stripeCustomerId` and `stripeSubscriptionId` are updated with the values from the session if they are non-null; otherwise, their existing values in the database are retained. The `updatedAt` timestamp is also refreshed.

*   **`customer.subscription.updated`**: This event is triggered when a customer's subscription changes, such as an upgrade or downgrade initiated through the Stripe customer portal. The handler checks if the subscription `status` is "active". It then identifies the `priceId` of the new subscription item and attempts to match it against the `PLANS` constant defined in `@/lib/stripe`. This constant provides a mapping between Stripe's internal price IDs and odys's internal plan keys (e.g., "basic", "pro"). If a matching plan is found, the professional's `plan` in the database is updated accordingly, along with the `updatedAt` timestamp.

*   **`customer.subscription.deleted`**: When a customer cancels their subscription, this event is fired. The handler sets the professional's `plan` back to "free" and nullifies their `stripeSubscriptionId` in the `professionals` table. This ensures that canceled subscriptions are correctly reflected in the application's state, revoking access to paid features. The `updatedAt` timestamp is also updated.

Finally, after processing the event, the endpoint returns a `200 OK` response with `{ received: true }`, acknowledging receipt of the webhook to Stripe.

## Design Decisions

The design of this Stripe webhook handler reflects several deliberate choices aimed at security, reliability, and maintainability:

*   **Raw Body for Verification**: The decision to read the request body as raw text (`req.text()`) before any parsing is fundamental. Stripe's signature verification mechanism requires the exact raw payload to compute and compare signatures. Parsing the body into JSON first would alter its structure and invalidate the signature, making verification impossible. This is a non-negotiable security requirement for Stripe webhooks.
*   **Immediate Signature Verification**: Placing the `stripe.webhooks.constructEvent` call at the very beginning of the processing flow ensures that only legitimate and untampered events are processed. This "fail fast" approach is a strong security practice, preventing any further resource consumption or potential data manipulation from unauthorized sources.
*   **Direct Database Updates**: The webhook directly interacts with the Drizzle ORM (`db.update(professionals)`) to modify the `professionals` table. This direct approach simplifies the data flow for critical subscription changes, ensuring that the application's view of a professional's plan is always in sync with Stripe's authoritative source. While this couples the webhook directly to the database, for these specific, high-priority updates, it minimizes latency and complexity compared to introducing an intermediary queue for simple state changes.
*   **Explicit Event Type Handling**: The handler explicitly checks for `event.type` using `if (event.type === "...")` blocks. This makes the logic clear and focused, processing only the events that are relevant to odys's subscription management. It avoids unnecessary computation for other Stripe events that might be sent but are not critical for updating professional plans.
*   **Centralized Plan Mapping**: The use of the `PLANS` constant from `@/lib/stripe` for mapping Stripe `priceId` values to internal `PlanKey` values is a good practice. It centralizes the configuration of subscription plans, making it easier to manage and update plan details without scattering Stripe IDs throughout the codebase.

## Potential Improvements

While the current implementation effectively handles Stripe webhooks, several areas could be enhanced for greater robustness, scalability, and maintainability:

1.  **Asynchronous Event Processing**: The current webhook processes events synchronously, performing database updates directly within the `POST` request handler. For more complex event handling or to mitigate the risk of Stripe retries due to slow responses, consider offloading the actual event processing to an asynchronous job queue. The webhook could simply verify the signature, enqueue the event payload, and immediately return a `200 OK` response. A separate worker process would then consume events from the queue and perform the database operations. This decouples the webhook's response time from the processing time, improving reliability.
2.  **Enhanced Idempotency Handling**: Although Stripe webhooks are designed to be idempotent, the current database update logic in `src/app/api/stripe/webhook/route.ts` does not explicitly track processed event IDs. While Drizzle's `update` operations are generally idempotent for setting specific values, adding a mechanism to store and check `event.id` in the database before processing could prevent unintended side effects if Stripe sends duplicate events (e.g., due to network issues or retries). This would involve adding a new table to store processed webhook event IDs.
3.  **Structured Logging and Alerting**: The error handling for signature verification currently uses `console.error("[STRIPE WEBHOOK] Invalid signature:", err)`. For a production environment, integrating a more structured logging solution (e.g., a dedicated logging library) that captures more context (like the `event.id` if available, or the request headers) would be beneficial. Furthermore, setting up alerts for critical errors, such as failed signature verifications or database update failures, would allow for proactive monitoring and faster incident response.
