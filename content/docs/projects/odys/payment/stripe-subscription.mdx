# Stripe Subscription Flow

This page details the internal mechanisms for managing professional subscriptions to Odys via Stripe. It covers how subscription plans are defined, the process for initiating a Stripe checkout session, and the high-level approach to handling Stripe webhooks for subscription lifecycle events.

## Subscription Model

The Stripe integration is used exclusively for professionals to subscribe to Odys's Basic, Pro, or Premium plans. It is not involved in client-to-professional payments, which are handled off-Stripe via PIX and tracked in the `appointments.paymentStatus` column. This design keeps Odys out of PCI scope for client transactions.

## Plan Definitions

Subscription plans are defined in `src/lib/stripe/plans.ts`. This file exports:

- `PLAN_FEATURES`: An ordered list of all available features, used for rendering plan comparison tables. Each feature has a `label` and an optional `comingSoon` flag.
- `PLANS`: An object mapping plan keys (`free`, `basic`, `pro`, `premium`) to their configurations. Each plan includes:
  - `label`: Display name.
  - `price`: Monthly price in Brazilian Reals.
  - `priceId`: The Stripe Price ID from environment variables (`STRIPE_BASIC_PRICE_ID`, etc.). The `free` plan has `null`.
  - `note`: Descriptive note, used for the `free` plan's limits.
  - `limits`: Resource limits (e.g., `clients`, `appointmentsPerMonth`). Paid plans use `Infinity`.
  - `features`: List of included features, referencing `PLAN_FEATURES` labels.

The `src/lib/stripe/index.ts` file initializes the Stripe client using `process.env.STRIPE_SECRET_KEY` and re-exports `PLAN_FEATURES`, `PLANS`, and `PlanKey` for server-side usage.

## Checkout Session Creation

The `src/app/api/stripe/checkout/route.ts` endpoint handles POST requests to create Stripe checkout sessions. The flow is:

1. **Authentication**: Uses Supabase to verify the user session. Returns `401` if no user is found.
2. **Plan Validation**: Receives `plan` from the request body. Rejects invalid or `free` plans with `400`.
3. **Professional Lookup**: Queries the `professionals` table using `user.id`. Returns `404` if not found.
4. **Trial Handling**: If the plan is `pro` and `trialDaysLeft(professional.trialEndsAt)` returns a value, `trial_end` is set in the session to delay payment.
5. **Session Creation**: Calls `stripe.checkout.sessions.create` with:
   - `mode: "subscription"`
   - `payment_method_types: ["card"]`
   - `line_items` containing the plan’s `priceId` and `quantity: 1`
   - `success_url` and `cancel_url` pointing to `/dashboard/plans`
   - `metadata` with `professionalId` and `plan` for webhook context
   - `customer_email` from the authenticated user
   - `locale: "pt-BR"`
   - `subscription_data.trial_end` if applicable
   - `payment_method_collection: "if_required"` during trial, `"always"` otherwise
6. **Response**: Returns `{ url: session.url }`, which the client uses to redirect to Stripe.

## Webhook Event Handling

The webhook handler is not included in the current code, but the intended behavior is:

1. **Signature Verification**: Uses `req.text()` to get the raw body and `constructEvent(body, sig, STRIPE_WEBHOOK_SECRET)` to verify authenticity. Invalid signatures return `400`.
2. **Event Dispatching**: On valid events:
   - `checkout.session.completed`: Activates the paid plan using metadata (`professionalId`, `plan`).
   - `customer.subscription.updated`: Handles plan changes (e.g., upgrades/downgrades).
   - `customer.subscription.deleted`: Reverts the professional’s plan to `free`.

Database updates are structured as `UPDATE ... SET plan = ? WHERE ...`, which is idempotent at the statement level.

## Known Gaps

Idempotency for webhook events relies on the structural idempotency of `UPDATE ... SET plan = ? WHERE ...` statements rather than an explicit `idempotency_keys` table. While sufficient for plan updates, this approach does not prevent duplicate processing of non-idempotent side effects if introduced later.

## Why this shape

Using Stripe-hosted checkout keeps Odys out of PCI scope, simplifying compliance. Including `professionalId` and `plan` in `metadata` ensures webhook handlers can act on events without needing to resolve Stripe customer IDs, which may not exist at checkout time for new subscribers. The trial logic is embedded in the session creation flow, enabling deferred payment collection without additional state management.
