# Stripe Subscription Flow

This document details the internal mechanics of the Stripe subscription system in odys, covering plan definitions, checkout integration, and webhook event handling. Understanding this flow is essential for maintaining billing integrity, supporting trial conversions, and ensuring reliable subscription state synchronization.

## Overview

The subscription system is built around four core components: defined pricing plans, Stripe Checkout session creation, Supabase authentication, and Drizzle ORM for database access. The platform offers four plans — `free`, `basic`, `pro`, and `premium` — priced at \$0, \$39, \$79, and \$149 respectively. Each paid plan maps to a Stripe Price ID via environment variables (`STRIPE_BASIC_PRICE_ID`, etc.), while the free plan enforces usage limits without requiring payment.

Stripe webhooks listen for three critical events: `checkout.session.completed`, `customer.subscription.updated`, and `customer.subscription.deleted`. These ensure subscription state in the database stays in sync with Stripe’s records, though webhook handlers themselves are not included in the current code sample.

## Plan Configuration and Feature Matrix

Plan metadata is centralized in `src/lib/stripe/plans.ts`, where both pricing and feature entitlements are declared. The `PLANS` object defines each tier’s label, price, Stripe priceId, and usage limits. Only the `free` plan enforces limits — capping clients at 10 and appointments at 20 per month. All paid plans grant unlimited access to core features.

The `PLAN_FEATURES` array drives the UI comparison table, listing capabilities like "Agendamentos recorrentes" and "Múltiplos profissionais" (the latter marked `comingSoon: true`). Each plan’s `features` array includes only the labels it supports, enabling a declarative approach to feature visibility.

## Checkout Session Creation

The API route `src/app/api/stripe/checkout/route.ts` handles checkout session creation. It first verifies user authentication via Supabase and retrieves the associated `professional` record. If the selected plan is invalid or `free`, it returns an error.

Notably, the `pro` plan supports trial continuation: if the professional has an active trial, the session is configured with `trial_end` and `payment_method_collection: "if_required"`, deferring payment collection. This logic avoids interrupting trial users while upgrading them to paid subscriptions seamlessly.

The session includes metadata (`professionalId`, `plan`) for post-checkout processing and uses localized checkout (`pt-BR`) for Brazilian users.

## Design decisions

The use of environment variables for Stripe Price IDs enables environment-specific configuration without code changes. The `priceId` field in `src/lib/stripe/plans.ts` directly accesses `process.env` variables (e.g., `process.env.STRIPE_BASIC_PRICE_ID ?? null`) to configure Stripe Price IDs. If a `STRIPE_PRICE_ID` environment variable for a paid plan is not set, its `priceId` will be `null`, which will cause a runtime error when attempting to create a Stripe Checkout session for that plan.

Feature flags are implemented via string matching against `PLAN_FEATURES`, reducing the need for boolean fields. This improves maintainability: adding a feature requires only appending to the list and updating plan entries.

The decision to embed trial logic in the checkout route — rather than a separate service — reduces round trips but couples trial state to payment initiation.

## Potential improvements

1. **Centralize environment variable names**: `src/lib/stripe/plans.ts` directly references `process.env` variables (e.g., `process.env.STRIPE_BASIC_PRICE_ID`) for Stripe Price IDs. Define these keys in one place (e.g., a config object) to prevent typos and improve testability.

2. **Validate Stripe Price IDs at startup**: Currently, `priceId` defaults to `null` if the env var is missing. This *will* cause a runtime error in Stripe API calls if a paid plan is selected without its corresponding `STRIPE_PRICE_ID` environment variable being set.

3. **Abstract trial logic**: The trial check in `src/app/api/stripe/checkout/route.ts` is tightly scoped to the `pro` plan. Extract `trialDaysLeft` and trial session logic into a reusable service to support future trial policies across plans.
